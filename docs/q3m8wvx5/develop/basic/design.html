0 STEP 0 — インプット確認

🎯
目的: 要件定義成果物が基本設計を始めるのに十分な状態かを確認する。ここで時間をかけることが後工程の手戻りを防ぐ最大の投資です。
👥
RACI確認（体制を最初に明確化する）
設計書の各成果物に対して「誰が作成（R）」「誰が最終承認（A）」「誰に相談（C）」「誰に通知（I）」かをPLと合意してから進めます。 特に「承認者（A）が1名か」「顧客確認が必要か」を事前に確認しないと、レビュー前に差し戻しが発生します。

8 STEP 8 — 取りまとめ・レビュー・承認

⚠️ よくある抜け漏れパターン

パターン症状対策
要件の読み飛ばし実装後に「この機能が設計書にない」機能要件一覧をすべてトレース
技術選定理由の未記録後から「なぜこれを選んだのか」不明ADRで必ず記録する
非機能要件の数値確認不足顧客が「性能が出ない」と言い始める数値を必ず顧客と合意
セキュリティ後回し実装完了後に大規模改修各STEPで同時に考える
データ設計の先行着手機能変更でテーブルを何度も作り直す主要機能が固まるまで待つ
✅
基本設計書の正式承認が取れたら 詳細設計フェーズ へ進みます。
