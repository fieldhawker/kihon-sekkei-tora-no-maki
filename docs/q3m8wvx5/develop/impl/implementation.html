タスク分割のコツ

粒度の目安理由
1タスク = 0.5〜2日進捗の「詰まり」が早期に発見できる
1PR = 1つの目的レビュアーが文脈を理解しやすい
APIエンドポイント単位フロント/バックの並行開発が可能になる
DB変更は単独PRマイグレーション適用順序を明確にする

STEP 2 — 実装フロー（1機能の流れ）

  1. Issueを取る: バックログからIssueをアサイン。設計書URLをIssueに貼る。
  2. ブランチを切る: feature/機能ID-概要（例: feature/F-001-create-request）
  3. テストを先に書く（TDD推奨）: 設計書の入力値・期待値からテストを先に書く。
  4. 実装する: テストが通る最小コードを書く。コーディング規約に従う。
  5. 自己レビュー: チェックリストを確認してからPR作成。
  6. PR作成・レビュー依頼: テンプレートに沿って記載。CIが全パスであること。
  7. マージ: 承認後、mainまたはdevelopにマージ。ブランチ削除。
