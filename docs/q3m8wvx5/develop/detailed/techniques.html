Phase 2 — 設計技法

詳細設計技法まとめ

詳細設計は、基本設計で定義した「システムが何をするか（What）」を元に、「それをプログラムでどう実現するか（How）」を具体化し、プログラマーが迷いなくバグの少ないコードを実装できる状態にするためのフェーズです。以下に、詳細設計で特に有用な設計技法と、それぞれの具体的な使用事例をまとめます。

🧭
どれを使うか迷ったら（早見表）
処理手順を文章で誤解なく伝えたい → IPO / フロー（アクティビティ/状態）
画面・API・DBのやり取りを誤解なく伝えたい → シーケンス図
分岐が多くてテスト漏れが怖い → デシジョンテーブル
目次
1. ロジックと処理 / 2. データと状態 / 3. 構造と連携 / 4. 外部IF / 5. 条件分岐

⚙️ 1. ロジックと処理の流れを整理する技法 {#logic}

複雑な計算やデータの加工処理を正確に伝えるための手法です。

1.1 IPO記述（Input - Process - Output）

モジュールや関数に対して、「何を入力し」「どのような処理を行い」「何を出力するか」を明確に定義する手法です。

📌 使用事例: 「割引料金計算関数」の設計
Input (入力) Process (処理) Output (出力)
  • ユーザーの年齢（数値）
  • 購入金額（数値）
  • クーポンコード（文字列）
  1. クーポンコードが有効かDBで検証する。
  2. 年齢が65歳以上ならシニア割引（10%オフ）を適用する。
  3. 購入金額が5000円以上なら送料無料フラグを立てる。
  • 最終請求金額（数値）
  • 送料無料フラグ（真偽値）

1.2 アクティビティ図（UML）

処理の分岐やループに加え、並行処理（マルチスレッド）などの複雑な処理の流れを視覚的に表現するフローチャートの発展形です。

📌 使用事例: 「ECサイトの注文確定バッチ処理」の設計
stateDiagram-v2 [*] --> 注文データ受信 注文データ受信 --> Fork_1 state Fork_1 <<fork>> Fork_1 --> 在庫引当API Fork_1 --> 与信確保API 在庫引当API --> Join_1 与信確保API --> Join_1 state Join_1 <<join>> Join_1 --> 判定 判定 --> 成功: 両方成功 判定 --> 失敗: いずれか失敗 成功 --> 注文確定・メール送信 失敗 --> エラー処理 注文確定・メール送信 --> [*] エラー処理 --> [*]

💾 2. データと状態を整理する技法 {#state}

データがどう保存され、どう変化していくかを定義します。

2.1 物理ER図 / テーブル定義書

システムで扱うデータを、実際のデータベースのテーブル構造、カラムのデータ型、制約（NOT NULL、外部キーなど）まで具体化した設計です。

📌 使用事例: 「ユーザー」と「注文」テーブルの設計
erDiagram users ||--o{ orders : "1人のユーザーは複数の注文を持つ" users { int id PK; string name } orders { int id PK; int user_id FK; date order_date }

2.2 状態遷移表（モジュールレベル）

特定のオブジェクトやデータが持つ状態（ステータス）が、どのようなイベントによってどう変化するかを網羅的に定義します。

📌 使用事例: 「釣行記録アプリの投稿ステータス」の設計
stateDiagram-v2 [*] --> 下書き : 新規作成 下書き --> 公開中 : 「公開する」イベント 下書き --> 削除済 : 「削除する」イベント 公開中 --> 下書き : 「非公開にする」イベント 公開中 --> 削除済 : 「削除する」イベント 削除済 --> [*] : 完全削除

🧩 3. プログラムの構造と連携を整理する技法 {#structure}

3.1 シーケンス図（UML）

オブジェクトやコンポーネント間で「いつ」「誰が」「どのAPIやメソッドを呼び出し」「どう返ってくるか」を時系列で図解する手法です。

📌 使用事例: 「スマホアプリからのログイン処理」の設計
sequenceDiagram participant UI as アプリ(UI) participant API as APIサーバー participant DB as データベース UI->>API: POST /login (ID, Pass) API->>DB: SELECT (パスワード照会) DB-->>API: 照会結果 API->>API: トークン発行 API-->>UI: 200 OK (トークン付与) UI->>UI: ホーム画面へ遷移

3.2 クラス図（UML）

システムを構成するクラス（データと振る舞いのまとまり）の構造と、クラス間の関係性を定義します。

📌 使用事例: 「釣り具の管理機能」の設計
classDiagram class Tackle { +String name; +String maker } class Rod { +Float length } class Reel { +Float gear_ratio } Tackle <|-- Rod : 継承 Tackle <|-- Reel : 継承

🌐 4. 外部とのインターフェースを取り決める技法 {#interface}

4.1 API仕様書（OpenAPI / Swagger 等）

システム間で通信するためのルール（エンドポイントURL、パラメータ、データ構造）を厳密に定義します。

📌 使用事例: 「お天気情報取得API」の設計
GET /api/v1/weather/{location_id}

# Request path param: location_id (integer, required)

# Response (200 OK) { "temp": 25.5, "weather": "sunny" }

🎯 5. 複雑な条件分岐の整理（テスト兼用） {#decision}

5.1 デシジョンテーブル（決定表）

複数の条件が複雑に絡み合うビジネスロジックにおいて、全組み合わせに対する動作をマトリクス状に整理します。

📌 使用事例: 「送料無料の判定ロジック」の設計
条件 / ルール1234
プレミアム会員か？YNNN
購入額5000円以上か？-YNN
キャンペーン中か？--YN
【結果】送料無料か？適用適用適用対象外

全ての組み合わせを可視化することで仕様の抜け漏れを防ぎ、そのままテストケースとしても活用できます。
