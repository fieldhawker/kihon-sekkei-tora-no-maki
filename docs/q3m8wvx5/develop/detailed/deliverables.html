詳細設計フェーズで作成する D-01〜D-18 の成果物について、使うツール・作り方・注意点をまとめています。

🧭
初心者向け：まず押さえる「必須」成果物
迷ったら最初はここを優先すると手戻りが減ります：D-03（OpenAPI）、D-06〜D-09（物理ER/インデックス/主要クエリ/マイグレーション）、D-12（セキュリティ）、D-13〜D-14（テスト設計）、D-15〜D-16（規約・環境）。

🗺️ 作成のおすすめ順（迷ったときの道順）

  1. 共通ライブラリ（C-01〜C-05） を先に固める（認証・エラー・レスポンス）
  2. D-03 OpenAPI を先に書く（モックでフロントが先行できる）
  3. D-01〜D-02 主要機能/画面の処理フロー・入力ルールを固める
  4. D-06〜D-09 DBを確定（型/桁/制約・インデックス・クエリ・マイグレーション）
  5. D-12〜D-16 セキュリティ・テスト・規約・環境（実装の品質を揃える）
  6. D-17〜D-18 取りまとめ（総括・トレーサビリティ）

🧩 D-01 詳細機能設計書 — 作成ガイド {#d01}

🧩

章立て（テンプレ）

  1. 対象機能（機能ID/名称/目的）とスコープ
  2. 前提（権限・状態・入力条件）
  3. 処理フロー（正常系/例外系）
  4. 入出力（リクエスト/レスポンス/DB更新/イベント）
  5. エラー（エラーコード、メッセージ、HTTP、復旧方針）
  6. 監査ログ（誰が/いつ/何を）
観点最低限の書き方よくある抜け
正常系入力→検証→永続化→レスポンストランザクション境界
例外系認証/認可/入力NG/整合性/外部失敗リトライ可否・通知
状態遷移ステータスと遷移条件並行更新（楽観/悲観ロック）
楽観ロック＝「更新時にバージョン番号が変わっていたらエラー」方式（衝突を後から検知）。悲観ロック＝SELECT FOR UPDATEで先にDBをロックする方式（衝突を事前に防止）。更新頻度が低いなら楽観ロックが一般的

📄 D-03 詳細API仕様書（OpenAPI）— 作成ガイド {#d03}

🔧
推奨ワークフロー
① openapi.yaml を作成 → ② Swagger UI でブラウザ確認 → ③ Prism でモックサーバー起動 → ④ フロントエンドがモックに対してUI開発 → ⑤ バックエンド実装完了後に本物に切り替え 
openapi: 3.0.3
info:
  title: 経費精算システム API
  version: 1.0.0

paths:
  /api/v1/expenses:
    post:
      summary: 申請書作成
      security:
        - bearerAuth: []
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ExpenseCreateRequest'
      responses:
        '201':
          description: 作成成功
        '400':
          description: バリデーションエラー
        '401':
          description: 認証エラー

components:
  schemas:
    ExpenseCreateRequest:
      type: object
      required: [title, amount, date]
      properties:
        title:
          type: string
          maxLength: 100
        amount:
          type: integer
          minimum: 1
        date:
          type: string
          format: date
ツール用途コマンド例
Swagger UIOpenAPIの可視化・確認docker run -p 8080:8080 -e SWAGGER_JSON=/openapi.yaml swaggerapi/swagger-ui
Prismモックサーバー起動npx @stoplight/prism-cli mock openapi.yaml
openapi-generatorクライアントコード自動生成openapi-generator-cli generate -i openapi.yaml -g typescript-axios

🗄️ D-06 物理ER図 — 作成ガイド {#d06}

🗄️

物理ERで"詳細設計"として必要な差分

  • 型/桁/NULL: 物理型（varchar(100)等）とNULL許容を確定。
  • 制約: PK/FK/UNIQUE/CHECK の方針を明記。
  • 監査項目: created_at/updated_at、作成者などの持ち方。
  • 削除方針: 論理削除・履歴テーブルの有無。
📌
物理ER単体では完結しません。D-07（インデックス）、D-08（主要クエリ）、D-09（マイグレーション）とセットで確定します。

🗄️ D-09 マイグレーションスクリプト（Flyway）— 管理方針 {#d09}

ファイル命名規則説明
V1__create_users.sqlV{バージョン}__{説明}.sql（バージョンは昇順）
V2__create_expenses.sqlテーブル作成 DDL
V3__add_index_expenses.sqlインデックス追加
V4__alter_users_add_column.sqlALTER TABLE でカラム追加
🚫
Flywayの鉄則: 一度適用したマイグレーションファイルは絶対に編集しないこと。変更が必要な場合は必ず新しいバージョンのファイルを追加してください。 
-- V4__alter_users_add_column.sql
-- ユーザーテーブルに部署IDカラムを追加
ALTER TABLE users
  ADD COLUMN department_id BIGINT,
  ADD CONSTRAINT fk_users_department
    FOREIGN KEY (department_id) REFERENCES departments(id);

COMMENT ON COLUMN users.department_id IS '所属部署ID';

🔗 D-14 結合テスト設計方針書 — 作成ガイド {#d14}

🔗

方針として決める項目

  • 範囲: API単体〜外部IF連携まで、どこまでを結合で見るか。
  • 環境: テスト環境の構成、外部はモック/本番相当のどちらか。
  • データ: テストデータ作成方針、後始末、個人情報の扱い。
  • 観点: 主要業務シナリオ、性能・耐障害の扱い、ログ/監査確認。
  • 完了条件: 実施率/合格率、未解決不具合の許容範囲。
