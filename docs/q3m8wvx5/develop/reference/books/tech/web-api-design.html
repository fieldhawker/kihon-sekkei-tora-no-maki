🌐 Web APIの設計

Arnaud Lauret（API Handyman）著。使いやすく・長く使われるWeb APIを設計するための実践的な知識を体系的にまとめた一冊。「動けばいい」APIではなく、利用者が迷わず・間違えず使えるAPIを作るための考え方とノウハウを学べます。

🧭
このページの使い方（初心者向け）
APIは「他の人（やシステム）が使う窓口」です。自分だけが使う関数と違い、一度公開したAPIは簡単に変更できません。だからこそ最初の設計が重要です。「APIとは何か」→「URLの設計」→「リクエスト・レスポンス」→「エラー設計」→「バージョン管理」の順に読み進めると、実務で使える知識が整います。

🤔 そもそもAPIとは何か

🔌

APIはシステム同士をつなぐ「契約」

API（Application Programming Interface）とは、異なるシステムやプログラムが互いにやりとりするための「決まりごと」です。レストランに例えると、お客さん（API利用者）とキッチン（システム内部）の間に立つ「メニューとウェイター」の役割です。

📋 メニュー（API仕様）

「こういう注文ができますよ」という約束。利用者はメニューを見て何が頼めるかがわかります。

🍽️ 料理（レスポンス）

注文通りに料理が運ばれてきます。キッチンの中がどうなっているかは知らなくていい。

💡
Web APIとは：インターネット（HTTP）を通じて使えるAPIです。スマホアプリが天気情報を取得するとき、ECサイトが決済サービスに問い合わせるとき、裏側ではWeb APIが使われています。

🏛️ REST APIの基本原則

📐

RESTとは「Webのルールに則った設計スタイル」

REST（Representational State Transfer）は、Web APIを設計するための考え方・スタイルです。難しく聞こえますが、本質は「HTTPの仕組みを正しく使おう」ということです。

REST の原則意味具体例
リソース指向操作ではなく「モノ（リソース）」でURLを設計する/users/123（ユーザーという「モノ」）
HTTPメソッドを正しく使うGET/POST/PUT/DELETE で操作の種類を表す取得はGET、作成はPOST
ステートレス各リクエストは独立していて、前の状態を引きずらないセッション情報に依存しない
統一インターフェースどのリソースも同じルールでアクセスできる一貫したURL構造・レスポンス形式

🔧 HTTPメソッドの正しい使い分け

📮

メソッドは「動詞」——操作の意図を正確に伝える

HTTPメソッドは「何をしたいか」を伝える動詞です。URLに /getUser や /deleteOrder のような動詞を入れるのはREST的ではありません。操作はメソッドで、対象はURLで表現します。

GET — 取得（データを変えない）

GET /users/123 → ユーザー123の情報を返す。何度呼んでも同じ結果。副作用なし。

POST — 作成（新しいリソースを生成）

POST /users → リクエストボディの内容で新ユーザーを作成。毎回新しいリソースが生まれる。

PUT — 完全更新（リソース全体を置き換え）

PUT /users/123 → リクエストボディでユーザー123全体を上書き。送らなかった項目はnullになる。

PATCH — 部分更新（一部だけ変更）

PATCH /users/123 → 送った項目だけ更新。名前だけ変えたいときはPATCHが適切。

DELETE — 削除

DELETE /users/123 → ユーザー123を削除。

⚠️
よくある間違い：GET /deleteUser?id=123 のようにGETで削除を行うのはNG。GETはデータを変えない操作に限定すべきです。ブラウザのキャッシュやクローラーに予期せず実行される危険があります。

🗺️ URLの設計原則

🔗

良いURLは「読むだけで意味がわかる」

URLはAPIの「住所」です。利用者はURLを見て何のリソースにアクセスしているかを理解します。一貫性があり・予測可能で・直感的なURLを目指しましょう。

😰 悪いURL例

  • /getUsers
  • /user_list
  • /Users/GetById/123
  • /api/v1/createNewOrder
  • /orders/123/deleteItem/456

😊 良いURL例

  • GET /users
  • GET /users（統一）
  • GET /users/123
  • POST /orders
  • DELETE /orders/123/items/456

URL設計のルール

① 名詞の複数形を使う

リソースは「モノ」なので名詞で表現します。/users、/orders、/products のように複数形で統一するのが一般的です。

② 階層はリソースの関係を表す

/users/123/orders は「ユーザー123の注文一覧」を意味します。親子関係があるリソースは階層で表現します。ただし深くなりすぎると読みにくいので、3階層程度が目安です。

③ 小文字・ハイフン区切りで統一

/product-categories ○、/productCategories や /product_categories は避ける。URLは大文字小文字が区別されるため、すべて小文字に統一します。

④ クエリパラメータはフィルタリング・ソートに使う

/users?status=active&sort=created_at のように、検索条件・絞り込み・並び替えはクエリパラメータで表現します。

📦 リクエスト・レスポンスの設計

💌

やりとりのデータ形式を統一する

現代のWeb APIではJSON形式が標準です。フィールド名・データ型・構造を一貫させることで、利用者が迷わず使えるAPIになります。

レスポンスの基本構造

✅ 単一リソースの取得

{
  "id": 123,
  "name": "田中太郎",
  "email": "tanaka@example.com",
  "createdAt": "2024-01-15T09:00:00Z"
}

✅ 一覧の取得（ページング付き）

{
  "items": [...],
  "total": 250,
  "page": 1,
  "perPage": 20,
  "hasNext": true
}
フィールド名はcamelCase（またはsnake_caseで統一）

createdAt か created_at かは好みがありますが、プロジェクト全体で統一することが最重要です。混在しているのが最もよくないです。

日付・時刻はISO 8601形式で

"2024-01-15T09:00:00Z" のようにUTC+ISO 8601形式を使います。「2024/01/15 18:00」のような独自形式は避けましょう。タイムゾーン問題を防げます。

不要なデータを返さない

パスワードハッシュや内部管理フラグなど、利用者に不要な情報は含めません。セキュリティリスクにもなります。

🚦 HTTPステータスコードを正しく使う

📊

ステータスコードは「処理結果の信号」

HTTPステータスコードは、リクエストの結果を数字で伝えます。常に200を返してエラー情報をレスポンスボディに入れるやり方は避けましょう。HTTPの仕組みを正しく使うと、クライアント側でのエラーハンドリングが楽になります。

コード意味使い所
200 OK成功GET・PUT・PATCHが成功したとき
201 Created作成成功POSTでリソースが作成されたとき
204 No Content成功・返すデータなしDELETEが成功したとき
400 Bad Requestリクエストが不正バリデーションエラー、必須項目がないとき
401 Unauthorized認証が必要ログインしていない（トークンがない）とき
403 Forbidden権限がないログインしているが操作する権限がないとき
404 Not Found見つからない指定したIDのリソースが存在しないとき
409 Conflict競合すでに存在するメールアドレスで登録しようとしたとき
500 Internal Server Errorサーバーエラー予期しないエラーが発生したとき
⚠️
「なんでもとりあえず200」は禁止：エラーが起きても200を返してボディに {"success": false, "error": "..."} を入れる実装をよく見かけます。これだとクライアント側でステータスコードを見るだけではエラー判定できず、常にボディを解析しなければなりません。

🚨 エラーレスポンスの設計

🔍

エラーは「なぜ失敗したか」を伝える

エラーが返ってきたとき、利用者が「何が問題で、どう直せばいいか」を理解できるように設計することが大切です。エラーメッセージは利用者への手紙です。

😰 悪いエラーレスポンス

{
  "error": true,
  "message": "Error"
}

何のエラーか全くわからない。

😊 良いエラーレスポンス

{
  "code": "VALIDATION_ERROR",
  "message": "入力内容に誤りがあります",
  "details": [
    {
      "field": "email",
      "message": "有効なメールアドレスを入力してください"
    }
  ]
}

エラーレスポンスに含めるべき情報

エラーコード（機械が読む用）

"code": "VALIDATION_ERROR" のような識別子。クライアントがコードで分岐処理を書けます。数値より文字列の方が意味が伝わります。

メッセージ（人間が読む用）

何が問題かを人間の言葉で説明します。開発中のデバッグに役立ちます。ただし本番環境では内部情報（スタックトレースなど）を含めないこと。

詳細（バリデーションエラーの場合）

どのフィールドに何の問題があるかをリストで返します。フォームのどの項目を直せばよいかがすぐにわかります。

🔐 認証と認可の設計

🛡️

「誰か」と「何ができるか」は別の問題

認証と認可は混同されがちですが、まったく別の概念です。

🪪 認証（Authentication）

「あなたは誰ですか？」を確認すること。ログイン・トークン検証がこれにあたります。

例：IDとパスワードを確認して「田中さんです」と証明する

🚪 認可（Authorization）

「その操作をしていいですか？」を確認すること。権限チェックがこれにあたります。

例：田中さんは一般ユーザーなので管理者ページは見られない

よく使われる認証方式

方式仕組み特徴
APIキー発行した固定の文字列をヘッダーで送るシンプル。ただしキーが漏れると誰でも使えてしまう
Bearer Token（JWT）ログイン後にトークンを発行し、以降はそれを送る現在の主流。Authorization: Bearer {token} ヘッダーで送る
OAuth 2.0「別サービスのアカウントで許可」を与える仕組み「Googleでログイン」「GitHubでログイン」はこれ
⚠️
401と403を混同しない：未ログイン（認証なし）は 401 Unauthorized、ログイン済みだが権限なし（認可失敗）は 403 Forbidden です。この区別はクライアント側の処理（再ログインを促すかどうか）に影響します。

📌 APIのバージョン管理

🔄

一度公開したAPIは簡単に変えられない

APIは「利用者との契約」です。突然URLの構造やフィールド名を変えると、すでに動いているクライアントが壊れます。変更が必要なときはバージョニングで古いAPIと新しいAPIを共存させるのが基本です。

バージョニングの方法（3パターン）

① URLパスに含める（最も一般的）

/api/v1/users、/api/v2/users

見てすぐわかる。ブラウザのアドレスバーでも確認しやすい。多くのAPIが採用。

② クエリパラメータで指定

/users?version=2

URLは変わらないが、バージョンが見えにくい。

③ カスタムヘッダーで指定

Accept: application/vnd.myapi.v2+json

URLを汚さないが、ヘッダーを意識して設定する必要があり複雑になりがち。

💡
破壊的変更と非破壊的変更：「フィールドの追加」は非破壊的変更（既存クライアントが壊れない）なのでバージョンアップ不要。「フィールドの削除・名称変更・型の変更」は破壊的変更なのでバージョンアップが必要です。

📄 ページングとフィルタリング

📋

大量データを扱うときの設計

データが数千・数万件あるとき、全件を一度に返すのはサーバー・クライアント双方に負荷がかかります。ページング（分割して返す）を設計に組み込みましょう。

オフセットベースのページング

GET /users?page=2&perPage=20

「2ページ目の20件」という指定。直感的でUIとの相性がよい。ただしデータが挿入・削除されると結果がずれることがある。

カーソルベースのページング

GET /users?cursor=eyJpZCI6MTIzfQ&limit=20

「この位置から20件」という指定。データの挿入・削除があっても重複・欠落が起きない。SNSの無限スクロールに適している。

フィルタリング・ソートのパラメータ例

GET /users?status=active&role=admin&sort=created_at&order=desc
GET /products?minPrice=1000&maxPrice=5000&category=electronics

フィルタ条件はクエリパラメータで渡します。パラメータ名は一貫したルール（snake_case など）で統一しましょう。

📖 APIドキュメントの重要性

✍️

使われるAPIには良いドキュメントがある

どれだけ良く設計されたAPIでも、ドキュメントがなければ利用者は使い方がわからず使ってもらえません。APIドキュメントはAPIと同じくらい重要な成果物です。

OpenAPI（Swagger）仕様で記述する

YAMLまたはJSONでAPI仕様を記述すると、自動的にインタラクティブなドキュメント（Swagger UI）を生成できます。「仕様＝ドキュメント」になるため、ズレが生じにくいです。

ドキュメントに含めるべき情報
  • エンドポイントのURL・HTTPメソッド
  • リクエストパラメータ・ボディの型と説明、必須/任意の区別
  • レスポンスの構造と各フィールドの説明
  • エラーコードの一覧と意味
  • 具体的なリクエスト・レスポンスの例（サンプル）
コードと一緒にドキュメントも更新する

実装が変わったのにドキュメントが古いままだと、利用者が混乱します。プルリクエストのチェックリストに「ドキュメント更新」を含めるなど、仕組みで担保しましょう。

💡
初心者へのアドバイス：APIを作ったら、まず「自分がこのAPIの利用者だったら使えるか」を考えてみましょう。ドキュメントなしに自分の作ったAPIを使ってみると、足りない情報がよくわかります。

🎯 良いAPI設計のチェックリスト

✅

設計・レビュー時に確認するポイント

APIを設計・実装したら、以下のポイントを確認しましょう。経験を積むにつれて、最初から意識できるようになります。

URL設計：名詞の複数形、小文字、動詞はメソッドで表現しているか
HTTPメソッド：GET/POST/PUT/PATCH/DELETE を正しく使い分けているか
ステータスコード：処理結果に応じた適切なコードを返しているか
エラーレスポンス：エラーコード・メッセージ・詳細が含まれているか
認証・認可：401と403を適切に使い分けているか
命名の一貫性：フィールド名・パラメータ名のケースが統一されているか
ページング：大量データを返す一覧APIにページング機能があるか
ドキュメント：全エンドポイントのリクエスト・レスポンス・エラーが文書化されているか
