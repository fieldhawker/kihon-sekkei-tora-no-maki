🌍 Webを支える技術

山本陽平 著（技術評論社）。HTTP・URI・HTMLなど、Webの根幹を支える技術を体系的に解説した一冊。「なぜWebはこんな仕組みなのか」を理解することで、設計の判断根拠を持ったエンジニアになるための基礎知識を学べます。

🧭
このページの使い方（初心者向け）
ブラウザでURLを開いたとき、裏側で何が起きているか知っていますか？Webは「HTTP」「URI」「HTML」という3つの技術の組み合わせで動いています。この順番に読み進めると、Webの全体像が体系的につながります：「URIとは」→「HTTPの仕組み」→「ステータスコード」→「RESTとは」→「実務への活かし方」。

🌐 Webとは何か

📖

Webは「リソースをつなぐ仕組み」

Web（World Wide Web）は、インターネット上に散らばる情報（リソース）をリンクでつなぐ仕組みです。メールや FTP とは異なり、Web はハイパーリンクによって情報が網の目（Web）のようにつながっているのが特徴です。

🔗

URI

リソースの「住所」。どこにある情報かを一意に指し示す

📡

HTTP

情報の「受け渡し方法」。クライアントとサーバーが会話するプロトコル

📄

HTML

情報の「表現形式」。構造と意味を持ったドキュメント

💡
ポイント：Webはこの3つ（URI・HTTP・HTML）が組み合わさって機能しています。API設計・システム設計においても、この基礎を知っているかどうかで判断の質が変わります。

🔗 URI — リソースを指し示す「住所」

📍

URIの構造を理解する

URI（Uniform Resource Identifier）は、Web上のリソース（情報）を一意に識別するための文字列です。よく使われる URL は URI の一種です。

https://example.com:443/users/123?sort=name#section1
スキーム　　ホスト名　ポート　パス　　　クエリ　フラグメント
構成要素役割例
スキーム通信プロトコルの種類https, http, ftp
ホスト名サーバーのドメイン名またはIPアドレスexample.com, 192.168.1.1
ポートサーバー上のサービス番号（省略可）:443（HTTPS）, :80（HTTP）
パスサーバー上のリソースの場所/users/123
クエリ追加パラメータ（?以降）?sort=name&page=2
フラグメントドキュメント内の位置（#以降）#section1

URIの設計原則

✅ 良いURI

  • 変わらない（永続的）
  • 何のリソースか一目でわかる
  • 短くシンプル
  • 小文字・ハイフン区切り
  • /users/123
  • /blog-posts/hello-world

❌ 悪いURI

  • 動詞が入っている
  • 実装の詳細が漏れている
  • 長くて読みにくい
  • 大文字・アンダースコア混在
  • /getUser?id=123
  • /blog_posts/HelloWorld.php
⚠️
Cool URIs don't change（良いURIは変わらない）
Tim Berners-Lee（Webの発明者）の言葉。一度公開したURIは変えてはいけません。URIを変えるとリンクが壊れ、検索エンジンのインデックスも失われます。システム設計の際は「このURIが5年後も使えるか」を考えましょう。

📡 HTTP — クライアントとサーバーの「会話ルール」

🤝

HTTPの基本的な仕組み

HTTP（HyperText Transfer Protocol）は、ブラウザ（クライアント）とサーバーが情報をやりとりするためのプロトコル（取り決め）です。HTTPの通信は必ず「リクエスト」と「レスポンス」のペアで成り立ちます。

💻
クライアント
（ブラウザ・スマホアプリ）
→ リクエスト →
📨
← レスポンス ←
🖥️
サーバー
（Webサーバー・APIサーバー）

HTTPリクエストの構成

GET /users/123 HTTP/1.1　　　← リクエストライン（メソッド + パス + バージョン）
Host: example.com　　　　　← ヘッダー
Authorization: Bearer token123
Accept: application/json
　　　　　　　　　　　　← 空行（ヘッダーとボディの区切り）
（ボディ：GETの場合は省略）

HTTPレスポンスの構成

HTTP/1.1 200 OK　　　　　← ステータスライン
Content-Type: application/json　← ヘッダー
Content-Length: 85

{ "id": 123, "name": "山田太郎" }　← ボディ（実際のデータ）

⚙️ HTTPメソッド — 操作の「動詞」

🎯

HTTPメソッドで操作の種類を表す

HTTPにはリソースに対して行う操作の種類を表す「メソッド」があります。URIが「どのリソースか」を示すのに対し、メソッドは「そのリソースに何をするか」を示します。

メソッド意味使用例冪等性安全性
GET リソースの取得 ユーザー情報を取得する ✅ あり ✅ 安全
POST リソースの作成 新しいユーザーを登録する ❌ なし ❌ 非安全
PUT リソースの完全更新（置き換え） ユーザー情報を全て書き換える ✅ あり ❌ 非安全
PATCH リソースの部分更新 ユーザーのメールだけ変更する ❌ なし ❌ 非安全
DELETE リソースの削除 ユーザーアカウントを削除する ✅ あり ❌ 非安全
📌
冪等性（べきとうせい）とは：
同じ操作を何度繰り返しても、結果が変わらない性質。例えば DELETE /users/123 は、1回でも10回でも「ユーザー123が存在しない状態」になります。
📌
安全性とは：
サーバーの状態を変更しない性質。GET はデータを読むだけで何も変えません。POST/PUT/PATCH/DELETE はサーバーのデータを変更するので「非安全」です。

🚦 HTTPステータスコード — 結果の「信号機」

✅

ステータスコードで処理結果を伝える

HTTPレスポンスには必ず3桁の数字（ステータスコード）が含まれます。このコードで「リクエストが成功したか・失敗したか・なぜか」を伝えます。信号機のように色（番号帯）で意味が分かれています。

番号帯意味代表的なコード
1xx 情報（処理継続中） 100 Continue
2xx 成功 200 OK / 201 Created / 204 No Content
3xx リダイレクト（別の場所へ） 301 Moved Permanently / 302 Found
4xx クライアントエラー（リクエストに問題） 400 Bad Request / 401 Unauthorized / 403 Forbidden / 404 Not Found
5xx サーバーエラー（サーバー側に問題） 500 Internal Server Error / 503 Service Unavailable

よく混乱するステータスコード

401 Unauthorized

「誰か教えてください（認証）」。ログインしていない、またはトークンが無効。名前の通り「認可されていない」ではなく実は「認証されていない」状態。

403 Forbidden

「あなたには権限がない（認可）」。ログインはしているが、そのリソースへのアクセス権がない。例：一般ユーザーが管理者ページにアクセス。

201 Created

POST でリソースを作成したときに返す。200 OK ではなく 201 を使うことで「新しく作られた」ことが明確になる。Location ヘッダーで新リソースのURIを返すとより親切。

204 No Content

DELETE や PUT の成功時に返す。「成功したけどボディはない」。200 との違いは「レスポンスボディが空」であること。

📋 HTTPヘッダー — メタ情報の「封筒」

✉️

ヘッダーはリクエスト・レスポンスの「付箋」

HTTPヘッダーは、ボディ（実際のデータ）とは別に付加されるメタ情報です。手紙に例えると「封筒の表書き」にあたります。

ヘッダー名役割例
Content-Type ボディのデータ形式 application/json, text/html
Accept クライアントが受け入れられる形式 application/json
Authorization 認証情報 Bearer eyJhbGc...
Cache-Control キャッシュの制御 no-cache, max-age=3600
Location リダイレクト先や作成リソースのURI /users/456
ETag リソースのバージョン識別子（キャッシュ制御に使用） "abc123"
💡
Content-Type は必ず指定しよう： サーバーは Content-Type を見てボディの解析方法を決めます。JSONを送るのに application/json を指定しないと、サーバーが正しく解釈できないことがあります。

🔄 コンテントネゴシエーション

🤝

クライアントとサーバーが「形式」を交渉する

同じURIのリソースでも、クライアントの要求に応じて異なる形式で返すことができます。これをコンテントネゴシエーション（内容交渉）と呼びます。

リクエスト（クライアント側）

GET /users/123 Accept: application/json

→ JSON形式で返してほしい

リクエスト（別のクライアント）

GET /users/123 Accept: text/xml

→ XML形式で返してほしい

同じ GET /users/123 でも、Accept ヘッダーによってJSONまたはXMLで返せます。URIを変えずに多様なクライアントに対応できるのがポイントです。

🏛️ REST — Webの「設計思想」

📐

RESTはWebの仕組みを正しく活かす設計スタイル

REST（Representational State Transfer）は、Roy Fielding が2000年の博士論文で提唱した、Webシステムのアーキテクチャスタイルです。難しいルール集ではなく、「Webの仕組みをそのまま活かして設計すれば、スケーラブルで使いやすいシステムになる」という考え方です。

RESTの制約意味なぜ重要か
クライアント・サーバー分離 UIとデータ管理を分けて独立させる フロントとバックエンドを別々に進化させられる
ステートレス 各リクエストは必要な情報を全て含む サーバーのスケールアウトが容易になる
キャッシュ可能 レスポンスをキャッシュできるかを明示する パフォーマンス向上・帯域節約
統一インターフェース URI・HTTPメソッド・ステータスコードを一貫して使う 誰でも使い方を推測できるAPIになる
階層型システム プロキシ・ゲートウェイを透過的に挟める セキュリティ・ロードバランシングが容易
💡
REST は仕様書ではなく思想： 「RESTful API」と言っても、厳密に全ての制約を満たしているシステムはほとんどありません。重要なのは「HTTPの仕組みを正しく使う」という考え方を持ち、URIでリソースを表し、メソッドで操作を表す設計をすることです。

⚡ キャッシュ — 賢い「手間の省き方」

🗂️

キャッシュで性能を向上させる

キャッシュとは、一度取得したデータを手元に保存し、次回から再取得を省く仕組みです。Webのパフォーマンス向上において非常に重要な技術です。

Cache-Control の値意味使いどころ
no-cache毎回サーバーに確認してからキャッシュ利用ユーザー固有データ（マイページなど）
no-storeキャッシュ禁止（保存しない）個人情報・決済情報など機密データ
max-age=36003600秒（1時間）はキャッシュを使い回す更新頻度の低い静的コンテンツ
publicプロキシ等でもキャッシュ可能誰でも同じ内容になるリソース
privateブラウザのみキャッシュ可（プロキシ不可）ログインユーザー固有のレスポンス

条件付きGET（効率的なキャッシュ更新）

リソースが変更されていない場合に無駄な転送を避ける仕組みです。

ETag を使った方法

サーバーはレスポンスに ETag: "abc123" を付与。次回クライアントは If-None-Match: "abc123" で問い合わせ。変更なければ 304 Not Modified が返る。

Last-Modified を使った方法

サーバーは Last-Modified: Wed, 21 May 2025 10:00:00 GMT を返す。クライアントは If-Modified-Since で同じ日時を送る。変更なければ 304 が返る。

🔒 HTTPS — 安全な通信を守る「暗号化」

🛡️

HTTPSでデータを守る

HTTPS（HTTP over TLS）は、HTTPにTLS（Transport Layer Security）による暗号化を加えたプロトコルです。現代のWebではHTTPSは必須です。

❌ HTTP（暗号化なし）

  • 通信内容が丸見え（盗聴可能）
  • データの改ざんが可能
  • なりすましが可能
  • パスワードや個人情報が危険

✅ HTTPS（TLS暗号化あり）

  • 通信が暗号化される（盗聴不可）
  • 改ざん検知が可能
  • サーバーの正当性を証明
  • SEOにも有利
⚠️
新規開発では必ずHTTPSを使う： 特に認証情報・個人情報・決済情報を扱う場合は絶対にHTTPSが必要です。Let's Encrypt などで無料でSSL証明書を取得できるため、HTTPSを使わない理由はありません。

📰 データフォーマット — JSON vs XML vs その他

📋

データを「何の形式」で送受信するか

Web APIがデータをやりとりするための形式（フォーマット）にはいくつかの選択肢があります。現代のWeb APIではJSON が事実上の標準です。

形式特徴使用シーン
JSON 軽量・JavaScriptと親和性が高い・人が読みやすい Web API のほぼ全て（現代の標準）
XML 厳密な構造・スキーマ定義・SOAP等で使用 レガシーシステム・企業間連携・RSS/Atom
CSV シンプル・Excelで開ける データ一括エクスポート・インポート
Protocol Buffers バイナリ形式・高速・スキーマ必須 マイクロサービス間通信（gRPC）

JSON の例

{
  "id": 123,
  "name": "山田太郎",
  "email": "yamada@example.com"
}

XML の例

<user>
  <id>123</id>
  <name>山田太郎</name>
  <email>yamada@example.com</email>
</user>

💼 実務での活かし方

🛠️

この知識を設計・開発に活かす

🔍 デバッグ・調査

  • ブラウザのDevTools → Network タブでHTTPの実際のやりとりを確認
  • ステータスコードで問題の原因を素早く特定（4xx か 5xx か）
  • curl コマンドでAPIの動作を手動確認

📐 API設計

  • URIはリソース（名詞）で設計し、操作はHTTPメソッドで表す
  • ステータスコードを正しく使い、クライアントが処理しやすいレスポンスを返す
  • Cache-Control を適切に設定してパフォーマンスを向上

🔒 セキュリティ

  • 必ずHTTPSを使う（特に認証・個人情報を扱う場合）
  • no-store で機密データのキャッシュを防ぐ
  • 401/403 を正しく使い分け、セキュリティ設計を明確にする

⚡ パフォーマンス

  • 静的ファイルに長めの max-age を設定してキャッシュ活用
  • ETag / Last-Modified で条件付きGETを実装し転送量を削減
  • CDNを活用してユーザーへの配信を高速化
