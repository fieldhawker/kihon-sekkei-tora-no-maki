初めて学ぶソフトウェアのテスト技法

Lee Copeland著。ソフトウェアテストの代表的な技法を体系的に解説した入門書。「どこをどうテストすればよいか」という疑問に答える具体的な手法が豊富に紹介されており、テスト初心者から中堅エンジニアまで幅広く活用できる一冊です。

🧭
このページの使い方（初心者向け）
「テストを書けと言われたけど、何をどうテストすればいい？」という疑問から始めましょう。「なぜテストが必要か」→「代表的な技法」→「実践での使い分け」の順に読むと、テスト設計の全体像が見えてきます。

❓ そもそもなぜテストが必要なのか

🔍

バグはいつ発見するのが一番安い？

テストの目的はバグを「できるだけ早く・安く」発見することです。バグの修正コストは、発見が遅れるほど指数的に大きくなります。

💻

開発中に発見

修正コスト：低
自分で気づいて直すだけ

🔧

テストで発見

修正コスト：中
開発に戻って修正が必要

🚨

本番で発見

修正コスト：最大
顧客影響・信頼損失も発生

⚠️
「全部テストすればいい」は幻想：すべての入力パターンを網羅的にテストすることは現実的に不可能です。だからこそ「効率よく重要なテストケースを選ぶ技法」が必要になります。

📊 同値分割（Equivalence Partitioning）

🗂️

似た入力をグループにまとめてテストを減らす

同じ処理結果になる入力値をグループ（同値クラス）に分け、各グループから代表値を1つだけ選んでテストする技法です。テストケース数を大幅に削減しながら、十分な品質を確保できます。

例：年齢入力フォーム（0〜120歳が有効）

無効クラス①
−∞〜−1
代表値：−5
有効クラス
0〜120
代表値：25
無効クラス②
121〜+∞
代表値：200

→ 無数にある入力パターンが、わずか3つのテストケースに集約されます。

💡
ポイント：「同じグループ内の値は、テスト結果が同じはず」という前提で成立します。有効クラスと無効クラスの両方を必ず作ることが重要です。

📏 境界値分析（Boundary Value Analysis）

🎯

バグは「境界」に集まる

プログラムのバグは、条件の境目（境界値）付近で起きやすいという経験則に基づく技法です。同値分割と組み合わせて使うことで、見逃しやすいバグを効率よく検出できます。

例：年齢入力フォーム（0〜120歳が有効）でのテストポイント

無効
−1
有効（下限）
0
有効（下限+1）
1
…
有効（上限−1）
119
有効（上限）
120
無効
121

→ 境界の「ちょうど」「一つ手前」「一つ先」を必ずテストします。

⚠️
よくある実装ミス：「0以上120以下」を実装するつもりが「0より大きく120より小さい」と書いてしまう（<と<=の混同）。境界値テストはこういったオフバイワンエラーを検出するために特に有効です。

📋 デシジョンテーブル（Decision Table Testing）

📊

複数の条件の組み合わせを整理する

複数の条件が絡み合う処理は、テストケースの漏れが起きやすいです。デシジョンテーブル（判定テーブル）を使うと、条件の組み合わせを表で整理し、抜け漏れなくテストできます。

例：ショッピングサイトの送料計算

条件 / ケース ケース① ケース② ケース③ ケース④
会員登録済み ✅ YES ✅ YES ❌ NO ❌ NO
購入金額 5,000円以上 ✅ YES ❌ NO ✅ YES ❌ NO
期待結果（送料） 無料 200円 300円 500円

→ 表にすることで「条件の組み合わせに漏れがないか」が一目でわかります。

💡
ポイント：条件が N 個あると最大 2N 通りの組み合わせが生まれます。多い場合はビジネス的に意味のある組み合わせに絞り込む「縮退」を行います。

🔄 状態遷移テスト（State Transition Testing）

🗺️

画面や処理の「状態」を図にしてテストする

システムが「状態」を持つ（ログイン中・ログアウト中、注文前・注文後など）場合、状態の遷移を図に起こしてからテストケースを作ると、見逃しが少なくなります。

例：注文の状態遷移

注文未確定
→ 注文確定ボタン →
注文確定済み
→ 発送処理 →
発送済み
→ 受取確認 →
完了
注文未確定
→ キャンセル →
キャンセル済み
注文確定済み
→ キャンセル →
キャンセル済み

テストすべきポイント

  • 正常遷移：期待通りの操作で状態が正しく変わるか
  • 逆方向の遷移：「発送済み」から「注文未確定」には戻れないか
  • 不正な遷移：「キャンセル済み」の注文をさらにキャンセルできないか
  • ガード条件：在庫なし状態で注文確定を押したらどうなるか

👤 ユースケーステスト（Use Case Testing）

🎬

ユーザーの操作シナリオをそのままテストにする

実際にユーザーがどう操作するかを「シナリオ」として記述し、それをテストケースにする技法です。個々の機能が単体で動いても、連携部分でバグが起きることはよくあります。

例：ECサイトでの購入フロー

  1. ユーザーがログインする
  2. 商品を検索して一覧を表示する
  3. 商品をカートに追加する
  4. カートを確認して注文手続きへ進む
  5. 支払い方法を選択して注文を確定する
  6. 注文確認メールが届くことを確認する

→ 各ステップを順番に実行し、全体として正しく動くことを確認します。

💡
基本フローと代替フロー：「正常に完了する」基本フローだけでなく、「支払いが失敗する」「在庫切れになる」などの代替フローもテストすることが重要です。

🔭 探索的テスト（Exploratory Testing）

🕵️

「テスト設計」と「テスト実行」を同時に行う

あらかじめテストケースを全部作るのではなく、実際に操作しながらテスト内容を考える技法です。仕様書だけでは想定できなかったバグを見つけるのに特に有効です。

😰 スクリプトテストだけの問題

  • 仕様書にない操作はテストされない
  • テスト設計者の思い込みが入る
  • ユーザーの実際の使い方と乖離しやすい
  • 同じテストを繰り返し、新たなバグを見逃す

😊 探索的テストの強み

  • 使ってみて気になった点を即テスト
  • 予期しない操作の組み合わせを試せる
  • テスターの経験と直感を活かせる
  • 仕様書がない/不完全でも実施できる
⚠️
注意：探索的テストは「闇雲に触る」のとは違います。テスト目的（チャーター）を決めて時間を区切り（タイムボックス）、実施内容を記録することで再現性と品質を確保します。

🗺️ テスト技法の使い分けガイド

🧰

状況に合った技法を選ぶ

テスト技法はそれぞれ得意な対象が異なります。「この機能には何が合うか？」を考えながら選択しましょう。

技法 こんな時に使う 見つけやすいバグ
同値分割 入力値の種類が多い時 無効入力の処理漏れ
境界値分析 数値・文字数などに上下限がある時 オフバイワン、境界の条件ミス
デシジョンテーブル 複数の条件が絡む処理がある時 条件の組み合わせの見落とし
状態遷移テスト 画面や処理が状態を持つ時 不正な状態遷移・ガード漏れ
ユースケーステスト 機能間の連携をテストしたい時 シナリオ全体での結合バグ
探索的テスト 仕様が不完全または変更が多い時 想定外の操作パターンのバグ
💡
組み合わせるのが正解：どれか一つの技法だけで完璧なテストはできません。機能の性質や開発フェーズに応じて複数の技法を組み合わせることで、より高い品質を効率よく実現できます。

📌 テストの7原則（ISTQB）

📜

テストに関する普遍的な考え方

ソフトウェアテストには国際的に認められた7つの原則があります。これらを知っておくことで、テストに対する誤解を防ぎ、より効果的なテストが設計できます。

1
テストはバグの存在を示せるが、バグがないことは証明できない
テストをどれだけ行っても、バグがゼロとは言えません。テストはリスクを減らすものです。
2
網羅的テストは不可能
全ての入力・条件の組み合わせをテストすることは現実的に不可能。リスクに応じた優先付けが必要です。
3
早期テストで節約
テストは早く始めるほど修正コストが低くなります。要件定義段階からのレビューも立派なテストです。
4
欠陥の偏在（バグは密集する）
バグは特定のモジュールや機能に集中しがちです。バグが多い場所をリスクとして重点的にテストします。
5
殺虫剤のパラドックス
同じテストを繰り返すと新たなバグを見つけられなくなります。定期的にテストケースを見直しましょう。
6
テストはコンテキスト依存
医療システムと社内ツールでは求められるテストの厳密さが異なります。何をどこまでテストするかは目的次第です。
7
「バグなし」は品質ではない
テストをパスしても、ユーザーが求めていない機能を正しく動かしていれば意味がありません。品質とはニーズを満たすことです。

🚀 明日からできること

🌱

まず3つから始める

テスト技法を一度に全部マスターしようとする必要はありません。まずは以下の3つを意識するだけで、テストの質は大きく上がります。

① 有効値と無効値を必ず両方テストする

入力フォームをテストするとき、「正しい値」だけでなく「おかしな値」（空欄、マイナス値、文字列など）も必ず試しましょう。同値分割の考え方です。

② 「ちょうど」「一つ手前」「一つ先」を試す

「10文字まで入力可能」なら、9文字・10文字・11文字を必ずテストしましょう。境界値分析の考え方です。プログラミングのバグの多くはここに潜んでいます。

③ ユーザーの操作を頭からおさらいする

「ユーザーがどう使うか」を最初から最後まで自分で操作して確認しましょう。単体テストでは見えない連携バグをユースケーステストで発見できます。

💡
テストを書くことは「未来の自分への投資」：今書くテストが、3か月後の仕様変更・リファクタリング時に自分を守ってくれます。「動いてるからいいや」で省略した分だけ、将来の修正リスクが積み上がります。
