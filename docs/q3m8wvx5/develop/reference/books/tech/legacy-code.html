レガシーコードからの脱却

David Scott Bernstein著。「なぜコードはすぐに負債になるのか」を問い直し、テスト・設計・習慣の9つのプラクティスを通じてレガシーコードを生み出さない開発スタイルを提唱しています。初心者から中堅エンジニアへの成長に直結する一冊です。

🧭
このページの使い方（初心者向け）
「動けばいいや」でコードを書いていると、気づかないうちに誰も触れない"レガシーコード"が生まれます。このページでは「レガシーコードとは何か」→「なぜ生まれるのか」→「9つのプラクティス」の順に読むと、明日からの開発習慣が変わります。

❓ そもそも「レガシーコード」とは何か

🏚️

「古いコード」ではなく「テストのないコード」

本書での定義はシンプルです。レガシーコードとは、テストのないコードのことです。昨日書いたコードでも、テストがなければレガシーです。逆に10年前のコードでも、テストがあれば安心して変更できます。

😰 レガシーコードの特徴

  • テストがない（または少ない）
  • 変更すると別の場所が壊れる
  • コードを読んでも意図がわからない
  • 「なぜこう書いたか」が不明
  • 修正が怖くて誰も触れない

😊 健全なコードの特徴

  • テストが意図を説明している
  • 変更の影響範囲が明確
  • 名前を読めば何をするかわかる
  • 小さく・シンプルに保たれている
  • 安心して改修・機能追加できる
⚠️
「後でリファクタリングする」は幻想：締め切りや仕様変更に追われる現場で「後で整理する」と思ったコードは、ほぼ100%そのまま残ります。最初からきれいに書く習慣こそが、将来の自分と仲間を守る唯一の方法です。

🌱 レガシーコードはなぜ生まれるのか

🤔

3つの根本原因

原因① 「動けばOK」という基準

機能が動くかどうかだけを基準にすると、「読みやすさ」「変更しやすさ」が置き去りになります。開発速度を上げるには、品質への意識が不可欠です。

原因② テストを後回しにする習慣

「テストは実装が終わってから書く」という順序だと、テストを書きやすい設計になりません。結果として、テストがないか、あっても壊れやすいコードが生まれます。

原因③ 巨大な関数・クラスを許容する文化

「とりあえずここに追加しよう」の積み重ねが、数百行の関数を生みます。巨大なコードはテストも読解も困難で、修正のたびに副作用が生じます。

📋 本の構成：9つのプラクティス

🗂️

レガシーコードを生み出さないための9の習慣

#プラクティス一言で言うと
1WHYを伝え、HOWを隠すコードは「何をするか」より「なぜするか」を語るべき
2DRYを適用する同じことを2度書かない
3テストを先に書く実装前にテストを書くと設計が良くなる
4小さなコードを書く関数・クラスは小さく保つ
5コードをシンプルに保つ余計な複雑さを排除する
6振る舞いを明確に示すコードが何をするかを名前と構造で伝える
7安定した依存関係の原則を適用する変わりやすいものに依存しない
8依存性を注入する依存するオブジェクトは外から渡す
9協力し、レビューし合うチームでコードの品質を守る

💬 プラクティス1：WHYを伝え、HOWを隠す

🗣️

コードは「意図」を語るべき

処理の手順（HOW）はコード自体が示しています。コードが語るべきは「なぜそうするのか（WHY）」です。良い名前・小さな関数・適切なコメントがWHYを伝える手段になります。

😰 HOWしか語らない例

// ユーザーリストをループして
// statusが1のものを抽出してnewListに追加
List newList = new List();
for (User u : users) {
  if (u.status == 1) {
    newList.add(u);
  }
}

😊 WHYが伝わる例

// アクティブユーザーのみメール送信対象にする
List<User> targets = getActiveUsers(users);
// 別の場所で定義
List<User> getActiveUsers(List<User> users) {
  return users.stream()
    .filter(u -> u.isActive())
    .collect(toList());
}

💡
良い名前のチェック：関数名や変数名を読んだとき「何のためにあるのか」がすぐ分かれば合格です。data、list、flag、tmpのような名前は「HOW」も「WHY」も伝えていません。

♻️ プラクティス2：DRYを適用する

🔁

Don't Repeat Yourself — 繰り返しはバグの温床

DRY（Don't Repeat Yourself）とは「同じ知識をコードの2箇所以上に書かない」原則です。コピペしたコードは、片方を直してもう片方を直し忘れるバグの温床になります。

DRYが大切な理由

仕様が変わったとき、同じロジックが3箇所に書かれていたら3箇所すべて直す必要があります。1箇所だけ直して2箇所を直し忘れると、バグが生まれます。変更は1箇所だけで済むように設計しましょう。

「コードの重複」だけがDRY違反ではない

コードだけでなく、知識・ロジック・データ定義の重複もDRY違反です。例えば、DBのカラム名と変数名が別々に管理されていて同期が必要な状態も、DRY違反の一形態です。

「偶発的な重複」には注意

コードが似ていても、別の概念・別の理由で変化するなら無理にまとめないことも重要です。異なる画面の「ユーザー名表示」が同じコードでも、それぞれ別の仕様で変わるなら共通化するとかえって複雑になります。

🧪 プラクティス3：テストを先に書く（TDD）

🔬

テストは「仕様書」であり「安全網」

TDD（テスト駆動開発）では「テストを書く → 失敗を確認する → 実装する → テストを通す → リファクタリング」のサイクルを回します。テストを先に書くと、設計が自然と整理されます。

TDDのサイクル（Red → Green → Refactor）

🔴
Red

まず失敗するテストを書く。仕様を先に定義する

→
🟢
Green

テストが通る最小限の実装を書く

→
🔵
Refactor

テストを維持しながらコードを整理する

テストを先に書くメリット

  • 「何を作るか」を先に明確にできる（仕様の確認）
  • テストしやすい設計（= 変更しやすい設計）になる
  • 実装後に「テストを書く時間がない」問題が起きない
  • リファクタリング時の安全網になる
💡
「テストを書く時間がない」は本当か：テストなしで実装してバグが出た場合、デバッグ・修正・再確認のコストは、最初からテストを書くコストより大きいことがほとんどです。テストはコストではなく投資です。

✂️ プラクティス4：小さなコードを書く

📏

関数もクラスも「1つのことだけ」

単一責任の原則（SRP）：関数・クラスは1つのことだけを担当すべきです。複数のことをやっている関数は、テストが難しく・読みにくく・変更するとバグが出やすくなります。

関数の長さの目安

本書では「関数は5〜10行程度」を理想としています。「長い関数は複数の責務を持っているサイン」です。関数が長くなったら、意味のある部分を別の関数に切り出しましょう。

「関数を分割すべき」シグナル

  • 関数名に「and」「or」が入っている（例: saveAndSendEmail()）
  • 20行を超えている
  • コメントで処理ブロックを区切っている
  • ネストが3段以上になっている

🧹 プラクティス5：コードをシンプルに保つ

🎯

YAGNI — 今必要でないものは書かない

YAGNI（You Ain't Gonna Need It）：「将来使うかもしれない」と思って追加した機能やコードは、ほぼ使われないまま複雑さだけを増やします。今の要件だけを満たす最もシンプルな実装を心がけましょう。

複雑さが生まれる典型パターン

  • 「将来のため」の汎用化・抽象化
  • 「念のため」のフラグやif文の追加
  • 「使うかもしれない」設定値・オプションの乱用
  • 未使用のコードやコメントアウトの放置

リファクタリングとシンプル化

機能追加・バグ修正のたびに、周辺のコードを少しずつ整理する「ボーイスカウトルール（来たときより綺麗にして帰る）」を実践しましょう。一度に全部リファクタリングする必要はありません。

🏷️ プラクティス6：振る舞いを明確に示す

📣

コードが「自分を語る」設計にする

良いコードは、読めばその振る舞いと意図が分かります。テスト・名前・構造の3つがセットで「このコードは何をするのか」を語るべきです。

テストが仕様書になる

テスト名を読むと「このクラスはこういう振る舞いをする」とわかる状態が理想です。例：空のリストを渡すと空のリストを返す、未ログイン時は401を返す。テストはコードの利用例であり、生きた仕様書です。

型・構造で意図を伝える

boolean isActiveよりUserStatus status（列挙型）のように、型と構造で取りうる状態を明示すると、コードを読む人が間違いを犯しにくくなります。コンパイラが仕様を守ってくれます。

🔗 プラクティス7：安定した依存関係の原則

⚖️

変わりやすいものに依存しない

安定した依存関係の原則（SDP）：変わりやすいモジュール（詳細・実装）が、変わりにくいモジュール（抽象・インターフェース）に依存する方向にすべきです。これにより、詳細が変わっても上位の概念に影響が出なくなります。

😰 悪い例：抽象が詳細に依存

ビジネスロジック（注文処理）が特定のDBライブラリ（MySQL直接呼び出し）に直接依存している。DBを変えたら注文処理も変更が必要になる。

😊 良い例：詳細が抽象に依存

ビジネスロジックは「注文リポジトリ（インターフェース）」に依存し、MySQL実装はそのインターフェースを実装する。DBを変えても注文処理は変更不要。

💡
初心者向けのイメージ：コンセントの形（インターフェース）が統一されていれば、家電（実装）はどのメーカーでも差し替えられます。コードも同じで、「使い方の約束（インターフェース）」に依存すると、「実装」を自由に替えられます。

💉 プラクティス8：依存性を注入する（DI）

🔧

依存するオブジェクトは外から渡す

依存性注入（DI: Dependency Injection）とは、クラスが必要とするオブジェクトを内部で生成せず、コンストラクタや引数として外から受け取る設計パターンです。テストのしやすさが劇的に向上します。

😰 DI なし（テスト困難）

class OrderService {
  // 内部でDBに直接依存
  private db = new MySQLDatabase();
  void createOrder(Order order) {
    db.save(order); // テスト時もDBが必要
  }
}

😊 DI あり（テスト容易）

class OrderService {
  // 外から渡してもらう
  constructor(private repo: OrderRepository) {}


  void createOrder(Order order) {
    repo.save(order);
    // テスト時はモックを渡せる
  }
}

✅
DIのメリット：テスト時に本物のDBやAPIの代わりに「偽物（モック）」を渡せます。本番と同じ構造を保ちながら、依存するサービスなしでロジックを検証できます。

👥 プラクティス9：協力し、レビューし合う

🤝

コード品質はチームで守るもの

レガシーコードとの戦いは個人ではなくチームで行います。コードレビュー・ペアプログラミング・モブプログラミングを通じて、チーム全体でコードの品質意識を高め合うことが重要です。

コードレビューの目的

  • バグや設計の問題を早期発見する
  • 知識・設計の意図をチームで共有する
  • チームのコーディング規約・品質基準を揃える
  • 書いた本人が見逃しがちなミスをチェックする

ペアプログラミングの効果

2人で1つのコードを書く手法。「一人が書き、もう一人がレビューしながら指摘する」のリアルタイム版です。バグの埋め込みを防ぎ、知識移転もリアルタイムで行われるため、長期的に見てコスト効率が高い手法です。

心理的安全性の大切さ

「このコードはまずい」と言える・言ってもらえる関係がないと、誰もレガシーコードを指摘できなくなります。コードを批判するのであって人を批判するのではないという文化が、チームをレガシーから救います。

📌 まとめ：9つのプラクティス早見表

🎓

明日から実践できること

#プラクティス明日から試せること
1WHYを伝える変数名・関数名を「目的」を表す名前にしてみる
2DRY同じ処理が2箇所にあったら関数に切り出す
3テストを先に書く次の実装の前に、期待する動作を1つテストに書いてみる
4小さく書く20行を超えたら「分割できないか」考えてみる
5シンプルに保つ「今は不要だが将来使うかも」のコードを書かない
6振る舞いを示すテスト名を「〇〇したとき〇〇を返す」という形にする
7安定した依存具体クラスではなくインターフェースに依存させる
8依存性を注入するクラス内でnewしている箇所をコンストラクタ引数に変える
9協力するPRレビューで「なぜそうしたか」を質問・説明し合う
📖
本書の核心メッセージ：レガシーコードは悪意から生まれるのではなく、小さな妥協の積み重ねから生まれます。9つのプラクティスはすべて「今この瞬間の選択」を変えることから始まります。完璧を目指す必要はありません。次のコミットから、少しだけ良い選択をすることが、レガシーコードからの脱却への第一歩です。
