チェックリスト

テストケース作成チェックリスト

テストケースを新規作成・レビューする際に確認すべき項目のチェックリストです。網羅性・命名・独立性・可読性・境界値・モックの観点を網羅しています。

🧭
このチェックリストの使い方
1) テスト設計時に「1. テスト設計・網羅性」「2. 命名」を確認 → 2) 実装時に「3. テストの独立性」「4. AAA構造」を意識 → 3) レビュー前に「5. 境界値・異常系」「6. モック」を確認。
迷ったら、テストの意図が一読で分かること・テストが単独で実行できることを最優先にします。
📌
初心者がハマりやすい3点
① 正常系だけテストして「完璧だ」と思う（→ バグは境界値・異常系・エッジケースに潜む）
② テストが他のテストの実行結果に依存する（→ テストは完全に独立して実行できること）
③ 1つのテストメソッドで複数のことを確認しようとする（→ 1テスト1アサーション が基本）

1. テスト設計・網羅性

テスト技法概要例
同値分割同じ動作をするグループからそれぞれ1つ選んでテストする年齢チェック（0〜17歳: 未成年、18〜64歳: 成人、65歳〜: 高齢者）の各グループから1件ずつテスト
境界値分析区切りの前後の値をテストする18歳以上が条件なら「17歳・18歳・19歳」でテスト
決定表テスト複数条件の組み合わせを表にして網羅する会員区分（VIP/一般）× 購入金額（1万以上/未満）→ 4通りの割引率をテスト
状態遷移テスト状態の遷移パターンをカバーする注文状態（未確定→確定→発送済→完了）の各遷移と不正遷移（完了→未確定）をテスト

2. テストの命名

命名パターン例適した場面
メソッド名_条件_期待結果calculateDiscount_withVipMember_returns20PercentOffユニットテスト全般
should_期待動作_when_条件should_throwException_when_emailIsNullBDD スタイル
given_前提_when_操作_then_期待結果given_emptyCart_when_addItem_then_cartHasOneItem振る舞い記述が重要な場合
日本語メソッド名VIPメンバーに対して20%割引が適用される()テスト内容を日本語で明示したい場合（JUnit5・pytest等で使用可）

3. テストの独立性

⚠️
テスト間の依存が引き起こす問題: テストAが作成したデータをテストBが前提としている場合、Bを単独で実行すると失敗します。また実行順序が変わると結果が変わる「不安定なテスト（Flaky Test）」になります。
✅ 各テストで @BeforeEach でデータを準備し、@AfterEach または @Transactional でクリーンアップする。

4. AAA構造（Arrange-Act-Assert）

フェーズ役割例
Arrange（準備） テストに必要なオブジェクト・データを準備する 
var user = new User("Taro", UserType.VIP);
var cart = new ShoppingCart(user);
cart.addItem(new Item("書籍", 3000));
Act（実行） テスト対象のメソッドを1回だけ呼び出す 
var discountedPrice = cart.calculateTotal();
Assert（検証） 結果が期待通りか確認する 
assertEquals(2400, discountedPrice,
  "VIP会員は20%割引のはず");

5. 境界値・異常系テスト

対象確認すべき値テスト例
数値0、負数、最小値、最大値、最大値+1、最小値-1割引率計算: quantity=0・quantity=1・quantity=99999・quantity=100000（上限超え）
文字列null、空文字("")、スペースのみ、最大長ちょうど、最大長+1文字ユーザー名バリデーション: ""・"a"・"a".repeat(50)・"a".repeat(51)
コレクションnull、空リスト、要素1件、大量件数合計金額計算: 空カート・1アイテム・1000アイテム
日付・時刻うるう年、月末、年末年始、タイムゾーン境界年齢計算: 2/29生まれのうるう年以外の誕生日
例外テスト（JUnit5） 
var ex = assertThrows(IllegalArgumentException.class,
    () -> service.createUser(null));
assertEquals("emailは必須です", ex.getMessage());

6. モック・スタブの使い方

種別定義使いどころ
Stub決まった値を返すだけのダミー実装テスト対象が依存するオブジェクトの戻り値を固定したい場合。when(repo.findById(1L)).thenReturn(Optional.of(user))
Mock呼び出しの記録・検証もできるダミー実装「このメソッドが1回だけ呼ばれたこと」を検証したい場合。verify(emailService, times(1)).send(any())
Spy実際のオブジェクトの一部だけをオーバーライドする大部分は実装を使いつつ、一部の外部呼び出しだけを差し替えたい場合
Fake実際に動くが本番用ではない実装（InMemory DBなど）H2などのインメモリDBを使った結合テスト
⚠️
モック過多の罠: Repositoryのモックに依存したServiceのテストだけでは、SQLクエリのバグ・トランザクションの問題を検出できません。
ユニットテスト（高速・モック多用）と結合テスト（実DB使用）を組み合わせて使うことが重要です。

7. テストの保守性

項目具体例・考え方
テストデータのBuilder 
// Builderを使って必要な部分だけ変更する
var user = UserTestBuilder.defaultUser()
    .withEmail("test@example.com")
    .withType(UserType.VIP)
    .build();
各テストで全フィールドをベタ書きしない（フィールド追加時の修正箇所を最小化）
Flaky Testの原因と対策① 時刻依存 → Clock を注入してテストで固定 ② テスト間の状態共有 → BeforeEachでリセット ③ 外部サービス依存 → モックまたはWireMockを使う ④ 実行順依存 → テストの独立性を確保
