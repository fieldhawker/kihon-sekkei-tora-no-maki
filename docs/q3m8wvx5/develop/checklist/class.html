クラスを新規作成・レビューする際に確認すべき項目のチェックリストです。責務・命名・カプセル化・継承・依存関係・SOLID原則の観点を網羅しています。

🧭
このチェックリストの使い方
1) 設計時に「1. 責務・設計」「2. 命名」を確認 → 2) 実装時に「3. カプセル化」「4. 継承・コンポジション」を意識 → 3) 完成後に「5. 依存関係・SOLID」「6. テスト容易性」を確認。
迷ったら、単一責任・適切なカプセル化・依存の方向を最優先にします。
📌
初心者がハマりやすい3点
① クラスに何でも詰め込んで「神クラス（God Class）」になる（→ 単一責任原則を徹底）
② すべてのフィールドをpublicにしてカプセル化を壊す（→ フィールドはprivate、getterは必要なものだけ）
③ 「is-a関係」でない継承を使う（→ コンポジション（has-a）を優先する）

1 責務・設計

レイヤー責務持つべきでないもの
Controller HTTPリクエストの受け取り・レスポンスの返却。リクエストのバリデーション ビジネスロジック・DB操作
Service ビジネスロジックの実装。トランザクション制御 HTTPの概念（Request/Response）・SQL
Repository データの永続化・検索。SQLまたはORMの使用 ビジネスロジック
Entity / Domain ビジネスの概念・ルールの表現。自身の状態を変更するメソッド インフラ層への依存（DB・外部API）

2 命名

種別良い命名例 ✅悪い命名例 ❌
ServiceOrderService・UserAuthenticationServiceOrderMgr・ServiceClass
RepositoryUserRepository・ProductRepositoryUserDAO2・DataAccess
ControllerOrderController・UserControllerOrderHandler（慣習と異なる）

3 カプセル化

⚠️
コレクションの漏洩（防衛的コピー）: フィールドのListをそのまま返すと、呼び出し側から直接変更できてしまいます。
❌ return this.items;（外部からリストを変更される）
✅ return Collections.unmodifiableList(this.items); または return new ArrayList<>(this.items);
項目具体例・考え方
setterを提供しない Entityは状態遷移メソッドで表現する。❌ order.setStatus("SHIPPED") → ✅ order.ship()（状態遷移の妥当性をクラスが担保できる）
不変クラス（Value Object） 金額・日付範囲・郵便番号などはImmutableなValue Objectとして設計する。コンストラクタで値を受け取り、全フィールドをfinalにする

4 継承・コンポジション

判断基準継承 ✅コンポジション ✅
関係性 Cat is-a Animal（CatはAnimalの一種） Car has-a Engine（CarはEngineを持つ）
適切なケース ポリモーフィズムが必要。親クラスのインターフェースを外部に公開したい コード再利用が目的。実装の詳細を隠蔽したい。継承階層を浅く保ちたい
悪い継承の例 Stack は Vector を継承（JavaのStack）→ Vector の全操作が公開されてしまい、LIFO規則を破れる。コンポジション（内部にListを持つ）が正しい

5 依存関係・SOLID原則

原則概要違反例 → 改善例
Single Responsibility クラスの変更理由は1つだけ UserクラスがDB保存・メール送信・PDF生成を行う → それぞれ別クラスに分離
Open/Closed 拡張に開き、修正に閉じる 支払い方法を追加するたびに既存のifを修正 → PaymentStrategyインターフェース＋実装クラスで拡張
Liskov Substitution サブクラスは親クラスと置き換え可能 ReadOnlyListがListを継承してaddで例外 → 継承でなくコンポジションを使う
Interface Segregation クライアントが使わないメソッドへの依存を強制しない 巨大なIWorkerを全クラスに実装させる → 読み取り専用IReader、書き込み専用IWriterに分割
Dependency Inversion 上位モジュールは抽象に依存する ServiceがMySQLRepositoryに直接依存 → UserRepositoryインターフェースに依存させ、実装は注入

6 テスト容易性

💡
「テストが書きにくい」はクラス設計の問題サイン
テストを書くのが難しい場合、それはクラスが大きすぎる・依存が多すぎる・責務が混在しているサインです。設計を見直すチャンスとして活用しましょう。
