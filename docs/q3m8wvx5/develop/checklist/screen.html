画面作成チェックリスト

画面（フロントエンド）を新規作成・レビューする際に確認すべき項目のチェックリストです。UX・セキュリティ・アクセシビリティの観点を網羅しています。

🧭
このチェックリストの使い方
1) 画面を作る前に「1. 要件・UX設計」を固める → 2) 実装しながら「2. バリデーション」「4. セキュリティ」も並行確認 → 3) 最後に「7. テスト」で実機・対象ブラウザまで確認。
UIの見た目だけでなく、サーバー側の認可・入力検証がセットで必要です。
📌
初心者が見落としがちな観点
① 「削除ボタン非表示＝権限制御」ではない（API側の認可が必須）
② 二重送信（連打・戻る・複数タブ）
③ 失敗時の導線（どこがエラーで、次に何をすればよいか）

1 要件・UX設計

項目具体例・考え方
画面の目的「注文管理画面」ではなく「担当者が当日の未出荷注文を一覧確認し、出荷指示を行う」と定義する。目的が明確でないと、表示すべき列・操作ボタンの優先度が決まらない
対象ユーザー「一般ユーザー」「管理者」「CSオペレーター」など、ロールごとに「見える項目」「操作できる項目」を洗い出す。同じ画面でもロールによって表示が変わる場合は仕様書に明記する
画面遷移「一覧 → 詳細 → 編集 → 確認 → 完了」の遷移を図に起こす。戻るボタンで前の画面に戻れるか、入力中にブラウザバックした場合の動作（確認ダイアログの要否）も定義する
初期値・デフォルト値日付選択は「本日」をデフォルト表示。ステータスドロップダウンは「全て」を初期選択。デフォルト値が未定義だと、実装者がそれぞれ判断してバラバラな動作になる

2 バリデーション

⚠️
クライアントサイドだけではダメな理由
ブラウザの開発者ツールや curl でバリデーションを迂回できるため、クライアントサイドのバリデーションはあくまで UX 向上のためのもの。サーバーサイドで必ず同等のチェックを実施すること。
バリデーション種別具体例NGなメッセージ例 → OKな例
必須チェック「会社名」「メールアドレス」など、DB の NOT NULL 制約と対応させる「入力してください」→「会社名は必須です」
文字数制限「氏名: 最大50文字」「備考: 最大500文字」。DB カラムの VARCHAR(n) と必ず合わせる「文字数オーバー」→「備考は500文字以内で入力してください（現在: 523文字）」
形式チェック電話番号: /^\d{10,11}$/、メール: RFC5322準拠、日付: YYYY-MM-DD かつ実在する日付「形式が違います」→「電話番号はハイフンなしの10〜11桁で入力してください」
相関チェック「開始日 ≤ 終了日」「数量 > 0 かつ在庫数以下」「パスワードと確認パスワードが一致」「エラーです」→「終了日は開始日以降の日付を入力してください」

3 データ表示・操作

項目具体例・考え方
ページネーション1ページあたり20件表示。全件数と現在位置を表示（「123件中 21〜40件」）。「全件CSV出力」と「画面表示」は別のAPI にする。画面で全件取得すると表示が重くなる
ソート・フィルタ一覧のヘッダークリックでソート（昇順↔降順）。フィルタ条件はURLパラメータに保持し、ページ再読み込みでも条件が維持されるようにする
楽観的ロック編集画面を開いた時点の updated_at を hidden に保持する。保存時にサーバーで「DB の updated_at = 画面の updated_at」を確認し、不一致なら「他のユーザーが更新しました。再読込してください」と表示する
削除の確認「本当に削除しますか？この操作は元に戻せません。」とダイアログで確認する。対象が何か（例: 「注文 #12345 を削除しますか？」）を明示する。論理削除か物理削除かを仕様に明記する

4 セキュリティ

項目具体例・考え方
認証・認可未ログイン状態でのアクセスはログイン画面にリダイレクト。ロール確認はサーバーサイドで実施する（フロントのボタン非表示だけでは不十分）
ロール別制御一般ユーザーには「削除ボタン」を非表示。しかしボタン非表示だけでは API を直接叩けば削除できてしまう。サーバーサイドの認可チェックが必須
XSS対策テンプレートエンジンの自動エスケープ機能を使う（React の JSX、Thymeleaf の th:text など）。innerHTML や v-html にユーザー入力を渡さない
CSRF対策セッションクッキーと CSRF トークンを組み合わせる。POST/PUT/DELETE リクエストのヘッダー（X-CSRF-Token）またはボディにトークンを含め、サーバーで検証する
パラメータ改ざんURL に ?user_id=999 を直接入力して他人のデータを閲覧できる「IDOR」に注意。サーバーサイドで「ログインユーザーがそのリソースにアクセスできるか」を必ず確認する
マスキングクレジットカード番号は下4桁のみ表示（**** **** **** 1234）。マイナンバーは参照権限を持つ画面のみ表示する

5 アクセシビリティ・ユーザビリティ

項目具体例・考え方
ローディング表示API 呼び出し中はボタンを disabled にしスピナーを表示する。二重送信防止にもなる。応答が3秒以上かかる場合は「処理中です。しばらくお待ちください」のメッセージも表示する
成功フィードバック保存成功後に「保存しました ✓」のトースト通知を3秒表示する。一覧に戻る場合は、保存したレコードが一覧に反映されていることを確認できる状態にする
レスポンシブ社内管理画面（PC専用）なのか、外部ユーザー向けのスマホ対応が必要なのかを明確にする。対応する場合は iPhone Safari / Android Chrome での動作確認も必須

6 非機能・運用

項目具体例・考え方
ブラウザ対応「Chrome 最新版・Firefox 最新版・Edge 最新版・Safari 16以上」と明記する。IE11対応が必要な場合は別途工数が大きくかかることを合意しておく
応答時間目標一覧表示: 2秒以内、詳細表示: 1秒以内、検索: 3秒以内。超過する場合は遅延箇所（API・DB・フロント）を特定し対策する
エラーページ404: 「お探しのページが見つかりません」＋トップへのリンク。500: 「システムエラーが発生しました。管理者にお問い合わせください」＋問い合わせ先。スタックトレースは絶対に見せない
操作ログ「誰が（user_id）・いつ（timestamp）・何を（action）」を記録する。個人情報を含む操作は特に監査目的で必須

7 テスト

テストケース確認内容
正常系（入力→送信→遷移）全必須項目を正しく入力し送信した場合、登録が完了し確認画面/一覧に遷移することを確認
バリデーションエラー表示必須項目を空にして送信、最大文字数超過、不正形式（メール・電話番号）のそれぞれでエラーメッセージが正しく表示されることを確認
権限外アクセス一般ユーザーで管理者専用URLに直接アクセスした場合、403エラーまたはログイン画面にリダイレクトされることを確認
複数タブ・並行操作同じレコードを2つのタブで同時に編集し、後から保存したほうで「他のユーザーが更新しました」エラーが表示されることを確認（楽観的ロック）
ブラウザ対応確認Chrome・Firefox・Edge・Safari（指定バージョン）でレイアウト崩れ・JSエラーがないことを確認。スマホ対応の場合は実機での動作確認も実施する
