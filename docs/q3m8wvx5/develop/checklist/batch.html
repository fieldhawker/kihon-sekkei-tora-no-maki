1 設計・要件定義

項目具体例なぜ必要か
実行トリガー 「毎日 02:00 JST に定時起動」「注文確定イベント受信時に起動」「手動実行コマンド ./run_batch.sh --date=YYYY-MM-DD」 トリガーが曖昧だと二重実行・実行漏れの原因になる
タイムゾーン cron設定は 0 17 * * *（UTC）= JST 翌 02:00。サーバー/コンテナのタイムゾーン設定を明記する UTC/JSTの混在はサマータイム切替時に処理時刻がずれる
処理対象条件 WHERE status = 'pending' AND created_at < NOW() - INTERVAL '1 hour' 条件が曖昧だと処理漏れ・二重処理が発生する
冪等性 処理前に processed_at IS NULL を確認し、処理後に processed_at = NOW() をセット。再実行しても処理済みはスキップされる 障害リランや誤実行でデータが重複するリスクを排除する
依存関係 「売上集計バッチ」は「注文取込バッチ」の完了を待ってから起動。完了チェックはステータステーブルまたはジョブスケジューラの依存設定で管理 先行バッチが未完了のまま集計すると集計漏れが発生する

2 エラーハンドリング

⚠️
中断 vs スキップの判断基準
  • 中断（ABORT）: 1件のエラーが後続処理全体に影響する場合。例：マスタ取込バッチで参照先データが壊れている
  • スキップ（SKIP）: 1件のエラーが他の件に無関係な場合。例：メール送信バッチで特定ユーザーのアドレスが不正。スキップした件はエラーログに残し、後続で個別対応する
項目具体例なぜ必要か
ログフォーマット [ERROR] YYYY-MM-DDThh:mm:ss+09:00 batch=order-summary record_id=12345 msg="DB接続失敗" err="connection refused" 構造化ログにすることで、ログ集約ツール（Datadog等）での絞り込みが可能になる
アラート通知 バッチ終了ステータスが FAILED の場合、Slackの #alert-batch チャンネルに通知。通知内容：バッチ名・実行日時・エラー件数・ログリンク エラーに気づかず翌日の業務に影響するリスクを防ぐ
タイムアウト バッチ全体の上限を 4時間 に設定。DB クエリ単体のタイムアウトは 30秒。タイムアウト時はアラートを上げてプロセスを強制終了する 無限ループ・デッドロックによるリソース占有を防ぐ
デッドロック対策 複数テーブルを更新する場合、常に同じ順序（例: orders → order_items）でロック取得する。デッドロック検知時は 3回まで自動リトライ（指数バックオフ） 更新順序が不定だとトランザクション間で循環待ちが発生する

3 パフォーマンス・スケーラビリティ

項目具体例・考え方
チャンク分割 100万件を一度に SELECT すると OOM やタイムアウトが発生する。LIMIT 1000 OFFSET n または WHERE id > last_processed_id LIMIT 1000（カーソル方式が更新系には安全）で分割処理する
DB負荷軽減 バッチ実行時間帯をオフピーク（深夜）に設定。大量 UPDATE は UPDATE ... WHERE id IN (select 1000件) の形で分割し、業務DBへの長時間ロックを避ける
インデックス バッチの抽出条件（例: status, created_at）に複合インデックスを設定。EXPLAIN ANALYZE で Seq Scan が出ていないか確認する
トランザクション粒度 「100件ずつコミット」が基本。全件を1トランザクションにすると、エラー時に全件ロールバックが必要になり、undo ログが肥大化する
並列処理 「ユーザーID の末尾で4分割して並列実行」など、データが独立している場合のみ並列化する。DBコネクション数の上限に注意し、接続プールを共有する
メモリ管理 全件をメモリに読み込むのではなく、ストリーム処理（カーソルや Iterator）で1件ずつ処理する。JVM の場合は -Xmx で上限を設定し、GC ログを監視する

4 実行管理・運用

項目具体例・考え方
実行ログ [INFO] batch=order-summary start_at=YYYY-MM-DDThh:mm:ss target_date=YYYY-MM-DD
[INFO] batch=order-summary end_at=YYYY-MM-DDThh:mm:ss processed=48320 skipped=5 errors=2 status=COMPLETED
実行状態管理 batch_execution_log テーブルに (batch_name, started_at, ended_at, status, processed_count) を記録。ジョブスケジューラ（JP1・Airflow等）が管理する場合はその機能を使う
二重起動防止 ① DB の batch_lock テーブルに排他レコードを INSERT し、終了時に DELETE（レコードが存在すれば起動しない）
② ファイルロック（flock）をプロセス起動時に取得する
③ ジョブスケジューラの「同一ジョブの重複起動禁止」設定を使う
進捗確認 [INFO] progress: 10000/48000 件処理完了 (20.8%) を一定件数ごとに出力。長時間バッチは「処理中の最終 record_id」をログに出すことで、どこまで進んだか把握できる
手動実行・停止 手動実行: ./batch.sh --target-date=YYYY-MM-DD --dry-run（dry-run オプションで副作用なし確認可能）
強制停止: プロセス ID を kill -TERM で送信→ SIGTERM ハンドラでチャンク境界まで処理してから終了

5 セキュリティ

項目具体例・考え方
実行ユーザー バッチ専用の OS ユーザー・DB ユーザーを作成し、必要最小限の権限のみ付与する。例：バッチ用 DB ユーザーは SELECT / INSERT / UPDATE のみで DROP は不可
個人情報のログ出力 ログに氏名・メールアドレスを出力しない。出力が必要な場合は user_id=12345 のように ID のみにする。デバッグログ（DEBUG レベル）を本番環境で有効にしない
ファイル入力値検証 CSV 取込バッチでは① ファイルサイズの上限チェック ② 文字コード確認 ③ カラム数・型の検証 ④ パストラバーサル防止（ファイル名に ../ が含まれないか確認）を実施する
接続情報の管理 DB パスワードや API キーをソースコードに直書きしない。環境変数（DB_PASSWORD）または Secrets Manager（AWS・Vault等）から実行時に取得する

6 テスト

テストケース確認内容
正常系（全件）対象データが複数件存在する状態で実行し、全件処理・ステータス更新・ログ出力が正しいことを確認
正常系（0件）対象データが0件の状態で実行し、エラーなく正常終了すること・処理件数=0のログが出ることを確認
異常系（データ不備）必須カラムが NULL のレコードを含めて実行し、スキップ or 中断が仕様どおり動作することを確認
異常系（DB障害）DB接続を途中で切断し、タイムアウト・アラート通知・リトライが正常に機能することを確認
境界値（チャンク境界）処理件数がチャンクサイズ（例: 1000件）の倍数になるデータで実行し、全件処理されることを確認
再実行（リラン）同じパラメータで2回実行し、2回目は処理済みレコードをスキップして結果が同一になることを確認
性能テスト本番想定の最大件数（例: 50万件）で実行し、制限時間内（例: 4時間以内）に完了することを確認
