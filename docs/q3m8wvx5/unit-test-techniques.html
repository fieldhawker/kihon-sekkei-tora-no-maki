🎯 単体テスト設計技法まとめ

単体テスト（Unit Test）は、プログラムを構成する最小単位（関数やメソッドなど）が、設計通りに正しく動作するかを検証するテストです。
テストケースを場当たり的に作成すると、考慮漏れによるバグの見逃しや、無駄なテストケースの増大を招きます。

以下に、単体テストを効率的かつ網羅的に行うための代表的なテスト設計技法（ブラックボックス・ホワイトボックス）をまとめます。

1. 入力データに着目した技法（ブラックボックステスト）

プログラムの内部構造を意識せず、仕様書（インプットとアウトプット）に基づいてテストケースを設計します。

1.1 同値分割法 (Equivalence Partitioning)

入力値を「同じように処理されるグループ（同値クラス）」に分け、各グループから代表値を1つ選んでテストする手法です。

使用事例: 「パスワードの文字数チェック（8文字以上、16文字以下）」
無効同値（短すぎる） 有効同値（正常） 無効同値（長すぎる）
〜7文字 8〜16文字 17文字〜
代表値: 5文字 代表値: 10文字 代表値: 20文字

1.2 境界値分析 (Boundary Value Analysis)

バグが発生しやすい「同値クラスの境界（しきい値とその前後の値）」を狙ってテストする手法です。同値分割法とセットで使われます。

使用事例: 「年齢による入場制限（18歳以上のみ入場可）」

境界となる値：18

  • 17歳 (境界のすぐ下：入場不可となるか)
  • 18歳 (境界値：入場可となるか)
  • 19歳 (境界のすぐ上：入場可となるか)

>= と > の書き間違い（オフバイワンエラー）などのバグを効果的に検出できます。

2. 条件と結果の組み合わせに着目した技法

2.1 デシジョンテーブルテスト (Decision Table Testing)

複数の入力条件の組み合わせによって出力（結果）が変化するロジックに対して、考えられる全ての組み合わせを表にしてテストする手法です。

使用事例: 「ECサイトの割引判定」
  • 会員=Yes, 5000円=Yes ➔ 結果: 割引適用
  • 会員=Yes, 5000円=No ➔ 結果: 割引適用外
  • 会員=No, 5000円=Yes ➔ 結果: 割引適用外
  • 会員=No, 5000円=No ➔ 結果: 割引適用外

2.2 ペアワイズテスト / 直交表 (Pairwise Testing)

多数のパラメータがある場合、「任意の2つのパラメータの組み合わせが必ず1回以上登場する」ようにケースを絞り込む手法です。

使用事例: 「OS、ブラウザ、言語の設定による画面表示」

パラメータ: OS(Win/Mac)、ブラウザ(Chrome/Edge/Safari)、言語(日/英)

全組み合わせは 12通りですが、ペアワイズツールを利用することで「OS×ブラウザ」「ブラウザ×言語」等の2ペア網羅をより少ない回数（6回程度）に絞り込めます。
ソフトウェアのバグの大半は「1〜2つのパラメータの組み合わせ」で発生するという経験則に基づいています。

3. プログラムの内部構造に着目した技法（ホワイトボックステスト）

3.1 制御フローテスト / パステスト (Control Flow Testing)

コード内の命令や分岐をどれだけ通過したか（カバレッジ：網羅率）を基準にテストを設計します。

flowchart TD A[開始] --> B{A > 0 || B > 0} B -- True --> C[処理X] B -- False --> D[処理Y] C --> E[終了] D --> E
  • C0 (命令網羅): すべての「行」を少なくとも1回実行。
  • C1 (分岐網羅): すべての「分岐(True/False)」を両方とも1回実行。（単体テストの目標水準）
    • 例：Trueルート A=1, B=0 / Falseルート A=0, B=0
  • C2 (条件網羅): 分岐の中の「複数条件」のTrue/False組み合わせを網羅。

4. 状態の変化や経験則に着目した技法

4.1 状態遷移テスト (State Transition Testing)

システムが特定の「状態」を持っており、イベントによって状態が変わる場合に、その遷移が正しく行われるかをテストします。

使用事例: 「動画プレイヤー（停止・再生・一時停止）」
  • 「停止中」に「再生ボタン」 ➔ 「再生中」になるか
  • 「一時停止中」に「再生ボタン」 ➔ 「再生中」になるか
  • 「停止中」に「一時停止ボタン」 ➔ 無視されるか（エラー処理されるか）

4.2 エラー推測 (Error Guessing)

エンジニアの経験や直感に基づき、「ミスしやすそうな箇所」「過去のバグパターン」を狙い撃ちにする手法です。

使用事例: 「入力フォームのテスト」

狙い撃ちデータ例：空文字列、半角スペースのみ、全角半角混在、超長文、特殊文字( ' や ; など )
