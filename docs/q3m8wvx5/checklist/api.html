このチェックリストの使い方
1) まず「1. 設計・インターフェース」だけ先に埋める → 2) 実装前に「3. セキュリティ」も確認 → 3) 実装後に「7. テスト」を実施。

1 設計・インターフェース

項目良い例 ✅悪い例 ❌ / 理由
URL命名 GET /api/v1/orders
GET /api/v1/orders/123
POST /api/v1/orders		 GET /getOrderList（動詞はメソッドで表現する）
GET /order（複数形にする）
HTTPメソッド 作成: POST、全体更新: PUT、部分更新: PATCH、削除: DELETE 削除に POST /deleteOrder（メソッドとURLで意味が重複・混乱する）
ステータスコード 作成成功: 201 Created、認証エラー: 401、権限なし: 403、リソースなし: 404、競合（重複登録）: 409 Conflict 全てのエラーに 200 OK でレスポンスボディにエラーを入れる（クライアントがエラー検知できない）

2 バリデーション

⚠️
SQLインジェクション対策の基本: SQL文字列に直接ユーザー入力を埋め込まず、必ずプリペアドステートメント（プレースホルダー）を使う。
❌ "SELECT * FROM users WHERE name = '" + name + "'"
✅ "SELECT * FROM users WHERE name = ?" + パラメータバインド
バリデーション種別具体例
必須チェック リクエストボディに name フィールドが存在しない場合: 400 Bad Request＋ "name は必須です"
型・形式チェック age に文字列が渡された場合・email が @ を含まない場合・date が YYYY-MM-DD 形式でない場合を弾く
文字数・値域チェック title は最大100文字、quantity は1以上99999以下、price は0以上など
相関チェック 予約APIで check_in_date >= check_out_date の場合エラー。決済APIで amount と currency の組み合わせが有効か確認する

3 認証・認可・セキュリティ

項目具体例・考え方
JWT の扱い アクセストークンの有効期限は15〜60分。リフレッシュトークンは7日。トークンはローカルストレージに保存しない（XSSリスク）→ HttpOnly クッキーに保存する
認可（IDOR対策） GET /api/v1/orders/123 で、注文 #123 が自分のものかをサーバーサイドで確認する。WHERE id = 123 AND user_id = current_user_id という条件を必ず付ける
HTTPS強制 HTTP でアクセスされた場合は HTTPS へリダイレクト（301）。Strict-Transport-Security: max-age=31536000 ヘッダーを設定する
レスポンスから除外する情報 ユーザー情報取得APIのレスポンスに password_hash・secret_key を含めない。ORM の hidden 属性や、専用の DTO に変換してから返す
CORS設定 Access-Control-Allow-Origin: https://app.example.com と許可するオリジンを明示する。*（全許可）は内部 API では絶対に使わない

4 パフォーマンス・信頼性

項目具体例・考え方
N+1問題 注文一覧を取得した後、各注文のユーザー名を取得するために N回の SELECT が発生するパターン。JOIN または IN句でまとめて取得する。ORM 使用時は eager loading（include/with）を指定する
キャッシュ戦略 マスタデータ（都道府県リスト・カテゴリ一覧）: TTL=1時間でサーバーキャッシュ。ユーザー固有データはキャッシュしない。キャッシュキーに検索条件を含める（例: products:category=3:page=1）
ページネーション offset 方式: ?limit=20&offset=40（実装簡単だが大量データで遅くなる）
カーソル方式: ?limit=20&after=cursor_xyz（無限スクロール・更新が多いデータに適切）
レート制限 外部公開 API: ユーザーごとに 100 req/min を上限とし、超過時は 429 Too Many Requests と Retry-After: 60 ヘッダーを返す
タイムアウト 外部 API 呼び出しのタイムアウトは接続 5秒・読み取り 30秒を設定。タイムアウト後はリトライせずに 503 Service Unavailable を返す（リトライによるカスケード障害を防ぐ）

5 データ整合性・トランザクション

項目具体例・考え方
トランザクション範囲 「注文作成」では ① orders テーブル INSERT ② order_items テーブル INSERT ③ 在庫テーブル UPDATE を1トランザクションで行う。どれか1つが失敗した場合に全てロールバックする
楽観的ロック 在庫更新のような競合が頻繁に起きないケース向け。WHERE id = 1 AND version = 3 の形で更新し、更新件数が0なら「他のリクエストに先を越された」として 409 Conflict を返す
悲観的ロック 残席数の確定予約のような、競合が必ず起きるケース向け。SELECT ... FOR UPDATE で行ロックを取得してから更新する。ロック保持時間は短くする
冪等性（POST の重複対策） クライアントが Idempotency-Key: uuid-1234 ヘッダーを付けて送信。サーバーは同じキーの2回目以降のリクエストに対して、最初のレスポンスをキャッシュして返す（決済APIで特に重要）
非同期処理 メール送信・PDF生成・外部サービス連携など、レスポンス時間が長い処理はジョブキュー（Redis Queue・SQS等）に積んで非同期実行する。APIは即座に 202 Accepted を返し、状態確認用のエンドポイントを提供する

6 ログ・モニタリング

項目具体例・考え方
リクエストログ 
{
  "request_id": "req-abc123",
  "method": "POST",
  "path": "/api/v1/orders",
  "user_id": 456,
  "status": 201,
  "duration_ms": 143,
  "timestamp": "YYYY-MM-DDThh:mm:ssZ"
}
リクエスト ID を全てのログに付与することで、1リクエストの処理を追跡しやすくなる
エラーログ 500エラー時はスタックトレースをサーバーログに記録する。レスポンスにはスタックトレースを含めない（内部構造の漏洩防止）。エラーログを Sentry 等に集約してアラートを設定する
処理時間計測 SLO（例: p95 で 500ms 以内）を定義し、超過リクエストをアラートで検知する。DB クエリ単体の実行時間も計測し、遅いクエリを特定できるようにする
ヘルスチェック GET /health → {"status": "ok", "db": "ok", "cache": "ok"}。ロードバランサーやコンテナオーケストレーター（k8s）がこのエンドポイントを使って死活監視する

7 テスト

テストケース確認内容
正常系有効なリクエストで期待するステータスコード・レスポンスボディが返ること。POST 後に GET で作成されたリソースが取得できることを確認
バリデーションエラー必須項目なし→ 400、不正形式→ 400、エラーメッセージのフィールド名・内容が正しいことを確認
認証・認可トークンなし→ 401、有効期限切れトークン→ 401、権限不足→ 403、他人のリソースへのアクセス→ 403 or 404（存在を教えない設計の場合）
境界値テスト文字数制限の上限値・下限値、数値の最大値・最小値、ページネーションの最終ページ（件数が端数の場合）
並行実行テスト残り1件の在庫に対して同時に2件の注文リクエストを送信し、1件のみ成功・もう1件はエラーになることを確認（在庫の二重引き当て防止）
冪等性テスト同じ Idempotency-Key で2回 POST し、DBレコードが1件のみ作成され、2回目は最初のレスポンスが返ることを確認
