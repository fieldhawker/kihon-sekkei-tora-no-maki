レガシーコードからの脱却¶

著者: David Scott Bernstein
原題: Beyond Legacy Code: Nine Practices to Extend the Life (and Value) of Your Software

レガシーコードとは何か¶

「レガシーコード」とは、単に古いコードのことではありません。
テストがなく、変更が怖いコードのことです。

レガシーコードの特徴： - 変更すると別の場所が壊れる（副作用が読めない） - 何をしているか理解しにくい - テストがない、または書きにくい - 誰も全体を把握していない

この本では、レガシーコードを生み出さないための 9つのプラクティス を紹介します。

9つのプラクティス¶

プラクティス1: 「何を・なぜ・誰のために」を伝える（How ではなく What/Why/Who）¶

悪い例：

「ユーザーテーブルにSQLでINSERTを実行してください」（How）

良い例：

「新規ユーザーが登録できるようにしてください。なぜなら、現在は管理者が手動で追加しており、オンボーディングに時間がかかっているからです」（What + Why + Who）

ポイント： - 実装方法（How）ではなく、目的・理由・対象者を伝える - 要件を「何をすべきか」レベルで定義することで、設計の自由度が生まれる - 「なぜ」を知らないと、間違った方向で正しく実装してしまう

プラクティス2: 小さなバッチで作る¶

大きな機能を一度に作ろうとするのは危険です。

小さく作るメリット： - 問題が早期に発見できる - フィードバックを受けやすい - 途中で方向修正しやすい - リリースのリスクが小さい

実践方法： - 1つのタスクを「1〜2日以内に完了できる粒度」に分割する - 動く状態を常に保つ（壊れたまま長時間作業しない） - 少しずつ統合してテストする

プラクティス3: 継続的インテグレーション（CI）¶

コードを書いたらすぐにチームの共有ブランチへ統合し、自動テストを走らせます。

なぜ必要か： - 長期間ブランチを分けていると、統合時に大量のコンフリクトが発生する - 「動いていたはずなのに壊れた」が早期に検知できる

最低限やること： 1. コードを書く 2. テストを書く 3. テストが通ることを確認する 4. メインブランチにマージする 5. 自動テストが全部通ることを確認する

プラクティス4: コラボレーション（協働する）¶

一人で抱え込まず、チームで協力して開発します。

ペアプログラミング： - 2人で1台のPCを使ってコードを書く - 「ドライバー」（コードを書く人）と「ナビゲーター」（確認・指示する人）を交代する - レビューの質が上がり、知識が共有される

コードレビューの目的： - バグ発見だけでなく、知識共有・設計品質の向上が主目的 - 批判ではなく学びの場として活用する

プラクティス5: CLEANなコードを書く¶

CLEANは以下の頭文字：

文字 英語 日本語 意味
C Cohesive 凝集性 1つのクラス・関数は1つのことだけ担当する
L Loosely Coupled 疎結合 部品同士の依存を最小化する
E Encapsulated カプセル化 内部の詳細を隠す
A Assertive 断定的 オブジェクト自身が責任を持って動く
N Nonredundant 非冗長 同じコードを繰り返さない（DRY原則）

例：凝集性（Cohesive）

# 悪い例：1つのクラスが何でもやっている
class UserManager:
    def create_user(self): ...
    def send_email(self): ...
    def generate_report(self): ...
    def save_to_database(self): ...

# 良い例：役割ごとにクラスを分ける
class UserRepository:
    def save(self, user): ...

class EmailService:
    def send_welcome(self, user): ...

例：非冗長（Nonredundant / DRY原則）

# 悪い例：同じ処理が複数箇所にある
def process_order():
    tax = price * 0.1
    ...

def calculate_total():
    tax = price * 0.1  # 同じ計算
    ...

# 良い例：1箇所にまとめる
def calculate_tax(price):
    return price * 0.1

プラクティス6: テストファーストで書く（TDD）¶

コードを書く前にテストを書きます。

TDD（テスト駆動開発）の3ステップ：

1. Red   → 失敗するテストを書く
2. Green → テストが通る最小限のコードを書く
3. Refactor → コードをきれいにする（テストが通ったまま）

なぜテストを先に書くのか： - 「どう使われるか」を先に考えることで、設計が改善される - テストがないコードは後からテストを追加しにくい - 安心してリファクタリングできる安全網になる

初心者向け：まずここから始めよう

# テストの例（pytest）
def test_tax_calculation():
    assert calculate_tax(1000) == 100

# テストが通るように実装する
def calculate_tax(price):
    return price * 0.1

プラクティス7: テストで振る舞いを明確にする¶

テストはドキュメントでもあります。
「このコードが何をするか」をテストを読めばわかるように書きます。

良いテスト名の付け方：

# 悪い例
def test_user():
    ...

# 良い例（振る舞いが分かる）
def test_未成年ユーザーはアルコール商品を購入できない():
    ...

def test_在庫が0の場合は注文できない():
    ...

テストの3A原則： 1. Arrange（準備） — テストに必要な状態を作る 2. Act（実行） — テスト対象の処理を呼ぶ 3. Assert（確認） — 結果が期待通りか検証する

プラクティス8: 設計を最後に実装する（創発的設計）¶

最初から完璧な設計を目指さず、小さく作りながら設計を育てます。

ウォーターフォールの問題： - 最初に完璧な設計を作ろうとする - 要件は変わるのに、設計が変えにくくなる - 後から変更するコストが非常に高い

創発的設計のアプローチ： 1. まず動くものを小さく作る 2. テストを書く 3. リファクタリングで設計を改善する 4. 繰り返す

「設計とは、コードを書きながら発見するものだ」

プラクティス9: レガシーコードをリファクタリングする¶

既存のレガシーコードを安全に改善する方法です。

リファクタリングの大原則： - 外から見た振る舞いを変えずに、内部構造を改善する - リファクタリング中に機能追加しない

安全なリファクタリングの手順： 1. 対象コードにテストを書く（まずテストで守りを固める） 2. 少しずつ変更する 3. テストが通ることを都度確認する 4. 完了

よく使うリファクタリングのパターン：

パターン名 説明
関数の抽出 長い関数の一部を別の関数に切り出す
変数名の改善 x → userAge など意味のある名前に
重複の除去 同じ処理を1つの関数にまとめる
条件の単純化 複雑なif文を読みやすくする

まとめ：9つのプラクティス一覧¶

# プラクティス 一言まとめ
1 What/Why/Whoを伝える 「何を・なぜ・誰のために」を定義する
2 小さなバッチで作る 小さく・早く・頻繁に届ける
3 継続的インテグレーション 常に統合してテストを回す
4 コラボレーション ペアプロ・レビューで品質を高める
5 CLEANなコード 凝集・疎結合・カプセル化・断定・非冗長
6 テストファースト コードより先にテストを書く（TDD）
7 テストで振る舞いを明確化 テストをドキュメントとして活用する
8 設計を最後に実装する 作りながら設計を育てる
9 レガシーコードをリファクタリング テストで守りを固めてから改善する

若手エンジニアへのメッセージ¶

レガシーコードは誰もが悪意を持って作ったわけではありません。
締め切りのプレッシャー、テストを書く時間のなさ、知識の不足——そういった積み重ねで生まれます。

あなたにできること： - 今日から小さなテストを1つ書く習慣をつける - コードを変更するときは、まず「何をしているか」を理解してから変える - 「動いているから触らない」ではなく、安全に少しずつ改善する

完璧なコードを最初から目指す必要はありません。
昨日より少し良いコードを書き続けることが、レガシーコードからの脱却への道です。
