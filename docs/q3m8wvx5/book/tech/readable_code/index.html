リーダブルコード 要点まとめ¶

対象：未経験〜若手エンジニア
参考書籍：『リーダブルコード ―より良いコードを書くためのシンプルで実践的なテクニック』（Dustin Boswell, Trevor Foucher 著）

この本が伝えたいこと¶

「コードは他の人（未来の自分を含む）が読むもの」 という前提で書くこと。

読みやすいコードは、バグが少なく、修正しやすく、チームで働きやすくなります。

第1部：表面上の改善（命名・書き方）¶

1. 名前に情報を詰め込む¶

変数名・関数名は「意味が伝わる名前」にする。

# 悪い例
d = 10

# 良い例
days_until_expiration = 10

ポイント - 省略しすぎない（tmp, data, val などは避ける） - 単位や性質を名前に含める（size_mb, is_valid, user_id） - 長くても意味が明確な方が良い

2. 誤解されない名前を選ぶ¶

名前から「他の意味に読めないか？」を考える。

# 悪い例：「フィルタする」は「除外」か「絞り込み」かわからない
results = filter(users)

# 良い例
results = exclude_inactive_users(users)
results = select_active_users(users)

ポイント - 反対語のある概念はペアで揃える（start/stop, begin/end, open/close） - min_/max_ は範囲の境界に使う - is_/has_/can_ で bool であることを示す

3. 美しく見やすく整える¶

コードの「見た目」も重要。一貫性があると読みやすい。

# 悪い例（揃っていない）
width=100
height = 200
background_color="red"

# 良い例（揃っている）
width            = 100
height           = 200
background_color = "red"

ポイント - 似た処理は似た形で書く（パターンを揃える） - 空白行でコードのブロックをグループ化する - 1行の長さは80〜100文字程度を目安にする

4. コメントは「なぜ？」を書く¶

コードを見れば「何をしているか」はわかる。コメントには なぜそうしているか を書く。

# 悪い例：コードの繰り返しにすぎない
# ユーザーのリストを繰り返す
for user in users:

# 良い例：なぜそうするかを説明している
# 退会済みユーザーはDBから削除されないため、ここで除外する必要がある
for user in active_users:

コメントを書くべき場面 - 仕様上の制約や背景（「〇〇の理由でこうしている」） - 将来の開発者への警告（「この順序を変えると〇〇が壊れる」） - TODO / FIXME（後で直す予定の箇所）

コメントを書かなくていい場面 - コードを読めばわかること - 変数名や関数名を変えれば済むこと

第2部：ロジックの単純化¶

5. 条件式はシンプルに保つ¶

読みやすい条件式のルール。

# 悪い例：否定が重なって読みにくい
if not (status != "active"):

# 良い例
if status == "active":

ポイント - 否定（not, !=）はできるだけ避ける - 「肯定の条件」を先に書く - 三項演算子はシンプルな場合のみ使う

6. 早期リターンでネストを減らす¶

条件が複雑なとき、「エラー条件を先に返す（ガード節）」と読みやすくなる。

# 悪い例：ネストが深い
def process(user):
    if user is not None:
        if user.is_active:
            if user.has_permission:
                # 処理
                return result

# 良い例：早期リターンでフラット
def process(user):
    if user is None:
        return None
    if not user.is_active:
        return None
    if not user.has_permission:
        return None
    # 処理
    return result

7. 変数は必要になった時点で作る¶

変数を先に宣言しすぎると、どこで使われるか追いにくくなる。

# 悪い例：使う場所と遠い
result = None
# ...（長い処理）...
result = calculate()
return result

# 良い例：使う直前に宣言
# ...（長い処理）...
result = calculate()
return result

8. 巨大な式を分割する¶

1行に詰め込みすぎず、変数に分けて意味を明確にする。

# 悪い例
if user.age >= 18 and user.country == "JP" and not user.is_banned:

# 良い例
is_adult        = user.age >= 18
is_domestic     = user.country == "JP"
is_not_banned   = not user.is_banned

if is_adult and is_domestic and is_not_banned:

第3部：コードの再構成¶

9. 関数は1つのことだけやる¶

関数が複数のことをしていたら、分割のサイン。

# 悪い例：検証・取得・保存を1関数でやっている
def handle_user(user_id):
    # バリデーション
    # DBからの取得
    # ファイルへの保存

# 良い例：役割ごとに分ける
def validate_user_id(user_id): ...
def fetch_user(user_id): ...
def save_user_to_file(user): ...

目安：関数名が「〜して、〜して、〜する」になったら分割を検討する

10. 汎用コードを切り出す¶

プロジェクト固有のロジックと、汎用的な処理を分ける。

# 悪い例：汎用処理がビジネスロジックに混在
def send_report(data):
    # JSONに変換
    json_str = json.dumps(data)
    # 改行コードを統一（汎用処理）
    json_str = json_str.replace('\r\n', '\n')
    # 送信（ビジネスロジック）
    api.send(json_str)

# 良い例：汎用処理を切り出す
def normalize_newlines(text):
    return text.replace('\r\n', '\n')

def send_report(data):
    json_str = normalize_newlines(json.dumps(data))
    api.send(json_str)

11. コードの量を減らす¶

書かないコードが最も読みやすい。

  • 標準ライブラリや既存の関数で代替できないか確認する
  • 使われていないコードは削除する（Gitで履歴が残る）
  • 「将来使うかも」で書かない。必要になったら書く

まとめ：読みやすいコードを書く習慣¶

カテゴリ やること
命名 意図が伝わる名前をつける
コメント 「なぜ」を書く。自明なことは書かない
条件式 シンプルに。否定を避ける
関数 1つのことだけやる
コード量 少ない方が良い。不要なコードは削除

「このコードを初めて読む人が理解できるか？」
を常に意識してコードを書くことが、リーダブルコードの本質です。

命名チェックリスト（対象別逆引き）¶

命名したあとに以下のリストで確認する。

変数名¶

  • 何の値を表しているかが名前だけでわかる（d → days_until_expiration）
  • 省略しすぎていない（tmp, data, val, buf を避けている）
  • 単位・型・性質を含めている（size_mb, is_valid, user_id, deadline_at）
  • bool 変数は is_ / has_ / can_ で始まっている（is_active, has_permission）
  • 範囲の境界には min_ / max_ を使っている（max_retries, min_age）
  • 別の意味に読めない（filter_users は「除外」か「絞り込み」か曖昧でないか）
  • 同じ概念に複数の名前を使っていない（user と account が混在していないか）

関数名・メソッド名¶

  • 「何をする関数か」が動詞＋目的語で表現されている（fetch_user, validate_input）
  • 副作用の有無が名前から推測できる（読み取りなら get_ / find_ / calculate_、書き込みなら save_ / update_ / delete_）
  • 戻り値の型が名前から推測できる（bool を返すなら is_ / has_ / can_）
  • 「〜して、〜して、〜する」と読める場合は分割を検討した
  • 誤解を生む動詞を避けている（process_, handle_, manage_ は意味が広すぎないか）
  • 対になる関数は対義語を使っている（open/close, start/stop, add/remove）

クラス名¶

  • 名詞または名詞句になっている（動詞で終わっていない）
  • 責務が1つに絞られた名前になっている（UserManager のように何でも入る名前でない）
  • 役割が具体的にわかる（Manager, Handler, Processor など汎用すぎる名前を避けている）
  • 継承・実装関係が名前から推測できる（AbstractRepository, UserRepository）
  • 他のクラス名と混同しない（User と UserInfo と UserData が乱立していないか）

定数・列挙値¶

  • ALL_CAPS（または言語慣習に合った形式）で書いている
  • マジックナンバーに意味のある名前をつけている（86400 → SECONDS_PER_DAY）
  • 列挙値は所属するグループを名前で示している（OrderStatus.PENDING, Color.RED）
  • 単位が名前に含まれている（TIMEOUT_MS, MAX_RETRIES）

引数名¶

  • 呼び出し側を見たときに意味がわかる（send(true) → send(notify_user=True)）
  • bool 引数は is_ / enable_ / with_ などで意図が明確になっている
  • 順序が逆でも意味が変わらないような名前になっていない（copy(src, dst) と copy(dst, src) を混同しないか）
