達人プログラマー 要点まとめ¶

対象読者：システム開発に携わる若手エンジニア（未経験〜数年目）
原著：The Pragmatic Programmer / Andrew Hunt・David Thomas 著

この本が伝えたいこと¶

「達人プログラマー」は、単なるコーディング技術書ではありません。
プロとして成長し続けるためのマインドセットと習慣を体系的にまとめた本です。
技術は変わっても、ここに書かれた原則は長く使い続けられます。

第1章：達人の哲学¶

1-1. 自分のキャリアに責任を持つ¶

  • 「会社がやってくれる」「誰かが教えてくれる」を待たない
  • 自分のスキルと成長は、自分でコントロールする
  • 失敗を認め、言い訳しない。問題が起きたら「言い訳」より「解決策」を持ってくる

若手へのヒント
「わかりません」で終わらせず、「調べて確認します」「○○という理解ですが合っていますか？」と能動的に動きましょう。

1-2. 壊れた窓を放置しない（割れ窓理論）¶

  • コードの中の「ちょっと気になる箇所」を放置すると、周囲も乱雑になっていく
  • 品質の低下は伝染する。小さな問題を早めに直す習慣をつける

若手へのヒント
バグを見つけたら、担当外でも報告する。コードの汚れに気づいたら、チームに共有する。

1-3. 変化の触媒になれ（石のスープ）¶

  • 大きな変化を一度に提案しても受け入れられにくい
  • 小さな改善を積み重ねて、周囲を巻き込んでいく

第2章：達人のアプローチ¶

2-1. DRY原則（Don't Repeat Yourself）¶

  • 同じ知識・ロジックを複数の場所に書かない
  • コピペしたコードは、片方を直したときもう片方が古くなる
  • 「一つの事実は、一箇所だけに存在する」
悪い例：同じバリデーション処理を3ファイルに書く
良い例：共通関数として切り出し、各ファイルから呼び出す

2-2. 直交性（Orthogonality）¶

  • 機能どうしが互いに影響しない設計を目指す
  • A の変更が B に影響しないようにする → テストしやすく、変更しやすい
  • 関心の分離（UIとビジネスロジックを混ぜない、など）が基本

2-3. 可逆性を意識する¶

  • 「これで決まり」はない。技術もビジネス要件も変わる
  • 特定のDBや特定のライブラリに強く依存しすぎない設計を意識する
  • 抽象化・インターフェースで依存を弱める

2-4. 曳光弾（Tracer Bullet）¶

  • 完璧な設計を最初から目指さず、動く最小限のものを早く作る
  • エンドツーエンドで動くものを早期に作り、フィードバックを受ける
  • プロトタイプとは違い、捨てずに本番コードとして成長させていく

若手へのヒント
まずスケルトン（骨格）を作り、動作確認してから肉付けしましょう。最初から完璧を目指すと、方向性がずれていても気づけません。

第3章：基本的なツール¶

3-1. テキスト操作を使いこなす¶

  • ファイル操作・テキスト変換・検索はコマンドライン（CLIツール）で効率化できる
  • シェルスクリプトや正規表現を覚えると、繰り返し作業を自動化できる

3-2. バージョン管理（Git）を必ず使う¶

  • コードの変更履歴は財産。「なぜこうしたか」が後でわかる
  • ひとつの変更はひとつのコミットに。コミットメッセージは意味のある内容を書く
  • バージョン管理に入っていないものは存在しないという意識で

若手へのヒント
作業前に必ずブランチを切る。こまめにコミットする。コミットメッセージは「何をしたか」だけでなく「なぜしたか」も書くと後で助かります。

3-3. デバッグのアプローチ¶

  • バグは恥ずかしいことではない。冷静に原因を特定することが大事
  • 「動かない理由」を仮説立てして検証する。感覚で直しにいかない
  • ゴムのアヒルデバッグ：誰かに（またはアヒルに）コードを説明するだけで気づくことがある
デバッグの手順例：
1. 現象を正確に把握する（何が、どのタイミングで、どう壊れるか）
2. 原因の仮説を立てる
3. 仮説を検証するコードやログを追加する
4. 原因を特定して修正する
5. なぜそうなったかを理解し、同じ問題を防ぐ

第4章：実用主義的なパラノイア¶

4-1. 契約による設計（Design by Contract）¶

  • 関数・メソッドには「前提条件」「事後条件」「不変条件」がある
  • 入力として何を期待するか、出力として何を保証するかを明確にする
  • これがドキュメントとテストの基礎になる

4-2. 早期クラッシュ（Crash Early）¶

  • 問題が起きたら、できる限り早い段階でエラーにする
  • 壊れた状態で処理を続けると、問題の発見が遅れ、影響範囲が広がる
  • 「怪しいと思ったら止める」 は正しい判断
# 悪い例：おかしな値でも続ける
def process(value):
    if value < 0:
        value = 0  # 無言で補正してしまう

# 良い例：想定外の入力を早期に拒否する
def process(value):
    if value < 0:
        raise ValueError(f"valueは0以上でなければなりません: {value}")

4-3. 表明（Assertion）を使う¶

  • 「ありえないはず」の状態をコードで明示する
  • テスト環境でバグを早期に発見できる

第5章：柔軟なコードを書く¶

5-1. デカップリング（結合度を下げる）¶

  • 「知りすぎているクラス」は危険。変更が波及しやすくなる
  • デメテルの法則：オブジェクトは直接の知人とだけ話す
  • a.b.c.doSomething() のような連鎖呼び出しは避ける

5-2. メタプログラミングと設定の外部化¶

  • ハードコーディング（コード中に直接値を書く）は避ける
  • 変わりやすい値（URLや閾値など）は設定ファイルや環境変数に出す
  • コードを変えずに振る舞いを変えられる設計を目指す

5-3. 偶発的プログラミングを避ける¶

  • 「なんとなく動いた」で終わらせない
  • なぜ動いているのか理解した上でコードを書く
  • 根拠のない決定は後で大きな問題になる

第6章：コードを書く前に¶

6-1. 要件を掘り下げる¶

  • ユーザーが言った言葉をそのまま実装しない
  • 「本当に必要なこと（本質的な要求）」を引き出す
  • 5回の「なぜ」 を繰り返して根本課題を探る

若手へのヒント
「ボタンを赤くしてほしい」という要望の裏には「重要なアクションを目立たせたい」という意図がある。本質を理解してから実装しましょう。

6-2. 仕様のない部分を想像する¶

  • 仕様書に書いていない状況は必ず起きる
  • 「書いていないからやらない」ではなく、チームで議論して決める

第7章：プロジェクトを進める¶

7-1. 実用的なチームワーク¶

  • コードはチームで共有する資産。自分だけのものではない
  • コードレビューは攻撃でなく学びの場
  • 知識のサイロ（属人化） を避ける。バスファクターを下げる

バスファクター：「この人がバスにひかれたら開発が止まる」人数のこと。1人に依存しすぎる状態は危険。

7-2. テストを書く習慣¶

  • テストはバグを見つけるためだけでなく、設計を改善するためにも使える
  • テストが書きにくいコードは、設計が悪いサインかもしれない
  • テストを書かないコードは、あなたの将来の敵になる

7-3. 自動化できることは自動化する¶

  • 手作業は間違えやすく、再現性が低い
  • ビルド・テスト・デプロイは自動化する
  • 繰り返し手作業をしていると気づいたら、自動化を検討する

第8章：達人の姿勢¶

8-1. 知識のポートフォリオを育てる¶

  • 毎年少なくとも1つの新しい言語を学ぶ
  • 技術書を読む、勉強会に参加するなど、継続的に学ぶ
  • 今役に立たない知識でも、引き出しを増やすことに価値がある

8-2. コミュニケーションも技術のひとつ¶

  • 良いコードを書くだけでなく、伝わるドキュメント・メッセージを書くことも重要
  • 相手（読者）を意識した文書を書く
  • 言いたいことを先に言い、理由や詳細を後に書く

まとめ：達人プログラマーになるための10の習慣¶

# 習慣 一言で
1 DRY原則を守る 繰り返しを排除する
2 バージョン管理を使う 変更履歴は財産
3 早期クラッシュを心がける 問題は早く発見する
4 テストを書く 将来の自分への贈り物
5 小さく動かす（曳光弾） 方向性を早く確認する
6 自動化する 手作業を最小限に
7 設計を疎結合にする 変更に強くする
8 要件の本質を掘り下げる 「なぜ」を5回聞く
9 継続的に学ぶ 知識のポートフォリオを育てる
10 責任を持つ 自分のキャリアは自分で作る
