UNIX的考え方 要点まとめ¶

対象読者：システム開発に携わる若手エンジニア（未経験〜数年目）
原著：The UNIX Philosophy / Mike Gancarz 著

この本が伝えたいこと¶

「UNIX的考え方」は、UNIXというOSが50年以上にわたって使い続けられている理由を、設計思想・哲学として言語化した本です。
書かれている内容は「UNIXの使い方」ではなく、良いソフトウェアを作るための普遍的な原則です。
プログラミング言語や開発環境が変わっても、ここに書かれた考え方は今も通用します。

9つの主要原則¶

原則1：スモール・イズ・ビューティフル（Small is beautiful）¶

  • プログラムは小さく、シンプルに作る
  • 小さいプログラムは理解しやすく、バグが少なく、直しやすい
  • 巨大な「なんでもできるシステム」は、結果的に誰も使いこなせなくなる

若手へのヒント
「1つの関数・1つのクラス・1つのモジュールが何をするか」を一言で説明できる状態を目指しましょう。
説明できないなら、まだ分割できるサインです。

原則2：1つのプログラムに1つのことをうまくやらせる（Make each program do one thing well）¶

  • プログラムの責任は1つに絞る
  • 複数の機能を詰め込むと、どれも中途半端になる
  • 「○○もできるし、△△もできる」ではなく「○○に特化している」が強み

若手へのヒント
関数を書くとき「この関数は何をするか？」と自問しましょう。
答えが「〜と、〜もする」になったら分割のタイミングです。

原則3：できるだけ早くプロトタイプを作る（Build a prototype as soon as possible）¶

  • 完璧な設計を目指して時間をかけるより、動くものをまず作る
  • 動かして初めてわかる問題がある
  • プロトタイプは「捨てる前提」で作ってよい

若手へのヒント
「動かしてみてから考える」は恥ずかしいことではありません。
早く動かして早くフィードバックをもらうほど、手戻りが少なくなります。

原則4：効率よりも移植性を選ぶ（Choose portability over efficiency）¶

  • 特定の環境・OSにしか動かないプログラムは、時代とともに消える
  • 多少遅くても、どこでも動くプログラムの方が長く使われる
  • 汎用性を意識した設計が、長期的な資産になる

若手へのヒント
「このコードはWindows専用になっていないか？」「このツールは自分のPCでしか動かないのでは？」と意識する習慣をつけましょう。

原則5：データはフラットなテキストファイルに格納する（Store data in flat text files）¶

  • バイナリ形式よりテキスト形式の方が、人間が読め、デバッグしやすく、他のツールと連携しやすい
  • CSVやJSONなどのテキスト形式は、どんな環境でも扱える
  • テキストは「最小公倍数」的な共通言語

若手へのヒント
ログファイルや設定ファイルをテキストで保持する設計は、トラブル時に大きく助かります。
「バイナリで保存するメリットはあるか？」を意識しましょう。

原則6：ソフトウェアの力を最大限に活かす（Use software leverage to your advantage）¶

  • 既存のツール・ライブラリを積極的に使う
  • 「車輪の再発明」をしない
  • 自分でゼロから作るより、組み合わせる方が速く確実

若手へのヒント
「これを実現するライブラリはないか？」を最初に調べましょう。
ただし、何をやっているか理解しないままブラックボックスとして使うのは危険です。

原則7：シェルスクリプトを使って移植性と効率を高める（Use shell scripts to increase leverage and portability）¶

  • 繰り返し作業はスクリプト化する
  • GUIで手動でやっていた作業をコマンドに落とし込むと、再利用・自動化ができる
  • スクリプトはドキュメントにもなる（何をやったかが残る）

若手へのヒント
「毎回同じコマンドを打っているな」と感じたら、スクリプト化のチャンスです。
最初は1行でも、自動化の習慣は大きな時間の節約につながります。

原則8：拘束的なユーザーインターフェースを作らない（Avoid captive user interfaces）¶

  • ユーザーを操作手順に縛り付けるUIは避ける
  • コマンドラインからも使えるようにしておくと、他のプログラムと連携しやすくなる
  • GUIは便利だが、自動化できないUIは弱い

若手へのヒント
「このツールはGUIだけで操作できる」という設計は、自動化の壁になります。
APIやCLIで操作できる設計を意識しましょう。

原則9：全てのプログラムをフィルターとして設計する（Make every program a filter）¶

  • プログラムは「入力を受け取り、変換して、出力する」フィルターとして考える
  • 入力と出力を標準化しておくと、プログラム同士をパイプでつなげる
  • UNIXの |（パイプ）はこの思想の象徴

若手へのヒント
「このプログラムは何を入力として受け取り、何を出力するか」を明確にしましょう。
入出力が明確なプログラムは、テストもしやすくなります。

補足原則（マイナーな掟）¶

原則 概要
沈黙は金なり 問題がなければ出力しない。不要なメッセージはノイズ
並列に考える 処理を直列ではなく並列で考える発想を持つ
部分の総和は全体より大きい 小さなツールを組み合わせると大きな価値が生まれる
90%解を狙う 100%を目指すより、90%を素早く届ける方が現実的
悪い方が良い（Worse is better） 完璧より「ちょっと足りないが動く」の方が普及する
階層的に考える 複雑な問題は階層（レイヤー）に分けて整理する

まとめ：UNIX哲学が教えてくれること¶

シンプルに作り、1つのことに集中し、組み合わせられるようにする

これがUNIX哲学の核心です。

現代のマイクロサービス、クラウドネイティブ、APIファースト設計にも、この考え方は息づいています。
「なぜこの設計がいいのか？」に迷ったとき、UNIX哲学に立ち返ることで判断の軸が生まれます。

開発現場での活かし方（チェックリスト）¶

  • この関数・クラスは1つの責任しか持っていないか？
  • 手動で繰り返している作業をスクリプト化できないか？
  • データをテキスト形式で保持しているか？（ログ・設定ファイルなど）
  • このツールは他のツールと組み合わせて使えるか？
  • 完璧を目指して停滞していないか？まず動くものを作れているか？
  • 既存のライブラリ・ツールを使えないか調べたか？
