エンジニアのためのGit・GitHubチーム実践 要点まとめ¶

対象：未経験〜若手エンジニア
参考書籍：『エンジニアのためのGit・GitHubチーム実践』

この本が伝えたいこと¶

「Gitはチームで安全に・効率よく開発するためのツール」 です。

一人で使うだけでなく、チームで使うことで真価を発揮します。
正しい使い方を身につけることで、ミスを防ぎ、レビューしやすく、リリースしやすい開発ができるようになります。

第1章：Gitの基本的な考え方¶

バージョン管理とは何か¶

ファイルの「変更履歴」を記録・管理する仕組みです。

  • いつ・誰が・何を変えたかを記録できる
  • 過去のバージョンにいつでも戻れる
  • 複数人が同時に作業しても変更をまとめられる

リポジトリとは¶

ファイルとその変更履歴を管理する「保管場所」のことです。

種類 説明
ローカルリポジトリ 自分のPC上にある
リモートリポジトリ GitHubなどサーバー上にある

第2章：基本的なGit操作¶

日常的な作業の流れ¶

1. git pull          → リモートの最新を取り込む
2. （ファイルを編集する）
3. git add           → 変更をステージングに追加
4. git commit        → ローカルに保存（スナップショットを記録）
5. git push          → リモートに送信

コミットの基本¶

コミットは「意味のある単位」でまとめる。

# ステージングに追加
git add src/feature.py

# コミット（わかりやすいメッセージをつける）
git commit -m "ログイン機能を追加"

良いコミットメッセージの書き方：

  • 何をしたかが一言でわかる
  • 「〜を追加」「〜を修正」「〜を削除」のように動詞で始める
  • 長すぎず、短すぎず（50文字程度を目安）

悪い例： 

# 何を変えたか全くわからない
git commit -m "修正"
git commit -m "作業中"

良い例： 

git commit -m "ユーザー登録フォームのバリデーションを追加"
git commit -m "ログイン時のエラーメッセージを日本語に修正"

第3章：ブランチを使った開発¶

ブランチとは¶

メインの作業を邪魔せずに「別の作業ライン」を作る機能です。

main      ●─────────────────●
              \             /
feature        ●───●───●

ブランチを使うことで： - 開発中の機能が本番環境に影響しない - 複数の機能を並行して開発できる - 問題があっても簡単に取り消せる

ブランチの基本操作¶

# ブランチを作成して移動
git checkout -b feature/ログイン機能

# 現在のブランチを確認
git branch

# ブランチを切り替える
git checkout main

# ブランチを削除
git branch -d feature/ログイン機能

ブランチ名の付け方¶

チームで統一したルールを使うことが重要です。

プレフィックス 用途 例
feature/ 新機能の開発 feature/user-login
fix/ バグ修正 fix/login-error
hotfix/ 緊急の本番修正 hotfix/payment-crash
release/ リリース準備 release/v1.2.0

第4章：GitHubを使ったチーム開発¶

プルリクエスト（PR）とは¶

自分のブランチの変更を「mainブランチに取り込んでほしい」とお願いする機能です。

PRを作ることで： - コードレビューができる（品質向上） - 変更内容を議論できる - 変更履歴が見える化される

良いPRの書き方¶

## 変更内容
ユーザーのパスワードリセット機能を追加しました。

## 変更理由
ユーザーからパスワードを忘れた場合の対応が必要というフィードバックがあったため。

## 確認方法
1. ログイン画面の「パスワードを忘れた方」リンクをクリック
2. メールアドレスを入力して送信
3. メールに届いたリンクからパスワードをリセット

## スクリーンショット
（画面キャプチャがあれば添付する）

コードレビューの心得¶

レビュアー（見る側）： - 批判ではなく、改善提案として伝える - 「なぜ？」を丁寧に質問する - 良いところも積極的に褒める

# 悪い例
このコードはわかりにくい。書き直して。

# 良い例
この処理は○○のケースで問題が起きそうです。
△△のように書くとどうでしょうか？

レビュイー（書いた側）： - 指摘を個人攻撃と受け取らない - 不明点は素直に質問する - 修正したら理由と一緒に返答する

第5章：コンフリクト（競合）の解決¶

コンフリクトとは¶

複数人が同じファイルの同じ行を変更すると発生する「競合」です。

<<<<<<< HEAD（自分の変更）
ボタンの色は青にする
=======
ボタンの色は赤にする
>>>>>>> feature/button-color（相手の変更）

解決手順¶

  1. コンフリクトしているファイルを開く
  2. <<<<<<<、=======、>>>>>>> の記号を探す
  3. どちらの変更を採用するか（または両方を組み合わせるか）決める
  4. 記号を含めて不要な行を削除する
  5. git add して git commit する

ポイント：コンフリクトは怖くない
どちらの変更を残すかを判断するだけです。迷ったら作業した人に相談しましょう。

第6章：チーム開発のブランチ戦略¶

Git Flow（代表的な戦略）¶

大規模な開発やリリースサイクルが明確なプロジェクト向け。

main        ●──────────────────────────●  ← 本番用
             \                        /
develop       ●──●──●──●──●──●──●   ← 開発用
                  \      /
feature            ●────●              ← 機能開発
ブランチ 役割
main 本番環境のコード（常に動く状態）
develop 次のリリースに向けた統合ブランチ
feature/* 機能ごとの開発ブランチ
release/* リリース前の最終調整
hotfix/* 本番の緊急修正

GitHub Flow（シンプルな戦略）¶

小規模チームや継続的デプロイ向け。

main        ●──────────────────●
             \                /
feature       ●───●───●───●
  1. main から feature ブランチを切る
  2. コミットを積み重ねる
  3. PRを作ってレビューしてもらう
  4. レビューOKになったら main にマージ

初心者にはGitHub Flowがおすすめ
シンプルなルールで迷いにくいため、まずこちらから始めると良いでしょう。

第7章：GitHubの便利な機能¶

Issues（イシュー）¶

バグ報告・機能要望・タスク管理に使います。

## バグの内容
ログインページでパスワードが正しいのにエラーになる

## 再現手順
1. ログインページを開く
2. 正しいIDとパスワードを入力
3. ログインボタンをクリック

## 期待する動作
ログインが成功してトップページに移動する

## 実際の動作
「パスワードが違います」というエラーが表示される

GitHub Actions（CI/CD）¶

コードをpushしたときに自動でテストやデプロイを実行する仕組みです。

# .github/workflows/test.yml の例
name: テスト自動実行
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: テスト実行
        run: npm test

メリット： - 人的ミスを防げる - レビュー前にテストが通っているか確認できる - デプロイ作業を自動化できる

第8章：よくあるミスと対処法¶

間違えてmainにコミットしてしまった¶

# 直前のコミットを取り消して変更をワーキングツリーに戻す
git reset HEAD~1

コミットメッセージを間違えた（まだpushしていない場合）¶

# 直前のコミットメッセージを修正
git commit --amend -m "正しいメッセージ"

間違えたファイルをaddしてしまった¶

# ステージングから取り消す（ファイルの変更は残る）
git reset HEAD ファイル名

pushしてしまったコミットを取り消したい¶

# 取り消し用の新しいコミットを作成（安全な方法）
git revert コミットID

⚠️ git push --force は他の人の作業を壊す可能性があるため、チーム開発では使わないのが基本です。

チェックリスト：チーム開発で守るべきこと¶

毎日の作業前¶

  • git pull で最新のコードを取り込む
  • 作業用のブランチを作成する（mainに直接コミットしない）

コミット時¶

  • 意味のある単位でコミットする
  • コミットメッセージは何をしたかが分かる内容にする
  • 不要なファイル（ログ・一時ファイルなど）をコミットしない

プルリクエスト作成時¶

  • PR説明に変更内容・理由・確認方法を記載する
  • セルフレビューで基本的なミスを事前にチェックする
  • 差分が大きすぎる場合は分割を検討する

レビュー時¶

  • 批判ではなく提案として書く
  • 良い点も伝える
  • 疑問点は遠慮なく質問する

まとめ¶

テーマ ポイント
コミット 小さく・意味のある単位で・わかりやすいメッセージ
ブランチ 作業ごとに必ず切る・名前はルールに従う
PR 説明を丁寧に書く・小さく作る
レビュー 批判しない・疑問は質問する
コンフリクト 焦らず・相談しながら解決する

Gitは「慣れ」が大切です。最初はわからないことも多いですが、
失敗しても必ず元に戻せるのがGitの強みです。
積極的に使って、チーム全体の開発を支える力を身につけましょう。
