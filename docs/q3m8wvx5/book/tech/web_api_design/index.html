Web APIの設計 — 要点まとめ¶

対象読者：未経験・若手エンジニア
参考書籍：「Web APIの設計」

1. Web APIとは何か¶

API（Application Programming Interface） とは、ソフトウェア同士が会話するための「窓口」です。
Web APIはその窓口をHTTP（インターネットの通信プロトコル）を使って提供します。

クライアント（スマホアプリ・Webブラウザ）
    ↓ HTTPリクエスト（「このデータをください」）
Web API（サーバー）
    ↓ HTTPレスポンス（「はい、どうぞ」）
クライアント

2. REST APIの基本¶

現在最も広く使われているWeb APIのスタイルが REST です。
REST APIは以下の考え方に基づいています。

リソース指向¶

APIは「もの（リソース）」を中心に設計します。

良い例 悪い例
/users /getUser
/orders/123 /fetchOrderById?id=123

URLはリソースの「住所」です。動詞（動き）はHTTPメソッドで表します。

HTTPメソッドの使い分け¶

メソッド 意味 例
GET データを取得する ユーザー一覧を取る
POST 新しいデータを作る ユーザーを登録する
PUT データを丸ごと更新する ユーザー情報を上書きする
PATCH データを一部更新する メールアドレスだけ変える
DELETE データを削除する ユーザーを削除する

3. URLの設計¶

基本ルール¶

  • 小文字を使う（/users ◎、/Users ✕）
  • 単語の区切りは ハイフン（-）（/user-profiles ◎、/user_profiles △）
  • リソース名は複数形（/users ◎、/user △）
  • URLに動詞を入れない（/getUsers ✕ → /users ◎）

階層構造¶

親子関係のあるリソースはパスで表します。

/users              # ユーザー一覧
/users/123          # IDが123のユーザー
/users/123/orders   # IDが123のユーザーの注文一覧
/users/123/orders/456 # そのユーザーの特定の注文

クエリパラメータ¶

絞り込みや並び替えにはクエリパラメータを使います。

/users?status=active          # 有効なユーザーのみ
/users?sort=created_at&order=desc  # 作成日降順で並び替え
/products?page=2&per_page=20  # ページング（2ページ目、1ページ20件）

4. HTTPステータスコード¶

レスポンスには必ず適切なステータスコードを返します。

よく使うコード¶

コード 意味 使う場面
200 OK 成功 GET・PUT・PATCHが成功した
201 Created 作成成功 POSTで新規作成に成功した
204 No Content 成功（返すデータなし） DELETEが成功した
400 Bad Request リクエストが不正 入力値が間違っている
401 Unauthorized 認証が必要 ログインが必要な操作
403 Forbidden アクセス禁止 権限がない操作
404 Not Found 見つからない 指定したリソースが存在しない
409 Conflict 競合 同じメールアドレスが既に存在する
500 Internal Server Error サーバーエラー サーバー側のバグ

5. リクエスト・レスポンスのデータ形式¶

JSONを使う¶

Web APIのデータのやり取りには JSON を使うのが標準です。

{
  "id": 123,
  "name": "田中太郎",
  "email": "tanaka@example.com",
  "created_at": "2024-01-15T09:00:00Z"
}

Content-Typeヘッダー¶

リクエスト・レスポンスともに必ず指定します。

Content-Type: application/json

日付・時刻の形式¶

日時は ISO 8601形式（UTC）で統一します。

"created_at": "2024-01-15T09:00:00Z"  ◎
"created_at": "2024/01/15 18:00"      ✕（タイムゾーン不明、形式が独自）

6. バージョン管理¶

APIは一度公開したら簡単に変更できません（使っているアプリが壊れるため）。
そのため、バージョン番号をURLに含めるのが一般的です。

/v1/users   # バージョン1
/v2/users   # バージョン2（新仕様）

7. 認証・認可¶

認証（Authentication）：「あなたは誰ですか？」¶

  • APIキー: シンプルな用途向け。Authorization: Bearer {APIキー}
  • OAuth 2.0: 外部サービスと連携する場合の標準方式

認可（Authorization）：「あなたはこれをしていいですか？」¶

  • ユーザーがアクセスできるリソースを制御する
  • 例：自分の注文は見られるが、他人の注文は見られない

8. エラーレスポンスの設計¶

エラー時は、なぜエラーになったかをクライアントに伝えます。

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力値が正しくありません",
    "details": [
      {
        "field": "email",
        "message": "メールアドレスの形式が正しくありません"
      }
    ]
  }
}
  • code: システムが識別するエラーコード（英字・大文字）
  • message: 人間が読めるエラー説明
  • details: どのフィールドで何が問題かの詳細（バリデーションエラー時）

9. ページング（大量データの分割取得）¶

一度に大量のデータを返すとサーバー・クライアントに負荷がかかります。
ページングで分割して返します。

オフセットページング（シンプルで使いやすい）¶

GET /users?page=1&per_page=20

レスポンスに件数情報を含める：

{
  "data": [...],
  "pagination": {
    "total": 150,
    "page": 1,
    "per_page": 20,
    "total_pages": 8
  }
}

カーソルページング（大量データ向け）¶

GET /users?cursor=eyJpZCI6MTAwfQ&limit=20

SNSのタイムラインなど、リアルタイムに更新されるデータに適しています。

10. セキュリティの基本¶

対策 内容
HTTPS必須 通信を暗号化する（HTTPは使わない）
入力値バリデーション 受け取ったデータは必ずサーバー側で検証する
レート制限 1ユーザーが短時間に大量リクエストできないよう制限する
CORS設定 ブラウザからのアクセス元を適切に制限する
機密情報をURLに入れない パスワード・トークンはURLでなくBodyやHeaderに

11. APIドキュメント¶

APIは使ってもらうためのもの。ドキュメントは必ず書きます。

OpenAPI（Swagger）¶

API仕様を標準形式で記述できるツールです。
openapi.yaml を書くと、自動でドキュメントサイトが生成できます。

最低限ドキュメントに含めるべき内容： - エンドポイントのURL・HTTPメソッド - リクエストのパラメータ・ボディの説明 - レスポンスの構造とフィールドの説明 - エラーコードの一覧

まとめ：良いWeb APIの条件¶

観点 ポイント
一貫性 命名規則・データ形式・エラー形式を統一する
わかりやすさ URLを見ただけで何のAPIかわかる
適切なHTTPの活用 メソッドとステータスコードを正しく使う
安全性 HTTPS・認証・バリデーションを徹底する
ドキュメント 使う人が迷わないよう丁寧に書く
