Webを支える技術 — 要点まとめ¶

対象読者：Web開発の経験が浅い若手エンジニア
参考書籍：『Webを支える技術 ── HTTP、URI、HTML、そしてREST』山本陽平 著（技術評論社）

1. Webとは何か¶

Webの三大要素¶

Webは3つの技術の組み合わせで成り立っています。

技術 役割 例
URI リソースの住所（どこにあるか） https://example.com/users/1
HTTP 通信のルール（どうやり取りするか） GET /users/1 HTTP/1.1
HTML 情報の表現形式（何を伝えるか） <h1>ユーザー情報</h1>

リソースとは¶

リソース = Web上で扱うすべての「もの」

  • ユーザー情報、商品データ、画像、動画など
  • 「名詞」で考える。「ユーザーを取得する」ではなく「ユーザーというリソースが存在する」

2. URI — リソースを指し示す仕組み¶

URIの構造¶

https://api.example.com:443/users/123?format=json#section1
  │         │              │    │       │            │
  │         │              │    │       │            └── フラグメント（ページ内の位置）
  │         │              │    │       └── クエリ（検索条件などの補足情報）
  │         │              │    └── パス（リソースの場所）
  │         │              └── ポート番号
  │         └── ホスト名（サーバーのアドレス）
  └── スキーム（通信方式）

良いURIの設計原則¶

  1. 変わらない — URIは一度公開したら変えない（リンク切れを防ぐ）
  2. シンプルで推測しやすい — /users/123 は分かりやすい、/getUserById?id=123 は冗長
  3. リソースを表す名詞を使う — 動詞はURIに入れない
  4. 階層構造を表現する — /departments/1/employees （部署1の従業員一覧）

悪い例 vs 良い例

# 悪い例（動詞がURIに含まれている）
POST /createUser
GET  /getUserList
POST /deleteUser?id=5

# 良い例（名詞でリソースを表現）
POST   /users          ← ユーザー作成
GET    /users          ← ユーザー一覧取得
DELETE /users/5        ← ユーザー削除

URIとURLの違い¶

  • URL (Uniform Resource Locator)：リソースの場所を示す
  • URI (Uniform Resource Identifier)：リソースを識別する（URLを含む上位概念）
  • 日常会話では混同しても問題ないが、技術的にはURIの方が広い概念

3. HTTP — Webの通信プロトコル¶

HTTPの基本：リクエストとレスポンス¶

Webの通信はすべて クライアント（ブラウザ等）がリクエストを送り、サーバーがレスポンスを返す 流れです。

クライアント                       サーバー
    │                                 │
    │  ──── HTTPリクエスト ────→      │
    │                                 │  処理
    │  ←─── HTTPレスポンス ────       │
    │                                 │

HTTPリクエストの構造¶

GET /users/1 HTTP/1.1          ← リクエストライン（メソッド・パス・バージョン）
Host: api.example.com          ← ヘッダ（付加情報）
Accept: application/json

                               ← 空行（ヘッダとボディの区切り）
                               ← ボディ（GETの場合は通常なし）

HTTPレスポンスの構造¶

HTTP/1.1 200 OK                ← ステータスライン（バージョン・ステータスコード・理由句）
Content-Type: application/json ← ヘッダ
Content-Length: 42

{"id": 1, "name": "山田太郎"}  ← ボディ（実際のデータ）

HTTPのバージョン¶

バージョン 特徴
HTTP/1.0 1リクエストごとに接続を切断
HTTP/1.1 接続の再利用（Keep-Alive）、パイプライン化
HTTP/2 複数リクエストの並列処理、ヘッダ圧縮
HTTP/3 UDPベース（QUIC）、さらなる高速化

4. HTMLとメディアタイプ¶

メディアタイプ（Content-Type）¶

HTTPでデータをやり取りするとき、「このデータは何形式か」をサーバーとクライアントに伝えるのがメディアタイプです。

Content-Type: text/html; charset=UTF-8
Content-Type: application/json
Content-Type: image/png
Content-Type: application/pdf

形式は タイプ/サブタイプ です。

タイプ 意味 例
text テキスト系 text/html, text/plain
image 画像 image/png, image/jpeg
application アプリケーション固有 application/json, application/xml

コンテントネゴシエーション¶

クライアントが「自分はこの形式が扱える」とサーバーに伝え、サーバーが最適な形式でレスポンスを返す仕組みです。

# クライアントのリクエスト（JSONとXMLを受け入れられる）
Accept: application/json, application/xml;q=0.9

# サーバーのレスポンス（JSONで返す）
Content-Type: application/json

5. REST — Webのアーキテクチャスタイル¶

RESTとは¶

REST (Representational State Transfer) = Webのアーキテクチャスタイル（設計思想）

「こう設計すると良いWebシステムができる」という考え方の集まりです。

RESTの6つの制約¶

制約 内容 なぜ重要か
クライアント/サーバー 役割を分離する それぞれが独立して進化できる
ステートレス サーバーはセッション状態を保持しない スケールアウトが容易になる
キャッシュ レスポンスにキャッシュ可否を示す 性能向上、負荷軽減
統一インターフェース URIとHTTPメソッドで統一する システム間連携が容易になる
階層化システム ロードバランサ等を挟める 柔軟なシステム構成が可能
コードオンデマンド JavaScriptなどを動的に送れる （オプション）機能の拡張性

ステートレスとは¶

サーバーがクライアントの「状態」を記憶しないということです。

# ステートフル（サーバーが状態を持つ）— RESTではない
リクエスト1: ログイン
リクエスト2: 商品を取得（サーバーは「誰がログインしているか」を覚えている）

# ステートレス（サーバーが状態を持たない）— REST
リクエスト1: ログイン → トークンを受け取る
リクエスト2: 商品を取得（リクエストにトークンを含める）

各リクエストが自己完結しているため、サーバーを複数台に増やしやすくなります。

6. HTTPメソッドの使い方¶

HTTPメソッドは「リソースに何をするか」を表します。

主要メソッド一覧¶

メソッド 操作 冪等性 安全性
GET リソースの取得 ○ ○
POST リソースの作成 ✕ ✕
PUT リソースの置き換え（全体更新） ○ ✕
PATCH リソースの部分更新 △ ✕
DELETE リソースの削除 ○ ✕
HEAD ヘッダのみ取得（GETと同じだがボディなし） ○ ○
OPTIONS 利用可能なメソッドの確認 ○ ○

冪等性（べきとうせい）：同じ操作を何度繰り返しても結果が同じになること
例：DELETE /users/1 を2回実行しても「ユーザー1が存在しない」状態になる（変わらない）

安全性：リソースの状態を変更しないこと
例：GETはデータを取得するだけで変更しない

CRUDとHTTPメソッドの対応¶

CRUD操作 HTTPメソッド URI例
Create（作成） POST POST /users
Read（取得） GET GET /users / GET /users/1
Update（更新） PUT / PATCH PUT /users/1
Delete（削除） DELETE DELETE /users/1

7. ステータスコード¶

レスポンスの「結果」を3桁の数字で表します。

ステータスコードの分類¶

範囲 分類 意味
1xx 情報 処理継続中
2xx 成功 リクエスト成功
3xx リダイレクト 別のURIへ転送
4xx クライアントエラー リクエストに問題がある
5xx サーバーエラー サーバー側で問題が発生

よく使うステータスコード¶

コード 名称 使いどころ
200 OK 成功（GET/PUT/PATCHなど）
201 Created リソース作成成功（POST）
204 No Content 成功だがボディなし（DELETE）
301 Moved Permanently 永続的なリダイレクト（URI変更時）
302 Found 一時的なリダイレクト
304 Not Modified キャッシュが有効（更新なし）
400 Bad Request リクエストの形式が不正
401 Unauthorized 認証が必要
403 Forbidden 認証済みだが権限なし
404 Not Found リソースが存在しない
409 Conflict 競合（重複登録など）
500 Internal Server Error サーバー内部エラー
503 Service Unavailable サーバーが一時的に利用不可

8. HTTPヘッダ¶

リクエストヘッダ（クライアント → サーバー）¶

ヘッダ名 役割 例
Host アクセス先のホスト名 Host: api.example.com
Accept 受け入れ可能なメディアタイプ Accept: application/json
Authorization 認証情報 Authorization: Bearer <token>
Content-Type ボディのメディアタイプ Content-Type: application/json
User-Agent クライアントの種類 User-Agent: Mozilla/5.0 ...

レスポンスヘッダ（サーバー → クライアント）¶

ヘッダ名 役割 例
Content-Type ボディのメディアタイプ Content-Type: application/json
Location リダイレクト先や作成されたリソースのURI Location: /users/123
Cache-Control キャッシュの制御 Cache-Control: max-age=3600
ETag リソースのバージョン識別子 ETag: "abc123"

キャッシュの仕組み¶

# 初回リクエスト
クライアント → GET /users/1
サーバー     ← 200 OK + ETag: "abc123" + Cache-Control: max-age=3600

# 2回目（キャッシュ有効期限内）
クライアントはサーバーにアクセスせずキャッシュを使用

# キャッシュ期限切れ後
クライアント → GET /users/1 + If-None-Match: "abc123"
サーバー     ← 304 Not Modified（変更なし → キャッシュ継続使用）
         または 200 OK + 新しいデータ（変更あり）

9. Webサービスの設計パターン¶

RESTful APIの設計ガイドライン¶

URIの設計¶

# コレクション（一覧）とメンバー（個別）
GET  /articles          # 記事一覧
POST /articles          # 記事作成
GET  /articles/1        # 記事1を取得
PUT  /articles/1        # 記事1を更新
DELETE /articles/1      # 記事1を削除

# ネストしたリソース
GET /articles/1/comments       # 記事1のコメント一覧
POST /articles/1/comments      # 記事1にコメント作成
GET /articles/1/comments/5    # 記事1のコメント5を取得

クエリパラメータの使いどころ¶

# 絞り込み・並び替え・ページング → クエリパラメータ
GET /articles?category=tech&sort=created_at&page=2&per_page=20

# リソースの識別（どのリソースか）→ パス
GET /articles/1

API バージョニング¶

APIを変更する場合、既存のクライアントを壊さないためにバージョン管理します。

# URIにバージョンを含める（シンプルで分かりやすい）
/v1/users
/v2/users

# ヘッダにバージョンを含める
Accept: application/vnd.example.v2+json

10. セキュリティの基礎¶

HTTPS（HTTP over TLS/SSL）¶

HTTPはデータが平文で流れるため盗聴・改ざんのリスクがあります。HTTPSはTLSによって通信を暗号化します。

HTTP  → データが平文（盗聴・改ざん可能）
HTTPS → データが暗号化（安全な通信）

現代のWebサービスでは必ずHTTPSを使用します。

認証・認可¶

概念 意味 例
認証（Authentication） 「あなたは誰？」を確認する ログイン、API Key確認
認可（Authorization） 「あなたは何ができる？」を確認する 管理者のみ削除可能

よく使われる認証方式¶

Basic認証 

Authorization: Basic base64(username:password)
- シンプルだがHTTPSなしでは危険

Bearer Token（JWT等） 

Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
- トークンをHTTPヘッダで送る - 現代のAPIでよく使われる

OAuth 2.0 - 外部サービスへのアクセス権限を安全に委譲する仕組み - 「Googleでログイン」などがこれ

CORS（Cross-Origin Resource Sharing）¶

異なるドメイン間のHTTPリクエストを制御する仕組みです。

# フロントエンド: https://app.example.com
# APIサーバー:    https://api.example.com

# ブラウザはデフォルトで異なるオリジンへのリクエストをブロック
# APIサーバー側でCORSを許可するヘッダを返す必要がある

Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE

まとめ：Webの基本を押さえるためのチェックリスト¶

  • URIはリソースを表す名詞で設計できる
  • HTTPメソッド（GET/POST/PUT/DELETE）の使い分けが分かる
  • ステータスコードの意味を200/400/500系で説明できる
  • RESTがステートレスである理由を説明できる
  • HTTPSとHTTPの違いを説明できる
  • 認証と認可の違いを説明できる
  • Content-Typeとは何かを説明できる
