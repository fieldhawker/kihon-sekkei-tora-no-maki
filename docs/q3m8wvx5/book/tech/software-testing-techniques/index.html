初めて学ぶソフトウェアのテスト技法 — 要点まとめ¶

対象読者：テストの経験が少ない若手エンジニア
目的：現場で使えるテスト技法の基礎を身につける

1. テストとは何か¶

テストの目的は「バグを見つけること」であり、「動くことを証明すること」ではありません。

  • テストで「バグがない」ことは証明できない（全網羅は不可能）
  • 限られた時間・コストの中で、リスクの高い箇所を重点的にテストする
  • バグを早期に発見するほど、修正コストは安くなる

2. テストの種類（レベル）¶

レベル 対象 誰がやるか
単体テスト（UT） 関数・クラス単体 開発者
結合テスト（IT） モジュール間の連携 開発者・テスター
システムテスト（ST） システム全体 テスター
受入テスト（UAT） 要件を満たすか ユーザー・顧客

3. ブラックボックステスト技法¶

内部の実装を見ず、入出力の仕様からテストを設計する技法です。

3-1. 同値分割（Equivalence Partitioning）¶

入力値を「同じ扱いをされるグループ（同値クラス）」に分けて、各グループから代表値を1つ選ぶ。

例：年齢入力（0〜120が有効）

クラス 値の範囲 テスト例
有効同値クラス 0〜120 30
無効同値クラス（下限外） 0未満 -1
無効同値クラス（上限外） 121以上 150

ポイント： 同じクラス内の値は全部テストしなくてよい。代表値1つで十分。

3-2. 境界値分析（Boundary Value Analysis）¶

バグは境界（端）に集中しやすいため、境界の前後を重点的にテストする。

例：年齢入力（0〜120が有効）

テストすべき値：-1, 0, 1, 119, 120, 121

ポイント： 同値分割と組み合わせて使うことで効果が高まる。

3-3. デシジョンテーブル（Decision Table）¶

複数の条件の組み合わせを表で整理し、漏れなくテストするための技法。

例：割引ロジック（会員かつ購入金額1万円以上で割引）

条件 ケース1 ケース2 ケース3 ケース4
会員である Y Y N N
1万円以上 Y N Y N
割引適用 Yes No No No

ポイント： 条件が多いほど組み合わせが爆発するため、重要な条件に絞る。

3-4. 状態遷移テスト（State Transition Testing）¶

システムが「状態」を持ち、イベントによって状態が変わる場合に使う技法。

例：ATMの状態遷移

[待機中] --カード挿入--> [暗証番号入力中] --正しいPIN--> [操作中]
                                          --誤りPIN×3--> [カード没収]

テスト観点： - 正常な状態遷移を通るか - 許可されていない遷移が発生しないか - 各状態での操作が正しく機能するか

3-5. ユースケーステスト（Use Case Testing）¶

ユーザーの操作シナリオ（ユースケース）に沿ってテストを設計する技法。

  • 正常系フロー（基本フロー）を必ずテストする
  • 代替フロー・例外フローもテストに含める
  • エンドツーエンドの動作確認に適している

4. ホワイトボックステスト技法¶

内部のソースコードを見て、どのコードが実行されたかを基準にテストを設計する技法。

4-1. カバレッジ（Coverage）¶

テストがコードをどれだけ網羅しているかを示す指標。

カバレッジ種別 意味 目安
C0（命令網羅） 全命令が最低1回実行 最低限
C1（分岐網羅） 全分岐（true/false）を実行 推奨
C2（条件網羅） 全条件の組み合わせを実行 高品質が必要な場合 
# 例：C1カバレッジを満たすには分岐の両方をテストする必要がある
def check_age(age):
    if age >= 18:       # True になるテストと
        return "adult"
    else:               # False になるテストが必要
        return "minor"

5. 経験ベースのテスト技法¶

5-1. エラー推測（Error Guessing）¶

経験や勘に基づいて、「バグが起きやすそうな箇所」を推測してテストする。

よくあるバグのパターン： - null / 空文字 / 空配列の入力 - 最大値・最小値の境界 - 0除算 - 特殊文字（', ", <, > など）の入力 - タイムゾーン・うるう年の日付

5-2. 探索的テスト（Exploratory Testing）¶

テストケースを事前に決めず、実際に操作しながらバグを探すテスト手法。

  • 仕様書に書かれていない振る舞いを発見しやすい
  • テスト担当者のスキルに依存する
  • 「チャーター（目的）」を決めてから実施すると効率的

6. テスト設計の進め方¶

1. テスト条件の洗い出し
   └── テスト対象の仕様・要件から「何をテストすべきか」を列挙する

2. テストケースの設計
   └── 上記の技法を使って、具体的な入力値と期待値を決める

3. テスト手順の作成
   └── 誰が実施しても同じ結果になるよう、手順を明確に記述する

4. テストの実施と結果記録
   └── 期待値と実際の結果を比較し、差異（バグ）を記録する

5. バグ報告
   └── 再現手順・環境・期待値・実際の値を明確に記述する

7. 良いバグ報告の書き方¶

若手がつまずきやすいポイントです。以下の情報を必ず含めてください。

【バグ報告テンプレート】

タイトル：〇〇画面で△△すると□□になる

環境：
  OS: Windows 11
  ブラウザ: Chrome 120
  バージョン: v1.2.3

再現手順：
  1. ログイン画面を開く
  2. メールアドレスに「test@example.com」を入力する
  3. パスワードを空白のまま「ログイン」ボタンを押す

期待結果：「パスワードを入力してください」とエラーメッセージが表示される

実際の結果：エラーメッセージが表示されず、画面が白くなる（フリーズ）

添付：スクリーンショット・ログなど

8. まとめ：テスト技法の選び方¶

状況 使うべき技法
入力値の範囲が決まっている 同値分割 ＋ 境界値分析
条件の組み合わせが多い デシジョンテーブル
画面遷移・ステータス管理がある 状態遷移テスト
ユーザー操作フローを確認したい ユースケーステスト
コードの網羅率を上げたい カバレッジ（C0/C1）
仕様の漏れ・ケアレスミスを探したい エラー推測・探索的テスト

若手へのアドバイス：
「どの技法を使えばいいか分からない」場合は、まず 同値分割 ＋ 境界値分析 を使いましょう。
ほとんどの入力系のテストはこの2つでカバーできます。
慣れてきたら、デシジョンテーブルや状態遷移テストを組み合わせていきましょう。
