SQLアンチパターン 要点まとめ¶

対象読者：データベースを使ったシステム開発に携わる若手エンジニア
原著：Bill Karwin「SQL Antipatterns: Avoiding the Pitfalls of Database Programming」

はじめに¶

「アンチパターン」とは、一見よさそうに見えるが、実は問題を引き起こしやすい実装パターンのことです。
この資料では、データベース設計・SQLの現場でよく起きる失敗パターンと、その正しい対処法を学びます。

第1部：論理的なデータベース設計のアンチパターン¶

1. ジェイウォーク（信号無視）¶

問題のあるパターン：
複数の値をカンマ区切りで1つのカラムに詰め込む。

-- ❌ 悪い例
CREATE TABLE products (
    id INT,
    tag_ids VARCHAR(100)  -- "1,2,3" のように格納
);

なぜ問題か？ - 特定のタグを持つ商品を検索するとき、LIKE '%1%' のような曖昧な検索になる - 件数を数えられない、外部キー制約が使えない

正しい対処法：
多対多の関係は中間テーブル（交差テーブル）を使う。

-- ✅ 良い例
CREATE TABLE product_tags (
    product_id INT,
    tag_id     INT,
    PRIMARY KEY (product_id, tag_id)
);

2. ナイーブツリー（素朴な木）¶

問題のあるパターン：
親子関係を持つデータ（カテゴリの階層など）を、parent_id だけで表現する。

-- ❌ 悪い例（隣接リスト）
CREATE TABLE categories (
    id        INT PRIMARY KEY,
    name      VARCHAR(100),
    parent_id INT  -- 親カテゴリのIDだけ持つ
);

なぜ問題か？ - 「ある親の下にある全ての子孫」を取得するには、再帰的なクエリが必要で複雑になる

正しい対処法（用途に応じて選択）：

手法 特徴
経路列挙（Path Enumeration） /1/3/7/ のようにパスを文字列で持つ。参照は簡単だが更新が難しい
入れ子集合（Nested Sets） 左右の番号でツリーを表現。検索が速いが更新が複雑
閉包テーブル（Closure Table） 祖先・子孫の全ペアを別テーブルに持つ。最も汎用的でおすすめ 
-- ✅ 良い例（閉包テーブル）
CREATE TABLE categories (
    id   INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE category_paths (
    ancestor   INT,
    descendant INT,
    depth      INT,
    PRIMARY KEY (ancestor, descendant)
);

-- 全ての子孫を取得（深さに関係なく1クエリで済む）
SELECT c.*
FROM categories c
JOIN category_paths cp ON c.id = cp.descendant
WHERE cp.ancestor = 3;  -- カテゴリID=3の全子孫

3. IDリクワイアド（とにかくID）¶

問題のあるパターン：
すべてのテーブルに機械的に id という名前の連番カラムを追加する。

なぜ問題か？ - 本来キーになるべき列（例：メールアドレス、注文番号）の一意性が保証されなくなる - 中間テーブルに不要な id が増えて複合主キーの意義が薄れる

-- ❌ 悪い例（中間テーブルに不要なidを追加）
CREATE TABLE product_tags (
    id         INT PRIMARY KEY AUTO_INCREMENT,  -- 不要
    product_id INT,
    tag_id     INT
);

-- ❌ 悪い例（メールアドレスの一意性が担保されない）
CREATE TABLE users (
    id    INT PRIMARY KEY AUTO_INCREMENT,
    email VARCHAR(255)  -- UNIQUE制約なし
);

正しい対処法：
主キーは自然キー（意味のある値）か複合主キーを検討する。連番IDが本当に必要かを考える。

-- ✅ 良い例（複合主キーで十分な中間テーブル）
CREATE TABLE product_tags (
    product_id INT,
    tag_id     INT,
    PRIMARY KEY (product_id, tag_id)
);

-- ✅ 良い例（自然キーに一意制約）
CREATE TABLE users (
    id    INT PRIMARY KEY AUTO_INCREMENT,
    email VARCHAR(255) UNIQUE NOT NULL
);

4. キーレスエントリ（外部キー嫌い）¶

問題のあるパターン：
「パフォーマンスが落ちる」「柔軟性が下がる」という理由で外部キー制約を使わない。

なぜ問題か？ - 参照先に存在しないIDが入り、データが壊れる（孤立レコード） - アプリ側で整合性チェックをしても、バグや並行処理の問題で抜け漏れが起きる

-- ❌ 悪い例（外部キー制約なし）
CREATE TABLE orders (
    id      INT PRIMARY KEY,
    user_id INT  -- 存在しないユーザーIDでも入れられる
);

正しい対処法：
外部キー制約を積極的に使う。整合性はデータベースに守らせる。

-- ✅ 良い例
CREATE TABLE orders (
    id      INT PRIMARY KEY,
    user_id INT NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE RESTRICT
);

5. EAV（エンティティ・アトリビュート・バリュー）¶

問題のあるパターン：
属性名と値をペアで持つ汎用テーブルで何でも管理しようとする。

-- ❌ 悪い例
CREATE TABLE attributes (
    entity_id  INT,
    attr_name  VARCHAR(50),   -- "color", "size", "weight" ...
    attr_value VARCHAR(200)
);

なぜ問題か？ - データ型の制約ができない（数値なのに文字列で入る） - 特定の属性を使った検索やJOINが非常に複雑になる

正しい対処法： - 属性が固定なら普通にカラムを追加する - 属性が可変なら JSON型（PostgreSQL/MySQL対応）またはサブタイプテーブルを検討する

-- ✅ 良い例①（属性が固定の場合：通常のカラム）
CREATE TABLE products (
    id     INT PRIMARY KEY,
    color  VARCHAR(50),
    size   VARCHAR(20),
    weight DECIMAL(8,2)
);

-- ✅ 良い例②（属性が可変の場合：JSON型）
CREATE TABLE products (
    id         INT PRIMARY KEY,
    name       VARCHAR(100),
    attributes JSON  -- {"color": "red", "size": "M"} のように格納
);

-- JSON内の特定属性での検索（MySQL 5.7+ / PostgreSQL）
SELECT * FROM products WHERE attributes->>'$.color' = 'red';

第2部：物理的なデータベース設計のアンチパターン¶

6. ファントムファイル（幻のファイル）¶

問題のあるパターン：
ファイルのパスだけをDBに保存し、実ファイルはDBの外で管理する。

なぜ問題か？ - DBとファイルシステムの同期がトランザクションで保護できない - ファイルだけ消えた、DBの参照だけ残った、という不整合が起きやすい

-- ❌ 悪い例（パスだけDBに保存）
CREATE TABLE attachments (
    id        INT PRIMARY KEY,
    file_path VARCHAR(255)  -- "/uploads/2024/report.pdf" など
    -- ファイル本体はDBの外のファイルシステムにある
);
-- → DBレコードは削除したがファイルは残る、逆にファイルを消したがDBに参照が残る、といった不整合が起きる

正しい対処法：
ファイルをDBに格納（BLOB型）するか、ファイル管理の仕組みを設計に組み込む。

-- ✅ 良い例①（BLOB型でDB内に格納）
CREATE TABLE attachments (
    id          INT PRIMARY KEY,
    file_name   VARCHAR(255),
    mime_type   VARCHAR(100),
    file_data   BLOB  -- ファイル本体をDB内に保持
);

-- ✅ 良い例②（外部ストレージを使う場合はステータス管理を設ける）
CREATE TABLE attachments (
    id          INT PRIMARY KEY,
    storage_key VARCHAR(255),   -- S3等のオブジェクトキー
    status      ENUM('pending', 'confirmed', 'deleted'),
    created_at  TIMESTAMP
    -- アップロード完了後にconfirmedへ変更するフローで整合性を保つ
);

7. インデックスショットガン（散弾銃インデックス）¶

問題のあるパターン：
「速くなるかも」と全カラムにインデックスを張る、または逆に全く張らない。

なぜ問題か？ - インデックスが多すぎると INSERT/UPDATE/DELETE が遅くなる - 少なすぎると SELECT が遅くなる

-- ❌ 悪い例①（全カラムにインデックスを張る）
CREATE INDEX idx_name    ON users(name);
CREATE INDEX idx_email   ON users(email);
CREATE INDEX idx_age     ON users(age);
CREATE INDEX idx_address ON users(address);  -- ほぼ検索に使わないカラムにも

-- ❌ 悪い例②（インデックスを全く張らない）
CREATE TABLE orders (
    id         INT PRIMARY KEY,
    user_id    INT,   -- JOINで頻繁に使うのにインデックスなし
    created_at TIMESTAMP
);

正しい対処法：
実際のクエリの WHERE 句・ORDER BY 句・JOIN 条件を確認して、必要な箇所だけに絞って張る。EXPLAIN でクエリの実行計画を確認する習慣をつける。

-- ✅ 良い例（使われるクエリに合わせて必要な箇所だけ）
CREATE INDEX idx_orders_user_id    ON orders(user_id);      -- JOIN条件
CREATE INDEX idx_orders_created_at ON orders(created_at);   -- 期間絞り込み

-- EXPLAINで実行計画を確認
EXPLAIN SELECT * FROM orders WHERE user_id = 42 AND created_at >= '2025-01-01';
-- type=ref や Using index が出ればインデックスが効いている

第3部：クエリのアンチパターン¶

8. フィア・オブ・ザ・アンノウン（NULLへの恐怖）¶

問題のあるパターン：
NULLを誤って扱う。

-- ❌ 悪い例（常にFALSEになる）
SELECT * FROM users WHERE deleted_at = NULL;

-- ❌ 悪い例（NULLの行が含まれない）
SELECT * FROM users WHERE name != 'admin';

なぜ問題か？ - NULL は「値が不明」であり、= NULL での比較は常に UNKNOWN（真でも偽でもない）になる - != の比較でもNULLの行は結果に含まれない

正しい対処法：

-- ✅ 良い例
SELECT * FROM users WHERE deleted_at IS NULL;
SELECT * FROM users WHERE name != 'admin' OR name IS NULL;

9. アンビギュアスグループ（曖昧なグループ）¶

問題のあるパターン：
GROUP BY を使ったクエリで、集計していないカラムを SELECT に含める。

-- ❌ 悪い例（MySQLは通るが意味が不定）
SELECT user_id, MAX(created_at), title
FROM posts
GROUP BY user_id;
-- titleはどの行のものか不明

なぜ問題か？ - title がどの行のものか確定しない（DBMSによって挙動が違う）

正しい対処法：

-- ✅ 良い例（サブクエリで最新の投稿IDを絞り込む）
SELECT p.user_id, p.created_at, p.title
FROM posts p
INNER JOIN (
    SELECT user_id, MAX(created_at) AS max_date
    FROM posts
    GROUP BY user_id
) latest ON p.user_id = latest.user_id AND p.created_at = latest.max_date;

10. ランダムセレクション¶

問題のあるパターン：
ランダムな1件を取得するために ORDER BY RAND() を使う。

-- ❌ 悪い例
SELECT * FROM products ORDER BY RAND() LIMIT 1;

なぜ問題か？ - 全レコードに対してランダム値を計算してからソートするため、データ量が増えると非常に遅い

正しい対処法：

-- ✅ 良い例（最大IDを取得してランダムなIDを計算）
SELECT * FROM products
WHERE id >= (SELECT FLOOR(MAX(id) * RAND()) FROM products)
ORDER BY id
LIMIT 1;

11. プアマンズサーチエンジン（貧者の全文検索）¶

問題のあるパターン：
LIKE '%キーワード%' で全文検索をする。

-- ❌ 悪い例
SELECT * FROM articles WHERE body LIKE '%データベース%';

なぜ問題か？ - 前方一致でないため、インデックスが使われない - データ量が増えると全件スキャンになり非常に遅くなる

正しい対処法： - DBの全文検索機能を使う（MySQL の FULLTEXT、PostgreSQL の tsvector など） - または Elasticsearch などの専用の検索エンジンを導入する

-- ✅ 良い例（MySQL FULLTEXT インデックス）
ALTER TABLE articles ADD FULLTEXT INDEX ft_body(body);

SELECT * FROM articles
WHERE MATCH(body) AGAINST('データベース' IN BOOLEAN MODE);

-- ✅ 良い例（PostgreSQL tsvector）
ALTER TABLE articles ADD COLUMN body_ts tsvector
    GENERATED ALWAYS AS (to_tsvector('japanese', body)) STORED;

CREATE INDEX idx_articles_body_ts ON articles USING GIN(body_ts);

SELECT * FROM articles
WHERE body_ts @@ to_tsquery('japanese', 'データベース');

12. スパゲッティクエリ¶

問題のあるパターン：
複雑な処理を1本の巨大なSQLで解決しようとする。

-- ❌ 悪い例（1本の巨大なクエリ）
SELECT u.name, COUNT(DISTINCT o.id) AS order_count, SUM(oi.price * oi.quantity) AS total_spent,
       MAX(o.created_at) AS last_order_date, t.tag_names
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status != 'cancelled'
LEFT JOIN order_items oi ON o.id = oi.order_id
LEFT JOIN (
    SELECT ut.user_id, GROUP_CONCAT(t.name) AS tag_names
    FROM user_tags ut JOIN tags t ON ut.tag_id = t.id GROUP BY ut.user_id
) t ON u.id = t.user_id
WHERE u.created_at >= '2024-01-01'
GROUP BY u.id, u.name, t.tag_names
HAVING total_spent > 10000
ORDER BY total_spent DESC;

なぜ問題か？ - 可読性が著しく低い - デバッグが困難 - 予期しない結果（重複行の増加など）が起きやすい

正しい対処法： - クエリをステップごとに分割し、一時テーブルや CTE（WITH 句）を活用する

-- ✅ CTE（共通テーブル式）を使った例
WITH active_users AS (
    SELECT id FROM users WHERE status = 'active'
),
recent_orders AS (
    SELECT user_id, COUNT(*) AS order_count
    FROM orders
    WHERE created_at >= '2025-01-01'
    GROUP BY user_id
)
SELECT u.id, r.order_count
FROM active_users u
JOIN recent_orders r ON u.id = r.user_id;

第4部：アプリケーション開発のアンチパターン¶

13. パスワードをプレーンテキストで保存¶

問題のあるパターン：
パスワードをそのままDBに保存する。

-- ❌ 絶対にやってはいけない
INSERT INTO users (email, password) VALUES ('user@example.com', 'mypassword123');

正しい対処法： - パスワードは必ず bcrypt、scrypt、Argon2 などのハッシュ関数でハッシュ化してから保存する - MD5やSHA-1は暗号学的に安全でないので使わない

# ✅ 良い例（Python + bcrypt）
import bcrypt

# 登録時
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
cursor.execute("INSERT INTO users (email, password_hash) VALUES (%s, %s)", (email, hashed))

# ログイン時
cursor.execute("SELECT password_hash FROM users WHERE email = %s", (email,))
row = cursor.fetchone()
if bcrypt.checkpw(password.encode(), row['password_hash']):
    # 認証成功
    pass

14. SQLインジェクション¶

問題のあるパターン：
ユーザーの入力をそのままSQLに文字列連結する。

# ❌ 悪い例（Python）
query = "SELECT * FROM users WHERE name = '" + user_input + "'"

user_input に ' OR '1'='1 と入力されると全件取得される。

正しい対処法：
必ずプレースホルダー（パラメータバインド）を使う。

# ✅ 良い例
cursor.execute("SELECT * FROM users WHERE name = %s", (user_input,))

15. 擬似キー潔癖症¶

問題のあるパターン：
削除されたレコードの連番IDを再利用しようとしたり、IDに「意味」を持たせる。

-- ❌ 悪い例①（削除したIDを再利用しようとするアプリ側の処理）
-- 欠番を埋めようとして最小の未使用IDを探す
SELECT MIN(t1.id + 1) FROM orders t1
LEFT JOIN orders t2 ON t1.id + 1 = t2.id
WHERE t2.id IS NULL;

-- ❌ 悪い例②（IDに意味を持たせる：例 "2024-001"）
INSERT INTO invoices (id, amount) VALUES ('2024-001', 5000);
-- → 2025年になったらどうする？ 年またぎの採番ロジックが複雑になる

なぜ問題か？ - IDが再利用されると、古いキャッシュや外部システムが誤ったレコードを参照する

正しい対処法：
IDは単なる識別子として扱い、意味を持たせない。削除されたIDは再利用しない。

-- ✅ 良い例（AUTO_INCREMENTやUUIDを使い、IDは採番に任せる）
CREATE TABLE invoices (
    id         INT PRIMARY KEY AUTO_INCREMENT,  -- 欠番があっても気にしない
    invoice_no VARCHAR(20) UNIQUE,              -- 請求書番号は別カラムで管理
    amount     DECIMAL(10,2)
);

-- または UUID を使えば分散環境でも衝突しない
CREATE TABLE invoices (
    id         CHAR(36) PRIMARY KEY DEFAULT (UUID()),
    amount     DECIMAL(10,2)
);
