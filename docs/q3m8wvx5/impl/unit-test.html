Phase 4 — テスト設計技法

🎯 単体テスト設計技法まとめ

単体テスト（Unit Test）は、プログラムを構成する最小単位（関数やメソッドなど）が、設計通りに正しく動作するかを検証するテストです。
テストケースを場当たり的に作成すると、考慮漏れによるバグの見逃しや、無駄なテストケースの増大を招きます。以下に、効率的かつ網羅的にテストを行うための代表的な設計技法をまとめます。

🧭
このページの使い方
1) まずブラックボックス（同値分割・境界値）で「仕様どおり」を固める → 2) 次にホワイトボックス（カバレッジ）で「抜け漏れ」を点検 → 3) 最後に状態遷移・エラー推測で事故りやすい所を潰す。
テストは「数を増やす」より、失敗したときに原因が1つに絞れる粒度（Arrange-Act-Assert）を優先します。

📦 1. 入力データに着目した技法（ブラックボックス）

プログラムの内部構造を意識せず、仕様書（インプットとアウトプット）に基づいてテストケースを設計します。

1.1 同値分割法 (Equivalence Partitioning)

入力値を「同じように処理されるグループ（同値クラス）」に分け、各グループから代表値を1つ選んでテストする手法です。

📌 使用事例: 「パスワードの文字数チェック（8文字以上、16文字以下）」
無効同値（短すぎる） 有効同値（正常） 無効同値（長すぎる）
〜7文字 8〜16文字 17文字〜
代表値: 5文字 代表値: 10文字 代表値: 20文字

無数にある入力パターンのうち重複するテストを省き、最小限のケースで効率よく全体をカバーできます。

1.2 境界値分析 (Boundary Value Analysis)

バグが発生しやすい「同値クラスの境界（しきい値とその前後の値）」を狙ってテストする手法です。同値分割法とセットで使われます。

📌 使用事例: 「年齢による入場制限（18歳以上のみ入場可）」
境界となる値：18

プログラムの >= と > の書き間違い（オフバイワンエラー）などのバグを効果的に検出できます。

🔀 2. 条件と結果の組み合わせに着目した技法

2.1 デシジョンテーブルテスト

複数の入力条件の組み合わせによって出力（結果）が変化するロジックに対して、考えられる全ての組み合わせを表にしてテストする手法です。（※詳細設計のデシジョンテーブルと同様です）

2.2 ペアワイズテスト / 直交表 (Pairwise Testing)

多数のパラメータがある場合、「任意の2つのパラメータの組み合わせが必ず1回以上登場する」ようにケースを絞り込む手法です。

📌 使用事例: 「OS、ブラウザ、言語の設定による画面表示」
  • OS: Windows, Mac
  • ブラウザ: Chrome, Edge, Safari
  • 言語: 日本語, 英語
12通り
全組み合わせ(2×3×2)
➔
6通り
ペアワイズで圧縮

ソフトウェアのバグの大半は「1〜2つのパラメータの組み合わせ」で発生するという経験則に基づき、テスト回数を大幅に削減しつつ高い欠陥検出率を維持します。

🔍 3. 内部構造に着目した技法（ホワイトボックス）

3.1 制御フローテスト / パステスト

コード内の命令や分岐をどれだけ通過したか（カバレッジ：網羅率）を基準にテストを設計します。

📌 カバレッジの基準 (例: if (A > 0 || B > 0))
flowchart TD A((開始)) --> B{A > 0
||
B > 0} B -- True --> C[処理X] B -- False --> D[処理Y] C --> E((終了)) D --> E
  • C0 (命令網羅 / ステートメントカバレッジ)
    すべての「行」を少なくとも1回実行。デッドコードが無いかを検証。
  • C1 (分岐網羅 / ブランチカバレッジ) 🎯 単体テストの基本目標
    すべての「分岐(True/False)」を両方とも1回実行。
    例: Trueルート(A=1, B=0), Falseルート(A=0, B=0)
  • C2 (条件網羅 / コンディションカバレッジ)
    分岐の中の「複数条件」のTrue/False組み合わせをすべて網羅。

🧠 4. 状態の変化や経験則に着目した技法

4.1 状態遷移テスト

システムが特定の「状態」を持っており、イベントによって状態が変わる場合に、その遷移が正しく行われるかをテストします。

📌 使用事例: 「動画プレイヤー（停止・再生・一時停止）」
  • 「停止中」に「再生ボタン」 ➔ 「再生中」になるか
  • 「一時停止中」に「再生ボタン」 ➔ 「再生中」になるか
  • 「停止中」に「一時停止ボタン」 ➔ 無視されるか（エラー処理されるか）

4.2 エラー推測 (Error Guessing)

エンジニアの経験や直感に基づき、「ミスしやすそうな箇所」「過去のバグパターン」を狙い撃ちにする手法です。

📌 使用事例: 「入力フォームのテスト」

よくある狙い撃ちデータの例：

空文字列（未入力） 半角スペースのみ 全角・半角混在 超長文（1万文字など） 特殊文字 ( ' ; < > )
