🔌 APIレスポンス設計サンプル

REST APIのレスポンス構造パターンと設計方針を示すサンプルです。チーム全体で統一することが重要です。

💡 APIのレスポンス構造はチーム全体で統一する： レスポンス形式がエンドポイントごとにバラバラだと、実装者が毎回仕様書を確認しなければならずミスも増えます。 このページのパターンをチームの標準として採用し、OpenAPI(Swagger)等でドキュメント化することを推奨します。

📐基本設計方針

✅ 4つの原則
  • 1
    一貫性：成功・エラーともに必ず同じトップレベル構造 { "data": ..., "meta": ..., "error": null } を返す。フィールドの有無でレスポンス型を判断しない。
  • 2
    機械可読なエラーコード：エラー時は "code": "USER_NOT_FOUND" のような文字列コードを返す。日本語メッセージだけでは実装者がif分岐を書けない。
  • 3
    日時はISO 8601："2026-05-20T12:00:00.000Z" 形式で返す。Unixタイムスタンプは人間が読めないため避ける（ログ・内部用途は除く）。
  • 4
    nullと省略の使い分け：フィールドが「値が存在するが空」→ null を返す。「そのフィールド自体が関係ない」→ フィールドを省略する（undefinedとして扱われる）。

📦レスポンスパターン集

タブを切り替えて各パターンのJSONサンプルを確認してください。

GET /api/v1/users/123 200 OK 
{
  "data": {

    "role": "editor",

    "updatedAt": "2026-05-20T09:00:00.000Z"
  },
  "meta": null,              // 単一リソースはmetaなし
  "error": null              // 成功時は常にnull
}
GET /api/v1/users?page=2&limit=20&sort=createdAt&order=desc 200 OK 
{
  "data": [
    { "id": "121", "name": "鈴木 花子", "email": "suzuki@example.com", "role": "viewer" },

  ],

POST /api/v1/users 201 Created

作成成功時は 201 Created を返し、作成されたリソース全体（サーバーが付与したID・日時を含む）をレスポンスボディに含めます。 クライアントが再度GETしなくてもリソースを取得できます。

リクエストボディ（POST body）

{
  "name": "佐藤 三郎",
  "email": "sato@example.com",
  "role": "viewer"
}

レスポンスボディ

{
  "data": {
    "id": "201",              // サーバーが採番したID
    "name": "佐藤 三郎",
    "email": "sato@example.com",
    "role": "viewer",
    "avatarUrl": null,
    "createdAt": "2026-05-20T12:34:56.789Z",
    "updatedAt": "2026-05-20T12:34:56.789Z"
  },
  "meta": null,
  "error": null
}
PUT /api/v1/users/123 200 OK

PUT（全体更新）・PATCH（部分更新）ともに、更新後のリソース全体を返します。 更新されたフィールドだけでなくリソース全体を返すことで、クライアントの状態管理が簡単になります。 

{
  "data": {
    "id": "123",
    "name": "山田 太郎（更新済み）",
    "email": "yamada-new@example.com",
    "role": "admin",
    "avatarUrl": "https://cdn.example.com/avatars/123.jpg",
    "createdAt": "2026-01-15T03:21:00.000Z",
    "updatedAt": "2026-05-20T15:00:00.000Z"  // 更新日時が変わる
  },
  "meta": null,
  "error": null
}
DELETE /api/v1/users/123 204 No Content
204 No Content について： 削除成功時はHTTPステータス 204 No Content を返し、レスポンスボディは空にします（JSONも返しません）。 クライアント側は response.status === 204 で判定します。

レスポンスボディ: なし（空）

// HTTP/1.1 204 No Content
// Content-Length: 0
// （ボディなし）
⚠️ 論理削除 vs 物理削除： 業務システムでは「削除フラグを立てる」論理削除が一般的です。論理削除の場合も204を返します。 削除後にリソースを返す設計（200 + 削除されたリソース）はチームで統一してください。
POST /api/v1/users/batch 200 OK

複数リソースの一括操作では、一部が成功・一部が失敗する「部分成功」が発生します。 この場合は 200 OK を返し、succeeded/failedの2配列で結果を返します。 

{
  "data": {
    "succeeded": [
      { "id": "101", "name": "田中 一郎" },
      { "id": "102", "name": "伊藤 二郎" }
    ],
    "failed": [
      {
        "id": "103",              // 失敗したリソースのID
        "code": "USER_NOT_FOUND",    // 機械可読なエラーコード
        "message": "ユーザーが見つかりません"
      }
    ]
  },
  "meta": {
    "total": 3,
    "succeededCount": 2,
    "failedCount": 1
  },
  "error": null
}

エラー時は data: null にし、error オブジェクトにコード・メッセージ・詳細を含めます。

🔢データ型の表現

型推奨形式NGパターン理由
日時 "2026-05-20T12:00:00.000Z"
ISO 8601 UTC文字列		 1748001234 (Unixタイムスタンプ) ISO形式は人間が読める。タイムゾーン情報も含まれる。Unixタイムスタンプはデバッグが困難
金額 12800 (整数・円単位) "¥12,800" (文字列・フォーマット込み) 文字列にすると計算・比較ができない。表示フォーマットはフロントで行う。小数点以下は避ける（浮動小数点誤差）
真偽値 true / false 1 / 0 または "1" / "0" JSONにはboolean型がある。数値や文字列で代替するとif文の条件が曖昧になる
列挙値（enum） "active" / "inactive" (文字列) 1 / 2 (数値) 数値は意味が分からない。文字列ならAPIレスポンスを見るだけで値の意味が理解できる
ID "123" (文字列) 123 (数値) JavaScriptのNumber型は2^53-1を超えるとUUID・DB連番で精度が失われる。文字列なら安全
null vs 省略 値が存在するが未設定 → null フィールドが存在しないことをnullで表現 「フィールドがないこと」と「フィールドが空であること」は意味が違う。フィールド自体が関係ない場合は省略（undefined）

🏷️バージョニング

💡 APIバージョニングの2つのアプローチ：

方法1 - URLパスに含める（推奨・シンプル）：
GET /api/v1/users → GET /api/v2/users
ブラウザのアドレスバー・curl・ログで一目でバージョンが分かる。最も一般的な方式。

方法2 - リクエストヘッダーで指定：
Accept: application/vnd.myapi.v2+json
URLをクリーンに保てるが、ヘッダーを意識する必要があり複雑。GraphQLなどで使われることがある。

チームではどちらかに統一すること。混在は最も悪い。

❌よくある設計ミス

NGパターン何が問題か正しい設計
エラー時も 200 OK を返す HTTPクライアントのエラーハンドリングが機能しない。監視ツールがエラーを検知できない 適切なHTTPステータスコード（4xx/5xx）を返す
エラーメッセージが日本語のみ if (error.message === 'ユーザーが見つかりません') という条件分岐は壊れやすい error.code に英語の定数（"USER_NOT_FOUND"）を必ず含める
成功・失敗でレスポンス構造が異なる 実装者が毎回「どっちのパターンか」を意識しなければならない 常に {data, meta, error} のトップレベル構造を維持する
コレクション取得で総件数を返さない フロントエンドでページネーション UIを実装できない meta.total に全件数を必ず含める
DELETEで削除済みリソースを返す / 返さないが統一されていない 実装者が混乱し、フロントで不要な処理が発生する DELETEは 204 No Content に統一する（ボディなし）
500エラーにスタックトレースを含める 内部実装・ライブラリバージョン・ファイルパスが露出し、攻撃者に情報を与える 本番では汎用メッセージのみ。requestId でサーバーログと紐付けられるようにする
