📋 ログ出力画面サンプル

ログレベルの使い分け・構造化ログの設計・良いログと悪いログの違いを具体例で学びます。適切なログは障害対応時間を大幅に短縮します。

⚠️
ログ設計の鉄則
「エラーが発生しました」では不十分です。障害時に役立つログには 誰が（userId/ip）・何を（対象リソース）・なぜ（エラーコード/原因）・どうなったか（結果/影響範囲） を含める必要があります。また、構造化ログ（JSON形式）にすることで、ログ集計ツールによる自動分析が可能になります。

1ログレベルの使い分け

ログレベルを正しく使い分けることで、本番環境でのノイズを減らし、重大な問題をすぐに発見できます。

レベル 用途 本番出力 具体例
DEBUG 開発・デバッグ用の詳細情報 ❌ 通常は無効 SQLクエリの内容、変数の中間値、処理の詳細フロー
INFO 正常な業務処理の記録 ✅ 出力する ログイン成功、注文完了、バッチ開始・終了
WARN 注意が必要だが処理は継続できる状態 ✅ 必ず出力 コネクションプール残数低下、外部API応答遅延、リトライ発生
ERROR 処理が失敗した。要対応 ✅ アラート連携 外部APIタイムアウト、バリデーションエラー、在庫不足
FATAL システム継続不能な致命的障害 ✅ 即時通報 DB接続不能、設定ファイル読み込み失敗、OOM
SECURITY セキュリティ関連イベント ✅ 監査用に永続保存 認証失敗、権限違反、アカウントロック、不審なアクセス

2ログエントリの理想的な構造（構造化ログ）

テキスト形式ではなく JSON形式 で出力することで、Elasticsearch・Datadog・CloudWatch Logs Insightsなどのツールで集計・検索が容易になります。

{
  "timestamp": "2026-05-20T14:32:01.523Z",  // ISO 8601形式。タイムゾーンはUTCで統一
  "level": "ERROR",                           // DEBUG / INFO / WARN / ERROR / FATAL / SECURITY
  "service": "order-service",                 // マイクロサービス名またはアプリ名
  "traceId": "abc123def456",                  // 分散トレーシング用ID（リクエスト全体を追跡）
  "spanId": "789xyz",                         // スパンID（個々の処理単位）
  "userId": "user_9821",                      // 操作者。匿名時は "anonymous"
  "sessionId": "sess_ab12cd34",               // セッション識別子
  "message": "在庫不足により注文処理に失敗しました",  // 人間が読む主メッセージ
  "context": {                                // 問題を再現・調査するための付加情報
    "orderId": "ord_20260520_001",
    "itemId": "item_456",
    "requiredStock": 5,
    "currentStock": 2
  },
  "error": {                                  // エラー時のみ。スタックトレースは開発環境のみ
    "code": "INSUFFICIENT_STOCK",
    "class": "InsufficientStockException"
  },
  "duration_ms": 42                           // 処理時間（パフォーマンス監視に活用）
}

4悪いログ vs 良いログ 対比

悪いログは障害時に「何も分からない」。良いログは「原因と影響範囲がすぐに分かる」。

シナリオ ❌ 悪い例 ✅ 良い例
ログイン失敗 ログインに失敗しました ログイン失敗 userId="user_42" ip="203.0.113.5" reason="PASSWORD_MISMATCH" attemptCount=2
DB エラー データベースエラーが発生しました DB クエリ失敗 table="orders" operation="INSERT" errorCode="23505" constraint="uk_order_no" duration_ms=12
外部API失敗 API呼び出しエラー 外部API失敗 url="https://api.payment.example/charge" status=503 retryCount=3 traceId="trace_abc" orderId="ord_001"
バッチ処理 バッチ処理が完了しました バッチ完了 batchId="batch_daily_01" processed=5000 skipped=12 failed=0 duration_ms=45230
権限エラー 権限がありません 権限不足アクセス拒否 userId="user_99" role="VIEWER" resource="/admin/users" requiredRole="ADMIN" ip="10.0.0.5"

5ログフィルタリングの考え方

🎯 本番環境のフィルタリング設定
✅
推奨設定
通常時: INFO 以上を出力
調査時: DEBUG を一時的に有効化
SECURITY は全件永続保存（監査要件）

DEBUGを本番で常時有効にすると、SQLクエリやパラメータが大量出力されてディスクを圧迫し、機密情報漏洩リスクもあります。

🔍 ログ集約ツールでの活用
  • 構造化ログ(JSON) → Elasticsearch でフィールド検索
  • traceId でリクエスト全体のログを串刺し検索
  • ERROR 件数をメトリクスとして Grafana で監視
  • WARN 急増をアラートのトリガーにする
  • duration_ms を集計してスロークエリを検出
💡
個人情報・機密情報はログに出力しない
パスワード・クレジットカード番号・マイナンバーなどの機密情報はログに絶対に含めてはいけません。メールアドレスも場合によってはマスキングが必要です（例: use***@example.com）。ログは多くのシステム担当者が参照できるため、機密情報の漏洩経路になります。
