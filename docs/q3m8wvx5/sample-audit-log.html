📜 監査ログ・操作履歴

誰が・いつ・何を・どう変えたか — トレーサビリティを確保する設計パターン

📌 監査ログとは何か
監査ログは「いつでもトレースできる状態」を保つためのもの。何か問題が起きたとき「誰が・いつ・何を・なぜ変えたか」を後から辿れることが目的。 法的要件（個人情報保護法・GDPR・SOX法など）や内部統制（J-SOX）で必須になる場合が多い。設計の早い段階で組み込む必要がある。

📋1. 記録すべき操作の考え方

全操作を記録するとデータ量が膨大になる。重要度に応じて記録レベルを決める。

操作種別記録必須記録推奨記録任意
LOGIN 認証操作 ログイン成功・失敗、ログアウト、パスワード変更 セッションタイムアウト、MFA認証 ページ遷移
CREATE 作成 マスターデータ・注文・ユーザー作成 ドラフト保存、テンプレート作成 一時データ作成
UPDATE 更新 個人情報・金額・ステータス変更、承認操作 メモ・コメント変更、設定変更 表示設定・ソート順変更
DELETE 削除 ほぼ全ての削除操作（論理・物理どちらも） 一時ファイル削除 キャッシュクリア
EXPORT エクスポート 個人情報・財務データのダウンロード、CSV出力 レポート出力 画面印刷
VIEW 参照 機密情報・個人情報の参照（金融・医療系） 管理画面へのアクセス 一般ページ閲覧

🏗️2. 監査ログの構造（理想形）

1件の監査ログに含めるべきフィールドを定義する。後から「あの情報があれば…」とならないよう最初から設計する。

{
  "audit_id":   "aud_20260520_001823",      // 監査ログの一意ID
  "timestamp":  "2026-05-20T14:23:11+09:00", // ISO 8601 タイムゾーン付き
  "actor": {
    "user_id":   "usr_00042",               // 操作者のID（削除されても残す）
    "user_name": "田中 太郎",               // 操作時点の表示名（変更前の名前でも記録）
    "role":      "admin",                   // 操作時点のロール
    "ip_address": "203.0.113.45",           // クライアントIP
    "user_agent": "Mozilla/5.0 ..."         // ブラウザ情報
  },
  "action": {
    "type":     "UPDATE",                   // CREATE / UPDATE / DELETE / VIEW / EXPORT / LOGIN
    "resource": "User",                     // 操作対象のリソース種別
    "resource_id": "usr_00108",             // 操作対象のID
    "description": "ユーザー情報を更新"     // 人が読める説明
  },
  "changes": {                              // UPDATEの場合: 変更前後の値
    "email": {
      "before": "old@example.com",
      "after":  "new@example.com"
    },
    "phone": {
      "before": null,
      "after":  "090-1234-5678"
    }
  },
  "request": {
    "method": "PUT",
    "path":   "/api/v1/users/usr_00108",
    "trace_id": "trace_abc123"              // 分散トレース用ID（ログとの紐付け）
  },
  "prev_hash": "sha256:a3f9c2...",          // 前のレコードのハッシュ（改ざん検知）
  "hash":      "sha256:b7e1d4..."           // このレコード全体のハッシュ
}

🔍4. 変更差分表示デモ

UPDATE操作の変更前後を視覚的に比較する。緑=追加・赤=削除・黄=変更のカラーコーディングで一目で把握できる。

ユーザー情報変更 — usr_00108（山田 花子） · UPDATE 2026-05-20 14:23:11 · 田中 太郎
フィールド 変更前 変更後 変更種別
email yamada_old@example.com yamada@example.com 🔄変更
phone （未設定） 090-2345-6789 ➕追加
department 総務部 人事部 🔄変更
manager_id usr_00003 （削除） ➖削除
name 山田 花子 山田 花子 変更なし

※ 変更なしのフィールドは通常は非表示にして変更があったものだけ表示するとログが読みやすくなる。

🔐5. 保存期間と改ざん防止

⚠️ 設計時に決めておくべき考慮点
  • 保存期間：業界・法律によって異なる（医療系は10年、金融系は7年など）。削除ポリシーも設計段階で決める。
  • RDB vs 追記専用DB：通常のDBはUPDATE/DELETEができてしまう。PostgreSQLのRow-level TTL、Amazon Timestream、Apache Iceberg等の追記専用構造を検討する。
  • ハッシュチェーン：各レコードに「前のレコードのハッシュ」を含めることで改ざん検知が可能。ブロックチェーンと同じ原理。
  • アクセス制限：監査ログ自体へのアクセスも監査する。無限ループに見えるが、監査ログDBへのアクセスログはOSレベルで記録する。
  • 検索性：user_id・timestamp・resource_idにはインデックスを必ず貼る。数百万件になっても検索できる設計が必要。

ハッシュチェーンのイメージ（改ざんするとチェーンが切れる）：

レコード #001 | prev_hash: genesis（なし） | hash: sha256:a3f9c2d1...
レコード #002 | prev_hash: sha256:a3f9c2d1... ← #001のhash | hash: sha256:b7e1d4f8...
レコード #003 | prev_hash: sha256:b7e1d4f8... ← #002のhash | hash: sha256:c9a2e5b3...
↑ #002を書き換えると hashが変わる → #003のprev_hashと一致しない → 改ざん検知！
