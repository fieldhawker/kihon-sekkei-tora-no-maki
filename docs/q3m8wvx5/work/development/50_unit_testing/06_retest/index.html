STEP 6: 修正・再テスト・デグレード確認¶

全体フロー: STEP 1 → STEP 2 → STEP 3 → STEP 4 → STEP 5 → STEP 6 → STEP 7

このSTEPの目的¶

不具合を修正したあと、修正が正しく反映されたかと他の機能が壊れていないかを確認します。

修正は「直った」で終わりではありません。
修正によって別の部分が壊れる（デグレード）ことがあるため、必ず再テストと周辺確認を行います。

修正・再テストの流れ¶

[テスト担当]          [開発担当]
不具合票を提出  →     原因を調査・特定
                      ↓
                      コードを修正
                      ↓
                      テスト環境にデプロイ
                      ↓
修正内容の確認  ←     「修正しました」と連絡
  ↓
再テスト実施
  ↓
OK → STEP 7へ
NG → 再度不具合票を作成して報告

再テストの手順¶

1. 修正内容を確認する¶

開発担当者から「修正しました」と連絡を受けたら、何が修正されたかを確認します。

確認すること: - どの不具合（BUG-ID）が修正されたか - どのような修正を行ったか（修正内容の概要） - テスト環境への反映が完了しているか

テスト環境への反映が完了していない状態で再テストを始めても意味がありません。
デプロイ完了をリーダーまたは開発担当者に確認してから再テストを開始してください。

2. 修正に対応するテストケースを再実施する¶

不具合が修正されたことを確認するため、NGになったテストケースを再度実施します。

再テストの対象
NGになったテストケースすべて
今回の修正に関連する処理のテストケース（リーダーに確認）

3. デグレードがないか確認する¶

デグレード（デグレ）とは?
修正・変更によって、以前は正常に動いていた機能が壊れてしまうことです。

コードを修正すると、意図しない箇所に影響が出ることがあります。
「直した箇所」だけでなく、関連する機能も再テストして確認します。

デグレード確認の範囲:

確認範囲 説明
修正箇所そのもの NGからOKになったか
修正箇所に関連する処理 同じ画面・処理を使っている他の機能が壊れていないか
影響範囲が広い場合 リーダーに確認して、追加で確認すべき範囲を決める

具体例:
ログイン機能のパスワード照合ロジックを修正した場合:

確認すること
ログイン（修正箇所）がOKになったか
パスワード変更機能が壊れていないか
ログアウト後の再ログインが正常に動くか

4. 再テスト結果を記録する¶

再テストの結果も、通常のテストと同様に記録します。

記録項目 内容
再テスト日時 再テストを実施した日時
再テスト結果 OK / NG
不具合票のステータス OK → 「修正済み」に更新 / NG → 「再発」として記録

再テスト結果の記録例¶

BUG-ID TC-ID 再テスト日時 結果 備考
BUG-001 TC-002 2026-05-26 10:00 OK 修正済みに更新
BUG-002 TC-007 2026-05-26 10:30 NG 修正後も同様に発生。BUG-003として再登録

再テストでまたNGになった場合¶

「修正したはずなのにまたNGになった」という場合は、慌てず以下を確認します。

  1. テスト環境に修正が反映されているか を確認する（デプロイ済みか）
  2. 前提条件・テストデータが正しいか を確認する
  3. それでもNGなら、新しい不具合票を作成してリーダーに報告する

前の不具合票を上書きしたり修正したりせず、必ず新しい不具合票を作成してください。
変更の履歴を残すことが大切です。

デグレードを防ぐための心がけ¶

修正担当者（開発担当）向けのポイントも参考にしてください。

ポイント 説明
修正範囲を最小限にする 不具合に直接関係ない箇所を触らない
修正内容を明確にする 何をどう直したかをメモ・コミットコメントに残す
関連処理を自分でも確認する テスト担当者に渡す前に、開発者本人も動作確認する

このSTEPの完了チェック¶

  • 修正がテスト環境に反映されていることを確認した
  • NGになったテストケースをすべて再テストした
  • 再テスト結果をテストケース一覧に記録した
  • 不具合票のステータスを更新した（修正済み / 再発 など）
  • 関連する機能のデグレードがないことを確認した
  • 再テストの結果をリーダーに報告した

【成果物例】社員管理システム — 再テスト結果記録¶

修正連絡の受領¶

BUG-ID 修正連絡日時 担当者 修正内容概要 テスト環境反映
BUG-001 2026-06-10 13:00 開発 次郎 社員登録処理にDBの一意制約チェックを追加。登録前に社員番号の重複確認クエリを実行するよう修正 v0.3.1（2026-06-10 13:30 デプロイ確認済み）
BUG-002 2026-06-10 13:00 開発 次郎 入社日フィールドにサーバーサイドのフォーマットバリデーションを追加（YYYY-MM-DD 形式以外はエラー） 同上

再テスト結果¶

BUG-ID TC-ID 再テスト日時 再テスト実施者 結果 備考
BUG-001 TC-003 2026-06-10 14:05 田中 OK 「この社員番号はすでに登録されています」と正しくエラー表示されることを確認
BUG-002 TC-010 2026-06-10 14:20 田中 OK 「正しい日付形式（YYYY-MM-DD）で入力してください」と正しくエラー表示されることを確認

デグレード確認¶

BUG-001の修正（登録前の重複チェック追加）は社員登録処理全体に影響する可能性があるため、以下を追加確認。

確認対象 確認内容 結果
TC-001（正常系登録） 新規社員番号での登録が引き続き正常に動作するか OK
社員情報更新機能（TC-020） 更新処理に影響がないか（社員番号変更なし） OK

BUG-002の修正（入社日バリデーション追加）について追加確認。

確認対象 確認内容 結果
TC-001（正常系登録） YYYY-MM-DD 形式の入力で引き続き正常登録できるか OK
社員情報更新機能（TC-021） 入社日の更新処理に影響がないか OK
