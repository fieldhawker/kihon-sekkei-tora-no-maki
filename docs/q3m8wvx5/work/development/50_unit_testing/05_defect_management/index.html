STEP 5: 不具合の記録・報告¶

全体フロー: STEP 1 → STEP 2 → STEP 3 → STEP 4 → STEP 5 → STEP 6 → STEP 7

このSTEPの目的¶

NGになった内容を整理し、担当者・リーダーに報告します。

不具合の記録は「後から誰でも再現・調査できる情報を残す」ことが目的です。
記録が曖昧だと、修正担当者が再現できずに調査に時間がかかり、プロジェクト全体が遅延します。

鉄則: 不具合を見つけたらすぐ記録する。記憶に頼ると詳細を忘れます。

不具合票の項目¶

項目 説明 記入例
不具合ID 管理用の番号 BUG-001
対応テストケースID どのテストケースで発見したか TC-002
発生日時 いつ発生したか 2026-05-25 14:30
発生環境 どの環境で発生したか テスト環境 / Chrome 124.0 / Windows 11
テスト対象 どの機能・処理で発生したか ログイン機能
重要度 どのくらい深刻か（後述） 高 / 中 / 低
発生手順 どの操作をしたらNGになったか（番号付きで） 1. ログイン画面を開く 2. …
期待した結果 本来どうなるべきだったか エラーメッセージが表示される
実際の結果 実際に何が起きたか エラーメッセージなしにホーム画面に遷移した
証跡 スクリーンショット・ログのファイル名 TC-002_NG_screenshot.png
修正担当者 修正を依頼する開発者 山田（開発担当）
ステータス 不具合の現在の状態 未対応 / 修正中 / 修正済み / 再テスト待ち

不具合の重要度の判断基準¶

不具合を発見したら、重要度を判断して記録します。
判断に迷った場合は、リーダーに確認してください。

重要度 基準 例
致命的 システムが起動しない・クラッシュする・データが壊れる ログイン画面が表示されない、DBが壊れる
高（Critical） 主要な機能が使えなくなる ログインできない、登録ボタンが動作しない
中（Major） 機能は動くが一部に問題がある・業務への影響が限定的 特定の入力値でエラーになる、メール通知が遅延
低（Minor） 軽微な表示ずれ・操作に支障はない テキストのズレ、誤字

補足: 結合テスト・運用テストフェーズでも同じ4段階を使います。「致命的」と「高」の判断に迷ったら、業務が完全に止まるかどうかで判断してください。止まるなら「致命的」、機能単位の問題なら「高」です。

不具合票の記入例¶

不具合ID    : BUG-001
TC-ID       : TC-002
発生日時    : 2026-05-25 14:30
発生環境    : テスト環境 / Chrome 124.0 / Windows 11
テスト対象  : ログイン機能
重要度      : 高

【発生手順】
1. ブラウザで http://test.example.com/login を開く
2. IDフィールドに「testuser」を入力する
3. パスワードフィールドに「wrongpass」（誤ったPW）を入力する
4. 「ログイン」ボタンをクリックする

【期待した結果】
「IDまたはパスワードが違います」というエラーメッセージが表示され、
ログイン画面に留まる。

【実際の結果】
エラーメッセージが表示されずに、ホーム画面（/home）に遷移した。
ログイン状態になっている（testuser として認識されている）。

【証跡】
- TC-002_NG_screenshot.png（ホーム画面に遷移した状態のスクリーンショット）
- TC-002_NG_console.log（ブラウザコンソールのログ）

修正担当者 : 山田
ステータス : 未対応

不具合の報告の仕方¶

誰に報告するか¶

  • まずリーダーに報告します
  • リーダーが開発担当者に修正を依頼します
  • プロジェクトによっては、不具合管理ツール（Redmine・Jira・BacklogなどのIssue管理）に登録します

報告のタイミング¶

タイミング 内容
不具合を発見したとき 不具合票に記録してリーダーへ口頭・チャット報告
その日のテスト終了時 その日のNGと不具合票の一覧を報告
緊急度が高い不具合 発見したらすぐに口頭で報告（重要度「高」の場合）

報告メッセージの例¶

リーダー、不具合を1件発見しました。報告します。

- 不具合ID: BUG-001
- テスト対象: ログイン機能（TC-002）
- 重要度: 高
- 内容: 誤ったパスワードでログインしても、エラーが出ずにホーム画面に遷移します
- 詳細は不具合票をご確認ください

やってはいけないこと¶

自己判断で「たぶん大丈夫」として不具合を流す¶

「これは自分の操作ミスかも」「些細なことだから報告しなくていいかな」という自己判断は危険です。
記録・報告するかどうかの判断はリーダーが行います。 迷ったら報告してください。

不具合を後でまとめて記録しようとする¶

テストを何件か実施してからまとめて記録しようとすると、手順や実際の結果の詳細を忘れます。
不具合はその場で記録するのが鉄則です。

修正まで自分で対応しようとする¶

テスト担当と開発担当が分かれているプロジェクトでは、テスト担当者がコードを直すのはルール違反になる場合があります。
修正は必ず開発担当者に依頼してください。

このSTEPの完了チェック¶

  • NGになったテストケースすべてについて不具合票を作成した
  • 発生手順・期待した結果・実際の結果を具体的に記録した
  • スクリーンショット・ログを保存して不具合票に添付した
  • 不具合票をリーダーに提出・報告した
  • 不具合管理ツールへの登録が必要な場合は登録した

【成果物例】社員管理システム — 不具合票¶

BUG-001¶

不具合ID    : BUG-001
TC-ID       : TC-003
発生日時    : 2026-06-05 10:42
発生環境    : テスト環境（http://emp-test.example.com）/ Chrome 124.0 / Windows 11
テスト対象  : 社員登録機能 — 社員番号重複チェック
重要度      : 高

【発生手順】
1. ブラウザで http://emp-test.example.com/employees/new を開く
2. 社員番号フィールドに「00000001」（既存の社員番号）を入力する
3. 氏名フィールドに「テスト 太郎」を入力する
4. 部署プルダウンから「営業部」を選択する
5. 入社日フィールドに「2026-04-01」を入力する
6. メールアドレスフィールドに「test.taro@example.com」を入力する
7. 「登録」ボタンをクリックする

【期待した結果】
「この社員番号はすでに登録されています」というエラーメッセージが表示され、
登録処理が行われない。

【実際の結果】
エラーメッセージが表示されず「登録が完了しました」と表示された。
社員一覧画面に遷移すると、社員番号「00000001」が2件表示されている。
DBを確認したところ employees テーブルに同じ社員番号のレコードが2件存在する。

【証跡】
- TC-003_NG_screenshot.png（登録完了画面のスクリーンショット）
- TC-003_NG_db_dump.txt（employees テーブルの重複レコード確認結果）

修正担当者 : 開発 次郎
ステータス : 未対応

BUG-002¶

不具合ID    : BUG-002
TC-ID       : TC-010
発生日時    : 2026-06-06 14:15
発生環境    : テスト環境（http://emp-test.example.com）/ Chrome 124.0 / Windows 11
テスト対象  : 社員登録機能 — 入社日フォーマットバリデーション
重要度      : 中

【発生手順】
1. ブラウザで http://emp-test.example.com/employees/new を開く
2. 社員番号フィールドに「00000099」を入力する
3. 氏名フィールドに「テスト 次郎」を入力する
4. 部署プルダウンから「営業部」を選択する
5. 入社日フィールドに「20260401」（YYYY-MM-DD 形式でなく YYYYMMDD 形式）を入力する
6. メールアドレスフィールドに「test.jiro@example.com」を入力する
7. 「登録」ボタンをクリックする

【期待した結果】
「正しい日付形式（YYYY-MM-DD）で入力してください」とエラーメッセージが表示され、
登録処理が行われない。

【実際の結果】
エラーメッセージが表示されず「登録が完了しました」と表示された。
DBの入社日フィールドに「20260401」という文字列がそのまま格納されている。

【証跡】
- TC-010_NG_screenshot.png（登録完了画面のスクリーンショット）
- TC-010_NG_db_record.txt（DBの入社日フィールド確認結果）

修正担当者 : 開発 次郎
ステータス : 未対応

リーダーへの報告メッセージ（例）¶

山田リーダー、不具合を2件発見しましたので報告します。

- BUG-001: 社員登録 — 社員番号の重複チェックが機能していない（重要度: 高）
  → 同じ社員番号で2件登録できてしまいます。

- BUG-002: 社員登録 — 入社日のフォーマットバリデーションが機能していない（重要度: 中）
  → YYYYMMDD 形式のまま登録されてしまいます。

詳細は不具合票（BUG-001, BUG-002）をご確認ください。
BUG-001 は重要度「高」のため、優先対応をお願いしたいです。
