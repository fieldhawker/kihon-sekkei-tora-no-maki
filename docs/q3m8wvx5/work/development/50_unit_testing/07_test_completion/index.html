STEP 7: 完了確認・成果物の提出¶

全体フロー: STEP 1 → STEP 2 → STEP 3 → STEP 4 → STEP 5 → STEP 6 → STEP 7

このSTEPの目的¶

すべてのテストケースがOKになったことを確認し、成果物をまとめてリーダーに提出します。

「全部OKにした」ではなく、「テストケース通りに確認してOKだった」という記録が大切です。
次の工程（結合テスト）の担当者が見ることを意識して、丁寧にまとめてください。

完了条件の確認¶

以下の項目をすべて確認してから「完了」を報告します。

確認項目 確認内容 確認結果
テストケースの網羅 全テストケースが実施済みか OK / NG
合否の記録 全件にOK/NGが記録されているか OK / NG
未解決不具合なし NGのまま放置されているテストケースがないか OK / NG
不具合票の整備 発生した不具合票が全件記録されているか OK / NG
証跡の添付 スクリーンショット・ログが揃っているか OK / NG

すべて「OK」になって初めて、テスト完了をリーダーに報告します。

提出する成果物¶

# 成果物 内容 対応STEP
1 テストケース一覧（結果記入済み） 全テストケースにOK/NGと実施日・実施者が記入されたもの STEP 2 / STEP 4
2 不具合票 発生した不具合の一覧（修正済みステータスを含む） STEP 5
3 テスト証跡 NGが発生したときのスクリーンショット・ログ STEP 4 / STEP 5
4 再テスト結果 修正後の再テスト結果の記録 STEP 6

成果物の整理の仕方¶

フォルダ構成の例¶

単体テスト_成果物/
├── 01_テストケース一覧.xlsx      ← 全テストケース・結果記入済み
├── 02_不具合票.xlsx              ← 発生した不具合一覧
└── 03_証跡/
    ├── TC-002_NG_screenshot.png
    ├── TC-002_NG_console.log
    └── TC-007_NG_screenshot.png

テストケース一覧の最終確認¶

提出前に以下をチェックしてください。

  • すべてのテストケースに実施日・実施者が記入されている
  • 結果（OK/NG）が全件に記入されている
  • NGになったものはすべて再テスト済みでOKになっている
  • NGのまま残っているものがある場合は、リーダーに確認している

不具合票の最終確認¶

  • 発生したすべての不具合が不具合票に記録されている
  • 全件のステータスが「修正済み」または「対応不要（リーダー承認済み）」になっている
  • 証跡ファイルが不具合票に正しく添付・参照されている

完了報告の仕方¶

リーダーへの報告メッセージ例¶

単体テストが完了しましたので報告します。

【実施期間】 2026-05-25 〜 2026-05-27
【テストケース数】 全30件
  - OK: 28件
  - NG（修正済み・再テストOK）: 2件
  - 未解決: 0件

【発生した不具合】 2件（BUG-001、BUG-002）いずれも修正・再テストOK

【成果物】共有フォルダ「単体テスト_成果物」に格納しました。

ご確認のほどよろしくお願いします。

完了できていない状態で「完了」と報告しない¶

プレッシャーを感じて「完了しました」と報告したくなることがあるかもしれません。
しかし、未完了のまま次の工程（結合テスト）に進むと、後で大きな問題になります。

正直に現状を報告することが、プロジェクト全体を助けます。

未完了の場合の報告例: 「TC-015〜TC-020が未実施です。現在の進捗は24/30件（80%）です。
明日中に完了予定ですが、スケジュールに影響が出る場合はご相談します。」

テスト完了後にやること¶

テストが完了したら、自分の作業を振り返りましょう。

振り返り項目 確認内容
抜け漏れがなかったか 見落としたテストケースや観点はなかったか
記録は十分だったか 後から見直したときに情報が足りない箇所はなかったか
次回に活かせることはあるか より効率的にできた手順はあったか

振り返りを次のテストに活かすことで、テストの品質と効率が向上します。

このSTEPの完了チェック¶

  • 全テストケースの実施・結果記録が完了した
  • 未解決の不具合がないことを確認した
  • 成果物（テストケース一覧・不具合票・証跡）を整理した
  • リーダーに完了報告をした
  • 成果物を指定の場所に提出した

お疲れ様でした！単体テスト完了です。

次の工程 → 結合テスト（テストケース一覧・不具合票を引き継ぎます）

【成果物例】社員管理システム — テスト完了報告¶

完了条件チェックシート¶

確認項目 確認内容 確認結果
テストケースの網羅 全テストケース（13件）が実施済みか OK
合否の記録 全件にOK/NGが記録されているか OK
未解決不具合なし NGのまま放置されているテストケースがないか OK（BUG-001, BUG-002 いずれも修正済み）
不具合票の整備 発生した不具合票（2件）が全件記録されているか OK
証跡の添付 スクリーンショット・ログが揃っているか OK（下記フォルダ参照）

成果物一覧¶

├── 02_不具合票.xlsx                  ← BUG-001, BUG-002（いずれも修正済み）
└── 03_証跡/
    ├── TC-003_NG_screenshot.png       ← BUG-001 発生時の画面
    ├── TC-003_NG_db_dump.txt          ← BUG-001 DB重複確認結果
    ├── TC-010_NG_screenshot.png       ← BUG-002 発生時の画面
    └── TC-010_NG_db_record.txt        ← BUG-002 DB日付フィールド確認結果

リーダーへの完了報告メッセージ（例）¶

山田リーダー、単体テストが完了しましたので報告します。

【実施期間】 2026-06-05 〜 2026-06-10
【テスト対象】 社員管理システム — 社員登録・検索・更新・削除機能
【テストケース数】 全13件
  - OK（初回）: 11件
  - NG（不具合あり）: 2件 → 修正・再テスト後、いずれもOKに解決
  - 未解決: 0件

【発生した不具合】 2件
  - BUG-001: 社員番号重複チェック未動作（重要度: 高）→ 修正・再テストOK
  - BUG-002: 入社日フォーマットバリデーション未動作（重要度: 中）→ 修正・再テストOK

【成果物】
  共有フォルダ「単体テスト_成果物_社員管理システム」に格納しました。

ご確認のほどよろしくお願いします。
田中 一郎

テスト振り返りメモ¶

振り返り項目 内容
抜け漏れがなかったか 社員番号重複チェック（BUG-001）は異常系として事前にテストケースに含めていたため早期に発見できた
記録は十分だったか DB状態を証跡として残したことで開発担当者が迅速に原因を特定できた
次回に活かせること 入力バリデーション系は「サーバーサイドでも検証しているか」を意識したテストケースを早めに作ると効果的
