基本設計 設計書レビューチェックリスト¶

このチェックリストの使い方¶

  1. 設計書を作成したら、レビュー依頼の前に自分でこのリストを確認する
  2. 各項目を確認し、OKなら [x]、該当しない場合は [N/A] と記入する
  3. 全項目が [x] または [N/A] になったらレビュー依頼を出してよい
  4. 「なぜ？」の補足がある項目は、意図を理解したうえで確認すること

新人向けメモ: 基本設計は「何を作るか」を決める工程。詳細設計（どう作るか）の前提となるため、抜け漏れがあると後工程で手戻りが発生する。

共通チェック項目¶

  • ファイル名・ドキュメントタイトルが命名規則に従っている
  • バージョン・作成日・作成者・更新履歴が記載されている

  • 誰がいつ何を変えたかを追跡できるようにするため

  • 関連ドキュメントへの参照（要件定義書・他設計書）がリンクまたはファイル名で明記されている
  • 用語が要件定義書と統一されている

  • 同じ概念に別の名前が混在すると、実装者が混乱する

  • 図・表に番号とキャプションが付いている
  • 図・表・文章の内容が互いに矛盾していない
  • 誤字・脱字がない

画面設計書チェック¶

一覧・構成¶

  • 全画面のリスト（画面ID・画面名・概要）が記載されている
  • 画面IDが全画面に対して一意に振られている
  • 画面遷移図が作成されており、最新の仕様を反映している
  • 画面遷移図に、遷移の条件（ボタン押下・ログイン状態・権限など）が記載されている
  • 権限・ロールによってアクセスできない画面の制御が遷移図または一覧に明記されている

個別画面¶

レイアウト¶

  • ワイヤーフレームまたはモックアップが添付されている
  • ワイヤーフレームと項目定義に齟齬がない（存在する項目が図に反映されている）

入力項目¶

  • 各入力項目について、以下がすべて定義されている
  • 項目名・項目ID
  • データ型（文字列・数値・日付など）と最大文字数
  • 必須 / 任意の区別
  • 初期値・プレースホルダー
  • 入力バリデーションルール（形式・範囲・文字種）
  • バリデーションエラー時のメッセージと表示位置

表示・操作¶

  • 表示項目のソート・フィルタ・ページネーション仕様が定義されている
  • 検索条件がある場合、デフォルト値と検索結果の上限件数が定義されている
  • ボタン押下時のアクション（遷移先・呼び出しAPI・確認ダイアログの有無）が定義されている
  • 処理中のローディング表示の有無が定義されている
  • 処理成功・失敗時のフィードバック（メッセージ表示など）が定義されている

権限・アクセス制御¶

  • 権限・ロールによる表示制御（表示 / 非表示 / 編集可否）が定義されている

アクセシビリティ・UX¶

  • 主要な操作のキーボード操作が考慮されている（対象の場合）
  • モバイル / レスポンシブ対応方針が明記されている（対象の場合）

API設計書チェック¶

全体¶

  • APIの一覧（エンドポイント・HTTPメソッド・概要）が記載されている
  • 認証・認可方式（Bearer Token・APIキーなど）が定義されている

  • どのAPIにも認証が必要かを明確にしないと、認証漏れが発生する

  • 共通リクエストヘッダー（Content-Type・Authorization等）が定義されている
  • 共通レスポンス形式（成功時・エラー時の構造）が定義されている
  • バージョニング方針（/v1/ 等のURL設計）が明記されている
  • レート制限の有無と上限が定義されている

個別エンドポイント¶

リクエスト¶

  • HTTPメソッドとURLパスが明記されている
  • リクエストパラメータ（Path / Query / Body）について以下が定義されている
  • パラメータ名・型
  • 必須 / 任意の区別
  • バリデーションルール（最大長・範囲・パターン等）
  • サンプル値

レスポンス¶

  • レスポンスボディのスキーマ（フィールド名・型・説明）が定義されている
  • 正常系のHTTPステータスコードが定義されている
  • 想定される異常系のHTTPステータスコードが網羅されている（400・401・403・404・500 等）
  • エラーレスポンスのフォーマットと code / message の値が定義されている
  • サンプルリクエスト・レスポンス（JSONの具体例）が記載されている

仕様詳細¶

  • ページネーション方式（オフセット / カーソル等）が定義されている（一覧系APIの場合）
  • ファイルアップロード / ダウンロードの場合、Content-Typeと最大サイズが定義されている
  • 冪等性（同じリクエストを複数回送っても結果が変わらないか）の扱いが明記されている

  • 特にPUT・DELETEで重要。ネットワーク障害時のリトライで二重処理が起きないようにするため

セキュリティ¶

  • 入力値のサニタイズ方針が記載されている
  • 個人情報・機密情報を含むパラメータのログ除外・マスク方針が定義されている
  • CORS（クロスオリジン）設定の方針が明記されている

データ設計書チェック¶

新人向けメモ: データ設計書はテーブル・ファイル等のデータ構造を定義する。基本設計段階では論理レベル（概念・エンティティ）の整理が主。

全体¶

  • ER図（エンティティ関連図）が作成されている
  • ER図が最新の仕様を反映している
  • エンティティ（テーブルの元となる概念）の一覧と説明が記載されている
  • エンティティ間の関連（1対多・多対多等）が明記されている

個別エンティティ¶

  • 各エンティティの属性（項目名・データ型・必須可否）が定義されている
  • 主キー・外部キーが定義されている
  • 多対多の関連がある場合、中間テーブルが定義されている
  • データの保持期間・削除方針（論理削除 / 物理削除）が定義されている

バッチ設計書チェック¶

全体¶

  • バッチ一覧（バッチID・バッチ名・概要）が記載されている
  • 実行環境・実行ユーザーが定義されている
  • ジョブスケジューラ・実行基盤（cron / ジョブキュー等）が明記されている
  • バッチ間の依存関係（実行順序）が定義されている

個別バッチ¶

  • 処理目的・対象データが明記されている
  • トリガー条件（スケジュール式 / イベント駆動）が定義されている
  • 処理フロー（フローチャートまたは連番手順）が記載されている
  • 入力データのソース（DB・ファイル・API等）と取得条件が定義されている
  • 出力データの宛先（DB・ファイル・通知等）と形式が定義されている
  • 処理件数の上限・バッチサイズ（チャンクサイズ）が定義されている
  • 排他制御・多重起動防止の方針が定義されている

  • 同じバッチが同時に2つ起動するとデータ二重更新になる恐れがある

  • リトライ方針（最大回数・間隔・リトライ対象エラー）が定義されている
  • エラー発生時の後続処理（中断 / 継続 / ロールバック）が定義されている
  • ログ出力内容（開始・終了・処理件数・エラー詳細）が定義されている
  • 監視アラートの条件（失敗・タイムアウト・しきい値超過）が定義されている
  • 想定実行時間と許容タイムアウト時間が定義されている
  • 前提条件（依存バッチ・依存テーブルの状態等）が記載されている
  • 本番データを更新する場合、バックアップ・リストア手順が定義されている

非機能要件との整合確認¶

新人向けメモ: 設計が要件定義書の非機能要件（性能・可用性など）を満たせているかを確認する。

  • 性能要件（レスポンスタイム・スループット）に対して、設計上の対応方針が検討されている
  • 可用性要件（稼働率・障害時の対応）に対して、設計上の対応方針が検討されている
  • セキュリティ要件に対して、設計上の対応方針が検討されている
  • 拡張性・保守性に関して、設計時点で考慮された点が記載されている

レビュー依頼前の最終確認¶

  • 上記チェックリストの全項目を自己レビュー済み
  • 要件定義書の全要件が設計書に反映されていることを確認した
  • レビュー依頼票（対象ドキュメント・確認期限・レビュアー）を作成した
  • レビュー対象ドキュメントを所定の場所に配置・共有した
