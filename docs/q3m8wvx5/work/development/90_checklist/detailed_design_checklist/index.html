詳細設計 設計書レビューチェックリスト¶

このチェックリストの使い方¶

  1. 設計書を作成したら、レビュー依頼の前に自分でこのリストを確認する
  2. 各項目を確認し、OKなら [x]、該当しない場合は [N/A] と記入する
  3. 全項目が [x] または [N/A] になったらレビュー依頼を出してよい
  4. 「なぜ？」の補足がある項目は、意図を理解したうえで確認すること

新人向けメモ: 詳細設計は「どう作るか」を決める工程。実装者がこの設計書だけを見て迷わずコーディングできる粒度になっていることが目標。

共通チェック項目¶

  • ファイル名・ドキュメントタイトルが命名規則に従っている
  • バージョン・作成日・作成者・更新履歴が記載されている
  • 関連ドキュメントへの参照（要件定義書・基本設計書・他詳細設計書）がリンクまたはファイル名で明記されている
  • 用語が上位設計書（要件定義・基本設計）と統一されている

  • 用語が揺れると実装者が混乱し、バグの原因になる

  • 図・表に番号とキャプションが付いている
  • 図・表・文章の内容が互いに矛盾していない
  • 誤字・脱字がない
  • 基本設計書からの差分・変更点がある場合、変更理由が記載されている

  • 「なぜ基本設計から変えたか」の根拠がないと、後から意図が失われる

画面設計書チェック¶

一覧・構成¶

  • 対応する基本設計の画面IDと紐づいている
  • 画面レイアウト（ワイヤーフレーム）が実装に十分な粒度で記載されている

  • 「このワイヤーフレームを見れば、デザイナーやフロントエンジニアが実装できる」レベルが目標

  • コンポーネント・部品の再利用方針が明記されている（共通ヘッダー・入力フォームなど）

入力・表示項目¶

  • 全入力項目について以下が定義されている
  • 項目名・項目ID・対応するDBカラム名
  • データ型・最大文字数・バイト数（DBの制約と整合しているか）
  • 必須 / 任意の区別
  • 初期値・デフォルト値の算出ロジック（固定値か、どのデータから計算するか）
  • バリデーションルール（形式・範囲・文字種・他項目との相関チェック）
  • バリデーションエラーメッセージと表示位置
  • 表示項目のフォーマットが定義されている（日付: YYYY/MM/DD、金額: カンマ区切り など）
  • 表示件数・ページネーション・ソートのデフォルト値と変更可否が定義されている
  • 権限・ロール別の表示 / 非表示 / 編集可否が項目単位で定義されている

  • 画面単位の制御だけでなく、項目レベルの制御も漏れなく定義する

イベント・アクション¶

  • 全ボタン・リンクのアクション（呼び出しAPI・遷移先・遷移条件）が定義されている
  • 確認ダイアログ・モーダルの表示条件とメッセージ内容が定義されている
  • 非同期処理中のローディング表示・操作ロック仕様が定義されている

  • ローディング中に再度ボタンを押せると二重送信が発生する

  • 処理成功・失敗時のフィードバック（トースト・メッセージ等）が定義されている
  • ブラウザバック・ページ離脱時の挙動（確認ダイアログの有無など）が定義されている
  • タイムアウト発生時の挙動（メッセージ表示・再ログイン誘導など）が定義されている

状態管理¶

  • 画面の各状態（初期・編集中・送信中・エラー・完了 等）が定義されている

  • 状態が明確でないと、UIの表示ロジックが実装者の判断に委ねられてしまう

  • 画面をまたぐ状態の保持方法（セッション・URLパラメータ・ストア等）が定義されている

API設計書チェック¶

全体¶

  • 対応する基本設計のAPI一覧と整合している
  • 内部呼び出し（フロント→バック）・外部呼び出し（外部サービス連携）の区別が明記されている

個別エンドポイント¶

リクエスト¶

  • HTTPメソッドとURLパスが明記されている
  • リクエストパラメータ（Path / Query / Body）について以下が定義されている
  • パラメータ名・型・必須 / 任意の区別
  • バリデーションルール（最大長・範囲・パターン・他パラメータとの相関チェック）
  • サンプル値（正常系・境界値の例）

レスポンス¶

  • レスポンスボディのスキーマ（全フィールドの名前・型・説明・null許容可否）が定義されている
  • 正常系・異常系の全HTTPステータスコードが網羅されている
  • エラーレスポンスのフォーマットと code / message の値が全パターン定義されている

  • エラーコードが定義されていないと、フロントエンド側でエラー判定ができない

  • サンプルリクエスト・レスポンス（JSONの具体例）が記載されている
  • ページネーション方式とレスポンスフィールド（total・next等）が定義されている（一覧系の場合）

処理ロジック¶

  • 処理フロー（シーケンス図またはフローチャート）が記載されている
  • 参照・更新するDBテーブルが明記されており、SQLの概要またはORMのクエリ方針が記載されている
  • トランザクション範囲（どこからどこまでが1トランザクションか）と分離レベルが定義されている

  • トランザクション範囲が曖昧だと、障害時にデータが不整合になる

  • 排他制御の方針が定義されている
  • 楽観的ロック（更新時にバージョン番号を確認）か悲観的ロック（DB行ロック）かが明記されている
  • 外部API・サービス呼び出しがある場合、タイムアウト時間とリトライ方針が定義されている
  • キャッシュを利用する場合、以下が定義されている
  • キャッシュキーの設計
  • TTL（有効期限）
  • キャッシュを無効化するタイミング（更新・削除時など）
  • 非同期処理（キュー投入等）がある場合、処理の流れと失敗時の扱いが定義されている

セキュリティ¶

  • 認証・認可チェックの実施箇所と方法が定義されている

  • 「このAPIは誰が呼べるか」を明確にする。認証漏れはセキュリティ上の重大な欠陥になる

  • 入力値のサニタイズ・エスケープ方針が記載されている
  • 個人情報・機密情報を含むパラメータのマスク・ログ除外が定義されている
  • 以下のOWASP Top10の主要な脅威に対する対策が考慮されている
  • SQLインジェクション（プレースホルダ・バインド変数の使用）
  • XSS（出力時のエスケープ）
  • 認証・セッション管理の不備

データ設計書チェック¶

新人向けメモ: 詳細設計段階では、基本設計のER図をもとに物理テーブル定義（実際のDBに作るテーブルの仕様）を作成する。

全体¶

  • 対応する基本設計のER図と整合している
  • テーブル一覧（テーブルID・テーブル名・概要）が記載されている

個別テーブル¶

  • テーブルの用途・格納するデータの概要が記載されている
  • 全カラムについて以下が定義されている
  • カラム名（物理名・論理名）
  • データ型・桁数
  • NOT NULL制約の有無
  • デフォルト値
  • 説明（特に用途が不明瞭なカラム）
  • 主キーが定義されている
  • 外部キー制約とその参照先が定義されている
  • インデックスが定義されており、検索条件・結合条件を考慮した設計になっている

  • インデックスがないと大量データ時に検索が極端に遅くなる

  • ユニーク制約が必要なカラムに設定されている
  • 論理削除を採用する場合、削除フラグ・削除日時カラムが定義されている
  • 作成日時・更新日時の管理方針（自動設定か手動設定か）が定義されている

バッチ設計書チェック¶

全体¶

  • 対応する基本設計のバッチ一覧と整合している
  • 実行環境・実行ユーザー・実行権限が定義されている
  • スケジュール式（cron式等）が具体的に記載されている（例: 0 2 * * * = 毎日午前2時）

個別バッチ¶

処理定義¶

  • 処理目的・対象データの条件が具体的に記載されている
  • 処理フロー（フローチャートまたは連番手順）が実装可能な粒度で記載されている

  • 「ステップ1: ○○テーブルから△△条件でデータを取得する」レベルの粒度が目標

  • 入力データのソース（テーブル名・ファイルパス・API）と取得SQL / 抽出条件が定義されている
  • 出力データの宛先（テーブル名・ファイルパス・通知先）と出力フォーマットが定義されている
  • 処理対象の抽出条件（日付範囲・ステータス等）が明確に定義されている

性能・信頼性¶

  • 処理件数の上限・チャンクサイズ（一度にコミットする件数）が定義されている

  • 全件を一括コミットすると大量データ時にメモリ不足やロック競合が起きる

  • 想定データ量と想定実行時間が記載されている
  • 許容タイムアウト時間と超過時の挙動が定義されている
  • 排他制御・多重起動防止の実装方法（ロックファイル・DBフラグ等）が定義されている
  • リトライ方針（最大回数・リトライ間隔・リトライ対象エラーの条件）が定義されている

エラー・リカバリ¶

  • エラー発生時の後続処理（中断 / 継続 / 部分ロールバック）が定義されている
  • エラーデータの退避・スキップ方針が定義されている

  • 1件エラーで全体が止まるのか、スキップして続けるのかを明確にする

  • リカバリ手順が記載されている（再実行方法・冪等性の保証方法）

  • 「途中で失敗した場合、どうやって安全に再実行するか」を定義する

  • 本番データを更新する場合、バックアップ・リストア手順が定義されている

監視・ログ¶

  • ログ出力内容が定義されている（開始・終了・処理件数・スキップ件数・エラー詳細）
  • ログファイルのパスとローテーション方針（保存期間・世代数）が定義されている
  • ログレベルの使い分けが定義されている（INFO / WARN / ERROR 等）
  • 監視アラートの条件（失敗・タイムアウト・しきい値超過）と通知先が定義されている

テスタビリティ確認¶

新人向けメモ: 実装後にテストしやすい設計になっているかを確認する。後から「テストできない」と気づくと大きな手戻りになる。

  • 単体テストで検証すべきロジックが明確になっている
  • 外部依存（DB・外部API・ファイルなど）をモックやスタブで差し替えられる設計になっている
  • テストデータの作成・クリーンアップ方針が考慮されている
  • バッチやAPIの処理結果を検証できる仕組みが設計に含まれている（ログ・検証用出力など）

レビュー依頼前の最終確認¶

  • 上記チェックリストの全項目を自己レビュー済み
  • 基本設計書との整合性を確認済み（基本設計から変更した点は理由を記載済み）
  • 実装者がこの設計書だけを見て、迷わず実装できる粒度になっているか確認した
  • レビュー依頼票（対象ドキュメント・確認期限・レビュアー）を作成した
  • レビュー対象ドキュメントを所定の場所に配置・共有した
