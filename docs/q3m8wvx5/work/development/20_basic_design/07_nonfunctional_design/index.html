非機能要件設計¶

このドキュメントの目的¶

非機能要件設計は、「何ができるか（機能）」ではなく「どれくらいの品質か」を設計する作業です。

機能要件 vs 非機能要件:
- 機能要件: 「注文ができる」「ユーザーを登録できる」（何をするか）
- 非機能要件: 「1秒以内に画面が表示される」「99.9%の稼働率を確保する」（どれくらいの品質か）

非機能要件の観点一覧¶

観点 概要
パフォーマンス レスポンスタイム、スループット
可用性 稼働率、ダウンタイム
拡張性・スケーラビリティ データ・ユーザー増加への対応
セキュリティ 不正アクセス対策、データ保護
運用・保守 バックアップ、監視、障害対応
ユーザビリティ 操作性、アクセシビリティ
法令・規制対応 個人情報保護法、各種コンプライアンス

パフォーマンス設計¶

設計の目的¶

システムが快適に動作するための性能目標を定義します。

主な指標¶

指標 説明
レスポンスタイム ユーザーが操作してから画面が表示されるまでの時間
スループット 単位時間あたりの処理件数（リクエスト数/秒 など）
同時接続数 同時にシステムを使えるユーザー数

要件定義書の確認事項¶

  • 想定ユーザー数は何人か？
  • ピーク時の同時アクセス数はどれくらいか？
  • 特定の時間帯に負荷が集中するか？（例: 月末の請求処理）

設計記載例¶

## パフォーマンス要件

### レスポンスタイム目標
| 処理種別 | 目標値 | 測定条件 |
|---|---|---|
| 画面表示（一覧系） | 3秒以内 | 同時100ユーザー時 |
| 画面表示（詳細・検索） | 2秒以内 | 同時100ユーザー時 |
| データ登録・更新 | 3秒以内 | 同時100ユーザー時 |
| ファイルダウンロード（PDF） | 5秒以内 | 同時10ユーザー時 |
| バッチ処理（月次請求書生成） | 30分以内 | 対象件数1,000件 |

### 処理量目標
| 項目 | 想定値 |
|---|---|
| 登録ユーザー数（初年度末） | 500名 |
| 同時アクセス数（ピーク） | 50名 |
| 1日あたりの注文件数（ピーク） | 200件 |
| 注文明細の蓄積データ量（3年後） | 約100万件 |

パフォーマンス改善のアプローチ¶

設計段階で対策を盛り込んでおくことが重要です。

問題 対策
一覧表示が遅い DBインデックスの設計、ページネーションの実装
同時アクセスで遅くなる キャッシュ（Redis等）の導入、ロードバランサーによる負荷分散
重い集計処理 集計用の別テーブル（サマリーテーブル）を用意し、バッチで事前集計する

可用性設計¶

設計の目的¶

システムが「使える状態」をどれくらい維持するかの目標を定義します。

稼働率の目安¶

稼働率 1年間のダウンタイム 用途の目安
99% 約87時間 社内ツール・低重要度システム
99.9% 約8.7時間 一般的な業務システム
99.99% 約52分 ミッションクリティカルなシステム

設計記載例¶

## 可用性要件

### 稼働率目標
- 目標稼働率: 99.9%（月間）
- 計画メンテナンス時間は稼働率計算から除外する

### 稼働時間
- 通常稼働: 月〜金 8:00〜22:00
- 緊急時は土日・深夜も対応する

### メンテナンス時間
- 計画: 毎週日曜日 02:00〜04:00
- 緊急メンテナンス: 事前に1日以上前にユーザーへ通知する

### 障害復旧目標
- RTO（Recovery Time Objective）: 障害発生から4時間以内に復旧する
- RPO（Recovery Point Objective）: 最大1時間分のデータ消失を許容する

### 冗長化方針
- Webサーバー: 2台構成（ロードバランサーで分散）
- DBサーバー: Primary/Replicaの2台構成（障害時は自動フェイルオーバー）

RTO / RPO とは？
- RTO（目標復旧時間）: 障害発生から何時間以内にシステムを復旧させるか
- RPO（目標復旧時点）: 障害発生時、どの時点までのデータを回復できればよいか（どれくらいのデータ消失を許容するか）

拡張性・スケーラビリティ設計¶

設計の目的¶

将来のユーザー増加・データ増加に対応できる設計方針を定義します。

設計記載例¶

## 拡張性要件

### スケールアウト方針
- Webサーバー・APサーバーは必要に応じてサーバー台数を増やせる構成とする
- サーバー台数の増減は設定変更のみで対応可能（アプリケーションの変更不要）

### データ増加への対応
- 5年後の想定データ量: 注文テーブル 約500万件
- 古いデータ（3年以上前）のアーカイブ方針: 別テーブルに移動し、通常クエリの対象から外す
- インデックス設計: データ量増加を見越して適切なインデックスを事前に設計する

### 機能拡張への対応
- APIをRESTful設計にし、将来のモバイルアプリ・外部連携に対応できる構成とする

セキュリティ設計¶

設計記載例¶

## セキュリティ要件

### 認証・認可
（→ システム方式設計書 セキュリティ方針を参照）

### 脆弱性対策
| 脅威 | 対策 |
|---|---|
| SQLインジェクション | プリペアドステートメントを使用する（ORMの利用）|
| XSS（クロスサイトスクリプティング） | 出力時のエスケープ処理を徹底する |
| CSRF（クロスサイトリクエストフォージェリ） | CSRFトークンを使用する |
| 不正ログイン | ログイン失敗回数制限、アカウントロック機能を実装する |
| 通信の盗聴 | HTTPS必須（TLS 1.2以上）|

### 個人情報保護
- 個人情報（氏名、メールアドレス等）は暗号化して保存する
- アクセスログに個人情報を含めない（マスキング）
- 個人情報を含むCSV/帳票の取り扱い手順を定める

### 監査ログ
- 個人情報へのアクセス・変更操作をログに記録する
- ログの保持期間: 1年間
- ログファイルの改ざん対策: ログへの書き込み権限を最小化する

運用・保守設計¶

設計記載例¶

## 運用・保守要件

### バックアップ方針
| 対象 | バックアップ頻度 | 保持期間 | 保存先 |
|---|---|---|---|
| DBデータ | フルバックアップ: 毎日03:00 / 差分: 毎時00分 | 30日間 | 別リージョンのストレージ |
| アップロードファイル | 毎日03:30 | 30日間 | 別リージョンのストレージ |
| ログファイル | リアルタイム（ログ集約サーバーへ転送） | 1年間 | ログ管理サービス |

### 監視方針
| 監視対象 | 監視間隔 | アラート条件 | 通知先 |
|---|---|---|---|
| サーバー死活監視 | 1分 | サーバーダウン | 管理者メール・Slackチャンネル |
| CPU使用率 | 5分 | 80%以上が5分継続 | 管理者メール |
| メモリ使用率 | 5分 | 85%以上が5分継続 | 管理者メール |
| DBのディスク使用率 | 1時間 | 80%以上 | 管理者メール |
| エラーログ | リアルタイム | ERRORレベル以上のログ発生 | Slackチャンネル |

### 障害対応手順（概要）
1. 監視アラートを受信する
2. 状況を確認する（サーバー・アプリ・DB）
3. 影響範囲を特定し、必要に応じてメンテナンスモードに切り替える
4. 原因を特定し、復旧対応を行う
5. 復旧後、動作確認を行い、正常化を確認する
6. 障害報告書を作成する（原因・対応内容・再発防止策）

### リリース手順（概要）
1. テスト環境での動作確認完了
2. リリース前の本番DBバックアップ取得
3. メンテナンスモードに切り替え
4. デプロイ実施
5. 動作確認・スモークテスト実施
6. メンテナンスモード解除

成果物チェックリスト¶

  • 要件定義書の非機能要件がすべて設計に反映されているか
  • 性能目標（レスポンスタイム・同時接続数）が数値で定義されているか
  • 稼働率目標・RTO/RPOが定義されているか
  • バックアップ方針が定義されているか
  • 監視方針が定義されているか
  • セキュリティ対策が主要な脅威に対して網羅されているか

よくある失敗¶

「なんとかなる」で性能目標を設定しない¶

性能目標がないと、テスト時に「これは合格なのかどうか」の判断ができません。曖昧でも「3秒以内」という数字を決めることが重要です。

バックアップを設計しない¶

「バックアップは後で考える」では遅いです。本番リリースと同時にバックアップが取れている状態にしておく必要があります。

監視を設計しない¶

「動かなくなってからユーザーから連絡が来て初めて気づく」という状態は防ぎましょう。最低限の死活監視はリリース前に必ず設定します。

成果物例: 社員管理システム¶

パフォーマンス要件¶

## レスポンスタイム目標

| 処理種別 | 目標値 | 測定条件 |
|---|---|---|
| 画面表示（社員一覧・部署一覧） | 3秒以内 | 同時30ユーザー時 |
| 画面表示（社員詳細・異動履歴） | 2秒以内 | 同時30ユーザー時 |
| 社員登録・編集 | 3秒以内 | 同時10ユーザー時 |
| 帳票出力（社員名簿PDF） | 10秒以内 | 同時5ユーザー時、対象500件 |
| バッチ（月次レポート生成） | 5分以内 | 社員数1,000名時 |

## 処理量目標

| 項目 | 想定値 |
|---|---|
| 登録社員数（初年度末） | 1,000名 |
| 同時アクセス数（ピーク） | 30名（月初・月末の人事処理集中時） |
| 社員データの蓄積量（5年後） | 約5,000名（退職者含む） |
| 異動履歴レコード数（5年後） | 約30,000件 |

可用性要件¶

## 稼働率目標
- 目標稼働率: 99.9%（月間）
- 計画メンテナンス時間は稼働率計算から除外する

## 稼働時間
- 通常稼働: 月〜金 7:00〜22:00
- 土日・祝日: 原則停止（緊急時は情報システム部が対応）

## メンテナンス時間
- 計画: 毎週日曜日 02:00〜04:00
- 緊急メンテナンス: 事前に1営業日以上前にユーザーへSlackおよびメールで通知する

## 障害復旧目標
- RTO（目標復旧時間）: 障害発生から4時間以内に復旧する
- RPO（目標復旧時点）: 最大1時間分のデータ消失を許容する

## 冗長化方針
- Webサーバー: 2台構成（ALBで分散）
- DBサーバー: Primary/Replicaの2台構成（Multi-AZ、障害時は自動フェイルオーバー）
- ADサーバー: 社内のAD冗長化構成に依存（情報インフラ部管理）

セキュリティ要件¶

## 認証・認可
- 認証: 社内Active Directoryによるシングルサインオン
- セッションタイムアウト: 60分（無操作時）
- ロールベースアクセス制御: システム管理者 / 人事担当者 / 管理職 / 一般社員

## 脆弱性対策

| 脅威 | 対策 |
|---|---|
| SQLインジェクション | Spring Data JPA（プリペアドステートメント）を使用する |
| XSS | Reactの自動エスケープ + Spring SecurityのCSPヘッダー設定 |
| CSRF | Spring SecurityのCSRFトークンを使用する |
| 不正アクセス | ADログイン失敗5回でアカウントロック（30分） |
| 通信の盗聴 | HTTPS必須（TLS 1.2以上）、社外アクセスはVPN経由 |

## 個人情報保護
- 社員個人情報（氏名・連絡先等）は権限のあるロールのみ参照可能とする
- 一般社員ロールは自分の情報のみ参照可能
- 個人情報を含む帳票の出力は人事担当者のみ可能
- 操作ログは個人情報をマスキングして記録する

## 監査ログ
- 個人情報（社員情報）への参照・変更・削除操作をすべてログに記録する
- ログ保持期間: 3年間
- ログへの書き込み権限: アプリケーションサービスアカウントのみ

運用・保守要件¶

## バックアップ方針

| 対象 | バックアップ頻度 | 保持期間 | 保存先 |
|---|---|---|---|
| DBデータ | フルバックアップ: 毎日03:00 / 自動スナップショット: 1時間ごと | 30日間 | AWS S3（別リージョン） |
| アプリログ | リアルタイム転送 | 1年間 | AWS CloudWatch Logs |
| 生成ファイル（帳票等） | 毎日04:00 | 90日間 | AWS S3 |

## 監視方針

| 監視対象 | 監視間隔 | アラート条件 | 通知先 |
|---|---|---|---|
| サーバー死活監視 | 1分 | サーバーダウン | 情報システム部Slackチャンネル + 担当者メール |
| CPU使用率 | 5分 | 80%以上が5分継続 | 情報システム部Slackチャンネル |
| DBのディスク使用率 | 1時間 | 80%以上 | 担当者メール |
| アプリエラーログ | リアルタイム | ERRORレベル以上のログ発生 | 情報システム部Slackチャンネル |
| バッチ処理失敗 | バッチ実行後 | 終了ステータスが異常 | 情報システム部 + 人事担当者メール |

## リリース手順（概要）
1. テスト環境での動作確認完了・人事担当者による受入テスト完了
2. リリース前の本番DBスナップショット取得
