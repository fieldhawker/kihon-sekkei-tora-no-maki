レビュー・承認手順¶

このドキュメントの目的¶

レビュー・承認は、基本設計の成果物に誤り・抜け漏れがないかを複数人で確認し、関係者の承認を得るプロセスです。

レビューは「形式的にハンコをもらう場」ではありません。
設計の品質を高める最後のチャンスです。指摘しやすい雰囲気を作り、実質的な確認を行いましょう。

レビューの全体フロー¶

1. セルフレビュー（自己チェック）
2. チーム内レビュー（リーダー・同僚によるレビュー）
3. 顧客レビュー（ユーザー・発注者による確認）
4. 指摘事項の修正・対応
5. 承認・完了

STEP 1: セルフレビュー（自己チェック）¶

目的¶

チームに見せる前に、自分で成果物を見直して明らかな誤りや抜けを修正します。
「恥ずかしいミス」を事前につぶしておくことで、チームレビューの質が上がります。

セルフレビューのチェックリスト¶

共通チェック項目¶

  • 要件定義書のすべての機能要件が設計に反映されているか
  • 要件定義書のすべての非機能要件が設計に反映されているか
  • 用語が全ドキュメントで統一されているか（例：「顧客」と「ユーザー」が混在していないか）
  • 各成果物のIDが重複・欠番なく採番されているか
  • 誤字・脱字がないか
  • 図や表が崩れていないか

成果物ごとのチェック項目¶

成果物 チェックポイント
システム構成図 全コンポーネントが記載されているか、通信方向が正しいか
画面一覧 全機能に対応する画面が洗い出されているか
画面遷移図 孤立した画面（到達できない・離脱できない）がないか
ワイヤーフレーム 全画面分作成されているか
画面入出力定義書 バリデーションが定義されているか、エラーメッセージが記載されているか
ER図 全エンティティが含まれているか、リレーションが正しいか
テーブル定義書 主キー・外部キーが定義されているか、created_at/updated_atがあるか
機能一覧 全機能要件が含まれているか
外部I/F設計書 エラー処理が定義されているか
非機能要件設計書 性能目標・稼働率が数値で定義されているか

整合性チェック（成果物間の矛盾確認）¶

これが最も重要です。成果物が独立して「正しい」だけでなく、相互に矛盾がないかを確認します。

  • 画面に表示する項目が、テーブル定義書に存在するか
  • 機能一覧の全機能が、画面設計・テーブル定義と対応しているか
  • 外部I/Fのデータ項目が、テーブル定義と整合しているか
  • 帳票の出力項目が、テーブル定義に存在するか
  • バッチ処理が参照・更新するテーブルが定義されているか

STEP 2: チーム内レビュー¶

目的¶

リーダーや同僚の視点で設計の妥当性・完全性を確認します。
自分では気づけない設計ミスや考慮漏れを発見することが目的です。

レビューの準備¶

レビュー依頼者（設計者）がやること: 1. 成果物一式を整理し、レビュー依頼メールまたはチケットを送る 2. 「どこを重点的に見てほしいか」を明示する 3. 事前に資料を共有し、レビュアーが事前に読めるようにする（当日ぶっつけは避ける）

依頼メールの例: 

件名: 【レビュー依頼】〇〇システム基本設計 - STEP3画面設計

田中さん

お世話になっております。
○○プロジェクトの画面設計レビューをお願いしたいです。

【対象成果物】
- 画面一覧
- 画面遷移図
- ワイヤーフレーム（全12画面）
- 画面入出力定義書

【資料リンク】（共有フォルダURL）

【特に見ていただきたい点】
- 注文フローの画面遷移（SCR-ORD-001〜003）が業務フローと合っているか
- 注文登録画面のバリデーションが要件を満たしているか

【希望レビュー期限】 〇月〇日（〇曜日）

お手数ですが、よろしくお願いします。

レビュー会の進め方¶

チームでレビューを行う場合の進め方の例です。

時間 内容
0〜5分 設計者が成果物の概要を説明する
5〜40分 各成果物をスクリーンに映しながらウォークスルーを行う
40〜50分 質問・指摘事項を出し合う
50〜60分 指摘事項を整理し、対応方針を確認する

ウォークスルーとは？
設計者が資料を見せながら「この画面はこういう動きをします」と説明し、レビュアーが気になった点を都度確認する方式です。最も一般的なレビュー方法です。

指摘の種類と対応¶

種類 意味 対応
必須対応（Critical） このままでは設計が成立しない 必ず修正する
要確認（Major） 認識が合っているか確認が必要 ユーザーまたはリーダーに確認して判断する
提案（Minor） より良くなる提案 設計者が判断して対応するかどうか決める

STEP 3: 顧客レビュー¶

目的¶

ユーザー（発注者・業務担当者）に設計内容を確認してもらい、認識のズレをこの段階でなくします。
「作ってみたら思ってたのと違う」という最悪の事態を防ぐためのステップです。

顧客レビューの準備¶

  • ユーザーが理解できる言葉・粒度で資料を準備する
  • ER図やシステム構成図は技術的すぎるので、ユーザー向けには噛み砕いて説明する
  • 画面ワイヤーフレームは非常に有効（具体的なので意見が出やすい）
  • 「承認を求める」のではなく「一緒に確認する」という姿勢で臨む

顧客レビューの重点確認項目¶

確認項目 なぜ確認するか
画面のレイアウト・項目 ユーザーが「使いやすいか」「項目が足りているか」を判断できる
業務フローとの整合性 実際の業務の流れと画面遷移が合っているかを確認する
帳票のレイアウト 既存の帳票と変わる場合、ユーザーが問題ないか確認する
権限設計（誰が何を操作できるか） 業務上の役割と権限が合っているかを確認する
非機能要件（性能・稼働率） 業務に支障ない性能かどうかを確認する

顧客への説明のコツ¶

  • 「何のために」から説明する: 技術的な説明の前に、「この機能は〇〇という業務課題を解決するためのものです」と目的を先に伝える
  • 具体例を使う: 「注文ボタンを押すと〜の順番で処理が動きます」と具体的なシナリオで説明する
  • 用語を合わせる: ユーザーが使っている言葉（業務用語）を使って説明する。設計書の用語と違う場合は用語集を作成する

STEP 4: 指摘事項の修正・対応¶

修正管理の方法¶

レビューで出た指摘事項を管理し、対応漏れをなくします。

指摘事項管理表の記載例¶

# 指摘日 指摘者 対象成果物 指摘内容 種別 対応方針 対応者 対応期限 対応状況
001 2024/12/01 田中（レビュアー） 画面入出力定義書 ユーザー登録画面に「確認パスワード」項目がない 必須 確認パスワード入力項目を追加する 山田（設計者） 12/05 対応済
002 2024/12/01 鈴木（顧客） ワイヤーフレーム ユーザー一覧の検索条件に「部署」でも絞り込みたい 要確認 PL確認の上、追加を検討 山田 12/07 確認中
003 2024/12/02 田中（レビュアー） テーブル定義書 ordersテーブルに配送先住所のカラムがない 必須 配送先住所用カラムを追加する 山田 12/05 対応済

ステータスの種類: - 未対応: まだ対応していない - 対応中: 現在修正中 - 確認中: 方針を確認中（顧客確認待ちなど） - 対応済: 修正・対応が完了した - 対応不要: 検討の結果、対応しないと判断した（理由を記載）

STEP 5: 承認・完了¶

承認の流れ¶

  1. 全指摘事項の対応が完了したことを確認する
  2. 修正後の成果物を関係者に共有する
  3. 承認者（リーダー・PMなど）から承認をもらう
  4. 顧客から承認のサインをもらう（重要）
  5. 基本設計書一式をバージョン管理する（日付・バージョン番号を記載）

承認書のポイント¶

  • 口頭承認だけでなく、文書（メールや署名付きの承認書）で残す
  • 「確認した」と「承認した」は違います。「この設計でシステムを作ります」という意味の承認を明示的にもらいます
  • 後から「そんな設計だとは知らなかった」という言い訳を防ぐためです

レビューに関するよくある失敗¶

指摘がない = 良い設計、ではない¶

指摘ゼロのレビューは、「レビュアーが理解できていない」「指摘しにくい雰囲気だった」可能性があります。指摘が出やすいよう、設計者自ら「特にここを見てほしい」「ここが不安なので意見をほしい」と積極的に問いかけましょう。

指摘事項を口頭で処理して記録しない¶

「その場で解決した」指摘でも、管理表に記録します。記録がないと、後から「その指摘はどう対応したんだっけ？」が追跡できなくなります。

顧客の承認を後回しにする¶

「設計はほぼ合意できているから、とりあえず開発を始めよう」は危険です。正式な承認前に開発を進めると、後から設計変更が生じた際に手戻りが大きくなります。

修正後の整合性チェックを忘れる¶

指摘に対応して成果物を修正すると、他の成果物との整合性が崩れることがあります。修正後は必ず「他の成果物に影響しないか」を確認します。
