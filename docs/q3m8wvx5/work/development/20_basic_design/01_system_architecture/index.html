システム方式設計¶

このドキュメントの目的¶

システム方式設計は、システム全体の骨格（アーキテクチャ）を決める作業です。
「どんなサーバーを使うか」「どんな技術で作るか」「セキュリティはどうするか」など、後続の全設計に影響する重要な意思決定を行います。

ここで決めたことは、後から変更するとコストが非常に大きくなります。慎重に、かつ根拠を持って決定しましょう。

作業の流れ¶

1. インプットの確認
2. システム構成の検討・決定
3. 技術スタックの選定
4. 環境構成の定義
5. セキュリティ方針の策定
6. 成果物の作成

STEP 1: インプットの確認¶

設計を始める前に、以下を必ず読み込んでおきます。

確認するもの 何を見るか
要件定義書 機能要件・非機能要件（特に性能・可用性・セキュリティ）
業務フロー図 誰が・何を・いつ操作するか
制約条件 既存システムとの連携有無、インフラの制約、予算
ユーザー環境 利用端末（PC/スマホ）、ネットワーク環境

チェックポイント: - [ ] 想定ユーザー数・同時アクセス数は把握できているか？ - [ ] 既存システムとのデータ連携が必要か？ - [ ] クラウド利用の可否・制約はあるか？ - [ ] 個人情報や機密情報を扱うか？

STEP 2: システム構成の検討・決定¶

システム構成パターン¶

代表的な構成を理解し、要件に合ったものを選びます。

パターン1: シンプルな2層構成¶

[ユーザーのブラウザ] ←→ [Webサーバー / アプリケーションサーバー] ←→ [データベース]
- 向いているケース: 小規模・シンプルな業務アプリ、社内ツール - 特徴: 構成がシンプルで管理しやすい

パターン2: 3層構成（標準的な業務システム）¶

[ユーザーのブラウザ]
        ↕
[フロントエンドサーバー（Webサーバー）]
        ↕
[アプリケーションサーバー（APIサーバー）]
        ↕
[データベースサーバー]
- 向いているケース: 中規模以上の業務システム - 特徴: 各層を独立して変更・スケールできる

パターン3: クラウドマネージドサービスを活用した構成¶

[ユーザーのブラウザ / モバイルアプリ]
        ↕
[CDN（コンテンツ配信）]
        ↕
[ロードバランサー]
        ↕
[アプリケーションサーバー（複数台）]
        ↕
[マネージドDB（RDSなど）] + [キャッシュ（Redisなど）]
- 向いているケース: アクセス数が多い・高可用性が必要なシステム - 特徴: 自動スケーリング・障害対応が容易

システム構成図の書き方¶

システム構成図は、サーバー・ネットワーク・ユーザー端末の関係を視覚的に表します。

含めるべき要素: - ユーザーの端末（ブラウザ、スマートフォンなど） - インターネット / イントラネットの境界 - ファイアウォール・ロードバランサー - Webサーバー / アプリケーションサーバー - データベースサーバー - 外部連携先（外部API、メールサーバーなど） - 各コンポーネント間の通信プロトコル（HTTPS、TCP など）

記載例（テキスト表現）: 

[インターネット]
    │ HTTPS(443)
[ファイアウォール]
    │
[ロードバランサー]
    ├── [Webサーバー #1]  ─── [APサーバー #1]
    └── [Webサーバー #2]  ─── [APサーバー #2]
                                    │ TCP(5432)
                              [DBサーバー（プライマリ）]
                                    │ レプリケーション
                              [DBサーバー（レプリカ）]

実際の作成には Draw.io、Miro、PowerPoint、Figma などのツールが便利です。

STEP 3: 技術スタックの選定¶

技術スタックとは¶

「どのプログラミング言語・フレームワーク・データベースを使うか」のセットのことです。

⚠️ 若手エンジニアへの注意: 技術選定は自分の判断だけで決めてはいけません。社内標準技術がある場合は原則それに従います。新しい技術を導入したい場合は、PLまたはアーキテクトに相談し、採用する理由を根拠書として残してから承認を得てください。「自分が使いたいから」という理由だけでは採用できません。

選定の観点¶

観点 確認内容
要件への適合性 機能要件・性能要件を満たせるか
チームのスキル 開発メンバーが扱えるか（習得コストは許容範囲か）
実績・信頼性 本番環境での利用実績があるか
ライセンス 商用利用に問題がないか
サポート・コミュニティ 問題が起きたときに情報を得られるか
コスト ライセンス費・インフラ費が予算内か

技術選定根拠書の記載項目¶

選んだ理由を必ず文書化します。「なんとなく使い慣れているから」ではなく、根拠を明示します。

## フロントエンド

**選定技術:** React 18 + TypeScript

**選定理由:**
- コンポーネントベースの開発でUIの再利用性が高い
- TypeScriptにより型安全性を確保できる
- 開発チームに習熟者が複数名いる

**代替案と比較:**
- Vue.js: 学習コストは低いが、チームの経験が少ない
- 素のHTML/JS: 複雑な状態管理が困難になる

---

## バックエンド

**選定技術:** Java 21 + Spring Boot 3

**選定理由:**
- 社内標準技術であり、既存資産を流用できる
- エンタープライズ向けの機能が充実している
- 長期的なサポートが保証されている（LTSバージョン）

---

## データベース

**選定技術:** PostgreSQL 16

**選定理由:**
- OSSで追加ライセンスコストが不要
- 複雑なJOINや集計処理に対応できる
- JSON型のサポートにより半構造データも扱える

STEP 4: 環境構成の定義¶

用意すべき環境¶

一般的なシステム開発では、目的に応じて複数の環境を用意します。

環境名 目的 利用者
開発環境（DEV） 開発者が自由に開発・テストを行う 開発チーム
テスト環境（STG/SIT） 結合テスト・システムテストを行う 開発チーム・テスター
ステージング環境（STG） 本番と同等の構成で最終確認を行う 開発チーム・顧客
本番環境（PRD） 実際のユーザーが使用する エンドユーザー

小規模プロジェクトでは開発・テスト・本番の3環境が一般的です。

環境構成の記載例¶

## 環境一覧

| 環境 | URL | サーバースペック | DB |
|---|---|---|---|
| 開発 | http://dev.example.local | t3.medium × 1台 | RDS t3.micro |
| テスト | https://stg.example.com | t3.medium × 1台 | RDS t3.small |
| 本番 | https://www.example.com | t3.large × 2台 | RDS t3.medium (Multi-AZ) |

## 環境間の違い

- 本番のみロードバランサーを使用
- 本番のみDBをMulti-AZ構成（障害時の自動フェイルオーバー）
- 本番・テスト環境はSSL/TLS必須

STEP 5: セキュリティ方針の策定¶

主要なセキュリティ観点¶

認証・認可¶

  • 認証（Authentication）: ユーザーが本人であることを確認する仕組み
  • 例: ID/パスワード、多要素認証（MFA）、SSO（シングルサインオン）
  • 認可（Authorization）: 認証済みユーザーに対して操作権限を制御する仕組み
  • 例: ロールベースアクセス制御（RBAC）

通信の暗号化¶

  • 外部との通信は必ずHTTPS（TLS 1.2以上）を使用する
  • 内部通信も機密情報を含む場合は暗号化を検討する

データ保護¶

  • パスワードはハッシュ化して保存（bcrypt、Argon2等）
  • 機密情報（クレジットカード番号等）は暗号化して保存
  • 個人情報の取り扱いは個人情報保護法・社内規定に準拠

アクセス制御¶

  • 管理画面へのアクセスはIPアドレス制限を検討する
  • 不要なポートは閉じる

セキュリティ方針記載例¶

## 認証方式
- ログイン: ID/パスワード + メール二要素認証
- パスワードポリシー: 8文字以上、英数字混在必須
- セッションタイムアウト: 30分（無操作時）
- ログイン失敗: 5回連続で15分間ロック

## 通信
- 全通信HTTPS必須（TLS 1.2以上）
- HTTPアクセスはHTTPSへリダイレクト

## ロール定義
| ロール名 | 権限 |
|---|---|
| 管理者 | 全機能にアクセス可能 |
| 一般ユーザー | 自分のデータのみ参照・編集可能 |
| 閲覧のみ | 参照のみ、更新・削除不可 |

成果物チェックリスト¶

設計完了時に以下を確認します。

  • システム構成図が作成され、全コンポーネントと通信経路が明示されている
  • 技術スタックの選定根拠が文書化されている
  • 開発・テスト・本番の環境構成が定義されている
  • 認証方式・ロール設計が定義されている
  • 通信の暗号化方針が記載されている
  • 要件定義書の非機能要件（性能・可用性・セキュリティ）が設計に反映されている

よくある失敗¶

「とりあえず動けばいい」で構成を決める¶

小規模でも、後からスケールする可能性を考慮しておかないと、ユーザーが増えたときに全面的な作り直しが必要になります。

技術選定の根拠を残さない¶

「なんでこの技術を使ったんだろう」と後から誰もわからなくなります。選定時の根拠を必ず文書化します。

環境の違いを設計しない¶

「本番だけで動く問題」が後から発覚します。本番に近い環境で早期にテストできるよう、環境差異を最小化する設計を心がけます。

参考: 用語集¶

用語 意味
アーキテクチャ システムの構造・設計思想
フロントエンド ユーザーが直接見る・操作する部分（ブラウザ側）
バックエンド ユーザーからは見えない処理・データ管理部分（サーバー側）
ロードバランサー 複数のサーバーへのアクセスを分散させる機器・ソフトウェア
CDN 画像・CSS等の静的ファイルを世界中のサーバーから高速配信する仕組み
HTTPS 通信を暗号化するWebのプロトコル
Multi-AZ 複数のデータセンターにDBを配置し、障害時に自動切り替えする構成

成果物例: 社員管理システム¶

システム構成図¶

[インターネット / 社内イントラネット]
    │ HTTPS(443)
[ファイアウォール]
    │
[ロードバランサー（AWS ALB）]
    ├── [Webサーバー #1（EC2 t3.medium）]
    └── [Webサーバー #2（EC2 t3.medium）]
            │ TCP(8080)
    [APサーバー（Spring Boot）]
            │ TCP(5432)
    [DBサーバー（RDS PostgreSQL 16 / Multi-AZ）]
            │ レプリケーション
    [DBサーバー（レプリカ）]

[社外メールサーバー（SendGrid API）] ← APサーバーから HTTPS で呼び出し
[Active Directory（社内認証）] ← APサーバーから LDAP(636) で接続

技術スタック選定根拠¶

フロントエンド¶

選定技術: React 18 + TypeScript

選定理由: - 社内の既存プロジェクトで採用実績があり、習熟者が複数名いる - TypeScriptにより入力フォームの型安全性を確保できる - コンポーネント単位の再利用で、各種マスター管理画面を効率よく開発できる

代替案と比較: - Vue.js: 学習コストは低いがチーム経験が浅い - 素のHTML/JS: 動的なフォームバリデーション・画面制御が煩雑になる

バックエンド¶

選定技術: Java 21 + Spring Boot 3

選定理由: - 社内標準技術であり既存資産を流用できる - Spring Securityで認証・認可の実装コストを削減できる - LTSバージョンのため長期サポートが保証されている

データベース¶

選定技術: PostgreSQL 16

選定理由: - OSSで追加ライセンスコストが不要 - 社員情報の複雑な検索・集計クエリに対応できる - RDS対応のためマネージドサービスとして運用コストを抑えられる

環境構成¶

環境 URL サーバースペック DB
開発 http://dev.emp-sys.local EC2 t3.medium × 1台 RDS t3.micro
テスト https://stg.emp-sys.example.com EC2 t3.medium × 1台 RDS t3.small
本番 https://emp-sys.example.com EC2 t3.large × 2台 RDS t3.medium（Multi-AZ）

環境間の差異: - 本番のみロードバランサー（ALB）を使用 - 本番のみDBをMulti-AZ構成（障害時の自動フェイルオーバー） - テスト・本番はSSL/TLS必須（TLS 1.2以上）

セキュリティ方針¶

## 認証方式
- ログイン: 社内Active DirectoryのLDAP認証（シングルサインオン）
- セッションタイムアウト: 60分（無操作時）
- ログイン失敗: 5回連続で30分間アカウントロック

## 通信
- 全通信HTTPS必須（TLS 1.2以上）
- HTTPアクセスはHTTPSへリダイレクト
- 社外からのアクセスはVPN接続を必須とする

## ロール定義
| ロール名 | 権限 |
|---|---|
| システム管理者 | 全機能にアクセス可能（マスター管理・ユーザー管理含む） |
| 人事担当者 | 全社員情報の参照・編集、帳票出力が可能 |
| 一般社員 | 自分の情報の参照・一部編集のみ可能 |
| 管理職 | 自部門の社員情報を参照可能（編集不可） |
