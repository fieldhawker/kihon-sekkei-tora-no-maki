画面設計¶

このドキュメントの目的¶

画面設計は、ユーザーが実際に操作する画面の構成・遷移・項目を定義する作業です。
「どんな画面があるか」「どの画面からどこへ移動できるか」「各画面に何が表示されるか」を具体的に決めます。

画面設計は、ユーザーにとっての「システムの顔」です。後から変更すると影響範囲が大きいため、ユーザーと合意を取りながら丁寧に進めましょう。

作業の流れ¶

1. 画面一覧の作成
2. 画面遷移図の作成
3. ワイヤーフレームの作成
4. 画面入出力定義書の作成

STEP 1: 画面一覧の作成¶

画面一覧とは¶

システムに存在するすべての画面をリストアップし、管理番号と名称を付けた一覧表です。

作成手順¶

  1. 要件定義書の機能要件を確認する
  2. 各機能に必要な画面を洗い出す
  3. 一覧表に整理し、ID・名称・概要を記載する

画面IDの採番ルール（例）¶

SCR-[カテゴリ]-[連番3桁]
例:
  SCR-AUTH-001: ログイン画面
  SCR-USR-001: ユーザー一覧画面
  SCR-USR-002: ユーザー詳細画面
  SCR-USR-003: ユーザー登録画面

IDのルールはプロジェクトで統一できれば形式は自由です。あとから画面が増えることを考えて、連番に余裕を持たせましょう。

画面一覧の記載例¶

画面ID 画面名 概要 対応機能 備考
SCR-AUTH-001 ログイン画面 ID/パスワードでログインする ログイン機能
SCR-AUTH-002 パスワードリセット画面 パスワードを再設定する パスワードリセット機能
SCR-DASH-001 ダッシュボード画面 ログイン後のトップ画面 ダッシュボード表示
SCR-USR-001 ユーザー一覧画面 登録ユーザーを一覧表示する ユーザー管理 管理者のみ
SCR-USR-002 ユーザー詳細画面 ユーザーの詳細情報を表示する ユーザー管理
SCR-USR-003 ユーザー登録画面 新規ユーザーを登録する ユーザー管理 管理者のみ
SCR-USR-004 ユーザー編集画面 ユーザー情報を編集する ユーザー管理

STEP 2: 画面遷移図の作成¶

画面遷移図とは¶

どの画面からどの画面へ移動できるかを図示したものです。ユーザーの操作フローが視覚的にわかります。

書き方のルール¶

  • 四角（□）: 画面
  • 矢印（→）: 遷移の方向
  • 矢印のラベル: 遷移のトリガー（ボタンのクリック、条件など）
  • 菱形（◇）: 条件分岐（例: 認証成功/失敗）

画面遷移図の記載例（テキスト表現）¶

[ログイン画面]
    │ 認証成功
    ▼
[ダッシュボード]
    ├─ 「ユーザー管理」クリック ──→ [ユーザー一覧]
    │                                     ├─ 「詳細」クリック → [ユーザー詳細]
    │                                     │                         └─ 「編集」クリック → [ユーザー編集]
    │                                     └─ 「新規登録」クリック → [ユーザー登録]
    │
    └─ 「設定」クリック ──→ [設定画面]

[ログイン画面]
    │ 「パスワードを忘れた方はこちら」クリック
    ▼
[パスワードリセット画面]

遷移図作成のポイント¶

  • すべての画面が遷移図に含まれているか確認する（孤立した画面がないか）
  • エラー時・キャンセル時の遷移先も明示する
  • ログインが必要な画面は、未ログイン時にログイン画面へリダイレクトされることを明示する

STEP 3: ワイヤーフレームの作成¶

ワイヤーフレームとは¶

各画面のレイアウト（項目やボタンの配置）を、デザインを省いたシンプルな図で表したものです。
「どこに何があるか」を確認するための設計図で、色・フォント・画像などの詳細デザインは含めません。

作成ツール¶

ツール 特徴 費用
Figma デザイン・プロトタイプ作成に広く使われる 無料プランあり
Draw.io（diagrams.net） シンプルな図作成に向く 無料
PowerPoint / Keynote 手軽に作れる 有料（Office契約）
Balsamiq ワイヤーフレーム専用ツール 有料
手書き 最速だが共有・修正が難しい 無料

まずは手書きやPowerPointで素早く作り、ユーザーに見せてフィードバックをもらってから清書するのが効率的です。

ワイヤーフレームに含めるべき情報¶

  • ページタイトル・ヘッダー・フッター
  • ナビゲーションメニュー
  • 表示する項目（テキスト、テーブル、画像の領域）
  • 入力フォーム（テキストボックス、セレクトボックス、ラジオボタンなど）
  • ボタン（「登録」「キャンセル」「検索」など）
  • エラーメッセージの表示位置

ワイヤーフレームの記載例（ユーザー一覧画面）¶

┌─────────────────────────────────────────┐
│ [ロゴ]  システム名              [ログアウト] │  ← ヘッダー
├──────────┬──────────────────────────────┤
│          │  ユーザー管理                   │  ← ページタイトル
│ ナビ     ├──────────────────────────────┤
│ ・ダッシュ│ [検索: 名前________] [検索ボタン]│  ← 検索エリア
│ ・ユーザー│                                │
│ ・設定   │ ┌──┬──────┬────────┬──────┐  │
│          │ │# │氏名  │メール  │操作  │  │  ← 一覧テーブル
│          │ ├──┼──────┼────────┼──────┤  │
│          │ │1 │山田太郎│yamada..│[詳細]│  │
│          │ │2 │鈴木花子│suzuki..│[詳細]│  │
│          │ └──┴──────┴────────┴──────┘  │
│          │                                │
│          │ ← 1 2 3 →   [新規登録ボタン]  │  ← ページネーション
└──────────┴──────────────────────────────┘

STEP 4: 画面入出力定義書の作成¶

画面入出力定義書とは¶

各画面の入力項目・出力項目それぞれの仕様を詳細に定義した文書です。
開発者がこれを見てプログラムを作れるレベルの詳細さが必要です。

定義が必要な項目¶

入力項目（ユーザーが入力・選択するもの）¶

定義項目 説明 例
項目名（論理名） 日本語の名称 氏名、メールアドレス
項目名（物理名） プログラム上の変数名 user_name、email
入力形式 入力コンポーネントの種類 テキストボックス、セレクトボックス
データ型 値の型 文字列、数値、日付
桁数・文字数 最大入力可能な文字数 最大50文字
必須/任意 入力が必須かどうか 必須、任意
初期値 初期表示時の値 空白、本日の日付
バリデーションルール 入力値の制約・チェック内容 メール形式、数値のみ
エラーメッセージ バリデーション違反時のメッセージ 「メールアドレスの形式が正しくありません」

出力項目（画面に表示されるもの）¶

定義項目 説明
項目名 表示するデータの名称
データソース どのテーブルのどのカラムから取得するか
表示形式 日付フォーマット、数値のカンマ区切りなど
表示条件 特定の条件下でのみ表示する場合はその条件

画面入出力定義書の記載例（ユーザー登録画面）¶

画面ID: SCR-USR-003
画面名: ユーザー登録画面
概要: 新規ユーザーの情報を入力し、システムに登録する

入力項目¶

# 項目名（論理） 項目名（物理） 形式 型 最大桁 必須 初期値 バリデーション エラーメッセージ
1 氏名 user_name テキスト 文字列 50 必須 空白 空白不可 「氏名を入力してください」
2 メールアドレス email テキスト 文字列 255 必須 空白 メール形式、重複不可 「有効なメールアドレスを入力してください」
3 パスワード password パスワード 文字列 100 必須 空白 8文字以上、英数混在 「8文字以上の英数字で入力してください」
4 ロール role セレクト コード - 必須 「一般」 選択必須 「ロールを選択してください」
5 部署 department テキスト 文字列 100 任意 空白 なし -

セレクトボックスの選択肢¶

項目 選択肢
ロール 管理者 / 一般 / 閲覧のみ

ボタン定義¶

ボタン名 処理内容 遷移先
登録 入力値を検証し、DBに保存する ユーザー一覧画面（SCR-USR-001）
キャンセル 入力内容を破棄する ユーザー一覧画面（SCR-USR-001）

成果物チェックリスト¶

作業完了チェック（各STEPの成果物確認）¶

  • 要件定義書のすべての機能に対応する画面が洗い出されているか
  • すべての画面に一意のIDが付与されているか
  • 画面遷移図に孤立した画面（どこからも遷移できない・どこへも遷移できない画面）がないか
  • ワイヤーフレームが全画面分作成されているか
  • 入力項目のバリデーションルールが定義されているか
  • エラーメッセージが具体的に記載されているか
  • 画面設計とデータ設計（テーブル定義）の整合性が取れているか

最終見直しチェックリスト¶

画面設計書の完成判断に使う総合チェックリストです。すべての項目にチェックが入れば完成とみなせます。

1. 網羅性¶

  • 要件定義書・機能要件に記載されたすべての機能に対応する画面が存在するか
  • 確認ダイアログ・モーダル・エラー画面など、補助的な画面も含めて洗い出されているか
  • 各画面に対してワイヤーフレームと入出力定義書の両方が作成されているか

2. 画面遷移の完全性¶

  • すべての画面が遷移図に含まれており、孤立した画面がないか
  • 正常系だけでなく、キャンセル・戻る・エラー時の遷移先が明示されているか
  • 未ログイン時・権限不足時のリダイレクト先が定義されているか
  • 登録・更新・削除後の遷移先（完了メッセージ表示を含む）が定義されているか

3. 入出力定義の品質¶

  • すべての入力項目に論理名・物理名・データ型・桁数・必須/任意が定義されているか
  • すべての入力項目にバリデーションルールが定義されているか（「後で決める」がないか）
  • バリデーション違反時のエラーメッセージが具体的な文言で定義されているか
  • セレクトボックス・ラジオボタン等の選択肢が列挙されているか（動的取得の場合はそのロジックが明示されているか）
  • 出力項目のデータソース（テーブル名・カラム名）と表示形式が定義されているか
  • 権限・ロールによって表示/非表示・編集可否が変わる項目が明示されているか

4. データ設計との整合性¶

  • 画面に表示・入力するすべての項目がテーブル定義に存在するか
  • 項目のデータ型・桁数がテーブル定義と一致しているか
  • 画面で操作するCRUD処理がすべてAPIまたはDB設計でカバーされているか

5. 非機能要件・例外ケースの考慮¶

  • 検索結果が0件の場合の表示内容が定義されているか
  • 一覧が大量件数になる場合のページネーション・件数上限が定義されているか
  • 必須のセレクトボックスで選択肢が0件（マスタ未登録等）の場合の扱いが定義されているか
  • 他ユーザーが同時に操作した場合（排他制御）の方針が検討されているか

6. ユーザー・ステークホルダーとの合意¶

  • 主要画面のワイヤーフレームについてユーザー（発注者・業務担当者）のレビューを受けているか
  • フィードバックを反映した修正が完了しているか
  • 最終版について承認を得ているか（口頭・メール・システム上のいずれかで記録が残っているか）

よくある失敗¶

画面数を過小評価する¶

「この機能は1画面でできる」と思っていても、確認ダイアログ・エラー表示・モーダルなどを含めると画面数が増えます。すべてのケースを洗い出しましょう。

バリデーションを設計しない¶

「後で決める」にしてしまうと、開発者がバリデーションを作らないか、バラバラな仕様になります。入力項目は必ずバリデーションを設計します。

データ設計と整合性を取らずに進める¶

画面に表示したい項目が、テーブル定義に存在しないことがあります。画面設計とデータ設計は必ず並行して整合性を確認しながら進めましょう。

# 項目名（論理） 項目名（物理） 形式 型 最大桁 必須 初期値 バリデーション エラーメッセージ

3 氏名（名） first_name テキスト 文字列 50 必須 空白 空白不可 「名を入力してください」
4 氏名カナ（姓） last_name_kana テキスト 文字列 50 必須 空白 全角カタカナのみ 「セイは全角カタカナで入力してください」
5 氏名カナ（名） first_name_kana テキスト 文字列 50 必須 空白 全角カタカナのみ 「メイは全角カタカナで入力してください」
6 メールアドレス email テキスト 文字列 255 必須 空白 メール形式、重複不可 「有効なメールアドレスを入力してください」
