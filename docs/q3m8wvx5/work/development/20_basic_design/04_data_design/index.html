データ設計¶

このドキュメントの目的¶

データ設計は、システムが扱うデータの構造（どんなデータを・どのように保持するか）を決める作業です。
後続の開発工程に最も影響するため、基本設計の中で最も重要な工程の一つです。

ここでの設計ミス（テーブルや関係性の誤り）は、後から修正すると大きなコストがかかります。丁寧に、関係者と合意しながら進めましょう。

作業の流れ¶

1. エンティティの洗い出し
2. ER図の作成
3. テーブル定義書の作成

STEP 1: エンティティの洗い出し¶

エンティティとは¶

エンティティ（Entity）とは、システムが管理すべき「もの」や「概念」のことです。
データベースのテーブルになる対象と考えると理解しやすいです。

例: ユーザー、注文、商品、請求書、部署、カテゴリ...

洗い出し方¶

以下の手順でエンティティを洗い出します。

方法1: 要件定義書・業務フローから名詞を拾う¶

業務フローや要件定義書を読み、「名詞」に注目します。

例文: 「ユーザーが商品を選んで注文し、請求書が発行される」
→ ユーザー、商品、注文、請求書 がエンティティ候補

方法2: 画面設計から確認する¶

ワイヤーフレームの入力項目・表示項目を確認し、どこかに保存が必要なデータがないか確認します。

方法3: 現行システム・現行ファイルを確認する¶

Excelで管理しているデータや既存のテーブル設計がある場合は、それを参考にします。

エンティティ一覧の記載例¶

# エンティティ名（論理） テーブル名（物理） 概要
1 ユーザー users システムを利用するユーザー情報
2 商品 products 販売する商品のマスター情報
3 カテゴリ categories 商品を分類するカテゴリ
4 注文 orders ユーザーが行った注文のヘッダー情報
5 注文明細 order_items 注文に含まれる商品の明細情報
6 請求書 invoices 注文に対して発行する請求書

STEP 2: ER図の作成¶

ER図とは¶

Entity-Relationship図（エンティティ関係図）の略で、エンティティ間の関係性（リレーション）を図示したものです。

リレーションの種類¶

エンティティ間の関係には3種類あります。

1対1（1:1）¶

1つのAに対して、Bが必ず1つ対応する関係。

例: 1人のユーザーは1つのプロフィール情報を持つ

[ユーザー] ─── [プロフィール]

1対多（1:N）¶

1つのAに対して、Bが複数対応する関係。最も多いパターン。

例: 1人のユーザーは複数の注文をすることができる

[ユーザー] ───< [注文]
              （1人が複数の注文を持つ）

多対多（M:N）¶

AもBも複数対応する関係。そのままDBに持てないため、中間テーブルを用いて1対多に分解します。

例: 1つの注文は複数の商品を含み、1つの商品は複数の注文に含まれる

[注文] ───< [注文明細] >─── [商品]
             （中間テーブル）

ER図の表記方法（IE記法）¶

よく使われる「IE記法（鳥の足記法）」の読み方：

[A] ──┤      ├──< [B]
  ↑                 ↑
 ゼロまたは1       1以上（多）

記号の意味:
  │ : 1
  < : 多（N）
  ○ : 0（ゼロ）
  ┤ : 1（必須）

ER図の記載例¶

[categories]
     │
     │ 1
     │
     ○──< [products]  ─── < [order_items] >─── [orders] >─── [users]
           商品              注文明細            注文           ユーザー

（読み方）
- 1つのカテゴリは0以上の商品を持つ
- 1つの商品は1以上の注文明細に含まれる
- 1つの注文明細は必ず1つの注文に属する
- 1つの注文は必ず1人のユーザーに属する
- 1人のユーザーは0以上の注文を持つ

実際のER図はDraw.io、dbdiagram.io、A5:SQL Mk-2などのツールで作成します。

ER図作成のポイント¶

  • 全エンティティを含める（図に登場しないエンティティがないか確認）
  • リレーションの方向と多重度を正確に書く（1か多か、0を含むかどうか）
  • 外部キー（FK）を意識する（多側のテーブルに、1側のIDカラムを持たせる）

STEP 3: テーブル定義書の作成¶

テーブル定義書とは¶

各テーブルのカラム名・データ型・制約を詳細に定義した文書です。
開発者がこれを見てCREATE TABLE文を書けるレベルの詳細さが必要です。

定義すべき項目¶

定義項目 説明
カラム名（論理） 日本語の項目名
カラム名（物理） DBの実際のカラム名（英語・スネークケース）
データ型 VARCHAR、INTEGER、DATE など
桁数・精度 VARCHAR(100)、DECIMAL(10,2) など
NOT NULL NULLを許容しないか（必須か）
PRIMARY KEY 主キーかどうか
UNIQUE 値の重複を許さないか
DEFAULT デフォルト値
FOREIGN KEY 外部キー（どのテーブルのどのカラムを参照するか）
概要 そのカラムが何を表しているかの説明

代表的なデータ型¶

型 用途 例
VARCHAR(n) 可変長文字列 氏名、メールアドレス
CHAR(n) 固定長文字列 郵便番号（7桁固定）
TEXT 長い文字列 コメント、本文
INTEGER / BIGINT 整数 数量、ID
DECIMAL(p,s) 小数を含む数値 金額、単価（p:全桁数、s:小数桁数）
BOOLEAN 真偽値 フラグ（有効/無効）
DATE 日付（年月日） 生年月日、注文日
TIMESTAMP 日時（年月日時分秒） 作成日時、更新日時

テーブル定義書の記載例¶

テーブル名（物理）: users¶

テーブル名（論理）: ユーザー
概要: システムを利用するユーザーの情報を管理する

# カラム名（論理） カラム名（物理） 型 桁数 NOT NULL PK UK DEFAULT FK 概要
1 ユーザーID id BIGINT - ○ ○ AUTO 主キー、自動採番
2 氏名 name VARCHAR 100 ○ ユーザーの氏名
3 メールアドレス email VARCHAR 255 ○ ○ ログインIDとして使用
4 パスワードハッシュ password_hash VARCHAR 255 ○ bcryptでハッシュ化して保存
5 ロール role VARCHAR 20 ○ 'general' admin / general / readonly
6 部署 department VARCHAR 100 NULL 任意入力
7 有効フラグ is_active BOOLEAN - ○ true false=論理削除
8 作成日時 created_at TIMESTAMP - ○ CURRENT_TIMESTAMP
9 更新日時 updated_at TIMESTAMP - ○ CURRENT_TIMESTAMP 更新時に自動更新

テーブル名（物理）: orders¶

テーブル名（論理）: 注文
概要: ユーザーが行った注文のヘッダー情報を管理する

# カラム名（論理） カラム名（物理） 型 桁数 NOT NULL PK UK DEFAULT FK 概要
1 注文ID id BIGINT - ○ ○ AUTO 主キー
2 注文番号 order_no VARCHAR 20 ○ ○ ORD-YYYYMMDD-XXXXの形式
3 ユーザーID user_id BIGINT - ○ users.id 注文したユーザー
4 注文ステータス status VARCHAR 20 ○ 'pending' pending/confirmed/shipped/cancelled
5 注文日 order_date DATE - ○ CURRENT_DATE
6 合計金額 total_amount DECIMAL 12,2 ○ 税込合計金額
7 備考 note TEXT - NULL
8 作成日時 created_at TIMESTAMP - ○ CURRENT_TIMESTAMP
9 更新日時 updated_at TIMESTAMP - ○ CURRENT_TIMESTAMP

テーブル名（物理）: order_items¶

テーブル名（論理）: 注文明細
概要: 注文に含まれる商品の明細情報を管理する（ordersとproductsの中間テーブル）

# カラム名（論理） カラム名（物理） 型 桁数 NOT NULL PK UK DEFAULT FK 概要
1 明細ID id BIGINT - ○ ○ AUTO 主キー
2 注文ID order_id BIGINT - ○ orders.id 親の注文
3 商品ID product_id BIGINT - ○ products.id 注文された商品
4 注文時商品名 product_name VARCHAR 200 ○ 注文時点の商品名（スナップショット）
5 注文時単価 unit_price DECIMAL 10,2 ○ 注文時点の単価（スナップショット）
6 数量 quantity INTEGER - ○
7 小計 subtotal DECIMAL 12,2 ○ unit_price × quantity

スナップショットとは？
注文明細の商品名・単価は、注文した「その時点」の値を保存します。
後から商品の価格が変わっても、注文時の価格が正しく保持されます。
productテーブルを参照せず、あえてコピーして持つのがポイントです。

テーブル設計の重要なルール¶

命名規則¶

対象 ルール 例
テーブル名 複数形・スネークケース users, order_items
カラム名 スネークケース user_name, created_at
主キー id id
外部キー 参照テーブル名_id user_id, product_id
フラグ系 is_〜 または has_〜 is_active, has_coupon
日時系 〜_at（TIMESTAMP）、〜_date（DATE） created_at, order_date

必ずすべてのテーブルに入れるカラム¶

id          BIGINT      主キー（AUTO INCREMENT）
created_at  TIMESTAMP   レコード作成日時
updated_at  TIMESTAMP   レコード最終更新日時

created_atとupdated_atは、問題発生時の調査やデータの追跡に必須です。後から追加するのは大変なので、最初から入れましょう。

論理削除 vs 物理削除¶

方式 説明 メリット デメリット
物理削除 DBからレコードを消す（DELETE） シンプル、DBが軽い 削除履歴が残らない、関連データが壊れる可能性
論理削除 削除フラグを立てて非表示にする（is_deleted=true） 履歴が残る、復元可能 DBが肥大化する、クエリが複雑になる

業務データは基本的に論理削除を採用します（注文履歴・ユーザー情報など）。
マスターデータも同様に is_active フラグで有効/無効を管理するのが一般的です。

成果物チェックリスト¶

  • すべての機能・画面に必要なデータがテーブルとして定義されているか
  • ER図にすべてのエンティティが含まれているか
  • リレーションの方向・多重度が正しく記載されているか
  • 主キー・外部キーが定義されているか
  • 全テーブルにcreated_at、updated_atが含まれているか
  • 画面設計の表示項目がデータ設計に存在するか（整合性確認）
  • 命名規則が統一されているか

よくある失敗¶

多対多をそのままテーブルにしようとする¶

多対多のリレーションはDBで直接表現できません。中間テーブルを作って1対多に分解します。

外部キーを設定しない¶

外部キーがないと、存在しないIDを参照するデータが生まれ、データの整合性が崩れます。

画面設計とズレたまま進める¶

「あの画面に表示したい項目がテーブルにない」という問題は非常によくあります。画面設計と並行して整合性を確認しながら進めましょう。

成果物例: 社員管理システム¶

エンティティ一覧¶

# エンティティ名（論理） テーブル名（物理） 概要
1 社員 employees 社員の基本情報を管理する
2 部署 departments 部署のマスター情報を管理する
3 役職マスター positions 役職のマスター情報を管理する
4 社員異動履歴 employee_transfers 社員の部署異動・役職変更の履歴を管理する
5 緊急連絡先 emergency_contacts 社員ごとの緊急連絡先を管理する

ER図¶

[departments]
     │ 1
     │
     ○──< [employees] ──< [employee_transfers] >── [departments]
           社員              異動履歴                部署
             │
             ○──< [emergency_contacts]
                    緊急連絡先

（読み方）
- 1つの部署は0以上の社員を持つ（現在の所属部署）
- 1人の社員は0以上の異動履歴を持つ
- 1つの異動履歴は必ず1人の社員・1つの部署に属する
- 1人の社員は0以上の緊急連絡先を持つ
