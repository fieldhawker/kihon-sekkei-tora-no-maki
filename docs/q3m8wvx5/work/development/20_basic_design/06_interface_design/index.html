外部インターフェース設計¶

このドキュメントの目的¶

外部インターフェース設計は、自システムと外部システム・サービスとのデータのやり取りを定義する作業です。

インターフェース（I/F）とは？
異なるシステム・サービス間でデータをやり取りするための「接続口」のことです。

外部I/Fの種類¶

種類 概要 例
REST API HTTP通信でJSON/XMLデータをやり取りする 外部サービスのAPI呼び出し、他システムへのAPI提供
ファイル連携 CSVやExcelなどのファイルでデータをやり取りする 基幹システムへのデータエクスポート
DB連携 直接データベースに接続してデータをやり取りする 旧システムのDBを参照
メール送信 SMTPサーバーやメール送信APIを使用する 注文確認メール、パスワードリセットメール
帳票出力 PDF生成ライブラリ・サービスを利用する 請求書PDF生成

作業の流れ¶

1. I/F一覧の作成
2. 各I/Fの詳細設計
3. エラー処理方針の定義

STEP 1: I/F一覧の作成¶

洗い出し方¶

要件定義書・業務フローから、外部システムとのデータのやり取りが発生する箇所を抽出します。

確認のポイント: - 他社・他部署のシステムとデータをやり取りするか？ - メール・SMS・プッシュ通知などの送信処理があるか？ - 決済サービス・地図サービスなどの外部APIを使うか？ - CSVやExcelでのデータ入出力があるか？

I/F一覧の記載例¶

I/F ID I/F名 連携先 方向 方式 概要 タイミング
IF-001 メール送信 SendGridメールAPI 送信 REST API 注文確認・パスワードリセットなどのメール送信 イベント発生時
IF-002 決済処理 Stripeカード決済API 送受信 REST API クレジットカード決済の実行・結果受信 注文確定時
IF-003 在庫データ受信 基幹システム 受信 CSVファイル 商品在庫数量の取り込み 毎日06:00
IF-004 売上データ送信 基幹システム 送信 CSVファイル 当日の売上データを基幹システムへ連携 毎日23:00
IF-005 住所検索 郵便番号APIサービス 受信 REST API 郵便番号から住所を自動入力 郵便番号入力時

方向の定義: - 送信: 自システム → 外部システム - 受信: 外部システム → 自システム - 送受信: 双方向（リクエスト送信 → レスポンス受信）

STEP 2: 各I/Fの詳細設計¶

REST API連携の設計¶

設計書の記載例（Stripe決済API）¶

## I/F ID: IF-002
## I/F名: 決済処理（Stripe API）

### 概要
注文確定時にStripeのAPIを呼び出し、クレジットカード決済を実行する。

### 連携先情報
- サービス名: Stripe
- API仕様書: https://stripe.com/docs/api
- 環境:
  - 開発・テスト: https://api.stripe.com（テストモードキー使用）
  - 本番: https://api.stripe.com（本番モードキー使用）

### 認証方式
- Bearer トークン認証（APIキーをAuthorizationヘッダーに設定）

### 呼び出しAPI

#### 1. 決済の実行（Payment Intent作成）

- メソッド: POST
- エンドポイント: /v1/payment_intents
- 呼び出しタイミング: ユーザーが「注文確定」ボタンを押したとき

**リクエスト（送信するデータ）:**
| パラメータ名 | 型 | 必須 | 説明 | 例 |
|---|---|---|---|---|
| amount | integer | ○ | 決済金額（円単位） | 12800 |
| currency | string | ○ | 通貨コード | "jpy" |
| payment_method | string | ○ | Stripe側のカードトークン | "pm_xxxxx" |
| confirm | boolean | ○ | 即時確定するか | true |

**レスポンス（受信するデータ）:**
| パラメータ名 | 型 | 説明 | 例 |
|---|---|---|---|
| id | string | 決済ID | "pi_xxxxx" |
| status | string | 決済ステータス | "succeeded" |
| amount | integer | 決済金額 | 12800 |

### エラー処理
| エラーコード | 意味 | 対応 |
|---|---|---|
| card_declined | カード拒否 | ユーザーに「カードが利用できませんでした」と表示し、再試行を促す |
| insufficient_funds | 残高不足 | ユーザーに「残高が不足しています」と表示する |
| expired_card | カード有効期限切れ | ユーザーに「カードの有効期限が切れています」と表示する |
| network_error | 通信エラー | 3回リトライ後、失敗したら注文を保留し管理者に通知する |

ファイル連携の設計¶

設計書の記載例（在庫データ受信）¶

## I/F ID: IF-003
## I/F名: 在庫データ受信（基幹システム連携）

### 概要
基幹システムが出力したCSVファイルを取り込み、商品の在庫数を更新する。

### 連携方式
- 方式: SFTPファイル転送
- 基幹システムが指定のSFTPサーバーにファイルを配置する
- 自システムのバッチ処理がファイルを取得して処理する

### ファイル仕様

- ファイル名: stock_YYYYMMDD.csv（例: stock_20241201.csv）
- 配置場所: /sftp/incoming/stock/
- 文字コード: Shift-JIS
- 改行コード: CRLF
- 区切り文字: カンマ（,）
- ヘッダー行: あり（1行目）

### データレイアウト

| カラム名 | 論理名 | 型 | 桁数 | 例 |
|---|---|---|---|---|
| product_code | 商品コード | 文字列 | 10 | "PRD-001" |
| stock_quantity | 在庫数量 | 数値 | 10 | 150 |
| updated_at | 更新日時 | 文字列 | 19 | "2024-12-01 06:00:00" |

### 処理内容
1. SFTPサーバーからCSVファイルをダウンロードする
2. ファイルの存在・フォーマットチェックを行う
3. 商品コードをキーにproductsテーブルの在庫数を更新する
4. 処理完了後、ファイルを処理済みフォルダに移動する

### エラー処理
| エラー | 対応 |
|---|---|
| ファイルが存在しない | 管理者にアラートメールを送信し、処理をスキップする |
| フォーマットエラー（列数不正など） | エラーログを記録し、管理者に通知する。データは更新しない |
| 存在しない商品コード | スキップして次のレコードを処理する。スキップ件数をログに記録する |

メール送信の設計¶

設計書の記載例¶

## I/F ID: IF-001
## I/F名: メール送信（SendGrid API）

### 利用APIサービス: SendGrid
### 認証: APIキー（Bearer認証）

### 送信メール一覧

| メールID | メール名 | 送信トリガー | 送信先 |
|---|---|---|---|
| MAIL-001 | 注文確認メール | 注文確定時 | 注文したユーザーのメールアドレス |
| MAIL-002 | パスワードリセットメール | リセット申請時 | 申請したメールアドレス |
| MAIL-003 | 請求書送付メール | 請求書生成時 | 顧客メールアドレス |

### 注文確認メール（MAIL-001）の仕様

**件名:** 【注文確認】ご注文番号: {注文番号}

**本文テンプレート:**
{氏名} 様

ご注文ありがとうございます。 以下の内容でご注文を承りました。

注文番号: {注文番号} 注文日時: {注文日時}

【ご注文内容】 {商品名} × {数量} ¥{金額} ...（明細の繰り返し）

合計金額: ¥{合計金額}（税込）

ご不明な点はサポートまでお問い合わせください。 

**差し込みデータ:**
| プレースホルダー | データソース |
|---|---|
| {氏名} | users.name |
| {注文番号} | orders.order_no |
| {注文日時} | orders.created_at |
| {商品名} | order_items.product_name |
| {数量} | order_items.quantity |
| {金額} | order_items.subtotal |
| {合計金額} | orders.total_amount |

STEP 3: エラー処理方針の定義¶

共通エラー処理方針¶

外部I/F全体で共通するエラー処理の方針を定義します。

## 共通エラー処理方針

### タイムアウト設定
- 接続タイムアウト: 10秒
- 読み取りタイムアウト: 30秒

### リトライ方針
- ネットワークエラー・5xxエラー: 3回リトライ（間隔: 1秒→2秒→4秒の指数バックオフ）
- 4xxエラー（クライアントエラー）: リトライしない

### ログ記録
- 外部I/F呼び出しのリクエスト・レスポンスをすべてログに記録する
- 個人情報・認証情報はマスキングしてログに記録する

### 障害時の対応
- 決済API: 障害時は注文処理を中断し、ユーザーにエラーを表示する
- メール送信API: 障害時は非同期キューに保存し、復旧後に再送する
- ファイル連携: ファイルが来ない場合は管理者にアラートを送り、前回データを維持する

完成判定チェックリスト¶

設計書の見直しを行う際に使用するチェックリストです。全項目にチェックが入った状態を「完成」と判断します。

I/F一覧¶

  • 要件定義書・業務フローに登場する全外部連携がI/F一覧に記載されている
  • I/F IDが採番されており、設計書全体で統一されている
  • 各I/Fに連携先・方向・方式・タイミングが記載されている
  • 同一連携先に複数のI/Fがある場合、それぞれ個別にIDを持っている

各I/Fの詳細設計（REST API）¶

  • 対象APIのエンドポイント・HTTPメソッドが明記されている
  • リクエストのパラメータ（名称・型・必須/任意・説明・例）が定義されている
  • レスポンスのパラメータ（名称・型・説明・例）が定義されている
  • 認証方式（Bearer・APIキー・OAuth等）が明記されている
  • 開発/本番の接続先URLが区別されている
  • 呼び出しタイミング（どの画面操作・バッチが呼ぶか）が明記されている

各I/Fの詳細設計（ファイル連携）¶

  • ファイル名の命名規則が定義されている
  • 文字コード・改行コード・区切り文字が定義されている
  • ヘッダー行の有無が定義されている
  • 全カラムのデータレイアウト（論理名・型・桁数）が定義されている
  • ファイルの配置場所・転送方式（SFTP等）が定義されている
  • 処理完了後のファイルの扱い（移動・削除等）が定義されている

各I/Fの詳細設計（メール送信）¶

  • 送信メール一覧（メールID・件名・送信トリガー・送信先）が定義されている
  • 件名・本文テンプレートが定義されている
  • 差し込みデータのプレースホルダーとデータソースが対応づけられている

エラー処理¶

  • 各I/Fに発生しうるエラーパターンと対応が定義されている
  • タイムアウト値（接続・読み取り）が定義されている
  • リトライ方針（回数・間隔・対象エラー種別）が定義されている
  • 外部サービス障害時にシステム全体への影響を局所化する方針が定義されている
  • エラー発生時のユーザーへの表示内容が定義されている
  • エラー発生時の管理者通知方法が定義されている

セキュリティ¶

  • APIキー・パスワード等の認証情報をソースコードにハードコードしない方針が明記されている
  • 認証情報の管理方法（環境変数・シークレット管理サービス等）が定義されている
  • ログに個人情報・認証情報をそのまま記録しないマスキング方針が定義されている
  • 暗号化通信（HTTPS・LDAPS・SFTP等）が使用されている

運用・保守¶

  • 外部APIのバージョンが明記されており、バージョン固定して呼び出す方針がある
  • ログの記録方針（何をログに残すか）が定義されている
  • 定期実行のI/Fについて実行タイミング・スケジュールが明記されている

よくある失敗¶

外部APIの仕様変更を考慮しない¶

外部サービスのAPIはバージョンアップされることがあります。バージョン指定して呼び出すようにし、変更時の影響範囲を小さくする設計を心がけます。

エラー時の挙動を設計しない¶

「外部APIが落ちたら？」「ファイルが来なかったら？」を設計しないと、本番障害時にシステム全体が止まります。

APIキーをソースコードに直書きする¶

APIキーなどの認証情報は環境変数や秘密管理サービスで管理します。GitHubなどに誤ってコミットすると、セキュリティインシデントになります。

成果物例: 社員管理システム¶

I/F一覧¶

I/F ID I/F名 連携先 方向 方式 概要 タイミング
IF-001 AD認証 社内Active Directory 送受信 LDAP（LDAPS:636） ログイン時のユーザー認証 ログイン時
IF-002 メール送信 SendGrid API 送信 REST API 社員登録通知・月次レポートのメール送信 イベント発生時・バッチ実行時
IF-003 社員データエクスポート 給与計算システム 送信 CSVファイル（SFTP） 月末に社員マスターを給与計算システムへ連携 毎月末日 23:00

I/F詳細設計¶

IF-001: Active Directory認証¶

## I/F ID: IF-001
## I/F名: AD認証（Active Directory LDAP認証）

### 概要
ユーザーがログイン画面に入力したIDとパスワードをActive Directoryに照合する。
AD認証成功後、ADから取得した属性情報をもとにシステム内のロールを決定する。

### 連携先情報
- サーバー: ldaps://ad.example.local:636
- ベースDN: OU=社員,DC=example,DC=local
- 認証方式: LDAPS（TLS暗号化）

### 認証フロー
1. ユーザーが入力したメールアドレス・パスワードでADへバインドを試みる
2. バインド成功 → AD属性（cn, mail, memberOf）を取得する
3. memberOfの値から社内グループを判定し、システムロールにマッピングする
4. セッションにユーザー情報・ロールを設定する

### ADグループ → システムロールのマッピング

| ADグループ | システムロール |
|---|---|
| CN=SysAdmin,OU=Groups,DC=example,DC=local | システム管理者 |
| CN=HRStaff,OU=Groups,DC=example,DC=local | 人事担当者 |
| CN=Manager,OU=Groups,DC=example,DC=local | 管理職 |
| 上記以外の全社員 | 一般社員 |

### エラー処理
| エラー | 処理 | ユーザーへの表示 |
|---|---|---|
| 認証失敗（パスワード誤り） | ログに記録、失敗回数インクリメント | 「IDまたはパスワードが正しくありません」 |
| 5回連続失敗 | アカウントを30分ロック | 「アカウントがロックされました。30分後に再試行してください」 |
| ADサーバー接続失敗 | システム管理者にアラートを送信 | 「現在ログインできません。システム管理者にお問い合わせください」 |

IF-003: 社員データエクスポート（給与計算システム連携）¶

## I/F ID: IF-003
## I/F名: 社員データエクスポート（給与計算システム連携）

### 概要
月末に社員マスターデータをCSVファイルとして生成し、SFTPサーバーに配置する。
給与計算システムがそのファイルを取り込む。

### 連携方式
- 自システムのバッチ処理がCSVを生成しSFTPサーバーにアップロードする
- SFTPサーバー: sftp.example.local / アップロード先: /outgoing/emp/

### ファイル仕様
- ファイル名: emp_master_YYYYMMDD.csv
- 文字コード: Shift-JIS（BOMなし）
- 改行コード: CRLF
- ヘッダー行: あり（1行目）

### データレイアウト

| カラム名 | 論理名 | データソース | 形式 |
|---|---|---|---|
| employee_no | 社員番号 | employees.employee_no | そのまま |
| last_name | 姓 | employees.last_name | そのまま |
| first_name | 名 | employees.first_name | そのまま |
| dept_code | 部署コード | departments.dept_code | そのまま |
| employment_type | 雇用形態 | employees.employment_type | そのまま |
| hire_date | 入社日 | employees.hire_date | YYYYMMDD |

### 対象データ
在籍中の社員（is_active = true）全件

### エラー処理
| エラー | 対応 |
|---|---|
| SFTP接続失敗 | リトライ3回後、システム管理者にアラートメール送信 |
| ファイル生成失敗 | システム管理者にアラートメール送信し、バッチを終了 |

共通エラー処理方針¶

## タイムアウト設定
- LDAPS接続タイムアウト: 5秒
- SendGrid APIタイムアウト: 10秒
- SFTP転送タイムアウト: 60秒

## リトライ方針
- ネットワークエラー・5xxエラー: 3回リトライ（1秒→2秒→4秒の指数バックオフ）
- 4xxエラー（クライアントエラー）: リトライしない

## 認証情報管理
- APIキー・SFTPパスワードは環境変数または AWS Secrets Manager で管理する
- ソースコードへのハードコードは禁止
