バッチ設計¶

このドキュメントの目的¶

バッチ設計は、画面操作を介さずにスケジュールや外部トリガーで自動実行される処理の仕様を設計する作業です。
オンライン処理（画面操作）と異なり、ユーザーが関与しないため、エラー検知・リカバリ・冪等性の設計が特に重要です。

バッチ処理の分類¶

種別 説明 例
定時バッチ 決まった日時にスケジューラーが自動起動 毎日深夜の売上集計、月次請求書生成
日次バッチ 1日1回、業務終了後などに実行 在庫の日次締め処理、ログのアーカイブ
オンデマンドバッチ 外部イベントや手動操作で起動 CSVファイル取り込み、データ移行
連携バッチ 外部システムとのデータ送受信 外部APIからのデータ取得・送信

作業の流れ¶

1. バッチ一覧の作成
2. 各バッチの設計書作成
3. ジョブネット（実行順序）の設計
4. 運用・監視方針の定義

STEP 1: バッチ一覧の作成¶

機能一覧と同様に、システムに存在するすべてのバッチをリストアップします。

バッチ一覧のフォーマット¶

バッチID バッチ名 種別 実行スケジュール 概要 優先度
BAT-SAL-001 日次売上集計 定時 毎日 01:00 前日の注文を集計してsales_summaryに保存 高
BAT-INV-001 月次請求書生成 定時 毎月末日 23:00 当月確定注文から請求書を自動生成 高
BAT-NTF-001 未ログインユーザー無効化 定時 毎日 02:00 90日以上未ログインのユーザーを無効化 中
BAT-IMP-001 商品マスタ取込 オンデマンド 手動起動 CSVから商品情報を一括登録 中

バッチIDの採番ルール¶

BAT - [業務分類コード] - [連番3桁]
例: BAT-SAL-001（SAL = Sales, INV = Invoice, NTF = Notification, IMP = Import）

STEP 2: バッチ設計書の作成¶

Cron式の読み方¶

Cron式は「いつ実行するか」をスケジュール指定するための書式です。

 分  時  日  月  曜日
 *   *   *   *   *
Cron式 意味
0 1 * * * 毎日 01:00
30 23 * * * 毎日 23:30
0 6 1 * * 毎月1日 06:00
0 9 * * 1 毎週月曜日 09:00

確認ツール: crontab.guru を使うとCron式をブラウザ上で確認できます（例: 0 1 * * * を入力すると「every day at 1:00 AM」と表示される）。

設計書テンプレート¶

## バッチID: BAT-〇〇〇-〇〇〇
## バッチ名: 〇〇バッチ

### 目的
（何のためにこのバッチを実行するか）

### 実行スケジュール
- 実行タイミング: （例: 毎日 01:00）
- Cron式: （例: 0 1 * * *）
- タイムアウト: （例: 30分）

### 処理概要
1. ...
2. ...

### 入力
| 種別 | 名称 | 内容 |
|------|------|------|
| テーブル | orders | 処理対象の注文データ |
| ファイル | /import/products.csv | 取込対象のCSVファイル |

### 出力
| 種別 | 名称 | 内容 |
|------|------|------|
| テーブル | sales_summary | 集計結果の保存先 |
| ファイル | /log/bat-sal-001_YYYYMMDD.log | 実行ログ |

### エラー処理
| エラー種別 | 処理内容 | 通知先 |
|-----------|---------|-------|
| 入力ファイルなし | バッチを中断 | 管理者にアラートメール |
| レコード単位の処理失敗 | スキップして継続 | ログに記録 |
| DB接続失敗 | バッチを中断 | 管理者にアラートメール |

### 冪等性
（同じバッチを2回実行した場合の動作を記述）

### 依存関係
（先行して完了している必要があるバッチ）

### リカバリ手順
（失敗した場合の再実行方法）

STEP 3: 設計の重要ポイント¶

冪等性（べきとうせい）¶

バッチ処理は障害や誤操作で再実行されることがあるため、「同じバッチを2回実行しても結果が変わらない」設計が必須です。

冪等性が壊れる悪い例：

-- ❌ 悪い例（毎回INSERTするため、2回実行すると重複レコードが発生する）
INSERT INTO sales_summary (date, amount)
SELECT order_date, SUM(total) FROM orders
WHERE order_date = CURDATE() - INTERVAL 1 DAY
GROUP BY order_date;

冪等性を確保した良い例：

-- ✅ 良い例（既存レコードがあれば更新、なければ挿入）
INSERT INTO sales_summary (date, amount)
SELECT order_date, SUM(total) FROM orders
WHERE order_date = CURDATE() - INTERVAL 1 DAY
GROUP BY order_date
ON DUPLICATE KEY UPDATE amount = VALUES(amount);

-- または処理前に削除してから挿入（DELETE + INSERT）
DELETE FROM sales_summary WHERE date = CURDATE() - INTERVAL 1 DAY;
INSERT INTO sales_summary (date, amount) ...;

エラー処理の設計方針¶

バッチ処理のエラーは大きく2種類に分けて設計します。

エラー種別 方針 具体例
致命的エラー バッチ全体を中断してアラートを通知 DB接続失敗、入力ファイル欠損
非致命的エラー 該当レコードをスキップして処理を継続し、後でまとめて報告 特定レコードの変換エラー

エラー処理の実装例（Python風の擬似コード）：

# ✅ 良い例（レコード単位のエラーはスキップして継続）
errors = []
success_count = 0

for record in target_records:
    try:
        process(record)
        success_count += 1
    except RecordProcessingError as e:
        errors.append({"id": record.id, "reason": str(e)})
        logger.warning(f"Skipped record {record.id}: {e}")

# 処理完了後にまとめてログ出力・通知
logger.info(f"Completed: {success_count} success, {len(errors)} errors")
if errors:
    notify_admin(errors)  # エラー件数が閾値を超えたらアラート

処理対象データの絞り込み¶

大量データを扱うバッチでは、処理対象の範囲を明確に定義します。

手法 説明 向いているケース
日付ベース WHERE created_at >= '昨日' のように日付で絞る 日次・月次バッチ
ステータスベース WHERE status = 'pending' のように処理状態で絞る 未処理レコードを順次処理する場合
バッチ管理テーブル どこまで処理したかを別テーブルで管理 大量データの分割処理、再開可能な設計

バッチ管理テーブルの例：

-- バッチ実行履歴を管理するテーブル
CREATE TABLE batch_executions (
    id           INT PRIMARY KEY AUTO_INCREMENT,
    batch_id     VARCHAR(20) NOT NULL,   -- 'BAT-SAL-001'
    executed_at  DATETIME NOT NULL,
    status       ENUM('running', 'success', 'failed') NOT NULL,
    target_date  DATE,                   -- 処理対象日（日次バッチの場合）
    success_count INT DEFAULT 0,
    error_count   INT DEFAULT 0,
    message      TEXT                    -- エラー詳細など
);

STEP 4: ジョブネット（実行順序）の設計¶

複数のバッチが存在する場合、実行順序の依存関係を整理します。

ジョブネット図の例¶

毎日 01:00
    │
    ▼
[BAT-ORD-001] 注文データ締め処理
    │
    ├──────────────────┐
    ▼                  ▼
[BAT-SAL-001]      [BAT-INV-001]
日次売上集計        日次在庫更新
    │                  │
    └──────┬───────────┘
           ▼
    [BAT-RPT-001]
    日次サマリーレポート送信

ジョブネット定義表¶

ジョブ順 バッチID バッチ名 先行バッチ 失敗時の動作
1 BAT-ORD-001 注文データ締め なし 後続ジョブをすべて停止
2 BAT-SAL-001 日次売上集計 BAT-ORD-001 後続ジョブを停止
2 BAT-INV-001 日次在庫更新 BAT-ORD-001 後続ジョブを停止
3 BAT-RPT-001 日次レポート送信 BAT-SAL-001, BAT-INV-001 独立して継続可能

STEP 5: 運用・監視の設計¶

監視項目¶

監視項目 方法 閾値の例
実行時間 開始〜終了時刻をログに記録 通常の3倍以上かかったらアラート
エラー件数 処理件数・エラー件数をログに記録 エラー率 1% 超えたらアラート
未実行検知 前日の実行履歴を確認 実行記録がなければアラート
処理件数の異常 前日比で大きく乖離していないか確認 前日比 50% 以下/以上でアラート

ログの設計¶

バッチのログには以下を記録します。

[2025-01-31 23:00:01] INFO  BAT-INV-001 開始
[2025-01-31 23:00:01] INFO  対象件数: 1,234件
[2025-01-31 23:00:05] INFO  処理中: 100/1234
[2025-01-31 23:00:15] WARN  レコードID=567 スキップ (金額が0円)
[2025-01-31 23:01:30] INFO  BAT-INV-001 完了 (成功: 1,233件 / エラー: 1件 / 経過時間: 89秒)

アラート通知の方針¶

状況 通知先 通知方法
バッチ正常終了 ログのみ（メール不要） ログファイル
非致命的エラーあり バッチ担当者 メール（日次サマリー）
致命的エラー・未実行 開発チーム・運用担当 メール即時通知 + チャット通知

設計書の記述例¶

日次売上集計バッチ¶

## バッチID: BAT-SAL-001
## バッチ名: 日次売上集計バッチ

### 目的
前日に確定した注文の売上を集計し、日次売上サマリーテーブルに保存する。

### 実行スケジュール
- 実行タイミング: 毎日 01:30（先行バッチ BAT-ORD-001 の完了後）
- Cron式: 30 1 * * *
- タイムアウト: 60分（超過した場合は強制終了してアラート）

### 処理概要
1. batch_executionsテーブルに実行開始レコードを登録する
2. 前日（実行日 - 1日）のステータスが'confirmed'の注文を全件取得する
3. 商品カテゴリ別・日別に売上金額・件数を集計する
4. sales_summaryテーブルに集計結果をUPSERT（存在すれば更新、なければ挿入）する
5. batch_executionsテーブルに完了レコードを更新する（件数・ステータス）
6. 処理結果をログファイルに記録する

### 入力
| 種別 | 名称 | 内容 |
|------|------|------|
| テーブル | orders | ステータス='confirmed'かつ前日分の注文 |
| テーブル | order_items | 注文明細（カテゴリ別集計のため） |
| テーブル | products | 商品カテゴリ情報 |

### 出力
| 種別 | 名称 | 内容 |
|------|------|------|
| テーブル | sales_summary | 日次売上集計結果（日付・カテゴリ別） |
| テーブル | batch_executions | バッチ実行履歴 |
| ファイル | /log/bat-sal-001_YYYYMMDD.log | 実行ログ |

### エラー処理
| エラー種別 | 処理内容 | 通知先 |
|-----------|---------|--------|
| DB接続失敗 | バッチを中断し、batch_executions.status を 'failed' に更新 | 開発チームにアラートメール |
| 集計クエリエラー | バッチを中断する | 開発チームにアラートメール |
| 集計件数が前日比50%以下 | 完了するが警告ログを出力 | 担当者にメール通知 |

### 冪等性
前日分のsales_summaryレコードが既に存在する場合はUPSERTで上書きする。
2回実行しても結果は同じになる。

### 依存関係
- 先行: BAT-ORD-001（注文データ締め処理）が成功していること
- 後続: BAT-RPT-001（日次レポート送信）が本バッチの完了を待つ

### リカバリ手順
1. 失敗原因を調査・修正する
2. 対象日を引数で指定して手動再実行する: `./run_batch.sh BAT-SAL-001 --date=2025-01-31`
3. batch_executionsテーブルで実行結果を確認する

成果物チェックリスト¶

バッチ設計書の完成判断に使うチェックリストです。
すべての項目が ✅ になれば、設計書として完成したとみなせます。

1. バッチ一覧の網羅性¶

  • 機能一覧・要件定義に記載されているすべてのバッチ処理が、バッチ一覧に転記されているか
  • 全バッチに一意のIDが採番ルールに従って付与されているか
  • バッチの種別（定時・日次・オンデマンド・連携）が正しく分類されているか
  • 実行スケジュール（Cron式・実行タイミング）が全バッチで定義されているか

2. 各バッチ設計書の完成度¶

目的・概要

  • バッチが「何のために」「何をする」処理なのか、1〜2文で明確に説明されているか
  • 処理概要がステップ単位で記述されており、実装者が手順を追えるか

入出力

  • 入力データ（テーブル・ファイル・API等）がすべて列挙されているか
  • 出力データ（テーブル・ファイル・ログ等）がすべて列挙されているか
  • 入力データの取得条件（WHERE句の条件・ファイルパス等）が明示されているか

スケジュール・タイムアウト

  • Cron式が記載されており、意図した時刻・頻度と一致しているか
  • タイムアウト時間が設定されており、超過時の対応（強制終了・アラート）が定義されているか

3. 信頼性設計¶

冪等性

  • 同じバッチを2回実行した場合の動作が明示されているか
  • 冪等性の担保方法（UPSERT・DELETE+INSERT・実行済みチェック等）が具体的に記述されているか
  • 再実行しても副作用が発生しないことを確認済みか

エラー処理

  • 致命的エラー（バッチを中断すべきケース）が定義されているか
  • 非致命的エラー（スキップして継続するケース）が定義されているか
  • エラー発生時の通知先・通知方法が定義されているか
  • エラー発生時にトランザクション（全件ロールバック vs 部分コミット）の方針が決まっているか

処理件数の検証

  • 処理対象件数が0件の場合の動作・警告が定義されているか
  • 前回実行比で件数が異常に少ない・多い場合の検知方法が定義されているか

4. 依存関係・ジョブネット¶

  • 先行して完了している必要があるバッチが明示されているか
  • 後続バッチへの影響（本バッチが失敗した場合に後続を止めるか）が定義されているか
  • ジョブネット図またはジョブネット定義表で、全体の実行順序が可視化されているか
  • 並列実行可能なバッチと直列実行が必要なバッチが整理されているか

5. 運用・監視¶

ログ

  • ログに記録する内容（開始・終了・処理件数・エラー詳細）が定義されているか
  • ログの出力先（ファイル・DBテーブル等）が定義されているか

監視・アラート

  • 実行時間の監視閾値（通常の何倍でアラート等）が定義されているか
  • 未実行検知の仕組み（前日の実行履歴確認等）が定義されているか
  • アラートの通知先・通知方法（メール・チャット等）が状況別に定義されているか

リカバリ

  • 失敗時の再実行手順（コマンド例・引数の指定方法）が記載されているか
  • 手動で対象日・対象範囲を指定して再実行できるか
  • 再実行後の結果確認方法が記載されているか

6. 横断的な確認¶

  • バッチ設計書の記述内容が、画面設計・テーブル設計と矛盾していないか
  • バッチが参照・更新するテーブルが、テーブル設計書に定義済みであるか
  • 処理量の見積もり（対象データ件数・想定実行時間）が記載されているか
  • 深夜バッチ等の実行ウィンドウ内に、全ジョブが完了できる見通しがあるか

よくある失敗¶

冪等性を考慮していない¶

再実行したら集計値が二重になった、という事故が起きやすい。設計の段階でUPSERT・DELETE+INSERT・実行済みチェックなど冪等性の担保方法を決める。

エラー時にロールバックするか継続するかが曖昧¶

1,000件中1件が失敗したとき、999件を反映するのか、全件ロールバックするのかを事前に決めておかないと、開発時・障害時に判断が揺れる。

タイムアウト設計がない¶

データ量が増えてバッチが終わらなくなっても気づけない。実行時間の上限とタイムアウト時の対応を設計に含める。

依存関係を暗黙にしている¶

「Aの後にBを実行する」という手順が担当者の頭の中にしかない状態は危険。ジョブネット図か依存関係テーブルで明文化する。

処理件数の検証がない¶

バッチが正常終了しても処理件数が0件だったとき（対象データの取得条件ミスなど）を検知できない設計になっている。処理件数が異常に少ない場合の警告を組み込む。
