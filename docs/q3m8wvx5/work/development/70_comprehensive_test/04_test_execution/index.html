STEP 4: テストの実施¶

このSTEPの目的¶

作成した業務シナリオを1件ずつ実施し、結果を正確に記録すること。

テストの「実施」と「記録」は同時に行います。「後でまとめて記録しよう」は禁止です。
記録が曖昧だと、不具合の再現・報告・再テストがすべて困難になります。

テスト実施の基本的な流れ¶

1. テストケースを1件選ぶ
   ↓
2. 前提条件を確認・設定する
   ↓
3. 操作手順を1ステップずつ実施する
   ↓
4. 期待する結果と実際の結果を比較する
   ↓
5. 結果を記録する（OK / NG）
   ↓
6. NGの場合は証跡を取得して次のSTEP 5（不具合票）へ
   ↓
7. 次のテストケースへ進む

結果の記録ルール¶

OK（合格）の場合¶

記録項目 記録内容
結果 OK
実施日時 YYYY/MM/DD HH:MM
実施者 氏名
備考 操作に迷った点・マニュアルと実際の画面の差異など

NG（不合格）の場合¶

記録項目 記録内容
結果 NG
実施日時 YYYY/MM/DD HH:MM
実施者 氏名
実際の結果 期待する結果と何が違ったか（エラーメッセージ・画面の状態など）
再現手順 同じNGを再現するための操作手順
証跡 スクリーンショット・ログのファイル名
業務への影響 この問題でどの業務が影響を受けるか

NGが出たら、まずリーダーに報告する。 自己判断で「軽微だから後回し」にしない。

テストケースID 業務シナリオ名 結果 実施日時 実施者 実際の結果（NG時） 不具合票ID 備考

運用テスト特有の確認ポイント¶

操作性・マニュアルの確認¶

「システムが動く」だけでなく「業務担当者がスムーズに使える」かを確認します。

確認ポイント 確認方法
マニュアル通りに操作できるか 操作マニュアルを見ながら操作し、手順通りに進められるか確認する
迷う箇所はないか 「次に何をすれば良いか」が画面を見て分かるか
エラーメッセージが分かりやすいか エラーが出たとき、業務担当者が次に何をすれば良いかが分かるか
処理時間が許容範囲か 処理が完了するまでの時間が業務として許容できる範囲か

気になった点はNG扱いでなくても「備考」に記録しておく。マニュアルや画面の改善に役立てる。

性能確認の記録方法¶

処理時間の計測方法：

1. 処理開始直前の時刻を記録する
2. 処理を実行する
3. 処理完了（画面の表示・結果の出力）後の時刻を記録する
4. 差分を計算して「処理時間」として記録する
テストケースID 計測箇所 処理時間 合否基準 結果
OT-020 受注一覧（1万件）表示 2.8秒 3秒以内 OK
OT-021 月末集計バッチ実行 5分12秒 10分以内 OK

バッチ処理の確認方法¶

確認項目 確認内容
実行時刻 指定された時刻にバッチが実行されたか
実行結果 バッチログにERROR・WARNINGが出ていないか
データの正確性 バッチ実行後のデータが期待通りに更新されているか

バッチログの確認例:

2024-03-15 02:00:00 INFO  バッチ開始: 日次在庫集計
2024-03-15 02:03:45 INFO  処理件数: 10,000件
2024-03-15 02:03:46 INFO  バッチ正常終了

上記のようにINFOのみで、ERRORがなければOK。

証跡（エビデンス）の取得方法¶

スクリーンショット¶

  • NGになった画面、エラーメッセージの画面を必ず取得する
  • ファイル名は「テストケースID + 内容」で管理する（例: OT-002_error_message.png）
  • 日時が確認できるよう、画面上の時刻や操作ログも含めて取得する

ログファイル¶

  • NGが発生した時刻前後のログを取得する
  • ログファイル名・取得方法はリーダーに確認する
  • ファイル名は「テストケースID + 取得日時」で管理する（例: OT-002_20240315_1100.log）

1日の終わりにやること¶

  1. その日に実施したテストケースの結果を確認する
    全件にOK/NGが記録されているか確認する。

  2. NGのテストケースがあれば不具合票を作成する（STEP 5参照）
    当日中に不具合票を作成してリーダーに報告する。

  3. 翌日の実施予定を確認する
    スケジュールと残りのテストケースを確認し、遅延が発生しそうな場合は早めにリーダーに相談する。

  4. テストデータのリセットが必要か確認する
    翌日のテストのために、テストデータを初期化する必要があれば実施する。

よくある問題と対処法¶

「前提条件を作るのに時間がかかって、テストが進まない」
データ準備の手順が不明確な場合は、STEP 3に戻ってデータ準備手順を整理する。繰り返し使うデータは初期化手順をまとめておく。

「NGが出たが、環境の問題なのかシステムの問題なのか分からない」
自己判断せず、リーダーに状況を報告して判断を仰ぐ。「再現手順」を丁寧に記録して共有する。

「テストケースの実施が遅れそうな場合」
スケジュールとの差異が出始めたら早めにリーダーに相談する。後回しにすると、リリース判断に影響する。

次のSTEPへ進む前のチェックリスト¶

  • 全テストケースを実施した
  • 全件にOK/NGと実施日時・実施者を記録した
  • NGのテストケースには証跡（スクリーンショット・ログ）を取得した
  • 操作に迷った点・マニュアルの不備も備考に記録した
  • 当日のNGはリーダーに報告した

NGのテストケースがある場合 → STEP 5: 不具合の記録・報告 へ進む
全件OKの場合 → STEP 7: 移行・引き継ぎ準備の確認 へ進む
