STEP 2: テストシナリオの作成¶

このSTEPの目的¶

「業務として成立するか」を確認するためのシナリオ（テストケース）を作成すること。

運用テストのシナリオは「開発者が機能を確認する」のではなく、「業務担当者が実際にやる操作の流れ」を再現します。
「システムが動くか」ではなく「業務として問題なく使えるか」を確認することを意識して作成してください。

テストシナリオとテストケースの違い¶

用語 説明 例
テストシナリオ 業務フロー全体の流れ 「受注から請求書発行までの一連の業務フロー」
テストケース シナリオの中の具体的な確認項目 「受注登録時に在庫不足の場合、エラーメッセージが表示されること」

1つのテストシナリオに、複数のテストケースが紐づきます。

テストケース一覧のテンプレート¶

以下のテンプレートを使ってテストケースを作成します。

テストケースID 業務シナリオ名 確認観点 前提条件 操作手順 期待する結果 関連要件ID 結果 備考
OT-001

各項目の説明¶

項目 説明 記載例
テストケースID 管理用の連番（OT = Operational Test） OT-001, OT-002
業務シナリオ名 どの業務フローを確認するか 受注登録～在庫確認フロー
確認観点 正常系・異常系・性能・バッチなどの種類 正常系 / 異常系 / 性能
前提条件 テスト開始前に用意すべきデータや状態 在庫数が10件登録されている状態
操作手順 実際の操作の流れ（番号付きで具体的に） 1.受注画面を開く 2.商品コードを入力する ...
期待する結果 何が起きれば合格か 「受注完了」画面が表示され、在庫数が9に更新される
関連要件ID 対応する要件定義書の項目番号 REQ-101
結果 テスト実施後に記入（OK/NG） （実施後に記入）
備考 気になった点・マニュアルの不備など （実施後に記入）

テストシナリオの種類と作成のポイント¶

1. 正常業務フロー（必須）¶

内容: 日常的な業務の流れを最初から最後まで通して確認する

作成のポイント: - 業務フロー図や操作マニュアルを参照して、実際の業務手順通りに作成する - 「最初から最後まで通して動くか」を1つのシナリオとして確認する（分割しすぎない） - 業務担当者が実際に行う操作の順番と同じにする

記載例:

テストケースID 業務シナリオ名 確認観点 前提条件 操作手順 期待する結果 関連要件ID
OT-001 受注から出荷指示までの業務フロー 正常系 商品コードAA001の在庫が10件ある状態 1. 受注画面を開く 2. 顧客コード・商品コード・数量（5）を入力する 3. 「登録」ボタンを押す 4. 在庫確認画面で在庫数（5）を確認する 5. 出荷指示ボタンを押す 受注番号が発行され、在庫が10→5に更新される。出荷指示一覧に表示される REQ-101

2. 例外・エラー対応（必須）¶

内容: 業務上で発生しうるエラーや例外ケースが適切に処理されるかを確認する

作成のポイント: - 「入力ミスをしたとき」「データが存在しないとき」「在庫不足のとき」など、業務担当者がよく経験するエラーをリストアップする - エラーメッセージが業務担当者にとってわかりやすい内容かを確認する - エラーが起きたあとの「回復手順」も確認する

記載例:

テストケースID 業務シナリオ名 確認観点 前提条件 操作手順 期待する結果 関連要件ID
OT-010 在庫不足時のエラー処理 異常系 商品コードBB001の在庫が2件ある状態 1. 受注画面を開く 2. 商品コードBB001・数量5を入力する 3. 「登録」ボタンを押す 「在庫が不足しています（在庫数：2件）」のエラーメッセージが表示される。受注は登録されない REQ-105

3. 大量データ・負荷（要確認）¶

内容: 本番に近いデータ量・同時アクセス数でも問題なく動作するかを確認する

作成のポイント: - 本番で想定される最大データ量・同時ユーザー数をテスト計画書で確認する - 「処理が遅すぎて業務が止まらないか」を業務の許容範囲（例: 3秒以内）と比較して確認する - 月末・年度末など、特に負荷が高くなるタイミングを想定する

記載例:

テストケースID 業務シナリオ名 確認観点 前提条件 操作手順 期待する結果 関連要件ID
OT-020 月末大量受注処理の性能確認 性能 受注データが1万件登録されている状態（本番相当のデータ量） 1. 受注一覧画面を開く 2. 当月の受注一覧を表示する 一覧が3秒以内に表示される REQ-201

4. バッチ・夜間処理（該当する場合）¶

内容: 定期バッチ・夜間処理が正しく動作するかを確認する

作成のポイント: - バッチの実行スケジュール（実行時刻・頻度）を確認する - バッチ実行後のデータが正しく更新されているかを確認する - バッチのログが正しく出力されているか確認する

記載例:

テストケースID 業務シナリオ名 確認観点 前提条件 操作手順 期待する結果 関連要件ID
OT-030 日次在庫集計バッチの動作確認 バッチ処理 当日の受注データが20件登録されている状態 1. 日次集計バッチを実行する（または指定時刻まで待機する） 2. バッチ実行ログを確認する 3. 在庫集計画面を開き、集計結果を確認する バッチが正常終了（ログにERRORなし）し、集計結果が受注データと一致している REQ-301

5. 障害・復旧（必須）¶

内容: 障害が発生したとき、復旧手順が正しく機能するかを確認する

作成のポイント: - 障害シナリオは「どんな障害を想定するか」をリーダーと確認してから作成する - 復旧手順書が存在するか、手順書通りに操作して復旧できるかを確認する - 復旧後のデータに欠損や不整合がないかを確認する

記載例:

テストケースID 業務シナリオ名 確認観点 前提条件 操作手順 期待する結果 関連要件ID
OT-040 サーバー再起動後のデータ整合性確認 障害復旧 受注データが5件登録された後にサーバーを再起動する 1. 受注データを5件登録する 2. サーバーを再起動する 3. 再起動後に受注一覧を開く 再起動前に登録した5件の受注データがすべて正しく表示される REQ-401

6. 引き継ぎ・移行（該当する場合）¶

内容: 旧システムからのデータ移行・切り替えが正しく機能するかを確認する

記載例:

テストケースID 業務シナリオ名 確認観点 前提条件 操作手順 期待する結果 関連要件ID
OT-050 旧システムの顧客データ移行確認 データ移行 旧システムの顧客データ（100件）を移行ツールで変換・移行済み 1. 新システムの顧客一覧を開く 2. 旧システムに存在した顧客コードで検索する 旧システムと同じ顧客情報（顧客名・住所・電話番号）が表示される REQ-501

シナリオ作成時の注意点¶

業務担当者に確認してもらう¶

作成したシナリオは、実際に業務を行う業務担当者に確認してもらいます。
「この操作手順は実際の業務と合っているか」「この確認項目で抜け漏れはないか」を聞く。

正常系だけで終わらせない¶

業務では必ずミスや例外が発生します。「エラーが出たときに業務担当者が適切に対処できるか」を確認する異常系のシナリオも必ず作成する。

前提条件を具体的に書く¶

「適切なデータを用意する」ではなく「商品コードAA001の在庫が10件ある状態」のように、誰でも同じ状態を再現できるように具体的に記載する。

操作手順は番号付きで書く¶

「受注登録をする」ではなく「1. 受注画面を開く 2. 顧客コードXXXを入力する...」のように、一つひとつの操作を番号付きで記載する。

完成チェックリスト¶

  • 正常業務フローのシナリオを作成した
  • 異常系・エラー対応のシナリオを作成した
  • 性能・負荷のシナリオを確認した（対象外の場合は理由を記録）
  • バッチ処理のシナリオを確認した（対象外の場合は理由を記録）
  • 障害復旧のシナリオを作成した
  • 移行・引き継ぎのシナリオを確認した（対象外の場合は理由を記録）
  • 業務担当者またはリーダーにシナリオを確認してもらった
  • 各テストケースに「前提条件」「操作手順（番号付き）」「期待する結果」が具体的に記載されている
  • 関連する要件IDが記載されている

上記がすべてチェックできたら → STEP 3: テスト環境・テストデータの準備 へ進む

成果物例：社員管理システム テストケース一覧（抜粋）¶

テストケースID 業務シナリオ名 確認観点 前提条件 操作手順 期待する結果 関連要件ID 結果 備考
OT-001 社員情報の新規登録 正常系 部署マスタに「営業部（dept001）」が登録されている状態 1. 社員管理メニューを開く 2. 「新規登録」ボタンをクリック 3. 社員番号「E1001」・氏名「鈴木 一郎」・部署「営業部」・入社日「2026/04/01」を入力 4. 「登録」ボタンをクリック 「社員情報を登録しました」メッセージが表示され、社員一覧に「E1001 鈴木 一郎」が追加される REQ-101
OT-002 社員情報の更新（部署異動） 正常系 社員「E1001 鈴木 一郎（営業部）」が登録済みの状態 1. 社員一覧で「E1001」を検索 2. 対象社員の「編集」をクリック 3. 部署を「営業部」→「人事部」に変更 4. 「更新」ボタンをクリック 「社員情報を更新しました」メッセージが表示され、社員一覧の部署が「人事部」に変わっている REQ-102
OT-003 退職社員の削除（論理削除） 正常系 社員「E1001 鈴木 一郎」が登録済みの状態 1. 社員一覧で「E1001」を検索 2. 「退職処理」ボタンをクリック 3. 退職日「2026/03/31」を入力 4. 「確定」ボタンをクリック 社員一覧から「E1001」が非表示になる。退職済み一覧には表示される。データは物理削除されない REQ-103
OT-010 重複社員番号での登録エラー 異常系 社員番号「E1001」がすでに登録済みの状態 1. 新規登録画面を開く 2. 社員番号「E1001」・氏名「田中 花子」を入力 3. 「登録」ボタンをクリック 「社員番号 E1001 はすでに使用されています」というエラーメッセージが表示され、登録されない REQ-201
OT-011 必須項目未入力での登録エラー 異常系 なし 1. 新規登録画面を開く 2. 氏名のみ「佐藤 次郎」を入力（社員番号・部署・入社日は未入力） 3. 「登録」ボタンをクリック 「社員番号は必須項目です」「部署は必須項目です」「入社日は必須項目です」の各エラーメッセージが表示される REQ-202
OT-020 社員一覧の性能確認（大量データ） 性能 社員データが5,000件登録されている状態（本番想定） 1. 社員一覧画面を開く 2. 絞り込み条件なしで全件表示する 一覧が3秒以内に表示される REQ-301
OT-030 月次社員レポートバッチの動作確認 バッチ処理 テスト環境で当月の社員データが登録済みの状態 1. 月次レポートバッチを手動実行する 2. バッチ実行ログを確認する 3. 出力フォルダにレポートファイルが生成されることを確認する バッチが正常終了（ログにERRORなし）し、レポートファイルが生成されている。レポート内の社員数が社員一覧の件数と一致している REQ-401
OT-040 サーバー再起動後のデータ整合性 障害復旧 社員情報を10件登録した後にサーバーを再起動する 1. 社員情報を10件登録する 2. アプリケーションサーバーを再起動する 3. 社員一覧を開く 再起動前に登録した10件がすべて正しく表示される REQ-501
OT-050 旧システムの社員データ移行確認 データ移行 旧システムの社員データ（3,000件）を移行ツールで変換・移行済みの状態 1. 社員一覧を開く 2. 旧システムに存在した社員番号「OLD-001」で検索する 旧システムと同じ社員情報（氏名・部署・入社日）が正しく表示される REQ-601
