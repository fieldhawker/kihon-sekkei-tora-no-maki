STEP 5: 不具合の記録・報告¶

このSTEPの目的¶

NGになった内容を正確に記録し、担当者・リーダーに適切に報告すること。

不具合の記録が曖昧だと、修正担当者が原因を調査できず、修正に時間がかかります。
「何が起きたか・いつ・どの操作で・どんな影響があるか」を具体的に記録することが最重要です。

不具合票の書き方¶

不具合票テンプレート¶

【不具合票】

不具合ID     : BUG-001
発生日時     : YYYY/MM/DD HH:MM
発生環境     : ステージング環境（v1.2.3）
報告者       : 氏名

業務シナリオ : どの業務フローを実施中に発生したか
テストケースID: OT-XXX

■ 発生手順（再現手順）
1. （操作1）
2. （操作2）
3. （操作3）→ ここで問題が発生

■ 期待した結果
（本来どうなるべきだったか）

■ 実際の結果
（実際に何が起きたか・エラーメッセージの内容など）

■ 業務への影響
（この問題でどの業務が止まるか・どの業務担当者が影響を受けるか）

■ 証跡
スクリーンショット: BUG001_error.png
ログファイル: BUG001_20240315_1100.log

■ 優先度
高 / 中 / 低
（優先度の根拠：）

■ 対応状況
未対応 / 調査中 / 修正中 / 修正完了 / 再テスト完了

不具合票の記載例¶

【不具合票】

不具合ID     : BUG-001
発生日時     : 2024/03/15 11:05
発生環境     : ステージング環境（v1.2.3）
報告者       : 田中 一郎

業務シナリオ : 受注登録～在庫確認フロー
テストケースID: OT-002

■ 発生手順（再現手順）
1. ブラウザで受注画面（https://stg.example.com/order）を開く
2. 顧客コード「C001」・商品コード「BB001」・数量「5」を入力する
3. 「登録」ボタンをクリックする
→ エラーメッセージが表示される

■ 期待した結果
「在庫が不足しています（在庫数：2件）」という日本語のエラーメッセージが表示される

■ 実際の結果
「Insufficient stock (available: 2)」という英語のエラーメッセージが表示された

■ 業務への影響
受注担当者（業務担当者）がエラーの内容を理解できない。
在庫不足の状況で適切な対処（代替品の案内・納期調整）ができなくなる可能性がある。

■ 証跡
スクリーンショット: BUG001_error_message.png
ログファイル: BUG001_20240315_1105.log

■ 優先度
中
（優先度の根拠：業務は止まらないが、業務担当者の対応に支障が出る。リリース前に修正が望ましい）

■ 対応状況
未対応

優先度の判断基準¶

優先度 基準 例
高 業務が止まる・データが壊れる・セキュリティに問題がある 受注登録がエラーで完了しない、データが欠損する
中 業務に支障が出るが回避策がある エラーメッセージが英語、操作手順が分かりにくい
低 軽微な表示崩れ・操作性の問題で業務への影響が小さい ボタンの位置がずれている、表記ゆれ

迷ったら「高」にして報告する。 リーダーと一緒に優先度を判断するのが正しいやり方です。
「大した問題じゃない」と自己判断して低くするのは危険です。

報告のタイミングと方法¶

即時報告が必要な場合（優先度：高）¶

  • 業務が止まる問題を発見したとき
  • データが壊れる・消える問題を発見したとき
  • セキュリティに関わる問題を発見したとき

→ 発見した時点でリーダーにすぐ声をかけ（またはメッセージを送り）、不具合票を送付する

当日中に報告（優先度：中・低）¶

  • その日のテストが終わったタイミングで、不具合票をまとめてリーダーに送付する

不具合管理台帳¶

発生した不具合を一覧で管理します。

不具合ID テストケースID 優先度 発生日時 内容（概要） 担当者 対応状況 修正完了日 再テスト結果
BUG-001 OT-002 中 2024/03/15 エラーメッセージが英語で表示される 鈴木 修正中
BUG-002 OT-010 高 2024/03/16 在庫0件でも受注登録ができてしまう 佐藤 修正完了 2024/03/18 OK

よくある失敗と注意点¶

再現手順が不明確で修正担当者が困る¶

悪い例: 

3. ボタンを押したらエラーが出た

良い例: 

3. 商品コード「BB001」・数量「5」を入力した状態で「登録」ボタンをクリックした
→ 画面上部に英語のエラーメッセージ「Insufficient stock (available: 2)」が表示された

「誰でも同じ状況を再現できる」だけの情報を書く。

「実際の結果」があいまい¶

悪い例: 

■ 実際の結果
エラーが出た

良い例: 

■ 実際の結果
「Insufficient stock (available: 2)」という英語のエラーメッセージが赤字で画面上部に表示された。
受注データは登録されていない（受注一覧に表示されないことを確認）。

エラーメッセージの内容・発生箇所・その後の状態を具体的に書く。

業務への影響を書かない¶

「業務への影響」は修正の優先度を判断するために必要です。
「なんかエラーが出た」ではなく「〇〇業務が止まる・〇〇担当者が対処できない」と書く。

次のSTEPへ進む前のチェックリスト¶

  • NGのテストケースすべてに不具合票を作成した
  • 不具合票に「再現手順」「期待した結果」「実際の結果」「業務への影響」「証跡」が記載されている
  • 優先度を設定し、リーダーに確認してもらった
  • 不具合管理台帳を最新の状態に更新した
  • 優先度「高」の不具合は即時報告した

次のSTEP → STEP 6: 修正・再テスト へ進む

成果物例：社員管理システム 不具合票¶

不具合票 BUG-001¶

【不具合票】

不具合ID     : BUG-001
発生日時     : 2026/06/03 13:35
発生環境     : ステージング環境（v1.0.0-rc2）
報告者       : 高橋 誠

業務シナリオ : 重複社員番号での登録エラー
テストケースID: OT-010

■ 発生手順（再現手順）
1. ブラウザで社員管理画面（https://stg.hr-system.example.com/employee/new）を開く
2. 社員番号「E9999」（登録済みデータ）・氏名「テスト 太郎」・部署「営業部」・入社日「2026/06/01」を入力する
3. 「保存」ボタンをクリックする
→ エラーメッセージが表示されるが、内容が汎用的すぎる

■ 期待した結果
「社員番号 E9999 はすでに使用されています」という具体的なエラーメッセージが表示され、登録されない

■ 実際の結果
「登録に失敗しました」という汎用メッセージが赤字で表示された。
社員番号の重複が原因であることがメッセージから判断できない。
登録は行われていない（社員一覧に追加されていないことを確認）。

■ 業務への影響
人事担当者が社員番号の重複を原因と特定できず、別の項目を修正しようとするなど、対処に時間がかかる可能性がある。
業務は継続可能だが、操作に支障が出る。

■ 証跡
スクリーンショット: OT-010_error.png

■ 優先度
中
（優先度の根拠：登録自体は失敗しており業務は止まらないが、担当者が原因を特定できないため業務効率が下がる。リリース前に修正が望ましい）

■ 対応状況
未対応

不具合票 BUG-002¶

【不具合票】

不具合ID     : BUG-002
発生日時     : 2026/06/04 14:35
発生環境     : ステージング環境（v1.0.0-rc2）
報告者       : 高橋 誠

業務シナリオ : 月次社員レポートバッチの動作確認
テストケースID: OT-030

■ 発生手順（再現手順）
1. テスト環境に社員データ50件が登録されている状態を確認する
2. 月次レポートバッチを手動実行する（コマンド: /opt/hr-system/batch/monthly_report.sh）
3. バッチ完了後に出力フォルダ（/output/reports/）のレポートファイルを確認する
4. レポートファイル内の「社員総数」欄を確認する

■ 期待した結果
レポート内の社員総数が「50」と表示される

■ 実際の結果
レポート内の社員総数が「49」と表示された。1件少ない。
バッチログにはERRORの出力はなく、正常終了のログが出力されていた。

■ 業務への影響
月次レポートに基づいて人員計画・採用計画を立てる際に、実際の社員数と異なるデータで判断してしまう可能性がある。
データの正確性に直結するため、リリース前に必ず解消が必要。

■ 証跡
バッチログ: OT-030_20260604_1430.log
レポートファイル: OT-030_monthly_report_20260604.xlsx

■ 優先度
高
（優先度の根拠：レポートの数値が実態と異なるため、業務判断に誤りが生じる可能性がある。データ正確性に関わる問題のためリリース前解消必須）

■ 対応状況
未対応
