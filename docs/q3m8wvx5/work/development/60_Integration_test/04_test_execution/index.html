STEP 4: テストの実施・結果記録¶

このSTEPの目的¶

作成したテストケースを1件ずつ実施し、機能間の連携結果を正確に記録します。
「なんとなく動いた」ではなく「テストケースに記載した期待する結果と一致したかどうか」を判断することが大切です。

テスト実施の基本ルール¶

1件ずつ、順番に実施する¶

テストケースは設計した順番で実施してください。
前のテストの結果が次のテストの前提条件になっている場合があります。

【実施の流れ（1件分）】

1. テストケースの内容を読む（前提条件・手順・期待する結果を確認）
2. 前提条件が整っているか確認する
3. 手順通りに操作する
4. 期待する結果と実際の結果を比較する
5. OK / NG を記録する
6. NG の場合は詳細を記録してから次のケースへ進む

重要: NGが出ても、その場でコードを修正しようとしないでください。
まず記録・報告を行い、修正はSTEP 6で行います。

結果の判断基準¶

判定 基準
OK（合格） 期待する結果の全項目が一致した
NG（不合格） 期待する結果のいずれか1つでも一致しなかった
保留 環境の問題等でテストが実施できなかった（理由を記録する）

「なんとなく動いた気がする」はOKではありません。
期待する結果に書かれた全項目を1つずつ確認し、全部一致したときだけOKと記録します。

結果の記録方法¶

OKの場合¶

記録すること 方法
テストケースIDに「OK」を記入する テストケース管理表の結果欄を更新
実施日時・担当者名を記入する
証跡が必要な場合はスクリーンショットを保存する ファイル名に「テストケースID_OK」を付ける

NGの場合¶

NGが発生したときは以下を必ず全部記録してください。1つでも欠けると原因調査が難しくなります。

記録する内容 具体例
発生したテストケースID IT-003
発生日時 2026-05-25 14:32
再現手順（どの操作でNGになったか） 「手順3の注文ボタン押下後にエラー画面が表示された」
期待した結果 「注文完了画面が表示されること」
実際の結果 「500エラー画面が表示された」
エラーメッセージ（全文） 「Internal Server Error: Cannot read property 'stock' of undefined」
影響を受けた機能 「注文機能・在庫管理機能」
証跡ファイル IT-003_NG_20260525_1432.png、server.log

スクリーンショット・ログの取り方¶

スクリーンショット¶

NGが発生したときは、エラーが表示された画面のスクリーンショットを必ず保存します。

ファイル名のルール: 

{テストケースID}_{OK/NG}_{日付}_{時刻}.png
例: IT-003_NG_20260525_1432.png

ログの保存¶

エラーが起きたとき、システムのログにヒントが残っていることがほとんどです。
NGが発生したら必ずログを確認・保存してください。

よく見るログの場所:

ログの種類 場所・取得方法
アプリケーションログ サーバーの /logs/app.log など（プロジェクトにより異なる）
DBログ DB管理ツールのクエリログ
ブラウザのコンソールログ F12（開発者ツール）→ Console タブ
ネットワークログ F12 → Network タブ

ログを保存するときのポイント: - NGが発生した直後のログを保存する（時間が経つとログが流れる） - ログはテキストファイルでも保存しておく（スクリーンショットだけだと検索しにくい） - 「どの機能からどの機能への処理でエラーが起きたか」が分かる部分を含める

「どこの繋ぎ目でNGになったか」を意識する¶

結合テストのNGは「どの機能が悪いか」がすぐにわからないことがあります。
記録するときは「どこからどこへの連携でNGになったか」を意識して書きます。

記録の例:

【良い記録】
「注文登録機能が注文データをDBに保存した後、在庫管理機能に通知する処理でエラーが発生した。
在庫数は更新されていない。注文データはDBに保存されている状態。」

【悪い記録】
「注文ボタンを押したらエラーになった。」

テスト実施中の確認方法¶

DBの状態を確認する方法¶

「DBに期待通りのデータが書き込まれているか」を確認するためにDBクライアントを使います。

例（在庫数の確認）: 

SELECT product_id, stock_qty FROM products WHERE product_id = 'P001';

プロジェクトで使用するDBクライアントツールと接続先はリーダーに確認してください。

メール送信の確認方法¶

テスト環境では、実際のメールアドレスに送信すると本番同様の副作用が起きる場合があります。
テスト環境ではMailHog・Mailtrap等のメールキャプチャツールを使うことが多いです。
接続先URLはリーダーに確認してください。

1日の終わりにやること¶

  • 本日実施したテストケースの結果を全件記録したか確認する
  • NGが発生した場合は不具合票に記録したか確認する（STEP 5）
  • 証跡（スクリーンショット・ログ）が保存されているか確認する
  • 翌日の実施予定テストケースを確認しておく

このSTEPの完了条件¶

  • 担当する全テストケースを実施した
  • 全件にOK/NGを記録した
  • NGの場合は詳細・証跡を記録した
  • 保留がある場合は理由を記録し、リーダーに報告した

次のSTEP¶

→ STEP 5: 不具合の記録・報告

NGテストケースの詳細記録例（IT-003）¶

発生状況:

IT-003「社員登録〜重複社員番号エラー」実施中、手順4の「登録」ボタン押下後に以下が発生。

確認項目 期待した結果 実際の結果 判定
エラーメッセージの表示 「この社員番号はすでに登録されています」と表示 エラーメッセージなしで登録完了画面に遷移 NG
DBへの未保存 重複レコードが保存されない 重複レコードがDBに保存された NG
メール非送信 メールが送信されない 重複した社員番号でメールが2件送信された NG

ブラウザコンソールのエラーログ（抜粋）: 

POST /api/employees 200 OK
Response: {"success": true, "employee_id": "EMP-001"}

サーバーログ（抜粋）: 

2026-06-02 14:23:15 [INFO] POST /api/employees - employee_no: EMP-001
2026-06-02 14:23:15 [INFO] INSERT INTO employees ... (重複チェックなしにINSERTが実行された)
2026-06-02 14:23:15 [INFO] Mail sent to manager@example.com

証跡ファイル: - IT-003_NG_20260602_1423_screen.png — 登録完了画面のスクリーンショット - IT-003_NG_20260602_1423_db.png — 重複レコードが保存されたDBの状態 - IT-003_NG_20260602_1423_server.log — サーバーログ全文

「どの繋ぎ目でNGになったか」の整理:

[社員登録画面] → APIを呼び出し（ここまでは動作）
                ↓
           [登録APIサーバー] → 重複チェックを行わずDBにINSERT（ここが問題）
                             ↓
                        [メール送信機能] → 誤って通知メールが送信された

→ 社員登録APIの重複チェック処理が機能していないと推定。BUG-001として起票。
