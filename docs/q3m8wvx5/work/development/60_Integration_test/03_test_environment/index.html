STEP 3: テスト環境・テストデータの準備¶

このSTEPの目的¶

テスト環境・データが整っていない状態でテストを始めると、「環境の問題なのか・コードの問題なのか」が切り分けられなくなります。
「準備完了」を確認してからテストを開始することが、無駄な調査時間を防ぐ最善策です。

結合テスト環境とは¶

結合テストは、単独の機能ではなく「複数の機能が繋がった状態」で確認します。
そのため、単体テスト環境より複雑な準備が必要です。

【単体テスト環境のイメージ】
  機能A単独 → テスト

【結合テスト環境のイメージ】
  機能A + 機能B + 機能C + DB + 外部API → テスト
  ↑ 全部が動いている状態が必要

準備するもの 4つ¶

1. テスト環境の確認¶

結合テスト用の専用環境があります。以下を必ず確認してください。

確認項目:

確認内容 確認方法 OK/NG
アプリケーションサーバーが起動しているか URLにアクセスしてトップ画面が表示されるか確認
データベースに接続できるか アプリ経由でデータを取得・表示できるか確認
外部APIとの接続ができるか 接続テスト用のエンドポイントを叩いて200が返るか確認
ログが出力されているか ログファイル・ログコンソールが稼働しているか確認
最新コードがデプロイされているか バージョン情報・最終デプロイ日時を確認

注意: 本番環境・開発環境・単体テスト環境と混同しないこと。
接続先URL・DBのホスト名・環境変数が結合テスト用になっているか必ず確認する。

2. テストデータの準備¶

テストデータとは「テストを実施するためにDBに事前に登録しておく必要があるデータ」です。

なぜテストデータが重要か:
例えば「注文→在庫減算」のシナリオをテストするには、テスト開始前に「商品データ」「在庫データ」「テストユーザーデータ」がDBに存在している必要があります。

テストデータの種類¶

種類 内容 例
マスターデータ 変わらない基礎データ 商品マスター・顧客マスター
初期登録データ テスト実施前に用意するデータ 在庫数・初期残高
シナリオ固有データ 特定のテストケースだけで使うデータ 「在庫0の商品」「無効な顧客アカウント」

テストデータ準備の手順¶

  1. テストケース一覧を見て、各シナリオに必要なデータをリストアップする
  2. SQLスクリプトやシードデータを使ってDBに登録する
  3. データが正しく登録されたか確認する（SELECT文で確認）

データ準備チェックシート（例）:

テストケースID 必要なデータ テーブル名 登録状態
IT-001 商品A（在庫数20） products, stock 登録済み
IT-002 商品B（在庫数0） products, stock 登録済み
IT-003 テストユーザーA users 登録済み

データ初期化の手順を確認しておく¶

テストを順番に実施するとDBのデータが変化します（例: 在庫が減る）。
テストを最初からやり直す場合や、別のシナリオを試す場合はデータを初期状態に戻す手順が必要です。

実施前に確認すること: - データの初期化スクリプトはどこにあるか - 初期化後に正しい状態になっているか確認する方法

3. テスト対象のビルド・デプロイ確認¶

「最新の状態でテストしているか」を確認します。古いバージョンのコードでテストすると、「修正したのにNGのまま」という混乱が起きます。

確認方法:

確認内容 方法
最新コードがマージされているか GitのブランチとコミットIDを確認
テスト環境にデプロイされているか デプロイログ・バージョン情報ページを確認
単体テストのNG修正が含まれているか デプロイされたコミットに修正が含まれるか確認

4. 外部システム連携の確認¶

外部APIや他システムと連携する機能がある場合は接続設定を確認します。

確認項目:

確認内容 確認方法
接続先URLが結合テスト用になっているか 設定ファイル・環境変数を確認
認証情報（APIキー・トークン）が有効か テスト用のAPIキーを使って接続テストを実施
連携先のテスト環境が稼働しているか 連携先の担当者に確認

注意: 外部サービス（決済・メール送信・SMS等）は本番環境に接続すると実際に処理が走ることがあります。
必ずテスト用のサンドボックス環境に接続しているか確認してください。

環境確認チェックリスト（テスト開始前に全項目を確認）¶

【環境】
- [ ] 結合テスト用の環境URLに接続できる
- [ ] DBに接続できる（アプリ経由でデータが表示される）
- [ ] 外部APIとの接続ができる（テスト用サンドボックス）
- [ ] ログが出力されている
- [ ] 最新コードがデプロイされている

【テストデータ】
- [ ] マスターデータが登録されている
- [ ] 各テストケースに必要な初期データが揃っている
- [ ] データ初期化の手順を把握した

【その他】
- [ ] テストで使う管理ツール（DBクライアント・ログビューア等）が使える
- [ ] 不明点はリーダーに確認し解消した

環境トラブル時の対応¶

環境の問題はよく起きます。慌てずに以下の順番で対応してください。

  1. まずエラーメッセージを確認する — ログを見て何が起きているかを把握する
  2. 設定を確認する — 接続先・認証情報・設定ファイルを再確認する
  3. リーダーや担当者に相談する — 環境構築はインフラ担当者の領域。一人で抱え込まない

環境の問題を「たぶん大丈夫」と放置してテストを進めると、テスト結果の信頼性が下がります。
疑問があればテスト開始前に必ず解決してください。

このSTEPの完了条件¶

  • テスト環境の全項目が確認できた
  • テストデータが正しく登録されている
  • 最新コードがデプロイされている
  • 外部連携の設定が確認できた
  • データ初期化の手順を把握した

データ初期化手順¶

テストを最初からやり直す場合は以下のSQLスクリプトを実行する。

-- データ初期化スクリプト（integration-test環境専用）
-- ★本番・開発環境では絶対に実行しないこと

-- 社員テーブルをリセット
DELETE FROM employees WHERE employee_no LIKE 'EMP-9%';  -- テスト用データのみ削除

-- 部署の人数をリセット
UPDATE departments SET staff_count = 5 WHERE id = 'D001';  -- 営業部
UPDATE departments SET staff_count = 8 WHERE id = 'D002';  -- 開発部

-- 田中　花子を営業部に戻す
UPDATE employees SET dept_id = 'D001' WHERE id = 'EMP-001';

-- 鈴木　一郎のステータスをアクティブに戻す
UPDATE employees SET status = 'active' WHERE id = 'EMP-050';

スクリプトファイルの場所: D:\projects\SelfNewsletter\test\integration\reset_data.sql
