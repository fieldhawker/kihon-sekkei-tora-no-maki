STEP 2: テストケース・テストシナリオの作成¶

このSTEPの目的¶

「テストケースを書く」とは「何を・どの順番で・どう確認するか」を事前に決めることです。
行き当たりばったりで確認すると、重要な連携パターンを見落とします。
他の人が見ても同じ手順で実施できるレベルで書くことがゴールです。

テストケースとテストシナリオの違い¶

用語 意味 例
テストシナリオ 確認したい「業務の流れ・ストーリー」 「ユーザーが商品を注文し、在庫が減り、確認メールが届く」
テストケース そのシナリオを実施するための具体的な手順・条件 「商品Aを1個注文する → 在庫数が19になること → メール受信を確認する」

シナリオは「何を確認したいか」のアイデア。テストケースは「どうやって確認するか」の具体的な手順書です。

テストシナリオの考え方¶

3種類のシナリオを必ず作る¶

種類 確認すること 重要度
正常系 正しい操作・正しいデータで、機能間の連携が期待通りに動くか 最優先
異常系 エラーが起きたとき、連携先も適切にエラー処理されるか 必須
境界値・特殊ケース 数量0・最大値・空文字など、境界のデータで連携が壊れないか 重要

よくある失敗: 正常系しか作らないパターン。
実際の業務では必ず想定外の操作やデータが発生します。「エラーが起きたとき隣の機能はどう動くか」も必ず確認してください。

シナリオの洗い出し方¶

  1. 基本設計書の「機能一覧」と「画面遷移図」を開く
  2. 連携している機能の組み合わせをリストアップする
  3. それぞれについて「正常系・異常系・境界値」を考える

例（注文システムの場合）:

シナリオID テストシナリオ 種類
SC-001 在庫あり商品を注文 → 在庫が減る → 確認メール送信 正常系
SC-002 在庫なし商品を注文 → エラーが表示 → 注文データが作られない 異常系
SC-003 数量0で注文 → バリデーションエラー → 在庫に影響しない 境界値
SC-004 顧客情報を更新 → 注文履歴の表示に最新情報が反映される データ連携

テストケースの書き方¶

テストケースの項目¶

項目 内容 記入のポイント
テストケースID 管理番号（例: IT-001） 連番で振る。後から追加する場合は IT-001-1 などサブ番号を使う
対象シナリオID 対応するシナリオID SC-001 など
テストシナリオ名 何を確認するシナリオか 「注文登録〜在庫減算〜メール送信の正常系」
確認観点 何を確認するか 「在庫数が1減ること」「メールが届くこと」など
前提条件 テスト前の状態 「商品Aの在庫数=20」「テストユーザーAでログイン済み」
テスト手順 操作の手順を番号で 「①商品Aを選択 ②数量1を入力 ③注文ボタンを押す」
期待する結果 どうなれば合格か 「在庫数が19になること」「注文確認メールが届くこと」
関連単体テストID 対応する単体テストID UT-023, UT-024

テストケース記入テンプレート¶

以下のテンプレートをコピーして使ってください。

## テストケース: IT-001

| 項目 | 内容 |
|---|---|
| テストケースID | IT-001 |
| 対象シナリオID | SC-001 |
| テストシナリオ名 | 注文登録〜在庫減算〜確認メール送信（正常系） |
| 確認観点 | ・在庫数が1減ること ・注文確認メールが届くこと |
| 関連単体テストID | UT-015, UT-023 |

### 前提条件

- 商品A（商品ID: P001）の在庫数が20に設定されている
- テストユーザーA（user_a@test.com）でログイン済み
- テスト用メール受信環境が準備されている

### テスト手順

1. 商品一覧画面から「商品A」を選択する
2. 数量に「1」を入力する
3. 「カートに追加」ボタンを押す
4. カート画面で「注文確認」ボタンを押す
5. 注文確認画面で内容を確認し「注文する」ボタンを押す

### 期待する結果

- [ ] 注文完了画面が表示される
- [ ] 商品Aの在庫数が「19」になっている（DB確認）
- [ ] テストユーザーAのメールアドレスに注文確認メールが届く
- [ ] メールの注文内容が注文した内容と一致している

### 実施結果

| 結果 | OK / NG |
|---|---|
| 注文完了画面の表示 | |
| 在庫数の減算 | |
| 確認メールの受信 | |
| メール内容の一致 | |

### 特記事項（NG時に記録）

（発生した問題・状況を記載）

テストケース作成時の注意点¶

「手順」は誰でも再現できるレベルで書く¶

悪い例:

「注文して在庫を確認する」

良い例:

「①商品一覧画面から商品A（商品ID: P001）を選択 ②数量「1」を入力 ③注文ボタンを押す ④DBで商品AのSTOCK_QTYを確認する」

「期待する結果」は具体的に書く¶

悪い例:

「在庫が減る」

良い例:

「商品A（商品ID: P001）のSTOCK_QTYが19になっている」

「前提条件」を正確に書く¶

テストを順番に実施するとDBのデータが変わります。「このテストを始める前にDBがどういう状態になっているか」を明確にしておかないと、2回目以降の実施で結果がおかしくなります。

テストケース作成後のレビュー¶

作成したテストケースは自分だけで完成とせず、必ずリーダー・先輩にレビューを依頼してください。

レビューで確認してもらうこと:

  • 対象の機能間連携が網羅されているか
  • 正常系・異常系・境界値が揃っているか
  • 手順が具体的で他者でも再現できるか
  • 前提条件が正確か
  • 期待する結果が明確か

このSTEPの完了条件¶

  • 担当範囲のテストシナリオを洗い出した
  • 正常系・異常系・境界値のテストケースを作成した
  • テストケースがリーダーにレビューされ承認された

テストシナリオ一覧（記入例）¶

シナリオID テストシナリオ 種類
SC-001 社員登録 → 部署の所属人数が増加 → 管理者にメール送信 正常系
SC-002 社員登録 → 必須項目が未入力 → エラーが表示・DB未保存 異常系
SC-003 社員登録 → 既存の社員番号で重複登録 → エラー・DB未保存 異常系
SC-004 社員情報を更新 → 社員一覧・詳細画面に最新情報が反映 データ連携

SC-006 社員削除（退職処理）→ ログイン権限が無効化 → 一覧に表示されない 正常系
SC-007 権限のない社員が管理者メニューにアクセス → アクセス拒否される 異常系
SC-008 社員一覧をCSV出力 → 登録済みの全社員データが含まれる データ連携

テストケース記入例¶

IT-001: 社員登録〜部署人数更新〜管理者メール送信（正常系）¶

項目 内容
テストケースID IT-001
対象シナリオID SC-001
テストシナリオ名 社員登録〜部署所属人数更新〜管理者メール送信（正常系）
確認観点 ・社員データがDBに保存されること ・部署の所属人数が+1されること ・部署管理者にメールが届くこと
関連単体テストID UT-011, UT-023, UT-034

前提条件: - 部署「営業部」（部署ID: D001）の所属人数が5人に設定されている - 部署「営業部」の管理者（manager@example.com）が登録されている - テスト用メール受信環境（MailHog: http://integration-test:8025）が起動している

テスト手順:

  1. 管理者アカウント（admin@example.com）でログインする
  2. メニューから「社員管理」→「新規登録」を選択する
  3. 以下の内容を入力する
  4. 社員番号: EMP-999
  5. 氏名: テスト　太郎
  6. 所属部署: 営業部（D001）
  7. 役職: 一般
  8. メールアドレス: test.taro@example.com
  9. 「登録」ボタンを押す

期待する結果:

  • 「登録が完了しました」のメッセージが表示される
  • 社員一覧画面に「テスト　太郎」が表示される
  • DBのemployeesテーブルに社員番号EMP-999のレコードが存在する
  • DBのdepartmentsテーブルの「営業部」のstaff_countが6になっている
  • MailHogの受信トレイに、manager@example.com宛の登録通知メールが届いている
  • メールの本文に「テスト　太郎」の名前と社員番号が記載されている

IT-002: 社員登録〜必須項目未入力エラー（異常系）¶

項目 内容
テストケースID IT-002
対象シナリオID SC-002
テストシナリオ名 社員登録時に必須項目（氏名）が未入力 → エラー・DB未保存
確認観点 ・バリデーションエラーが表示されること ・DBに不完全なデータが保存されないこと
関連単体テストID UT-012

前提条件: - 管理者アカウントでログイン済み - DBのemployeesテーブルのレコード数を事前に確認しておく

テスト手順:

  1. メニューから「社員管理」→「新規登録」を選択する
  2. 社員番号に「EMP-998」を入力する（氏名は空白のまま）
  3. 所属部署に「営業部」を選択する
  4. 「登録」ボタンを押す

期待する結果:

  • 「氏名は必須項目です」のエラーメッセージが画面に表示される
  • 入力画面のまま遷移しない
  • DBのemployeesテーブルのレコード数が変わっていない
  • メールが送信されていない（MailHogに新着メールなし）

IT-005: 部署変更〜旧部署・新部署の人数同時更新（正常系）¶

項目 内容
テストケースID IT-005
対象シナリオID SC-005
テストシナリオ名 社員の所属部署変更 → 旧部署・新部署の所属人数が正しく更新される
確認観点 ・旧部署のstaff_countが-1されること ・新部署のstaff_countが+1されること ・両方が同時に更新されること（片方だけ更新される状態にならないこと）
関連単体テストID UT-041, UT-042

前提条件: - 社員「田中　花子」（社員ID: EMP-001）が「営業部」（D001）に所属、営業部のstaff_count=5 - 「開発部」（D002）のstaff_count=8

テスト手順:

  1. 社員一覧画面で「田中　花子」を選択する
  2. 「編集」ボタンを押す
  3. 所属部署を「営業部」から「開発部」に変更する
  4. 「更新」ボタンを押す

期待する結果:

  • 「更新が完了しました」のメッセージが表示される
  • 社員詳細画面の所属部署が「開発部」に変わっている
  • DBの営業部（D001）のstaff_countが4になっている
  • DBの開発部（D002）のstaff_countが9になっている
