STEP 1: テスト計画の確認・理解¶

このSTEPの目的¶

テスト計画の確認・理解は「地図を持たずに旅に出ない」ための工程です。
何をどこまでテストするか・誰がどの範囲を担当するか・いつまでに終わらせるかを全員が正しく把握してから作業を始めます。

なぜ最初に確認するのか?
計画を理解せずにテストを始めると「やったが対象外だった」「重複して実施した」「期限に間に合わなかった」が起きます。
1時間の確認で、後の数日分の手戻りを防げます。

確認するドキュメント一覧¶

ドキュメント 確認する目的 どこにあるか
テスト計画書 テストの方針・対象範囲・スケジュールを把握する プロジェクトの共有フォルダ / リーダーに確認
基本設計書 機能間の連携・データの流れを把握する src/work/development/20_basic_design/
詳細設計書 各機能の処理内容を確認する src/work/development/30_detailed_design/
単体テスト結果 未解決のNGがないか確認する src/work/development/50_unit_testing/
担当割り当て表 自分が担当するシナリオ・機能を把握する リーダーに確認

確認チェックリスト¶

テスト方針の確認¶

  • テストの目的（何を確認するためのテストか）を理解した
  • テスト対象の機能・画面の範囲を把握した
  • テスト対象外の範囲（スコープ外）を把握した
  • テストの合格基準（どうなれば完了とみなすか）を確認した

スケジュールの確認¶

  • テスト開始日・終了日を確認した
  • 中間報告・最終報告のタイミングを把握した
  • 不具合修正のための予備日が確保されているか確認した

担当範囲の確認¶

  • 自分が担当するテストシナリオ（機能間連携）を把握した
  • 他のメンバーとの担当の境界線を確認した（重複・漏れがないか）
  • 不明点をリーダーに確認し、解決した

単体テストの引き継ぎ確認¶

  • 単体テストでNGが残っていないか確認した
  • NGが残っている場合は、リーダーと「結合テスト開始の可否」を判断した

重要: 単体テストでNGが残っている状態で結合テストを開始すると、 結合テストのNGが「単体の問題か・連携の問題か」を切り分けられなくなります。 原則として単体テストのNGを解消してから結合テストを開始してください。

基本設計書で確認すべき「機能間連携」の読み方¶

結合テストで最も重要なのは「機能間の繋がり」です。基本設計書を読む際は以下の点を意識してください。

確認ポイント¶

1. データの流れを追う

例: 注文機能の場合

[注文登録画面] → 注文データをDBに保存
                ↓
           [在庫管理機能] → 在庫数を自動で減算
                ↓
           [メール送信機能] → 注文確認メールを送信

このような「データがどこからどこへ流れるか」を矢印で整理するとわかりやすいです。

2. 連携の「入口」と「出口」を把握する

機能A（送り手） 渡すデータ 機能B（受け手） 期待する動作
注文登録 注文ID・商品ID・数量 在庫管理 在庫数を減らす
在庫管理 在庫不足フラグ 注文登録 エラーメッセージを表示

3. エラー時の連携を確認する

正常に動いた場合だけでなく、「エラーが起きたとき、連携先はどう動くか」も設計書で確認します。

不明点の確認方法¶

テスト計画書や設計書を読んでわからないことがあれば、その場で確認することが大切です。

確認すべき相手¶

疑問の内容 確認先
テストの進め方・スケジュール テストリーダー
機能の動作仕様 機能の開発担当者
設計書の記載内容 設計書を書いた人
環境・ツールの使い方 インフラ担当者・先輩エンジニア

確認するときのポイント¶

  • 「なんとなくわかった」で進めない。理解できるまで聞く
  • 疑問は具体的に伝える（「設計書のp.5の図3がわからない」など）
  • 確認した内容はメモしておく（後で「言った・言わない」を防ぐ）

このSTEPの完了条件¶

以下がすべて揃った状態でSTEP 2へ進む。

  • テスト対象の範囲・自分の担当範囲を口で説明できる
  • テストの完了条件（合格基準）を理解している
  • スケジュールを把握し、作業計画のイメージができている
  • 単体テストのNG状況を確認し、開始可否をリーダーと合意した
  • 疑問点を解消した

次のSTEP¶

→ STEP 2: テストケースの作成

成果物例: 社員管理システム¶

以下は「社員管理システム」の結合テストを想定した、STEP 1 の成果物記入例です。

テスト計画確認チェックリスト（記入例）¶

テスト方針の確認¶

  • テストの目的を理解した
    → 社員登録・更新・削除・検索の各機能が正しく連携し、データが一貫して管理されることを確認する
  • テスト対象の機能・画面の範囲を把握した
    → 社員登録画面 / 社員一覧画面 / 社員詳細画面 / 部署管理画面 / メール通知機能
  • テスト対象外の範囲を把握した
    → 給与計算機能・勤怠管理機能はシステムテストで確認（今回はスコープ外）
  • 合格基準を確認した
    → 全テストケースOK・未解決不具合0件

スケジュールの確認¶

  • テスト開始日: 2026-06-01　終了日: 2026-06-10
  • 中間報告: 2026-06-05（リーダーに進捗を口頭報告）
  • 不具合修正の予備日: 2026-06-09〜10

担当範囲の確認¶

  • 自分の担当: 社員登録〜部署連携・メール通知のシナリオ（SC-001〜SC-005）
  • 他メンバーとの境界: 検索・CSV出力機能は山田さんが担当（境界線を確認済み）

単体テストの引き継ぎ確認¶

  • 単体テストのNG: 0件（全件解決済みを確認、2026-05-31付けの単体テスト完了報告書で確認）

機能間連携の整理（記入例）¶

社員管理システムの主な連携フローを整理した。

【社員登録フロー】
[社員登録画面] → 社員データをDBに保存
                ↓
           [部署管理機能] → 所属部署の社員数カウントを更新
                ↓
           [メール通知機能] → 所属部署の管理者にメール送信

【社員削除フロー】
[社員詳細画面] → 社員ステータスを「退職」に変更
                ↓
           [部署管理機能] → 所属部署の社員数カウントを減算
                ↓
           [権限管理機能] → 対象社員のログイン権限を無効化
機能A（送り手） 渡すデータ 機能B（受け手） 期待する動作
社員登録 社員ID・部署ID 部署管理 部署の所属人数を+1する
社員登録 社員ID・部署ID メール通知 部署管理者に登録通知メールを送信
社員削除 社員ID 権限管理 対象社員のログイン権限を無効化する
部署変更 旧部署ID・新部署ID 部署管理 旧部署-1、新部署+1の人数を更新する
