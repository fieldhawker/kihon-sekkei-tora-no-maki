エラー・例外設計¶

このドキュメントの目的¶

エラー・例外設計は、システムで発生しうるエラーを網羅的に洗い出し、ユーザーへの表示・ログの出力・システムの動作を統一する作業です。

なぜ詳細設計で統一するのか: エラー処理を各機能の担当者がバラバラに実装すると、画面によってエラーの見せ方が異なったり、ログが残らなかったり、内部情報が漏洩するリスクがあります。設計段階でルールを決めることで品質と安全性が揃います。

作業の流れ¶

1. エラーの種類を整理する
2. エラーコード一覧を定義する
3. ログ設計を行う
4. セキュリティ上の注意点を確認する

STEP 1: エラーの種類を整理する¶

エラーは大きく以下の3種類に分類できます。分類することで、適切な対処方法を選択できます。

分類 説明 例 ユーザーへの対応
業務エラー（クライアントエラー） ユーザーの操作・入力に起因するエラー 必須項目の未入力・存在しないデータへのアクセス エラーメッセージを表示し、ユーザーに修正を促す
システムエラー（サーバーエラー） サーバー・DBなどシステム側の問題 DB接続失敗・外部APIのタイムアウト 「しばらくしてから再度お試しください」と表示し、内部をログに残す
セキュリティエラー 不正アクセス・認証失敗など 未ログインでの操作・権限のないデータへのアクセス ログイン画面へリダイレクト等の適切な処理

STEP 2: エラーコード一覧の定義¶

エラーコードとは¶

エラーの種類を識別するための番号・コードです。ユーザーには見せず、ログや開発者向けのエラーレスポンスに含めます。

なぜコード化するのか: コードを見るだけで「どんなエラーか」を素早く特定できるためです。サポート対応でも「エラーコードE002が出ていました」と言えば、どの処理で何が起きたか素早く判断できます。

エラーコード一覧¶

業務エラー（E1xx系）¶

エラーコード HTTPステータス 発生条件 ユーザーへの表示文言 ログレベル
E101 400 必須項目の未入力 「入力内容に誤りがあります。赤字の項目を確認してください」 INFO
E102 400 入力値の形式エラー 「入力内容の形式が正しくありません」 INFO
E103 404 指定されたデータが存在しない 「指定されたデータが見つかりません」 INFO
E104 409 在庫不足・重複登録など業務上の制約違反 エラーの内容に応じたメッセージ（例: 「在庫が不足しています」） INFO

セキュリティエラー（E2xx系）¶

エラーコード HTTPステータス 発生条件 ユーザーへの表示文言 ログレベル
E201 401 未認証（ログインしていない状態での操作） 「セッションが切れました。再度ログインしてください」 WARN
E202 403 権限なし（ログインしているが操作権限がない） 「この操作を行う権限がありません」 WARN
E203 401 認証失敗（ログイン失敗） 「メールアドレスまたはパスワードが正しくありません」 WARN

重要: E203の文言は意図的に「メールアドレスが間違っています」と「パスワードが間違っています」を分けて表示しません。分けると「このメールアドレスは登録済み」と攻撃者に教えることになるためです。

システムエラー（E3xx系）¶

エラーコード HTTPステータス 発生条件 ユーザーへの表示文言 ログレベル
E301 500 DB接続エラー・クエリ実行エラー 「システムエラーが発生しました。しばらくしてから再度お試しください」 ERROR
E302 503 外部APIのタイムアウト・接続エラー 「外部サービスとの通信に失敗しました。しばらくしてから再度お試しください」 ERROR
E303 500 予期しない例外（想定外エラー） 「システムエラーが発生しました。しばらくしてから再度お試しください」 ERROR

エラーレスポンスのフォーマット（APIの場合）¶

APIのエラー時のレスポンスの形式を統一します。

{
  "error": {
    "code": "E101",
    "message": "入力内容に誤りがあります。赤字の項目を確認してください",
    "details": [
      {
        "field": "email",
        "message": "メールアドレスを入力してください"
      }
    ]
  }
}
フィールド 説明
code エラーコード（上記の一覧のコード）
message ユーザーに表示する概要メッセージ
details 項目ごとの詳細エラー（バリデーションエラーの場合のみ）

絶対に含めてはいけない情報: スタックトレース・SQL文・テーブル名・DBのエラーメッセージをそのままレスポンスに含めない。

STEP 3: ログ設計¶

ログとは¶

システムの動作・エラーをファイルやDBに記録するものです。障害発生時の原因調査（デバッグ）に使います。

ログがないと何が困るか: 本番環境でエラーが起きたとき「どんな入力値で・どの処理で・何が起きたか」を再現できず、原因調査に何時間もかかります。

ログレベルの定義¶

レベル 用途 例
ERROR システムの動作に支障が出る深刻なエラー DB接続失敗・外部APIエラー
WARN 問題は起きていないが注意が必要な事象 認証失敗・権限エラー
INFO 正常な動作の記録 処理開始・終了・入力バリデーション失敗
DEBUG 開発・デバッグ用の詳細情報 本番環境では出力しない

ログの出力項目¶

項目 説明 例
タイムスタンプ ログを出力した日時（ISO8601形式） 2024-03-15T09:30:00+09:00
ログレベル ERROR / WARN / INFO / DEBUG ERROR
ログID ログの一意識別番号（追跡用） LOG-xxxxxxxx
リクエストID APIリクエストの追跡ID req-xxxxxxxx
ユーザーID 操作したユーザーのID（ログイン時のみ） 12345
処理名 どの処理でログを出力したか UserService.login
メッセージ エラーの内容 DB connection failed
エラーコード エラーコード（エラー時のみ） E301

ログのフォーマット例（JSON形式）¶

{
  "timestamp": "2024-03-15T09:30:00+09:00",
  "level": "ERROR",
  "log_id": "LOG-abc12345",
  "request_id": "req-def67890",
  "user_id": 12345,
  "process": "UserService.login",
  "message": "DB connection failed",
  "error_code": "E301",
  "error_detail": "Connection timeout after 30000ms"
}

個人情報のログ出力禁止: パスワード・クレジットカード番号・マイナンバーなどはログに出力しない。

チェックリスト¶

  • ログレベルの使い方の基準を定義したか
  • ログに含める項目を定義したか
  • パスワードなどの機密情報をログに出力しない設計になっているか
  • ログの保存期間・保存場所を確認したか（運用担当者と確認）

STEP 4: セキュリティ上の注意点¶

詳細設計の段階で確認すべきセキュリティ観点を以下に示します。

エラーメッセージによる情報漏洩の防止¶

やってはいけないこと 理由 代わりにすること
DBのエラーメッセージをそのまま表示 テーブル名・カラム名・SQLがわかる 「システムエラーが発生しました」と表示
スタックトレースをレスポンスに含める 内部の処理構造が攻撃者にわかる ログに記録するだけにする
「このメールは登録されていません」と表示 メールアドレスの存在確認ができてしまう 「メールアドレスまたはパスワードが正しくありません」と表示

その他のセキュリティチェック¶

観点 確認内容
SQLインジェクション バインド変数を使用しているか（03_data_detail_design.md 参照）
XSS（クロスサイトスクリプティング） ユーザー入力をHTMLに出力する際にエスケープしているか
CSRF（クロスサイトリクエストフォージェリ） フォーム送信にCSRFトークンを使用しているか
認証の強制 ログインが必要なページに未ログインでアクセスした場合、ログイン画面にリダイレクトするか
権限チェック URL直打ちで他人のデータにアクセスできないか

成果物チェックリスト¶

  • エラーコード一覧を作成したか（業務エラー・セキュリティエラー・システムエラー）
  • エラーレスポンスのフォーマットを定義したか
  • ログに出力する項目・レベルを定義したか
  • エラーメッセージに内部情報を含めない設計になっているか
  • セキュリティ上の主要リスク（SQLインジェクション・XSS等）の対策を確認したか

成果物例：社員管理システム¶

STEP 1: エラーの種類整理¶

分類 発生シナリオ（社員管理システムの例） 対応方針
業務エラー 社員番号の重複・必須項目の未入力・存在しない社員IDへのアクセス エラーメッセージを画面に表示し、ユーザーに修正を促す
システムエラー DB接続失敗・勤怠管理システムAPIのタイムアウト 「しばらくしてから再度お試しください」を表示し、詳細をログに記録
セキュリティエラー 未ログインでの社員情報アクセス・一般社員が他社員の給与情報にアクセス ログイン画面へリダイレクト、または権限エラーページを表示

STEP 2: エラーコード一覧¶

業務エラー（E1xx系）¶

エラーコード HTTPステータス 発生条件 ユーザーへの表示文言 ログレベル
E101 400 必須項目（社員番号・氏名・入社日等）の未入力 「入力内容に誤りがあります。赤字の項目を確認してください」 INFO
E102 400 社員番号の形式エラー（E＋6桁数字以外） 「社員番号はEから始まる7文字で入力してください」 INFO
E103 404 指定した社員IDが存在しない 「指定された社員情報が見つかりません」 INFO
E104 409 社員番号の重複 「この社員番号はすでに使用されています」 INFO
E105 409 メールアドレスの重複 「このメールアドレスはすでに登録されています」 INFO
E106 400 退職日が入社日より前の日付 「退職日は入社日以降の日付を入力してください」 INFO
E107 409 すでに退職処理済みの社員への退職処理 「この社員はすでに退職処理が完了しています」 INFO

セキュリティエラー（E2xx系）¶

エラーコード HTTPステータス 発生条件 ユーザーへの表示文言 ログレベル
E201 401 未ログインでの社員情報へのアクセス 「セッションが切れました。再度ログインしてください」 WARN
E202 403 一般社員が給与情報・管理者機能にアクセス 「この操作を行う権限がありません」 WARN
E203 401 ログイン失敗（メールアドレスまたはパスワード不一致） 「メールアドレスまたはパスワードが正しくありません」 WARN
E204 403 他社員の情報への不正アクセス（URL直打ち等） 「この操作を行う権限がありません」 WARN

システムエラー（E3xx系）¶

エラーコード HTTPステータス 発生条件 ユーザーへの表示文言 ログレベル
E301 500 DB接続エラー・クエリ実行エラー 「システムエラーが発生しました。しばらくしてから再度お試しください」 ERROR
E302 503 勤怠管理システムAPIのタイムアウト・接続エラー 「外部サービスとの通信に失敗しました。しばらくしてから再度お試しください」 ERROR
E303 503 メール配信サービスAPIのタイムアウト・接続エラー 表示なし（メール送信は非同期扱い。内部でログ記録・管理者通知のみ） ERROR
E304 500 予期しない例外（想定外エラー） 「システムエラーが発生しました。しばらくしてから再度お試しください」 ERROR

エラーレスポンスのフォーマット¶

{
  "error": {
    "code": "E104",
    "message": "この社員番号はすでに使用されています",
    "details": [
      {
        "field": "emp_code",
        "message": "社員番号「E000042」はすでに登録済みです"
      }
    ]
  }
}

絶対に含めてはいけない情報: DBのエラーメッセージ・スタックトレース・SQLクエリ・テーブル名・カラム名

STEP 3: ログ設計¶

ログ出力項目¶

項目 説明 例
タイムスタンプ ログ出力日時（ISO8601形式） 2025-05-25T09:30:00+09:00
ログレベル ERROR / WARN / INFO ERROR
ログID ログの一意識別番号 LOG-abc12345
リクエストID APIリクエストの追跡ID req-def67890
ユーザーID 操作したユーザーのID 42
処理名 エラーが発生した処理名 EmployeeService.register
メッセージ エラーの内容 Duplicate entry for emp_code
エラーコード エラーコード E104

ログフォーマット例（JSON）¶

{
  "timestamp": "2025-05-25T09:30:00+09:00",
  "level": "WARN",
  "log_id": "LOG-abc12345",
  "request_id": "req-def67890",
  "user_id": 42,
  "process": "EmployeeService.register",
  "message": "社員番号重複エラー: emp_code=E000042",
  "error_code": "E104"
}

個人情報のログ出力禁止: パスワード・マイナンバー・給与情報の具体的な金額はログに出力しない。

STEP 4: セキュリティ上の注意点（社員管理システム固有）¶

観点 確認内容
権限チェック（URL直打き対策） /employees/{id}/salary 等の給与情報URLに管理者以外がアクセスした場合、E202を返すこと
退職社員のデータアクセス制限 退職処理後はログイン不可にする。退職社員のデータは管理者のみ参照可能にする
CSVエクスポートの権限制限 社員データエクスポート（RPT-EMP-002）は管理者・人事担当者ロールのみ実行可能とする
SQLインジェクション 社員検索のキーワード入力など全てのユーザー入力をバインド変数で処理すること
XSS 社員の氏名・部署名など全ての表示項目をHTMLエスケープすること
認証の強制 ログインが必要な全ページ・全APIエンドポイントで未ログイン時はE201を返しログイン画面にリダイレクトすること
