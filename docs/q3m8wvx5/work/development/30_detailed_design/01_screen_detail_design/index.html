画面詳細設計¶

このドキュメントの目的¶

画面詳細設計は、基本設計で定義した画面レイアウト・項目をもとに、「実際にどう動かすか」をコード化できるレベルまで詳細化する作業です。

製造担当者がこのドキュメントを見て「何を実装すればよいか迷わない」状態を目指します。

基本設計との違い: 基本設計は「ログインボタンをクリックしたらログインする」、詳細設計は「ログインボタンクリック時に①バリデーション実行→②API呼び出し→③成功なら画面遷移、失敗ならエラー表示」まで定義する。

作業の流れ¶

1. 入力バリデーション定義
2. イベント処理定義
3. 画面表示制御定義
4. 初期表示・データ取得定義

STEP 1: 入力バリデーション定義¶

バリデーションとは¶

ユーザーが入力した値が正しい形式・内容かどうかをチェックする処理です。

なぜ必要か: 不正な値がシステムに入ると、DBが壊れたり、セキュリティ上の問題が起きたりします。入力の段階で正しい値だけを通すのがバリデーションの役割です。

定義する内容¶

各入力項目に対して以下を定義します：

定義項目 説明 例
チェック種別 何をチェックするか 必須・形式・桁数・範囲
チェック条件 具体的な条件値 8文字以上、半角英数字のみ
チェック順序 複数チェックがある場合の優先順位 必須チェック → 形式チェック
エラーメッセージ 失敗時にユーザーに表示する文言 「パスワードは8文字以上で入力してください」
チェックタイミング いつチェックするか 送信時・フォーカスアウト時

記入例：ログイン画面¶

項目ID 項目名 チェック種別 チェック条件 エラーメッセージ タイミング
email メールアドレス 必須 空でないこと 「メールアドレスを入力してください」 送信時
email メールアドレス 形式 @を含む正規表現 「メールアドレスの形式が正しくありません」 送信時
password パスワード 必須 空でないこと 「パスワードを入力してください」 送信時
password パスワード 桁数 8文字以上 「パスワードは8文字以上で入力してください」 送信時
password パスワード 形式 英字と数字を両方含む 「パスワードは英字と数字を含めてください」 送信時

チェックリスト¶

  • すべての入力項目にバリデーションを定義したか
  • エラーメッセージは利用者にとってわかりやすい言葉か（技術的な表現になっていないか）
  • チェック順序は明確か（必須チェックを最初に行うなど）
  • 境界値（最大文字数ちょうど、最小文字数ちょうどなど）のケースを考慮したか

STEP 2: イベント処理定義¶

イベント処理とは¶

ユーザーの操作（クリック・入力・フォーカス移動など）をトリガーとして実行される処理の定義です。

イメージ: 「電源ボタンを押したら電気がつく」という動作と同じ。「何を操作したら → 何が起きるか」を定義します。

定義する内容¶

定義項目 説明
イベント種別 click / change / blur / submit など
対象要素 どのボタン・入力欄に対するイベントか
処理内容 イベント発生時に呼び出す処理の名前と手順
前提条件 そのイベントが有効になる条件

記入例：ログイン画面¶

イベントID 対象要素 イベント種別 処理内容 備考
EVT-001 ログインボタン click ① バリデーション実行（STEP1の全チェック）
② エラーがあればエラーメッセージ表示して処理終了
③ ログインAPI（POST /api/login）を呼び出す
④ 成功 → ダッシュボード画面に遷移
⑤ 認証失敗 → エラーメッセージ「メールアドレスまたはパスワードが間違っています」を表示		 API仕様は外部I/F設計書参照
EVT-002 パスワード入力欄 blur（フォーカスが外れたとき） パスワードの桁数チェックを実行し、エラーがあればリアルタイムで表示 —
EVT-003 パスワードを表示するチェックボックス change チェックON：パスワード欄のtype属性を「text」に変更
チェックOFF：type属性を「password」に変更		 —

チェックリスト¶

  • すべてのボタン・リンクに対してイベント処理を定義したか
  • API呼び出しが含まれる場合、成功・失敗それぞれの処理を定義したか
  • 二重送信防止（送信中はボタンを非活性にするなど）を考慮したか

STEP 3: 画面表示制御定義¶

画面表示制御とは¶

ユーザーの権限・状態・入力内容によって、画面の要素を表示/非表示・活性/非活性に切り替える仕組みです。

例: 管理者ユーザーには「削除ボタン」が表示されるが、一般ユーザーには表示されない。

定義する内容¶

定義項目 説明
対象要素 制御するボタン・入力欄・エリアなど
制御種別 表示/非表示（display） or 活性/非活性（disabled）
制御条件 どんな状態のときに制御するか
初期状態 画面表示時のデフォルト状態

記入例：注文一覧画面¶

要素ID 要素名 制御種別 条件 初期状態
btn-delete 削除ボタン 表示/非表示 ユーザーロールが「管理者」の場合のみ表示 非表示
btn-submit 注文確定ボタン 活性/非活性 必須入力項目がすべて入力済みの場合に活性 非活性
area-admin 管理者メニュー 表示/非表示 ログインユーザーの権限が「admin」の場合のみ表示 非表示

チェックリスト¶

  • 権限によって見せてはいけない要素を定義したか
  • データが0件のとき（空状態）の表示はどうなるか定義したか
  • ローディング中の表示（スピナー等）を考慮したか

STEP 4: 初期表示・データ取得定義¶

初期表示とは¶

画面を開いたときに、どのデータをどこから取得して表示するかを定義します。

例: 会員情報編集画面を開いたとき、現在の会員情報をDBから取得してフォームに初期表示する。

定義する内容¶

定義項目 説明
取得データ 何のデータを取得するか
取得元 API・DB・セッションなど
取得条件 どんな条件で取得するか（URLパラメータなど）
表示先 どの画面要素に表示するか
データがない場合の処理 データが取得できなかった場合の動作

記入例：会員情報編集画面¶

# 取得データ 取得元 取得条件 表示先 データなし時の処理
1 会員情報 GET /api/members/{id} URLパラメータのmember_id 各入力欄に初期値として設定 エラーメッセージ表示→一覧画面に戻る
2 都道府県リスト GET /api/prefectures なし（全件取得） 都道府県プルダウンの選択肢 「データを取得できませんでした」と表示

チェックリスト¶

  • 画面表示に必要なすべてのデータの取得元を定義したか
  • データ取得に失敗した場合の処理を定義したか
  • 取得したデータをどの順番で表示するか定義したか

成果物チェックリスト¶

詳細設計書として提出する前に、以下を確認してください。

網羅性（抜け漏れがないか）¶

  • 基本設計書に記載されたすべての画面・項目を詳細化したか
  • すべての入力項目にバリデーションを定義したか
  • すべてのボタン・リンク・操作にイベント処理を定義したか
  • 権限・状態による表示制御をすべての要素に対して定義したか
  • 初期表示時に必要なデータ取得をすべて定義したか

正確性（内容が正しいか）¶

  • 基本設計書の内容と矛盾がないか（画面項目名・遷移先など）
  • API仕様書・外部I/F設計書と呼び出し方法・パラメータが一致しているか
  • バリデーション条件がDB定義（桁数・型・NULL制約）と整合しているか
  • 権限の定義がシステム要件・認証設計と一致しているか

異常系（エラーケースを考慮しているか）¶

  • すべてのAPI呼び出しに対して失敗時の処理を定義したか
  • データが0件のとき（空状態）の表示を定義したか
  • データ取得に失敗した場合のフォールバック処理を定義したか
  • ネットワークエラー・タイムアウト時の挙動を定義したか
  • 権限のないユーザーがURLを直接入力した場合の挙動を定義したか

ユーザビリティ（利用者視点で問題がないか）¶

  • エラーメッセージは技術的な表現を避け、利用者にわかりやすい文言か
  • 二重送信防止（送信中のボタン非活性など）を考慮したか
  • 入力途中でページを離れる場合の確認ダイアログを考慮したか
  • ローディング中の表示（スピナー等）を定義したか

実装可能性（製造担当者が迷わないか）¶

  • イベント処理の手順が「①②③…」の形式で順序立てて書かれているか
  • 条件分岐（成功/失敗、権限あり/なし）がすべて網羅されているか
  • 用語・ID・名称が設計書内で統一されているか（表記ゆれがないか）
  • 曖昧な表現（「適切に」「必要に応じて」など）が残っていないか

1 社員情報 GET /api/employees/{id} URLパラメータの employee_id（編集モードのみ） 各入力欄に初期値として設定 「社員情報が見つかりません」と表示し、社員一覧画面に戻る
2 部署リスト GET /api/departments なし（有効な部署のみ：deleted_at IS NULL） 部署プルダウンの選択肢 「データを取得できませんでした」と表示しページをリロード促す
3 雇用区分リスト GET /api/employment-types なし（全件） 雇用区分プルダウンの選択肢 「データを取得できませんでした」と表示しページをリロード促す
4 ログインユーザー権限 セッション情報 — 表示制御（STEP3）に使用 ログイン画面にリダイレクト
