データ詳細設計¶

このドキュメントの目的¶

データ詳細設計は、基本設計で定義したテーブル構造・ER図をもとに、「実際にどんなSQLを発行するか」「トランザクションの範囲はどこか」「インデックスは何が必要か」を設計する作業です。

基本設計との違い: 基本設計は「usersテーブルにname・emailカラムがある」という構造の定義。詳細設計は「ユーザー一覧を取得するには SELECT id, name, email FROM users WHERE deleted_at IS NULL ORDER BY created_at DESC を実行する」という具体的な操作の定義。

作業の流れ¶

1. SQL設計
2. トランザクション設計
3. インデックス設計
4. データ変換・マッピング定義

STEP 1: SQL設計¶

SQL設計とは¶

各機能で実行するSQL文（データの検索・登録・更新・削除）を設計します。

なぜ設計が必要か: SQLを製造担当者に任せると、パフォーマンスが悪いSQLや、バグの原因になるSQL（NULLの扱いが間違っているなど）が書かれることがあります。詳細設計で正しいSQLを定義しておくことで品質が安定します。

定義する内容¶

定義項目 説明
SQL-ID SQLの識別番号
対応機能 このSQLを使う機能名
SQL種別 SELECT / INSERT / UPDATE / DELETE
SQL文 実際のSQL（または擬似SQL）
バインド変数 SQLに渡すパラメータ
結果の使い方 取得したデータをどう使うか

記入例¶

SQL-001：ユーザー一覧取得¶

SELECT
    id,
    name,
    email,
    created_at
FROM
    users
WHERE
    deleted_at IS NULL   -- 論理削除されていないもののみ
ORDER BY
    created_at DESC
LIMIT :limit OFFSET :offset;
バインド変数 型 説明
:limit INTEGER 1ページあたりの表示件数（例: 20）
:offset INTEGER 取得開始位置（ページ番号 × limit）

結果の使い方: ユーザー一覧画面のテーブルに表示する。

SQL-002：注文登録¶

INSERT INTO orders (
    user_id,
    product_id,
    quantity,
    total_price,
    status,
    order_date,
    created_at,
    updated_at
) VALUES (
    :user_id,
    :product_id,
    :quantity,
    :total_price,
    '受付済み',
    NOW(),
    NOW(),
    NOW()
);
バインド変数 型 説明
:user_id INTEGER ログインユーザーのID
:product_id INTEGER 注文する商品のID
:quantity INTEGER 注文数量
:total_price DECIMAL 合計金額（単価 × 数量）

SQL-003：在庫数更新¶

UPDATE products
SET
    stock = stock - :quantity,
    updated_at = NOW()
WHERE
    id = :product_id
    AND stock >= :quantity;  -- 在庫が足りない場合は更新しない（0件更新になる）

注意: 更新件数が0件の場合は「在庫不足」と判断してエラーを返す。

SQLのよくある注意点¶

注意点 悪い例 良い例
NULLの比較 WHERE deleted_at = NULL WHERE deleted_at IS NULL
SQLインジェクション対策 WHERE id = ' + userId + ' WHERE id = :userId（バインド変数を使う）
件数制限なしの全件取得 SELECT * FROM users SELECT * FROM users LIMIT 100
必要以上のカラム取得 SELECT * SELECT id, name, email（必要なカラムだけ）

チェックリスト¶

  • すべての機能のSQL（SELECT/INSERT/UPDATE/DELETE）を定義したか
  • バインド変数を使用してSQLインジェクション対策をしているか
  • SELECT文に WHERE条件・ORDER BY・LIMIT を適切に設定したか
  • NULL の扱い（IS NULL / IS NOT NULL）が正しいか

STEP 2: トランザクション設計¶

トランザクションとは¶

「複数のDB操作を、まとめて成功・失敗させる仕組み」です。

身近な例: ATMでの振り込みを考えます。「A口座からお金を引く」と「B口座にお金を足す」の2つの操作があります。もし「引いた後に追加が失敗」したら、お金が消えてしまいます。トランザクションを使うことで、両方成功したときだけ確定（コミット）、どちらかが失敗したら両方取り消し（ロールバック）できます。

定義する内容¶

定義項目 説明
トランザクションID 識別番号
対応機能 このトランザクションを使う機能
トランザクションに含むSQL まとめて実行するSQL一覧
コミット条件 すべてのSQLが成功した場合
ロールバック条件 いずれかのSQLが失敗した場合

記入例：注文登録処理¶

TRX-ID TRX-001
対応機能 注文登録
含むSQL SQL-002（注文テーブルINSERT）、SQL-003（在庫テーブルUPDATE）
コミット条件 SQL-002とSQL-003が両方成功した場合
ロールバック条件 いずれかが失敗した場合（在庫不足で0件更新の場合も含む） 
トランザクション開始
  ↓
  注文テーブルにINSERT（SQL-002）
  ↓
  在庫テーブルをUPDATE（SQL-003）
  ↓
  更新件数が0件（在庫不足） → ロールバック → エラー返却
  ↓
  両方成功 → コミット → 完了

チェックリスト¶

  • 「一連の処理として失敗したら全部取り消すべき」操作にトランザクションを設定したか
  • トランザクション範囲が広すぎないか（DBへの接続時間が長くなるとパフォーマンスに影響する）
  • ロールバック時の挙動（エラーメッセージ・ログ）を定義したか

STEP 3: インデックス設計¶

インデックスとは¶

テーブルの特定のカラムに付ける「索引」のことです。索引があると、データを検索するときに全件確認しなくて済むため、検索が高速になります。

イメージ: 本の巻末にある「索引」と同じです。「認証」という言葉を探すとき、本を最初から読まなくても索引を見ればすぐにページを見つけられます。

インデックスが必要なカラムの見極め方¶

以下に該当するカラムはインデックスが有効です：

  • WHERE句で検索条件に使うカラム（例: WHERE user_id = :id）
  • JOIN（テーブル結合）のキーになるカラム（例: ON orders.user_id = users.id）
  • ORDER BYで並び替えに使うカラム（例: ORDER BY created_at DESC）

逆に、データ件数が少ないテーブルや、値の種類が少ないカラム（true/false など）はインデックスの効果が薄いです。

記入例¶

インデックスID テーブル名 カラム名 インデックス種別 追加理由
IDX-001 orders user_id 通常 ユーザーごとの注文一覧検索で使用（WHERE user_id = ?）
IDX-002 orders created_at 通常 日付範囲での注文検索・並び替えで使用
IDX-003 users email ユニーク メールアドレスでのログイン検索（一意制約も兼ねる）

チェックリスト¶

  • 検索条件（WHERE句）で使うカラムにインデックスを追加したか
  • 既存のインデックスで対応できるものを重複して追加していないか
  • インデックスの追加によってINSERT/UPDATEが遅くなることを許容範囲か確認したか

STEP 4: データ変換・マッピング定義¶

データ変換・マッピングとは¶

「画面の入力値 → DBへの保存形式」および「DBの値 → 画面の表示形式」への変換ルールを定義します。

例: 画面では「2024年3月15日」と表示するが、DBには「2024-03-15 00:00:00」という形式で保存する。

定義する内容（画面入力 → DB保存）¶

画面項目 入力値の例 DB カラム 保存値 変換処理
生年月日 2000/03/15 birth_date 2000-03-15 スラッシュ区切り → ハイフン区切りに変換
パスワード password123 password_hash $2b$10$... bcryptでハッシュ化して保存
金額 1,000 price 1000 カンマを除去して数値に変換

定義する内容（DB → 画面表示）¶

DBカラム DB値の例 画面項目 表示値 変換処理
created_at 2024-03-15 09:30:00 登録日時 2024年03月15日 09:30 日時フォーマットを変換
status 1 ステータス 受付済み コード値を名称に変換（コード定義参照）
price 1000 金額 ¥1,000 通貨記号・カンマ区切りを付与

チェックリスト¶

  • パスワードなどの機密情報を平文でDBに保存していないか（ハッシュ化しているか）
  • 日付・数値のフォーマット変換を定義したか
  • コード値（ステータスの数値など）の名称変換ルールを定義したか

成果物チェックリスト¶

  • すべての機能のSQL（SELECT/INSERT/UPDATE/DELETE）を設計したか
  • トランザクションが必要な箇所を特定し設計したか
  • 必要なインデックスを洗い出したか
  • データ変換・マッピングを定義したか
  • 基本設計のテーブル定義と矛盾がないか

成果物例：社員管理システム¶

STEP 1: SQL設計¶

SQL-EMP-001：社員一覧取得¶

SELECT
    e.id,
    e.emp_code,
    e.last_name,
    e.first_name,
    e.email,
    e.hire_date,
    e.employment_type,
    e.status,
    d.name AS department_name
FROM
    employees e
    LEFT JOIN employee_departments ed ON e.id = ed.employee_id
        AND ed.end_date IS NULL          -- 現在の所属部署のみ
    LEFT JOIN departments d ON ed.department_id = d.id
WHERE
    e.deleted_at IS NULL
    AND (:status IS NULL OR e.status = :status)
    AND (:department_id IS NULL OR ed.department_id = :department_id)
    AND (:keyword IS NULL
        OR e.last_name LIKE CONCAT('%', :keyword, '%')
        OR e.first_name LIKE CONCAT('%', :keyword, '%')
        OR e.emp_code LIKE CONCAT('%', :keyword, '%'))
ORDER BY
    e.emp_code ASC
LIMIT :limit OFFSET :offset;
バインド変数 型 説明
:status VARCHAR 在籍ステータスによる絞り込み（NULL=全件）
:department_id INTEGER 部署による絞り込み（NULL=全部署）
:keyword VARCHAR 氏名・社員番号の部分一致検索（NULL=指定なし）
:limit INTEGER 1ページあたりの件数（デフォルト: 20）
:offset INTEGER 取得開始位置

結果の使い方: 社員一覧画面のテーブルに表示する。

SQL-EMP-002：社員情報登録¶

INSERT INTO employees (
    emp_code,
    last_name,
    first_name,
    email,
    hire_date,
    employment_type,
    status,
    created_at,
    updated_at
) VALUES (
    :emp_code,
    :last_name,
    :first_name,
    :email,
    :hire_date,
    :employment_type,
    '在籍中',
    NOW(),
    NOW()
);
バインド変数 型 説明
:emp_code VARCHAR(7) 社員番号（E＋6桁数字）
:last_name VARCHAR(20) 姓
:first_name VARCHAR(20) 名
:email VARCHAR(255) メールアドレス
:hire_date DATE 入社日
:employment_type VARCHAR(20) 雇用区分（正社員・契約社員・パートなど）

SQL-EMP-003：社員情報更新¶

UPDATE employees
SET
    last_name = :last_name,
    first_name = :first_name,
    email = :email,
    employment_type = :employment_type,
    updated_at = NOW()
WHERE
    id = :employee_id
    AND deleted_at IS NULL;

注意: 更新件数が0件の場合は「対象社員が存在しない、または削除済み」と判断してエラー（404）を返す。

SQL-EMP-004：退職処理（論理削除）¶

UPDATE employees
SET
    status = '退職',
    retire_date = :retire_date,
    updated_at = NOW(),
    deleted_at = NOW()
WHERE
    id = :employee_id
    AND status = '在籍中'
    AND deleted_at IS NULL;
バインド変数 型 説明
:retire_date DATE 退職日
:employee_id INTEGER 対象社員のID

STEP 2: トランザクション設計¶

TRX-EMP-001：社員登録トランザクション¶

TRX-ID TRX-EMP-001
対応機能 社員情報登録
含むSQL SQL-EMP-002（employeesテーブルINSERT）、employee_departmentsテーブルINSERT
コミット条件 両方のINSERTが成功した場合
ロールバック条件 いずれかのINSERTが失敗した場合 
トランザクション開始
  ↓
  employeesテーブルにINSERT（SQL-EMP-002）
  ↓
  employee_departmentsテーブルにINSERT
  ↓
  いずれかが失敗 → ロールバック → エラー（500）返却
  ↓
  両方成功 → コミット → 完了（201 Created）

STEP 3: インデックス設計¶

インデックスID テーブル名 カラム名 インデックス種別 追加理由
IDX-EMP-001 employees emp_code ユニーク 社員番号での検索・重複チェック（一意制約も兼ねる）
IDX-EMP-002 employees email ユニーク メールアドレスでの重複チェック（一意制約も兼ねる）
IDX-EMP-003 employees status 通常 在籍ステータスによる絞り込み検索で使用

画面項目 入力値の例 DBカラム 保存値 変換処理
