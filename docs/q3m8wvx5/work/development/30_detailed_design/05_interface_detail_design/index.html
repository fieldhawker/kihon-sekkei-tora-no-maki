外部インターフェース詳細設計¶

このドキュメントの目的¶

外部インターフェース詳細設計は、外部システムとのやり取り（API通信・ファイル連携など）のリクエスト・レスポンスの内容・認証方式・エラー処理・リトライ設計を詳細化する作業です。

外部インターフェースとは: 自社システムが外部のサービスやシステムと通信する窓口のことです。例えば「決済サービスへのAPI呼び出し」「外部システムへのCSVファイルの受け渡し」などが該当します。

作業の流れ¶

1. I/F一覧の確認
2. APIリクエスト・レスポンスの項目定義
3. 認証方式の詳細定義
4. リトライ設計
5. タイムアウト設定の定義

STEP 1: I/F一覧の確認¶

基本設計書の外部I/F一覧をもとに、詳細設計の対象を確認します。

I/F-ID 連携種別 相手システム 方向 概要
IF-001 REST API呼び出し 決済サービス 自システム → 外部 クレジットカード決済処理
IF-002 REST API呼び出し 外部通知サービス 自システム → 外部 メール・SMS通知の送信
IF-003 ファイル受け取り 取引先システム 外部 → 自システム 注文データのCSVインポート

STEP 2: APIリクエスト・レスポンスの項目定義¶

目的¶

外部APIを呼び出す際に送るリクエストの内容と、受け取るレスポンスの内容を定義します。

なぜ定義が必要か: 外部システムのAPI仕様書だけでは「どの項目を使うか」「どう変換するか」が製造担当者には判断できません。自システムとの接続部分を詳細設計で定義します。

記入例：決済API（IF-001）¶

基本情報¶

項目 内容
I/F-ID IF-001
相手システム 〇〇決済サービス
エンドポイント POST https://api.payment-service.example.com/v1/charges
呼び出し契機 注文確定ボタンクリック時
タイムアウト 30秒（STEP 5参照）

リクエスト定義¶

項目名 型 必須 説明 値の例 自システムのデータ元
amount integer 必須 請求金額（円） 1500 orders.total_price
currency string 必須 通貨コード "JPY" 固定値
card_token string 必須 カードトークン（画面から受け取る） "tok_xxxxxxxx" フロントから送信された値
order_id string 必須 注文ID（自システム側の管理番号） "ORD-2024-001" orders.order_number
description string 任意 決済内容の説明 "商品購入" 固定値

リクエストヘッダー:

ヘッダー名 値
Content-Type application/json
Authorization Bearer {APIキー}（STEP 3参照）

リクエストボディ（JSON例）:

{
  "amount": 1500,
  "currency": "JPY",
  "card_token": "tok_xxxxxxxx",
  "order_id": "ORD-2024-001",
  "description": "商品購入"
}

レスポンス定義¶

成功時（HTTP 200）:

項目名 型 説明 自システムでの使い方
charge_id string 決済サービス側の決済ID payments.external_charge_id に保存
status string 決済ステータス "succeeded" なら決済成功として処理
amount integer 実際に請求した金額 金額の検証に使用
created_at string 決済日時（ISO8601形式） payments.charged_at に保存

失敗時（HTTP 4xx / 5xx）:

エラーコード HTTPステータス 意味 自システムでの処理
card_declined 402 カードが拒否された ユーザーに「カードが使えません」と表示
insufficient_funds 402 残高不足 ユーザーに「残高が不足しています」と表示
invalid_token 400 無効なカードトークン ユーザーに「カード情報を再入力してください」と表示
service_error 500 外部サービスのエラー リトライ実行（STEP 4参照）

チェックリスト¶

  • リクエストのすべての項目（型・必須/任意・説明）を定義したか
  • レスポンスの成功時・失敗時それぞれの処理を定義したか
  • 外部APIのエラーコードと自システムでの処理を対応付けたか

STEP 3: 認証方式の詳細定義¶

目的¶

外部APIを呼び出す際の認証（「正規のシステムからのアクセスであること」を証明する仕組み）の詳細を定義します。

主な認証方式の種類: - APIキー認証: 固定のキー文字列をヘッダーに付与する。シンプルだが鍵が漏れるとリスクが高い。 - OAuth 2.0: トークンを定期的に取得・更新する方式。セキュリティが高い。 - HMAC署名: リクエスト内容から署名を生成して付与する。改ざん検知ができる。

記入例：決済APIのAPIキー認証¶

定義項目 内容
認証方式 APIキー認証（Bearer Token）
APIキーの保管場所 環境変数 PAYMENT_API_KEY（コードに直書き禁止）
ヘッダーへの付与方法 Authorization: Bearer {PAYMENT_API_KEY}
APIキーのローテーション 年1回（担当者が手動で更新）

重要: APIキーやシークレットは絶対にソースコードに直接書かない。環境変数・シークレット管理サービスを使う。

記入例：OAuth 2.0認証の場合¶

1. アクセストークンの取得
   POST https://auth.example.com/oauth/token
   Body: { grant_type: "client_credentials", client_id: xxx, client_secret: xxx }
   → レスポンスの access_token を取得する

2. アクセストークンの有効期限確認
   有効期限（expires_in）が切れていたら再取得する

3. APIリクエスト時にトークンをヘッダーに付与
   Authorization: Bearer {access_token}

チェックリスト¶

  • APIキー・シークレットを環境変数で管理しているか（ソースコードに直書きしていないか）
  • トークンに有効期限がある場合、期限切れ時の再取得処理を定義したか

STEP 4: リトライ設計¶

目的¶

外部APIは通信エラーや一時的な障害で失敗することがあります。自動的に再実行（リトライ）する条件と回数を定義します。

リトライすべきケースとすべきでないケース: - リトライすべき: 一時的なネットワーク障害、外部サービスの5xx系エラー - リトライすべきでない: リクエスト内容が間違っている（4xx系エラー）← 同じリクエストを送り直しても意味がない

記入例¶

I/F-ID リトライ対象エラー 最大リトライ回数 リトライ間隔 リトライ超過時の処理
IF-001 HTTP 500, 503・タイムアウト 3回 5秒・10秒・30秒（指数バックオフ） エラーログ出力・管理者にアラート通知・ユーザーにエラー表示
IF-002 HTTP 500, 503・タイムアウト 2回 10秒 エラーログ出力・手動再送キューに追加

指数バックオフとは: 1回目は5秒後、2回目は10秒後、3回目は30秒後、のようにリトライ間隔を指数的に増やすことで、外部サービスへの負荷を分散する方式。

チェックリスト¶

  • リトライ対象のエラー（HTTP 5xx・タイムアウト）を定義したか
  • リトライ回数の上限を定義したか（無限リトライにしていないか）
  • リトライを超過した場合の処理（通知・ログ・代替処理）を定義したか
  • 決済など「2重実行すると問題が起きる処理」のリトライは冪等性（何度実行しても同じ結果になること）を確保しているか

STEP 5: タイムアウト設定の定義¶

目的¶

外部APIの応答を何秒まで待つかを定義します。タイムアウトを設定しないと、外部サービスが応答しないとき、自システムがいつまでも待ち続けて他の処理ができなくなります。

記入例¶

I/F-ID 接続タイムアウト 読み取りタイムアウト 備考
IF-001 5秒 30秒 決済処理は時間がかかることがある
IF-002 5秒 10秒 —
  • 接続タイムアウト: 外部サーバーへの接続確立を何秒まで待つか
  • 読み取りタイムアウト: 接続後、レスポンスが返ってくるまで何秒まで待つか

チェックリスト¶

  • すべての外部I/Fにタイムアウト値を定義したか
  • ユーザーが待てる時間（UX的な許容時間）を考慮したタイムアウト値になっているか

成果物チェックリスト¶

  • すべての外部I/FについてリクエストとレスポンスのI/F定義を作成したか
  • 認証方式を定義し、APIキーの安全な管理方法を定義したか
  • リトライ設計（回数・間隔・超過時の処理）を定義したか
  • タイムアウト設定を定義したか
  • 外部サービスが障害のとき、自システムはどう動くかを定義したか

完成判定チェックリスト（設計書レビュー用）¶

設計書が完成したと判断する前に、以下の観点で見直しを行います。

I/F網羅性¶

  • 基本設計書のI/F一覧に記載されたすべての外部I/Fについて定義しているか
  • 各I/FにI/F-IDが付与されており、基本設計書と対応しているか
  • 自システム→外部・外部→自システムの両方向を考慮しているか

リクエスト定義¶

  • すべてのリクエスト項目に「型・必須/任意・説明・自システムのデータ元」を定義しているか
  • リクエストヘッダー（Content-Type・認証ヘッダー等）を定義しているか
  • リクエストボディの具体的なJSON例・フォーマット例を記載しているか
  • 外部API仕様書で「必須」とされている項目をすべてカバーしているか
  • 値の変換・加工が必要な項目（名前の結合・日付フォーマット変換など）を明記しているか

レスポンス定義¶

  • 成功時のレスポンス項目を定義し、自システムでの使い方（保存先カラムなど）を記載しているか
  • 外部APIが返しうるすべてのエラーコード・HTTPステータスについて自システムでの処理を定義しているか
  • ユーザーへの表示メッセージ（エラー文言）が必要なケースを定義しているか
  • レスポンスの値を検証・バリデーションする必要があるか検討したか（例：返却された金額が一致するか確認する）

認証¶

  • すべての外部I/Fについて認証方式を定義しているか
  • APIキー・シークレットを環境変数またはシークレット管理サービスで管理する方針を明記しているか
  • OAuth 2.0を使う場合、アクセストークンの取得・有効期限切れ時の再取得フローを定義しているか
  • APIキーのローテーション（更新）頻度・手順を定義しているか

リトライ設計¶

  • リトライ対象とするエラー（5xx・タイムアウト）を明確にしているか
  • 4xxエラー（リクエスト起因のエラー）をリトライ対象から除外しているか
  • リトライ回数の上限を設けているか（無限リトライになっていないか）
  • リトライ間隔（固定 or 指数バックオフ）を定義しているか
  • リトライ上限を超えた場合の処理（ログ・通知・代替処理・手動対応手順）を定義しているか
  • 決済など冪等性が必要な処理について、重複実行を防ぐ仕組みを検討したか

タイムアウト¶

  • すべての外部I/Fに接続タイムアウト・読み取りタイムアウトを定義しているか
  • タイムアウト値がUX（ユーザーの待機許容時間）を考慮した値になっているか
  • タイムアウト発生時の自システムの動作（エラー表示・ロールバック・継続処理）を定義しているか

障害時の振る舞い¶

  • 外部サービス障害時に自システムのどの処理が止まるか・止まらないかを整理しているか
  • 外部I/Fの失敗が自システムのトランザクション（DB更新）に影響するかを明確にしているか（ロールバックするか・しないか）
  • 障害時のユーザー向けメッセージ・管理者向け通知を定義しているか
  • 手動での再実行・リカバリ手順が必要な場合、その手順を記載しているか

全体整合性¶

  • 同一の外部I/Fを複数の画面・バッチから呼び出す場合、定義が統一されているか
  • プロセス設計書（処理フロー）の「外部I/F呼び出し」ステップと、本設計書のI/F-IDが対応しているか
  • 外部APIのバージョン（v1・v2等）を明記しており、将来の変更時に影響箇所を特定できるか

成果物例：社員管理システム¶

STEP 1: I/F一覧¶

I/F-ID 連携種別 相手システム 方向 概要
IF-EMP-001 REST API呼び出し 社内メール配信サービス 自システム → 外部 社員登録時・退職時のメール通知送信
IF-EMP-002 REST API呼び出し 勤怠管理システム 自システム → 外部 社員登録・退職時に勤怠管理システムへ社員情報を連携
IF-EMP-003 ファイル受け取り 採用管理システム 外部 → 自システム 内定者情報のCSVインポート（入社前準備）

STEP 2: APIリクエスト・レスポンスの項目定義¶

社員登録通知メール（IF-EMP-001）¶

基本情報

項目 内容
I/F-ID IF-EMP-001
相手システム 社内メール配信サービス（MailSend API）
エンドポイント POST https://mail.internal.example.com/api/v1/send
呼び出し契機 社員情報登録処理の完了後（PROC-001の手順9の後）
タイムアウト 接続：5秒 / 読み取り：15秒

リクエスト定義

項目名 型 必須 説明 値の例 自システムのデータ元
to string 必須 送信先メールアドレス "tanaka@example.com" employees.email（登録社員）
