処理詳細設計（ロジック設計）¶

このドキュメントの目的¶

処理詳細設計は、各機能の内部ロジック（処理の流れ・分岐・例外処理）をプログラムに落とし込める粒度まで設計する作業です。

基本設計では「ログイン機能」として概要を定義しましたが、詳細設計では「リクエストの検証→DBへの問い合わせ→認証トークンの発行→レスポンス返却」という具体的な手順まで設計します。

製造担当者へのメッセージ: このドキュメントを読めば、コードを「どう書くか」ではなく「何を書くか」が明確にわかる状態を目指します。

作業の流れ¶

1. 処理フロー図の作成
2. 擬似コード（プログラム設計書）の作成
3. バッチ処理設計（バッチがある場合のみ）

STEP 1: 処理フロー図の作成¶

処理フロー図とは¶

処理の開始から終了までの流れを、図で表現したものです。分岐条件・ループ・例外処理を視覚的に示します。

なぜ図で表現するか¶

文章だけでは「どの条件のとき何をするか」が読み手によって解釈が異なることがあります。図にすることで、処理の流れを誰でも同じように理解できます。

フローチャートの基本記号¶

記号の形 意味 使い方
角丸長方形（開始・終了） 処理の始まりと終わり フローの最初と最後
長方形（処理） 実行する処理内容 DBアクセス・計算・変換など
ひし形（分岐） Yes/No の条件分岐 if 文に相当する判断
平行四辺形（入出力） データの入力・出力 画面入力値の受け取り・結果の返却
矢印 処理の流れ 次の処理への順番を示す

ツールの候補: draw.io（無料）・Mermaid（テキストで書ける）・PowerPoint などが使いやすいです。

Mermaidによるフロー図の例：ログイン処理¶

flowchart TD
    A([開始]) --> B[リクエストからメールアドレス・パスワードを取得]
    B --> C{メールアドレスが空か？}
    C -- Yes --> D[エラー返却: 400 Bad Request]
    C -- No --> E{パスワードが空か？}
    E -- Yes --> D
    E -- No --> F[DBからメールアドレスでユーザーを検索]
    F --> G{ユーザーが存在するか？}
    G -- No --> H[エラー返却: 401 Unauthorized]
    G -- Yes --> I{パスワードが一致するか？}
    I -- No --> H
    I -- Yes --> J[認証トークンを発行する]
    J --> K[レスポンスにトークンをセットして返却]
    K --> L([終了])
    D --> L
    H --> L

チェックリスト¶

  • 正常系（うまくいった場合）の流れが書かれているか
  • 異常系（エラー・例外）の流れが書かれているか
  • すべての分岐に行き先（Yes/No両方）があるか
  • 処理の開始と終了が明確か

STEP 2: 擬似コード（プログラム設計書）の作成¶

擬似コードとは¶

実際のプログラミング言語ではなく、日本語と構造化された記述でロジックを表現したものです。

なぜ実際のコードを書かないのか: 詳細設計の段階では「何をするか」を確定させることが目的です。「どう書くか（言語・ライブラリの選択）」は製造担当者に委ねます。また、設計者がコードを書いてしまうと、レビューが難しくなります。

擬似コードの書き方ルール¶

  1. インデントでネスト（入れ子）の深さを表現する
  2. IF / ELSE / ELSE IF で分岐を表現する
  3. FOR / WHILE でループを表現する
  4. 処理の単位は「動詞＋目的語」で書く（「ユーザーを取得する」「エラーを返す」など）
  5. 条件は具体的な値・状態で書く（「正常な場合」ではなく「ステータスコードが200の場合」）

擬似コードの例：ユーザー情報取得処理¶

関数: ユーザー情報取得（ユーザーID）

1. ユーザーIDが空の場合
   → エラー（400 Bad Request）「ユーザーIDは必須です」を返す。処理終了。

2. DBの usersテーブルからユーザーIDが一致するレコードを1件取得する

3. レコードが存在しない場合
   → エラー（404 Not Found）「指定されたユーザーが見つかりません」を返す。処理終了。

4. 取得したレコードをレスポンス用の形式に変換する
   - user_id → id
   - user_name → name
   - created_at → registeredAt（ISO8601形式）
   ※ パスワードハッシュはレスポンスに含めない

5. 変換後のデータをレスポンス（200 OK）として返す。処理終了。

擬似コードの例：注文登録処理（トランザクションあり）¶

関数: 注文登録（注文データ）

1. 入力値バリデーション
   1-1. 商品IDが空の場合 → エラー（400）を返す
   1-2. 数量が1未満の場合 → エラー（400）「数量は1以上を指定してください」を返す

2. DBから商品情報を取得する（商品ID で検索）
   2-1. 商品が存在しない場合 → エラー（404）を返す
   2-2. 商品の在庫数 < 注文数量 の場合 → エラー（409）「在庫が不足しています」を返す

3. トランザクション開始

4. ordersテーブルに注文レコードをINSERTする
   - order_date：現在日時
   - user_id：ログインユーザーのID
   - status：「受付済み」

5. productsテーブルの在庫数を（現在の在庫数 - 注文数量）にUPDATEする

6. 手順4・5が両方成功した場合 → コミット
   いずれかが失敗した場合 → ロールバックし、エラー（500）を返す

7. 登録した注文IDをレスポンスとして返す

チェックリスト¶

  • 処理の手順に番号を振り、順番が明確か
  • 分岐条件は「具体的な値・状態」で書かれているか（「正常な場合」などの曖昧な記述がないか）
  • すべての分岐でエラー時の処理も書かれているか
  • トランザクションが必要な箇所を明示しているか
  • レスポンスに含めてはいけない情報（パスワードなど）を明記しているか

STEP 3: バッチ処理設計（バッチがある場合のみ）¶

バッチ処理とは¶

人が操作せず、一定の時刻や条件で自動的に大量のデータを処理するプログラムのことです。

例: 毎日深夜0時に「当日の売上を集計してレポートを生成する」処理。

定義する内容¶

定義項目 説明 例
バッチID 識別名 BATCH-001
バッチ名 処理の名前 売上日次集計バッチ
実行スケジュール いつ実行するか 毎日 00:05
実行契機 自動実行 or 手動実行 自動（cron）
処理対象データ 何のデータを処理するか 当日（処理日）の注文データ
処理内容 何をするか（擬似コードで記述） 注文を集計して日次売上テーブルに格納
想定件数・処理時間 性能の見積もり 1万件・5分以内
リトライ方針 エラー時の再実行ルール 3回まで・5分間隔でリトライ
排他制御 同時実行防止の方法 ロックファイルを使用して多重起動を防ぐ
正常完了の確認方法 正常に終わったかを確認する方法 終了ステータス0・完了ログをSlackに通知

チェックリスト¶

  • バッチの実行スケジュールと契機を定義したか
  • リトライ方針（回数・間隔・上限）を定義したか
  • 多重起動防止（排他制御）の方法を定義したか
  • 大量データを処理する場合の性能（件数・時間）を見積もったか
  • バッチ失敗時の通知・アラートを定義したか

成果物チェックリスト¶

網羅性（抜け漏れがないか）¶

  • 基本設計書に記載されたすべての機能・処理に対して設計したか
  • すべての機能に処理フロー図を作成したか
  • すべての機能に擬似コード（プログラム設計書）を作成したか
  • バッチ処理がある場合はバッチ処理設計を作成したか
  • 処理フロー図と擬似コードの内容が対応しているか（どちらかだけに存在するロジックがないか）

正確性（内容が正しいか）¶

  • 基本設計書の内容と矛盾がないか（機能名・処理概要・画面との連携など）
  • API仕様書・外部I/F設計書と呼び出し方法・パラメータ・レスポンス形式が一致しているか
  • バリデーション条件がDB定義（桁数・型・NULL制約・ユニーク制約）と整合しているか
  • テーブル名・カラム名・ステータス値がDB設計書と一致しているか

処理フロー図（分岐・流れが正しいか）¶

  • 正常系（すべての処理が成功した場合）の流れが書かれているか
  • すべての異常系（バリデーションエラー・DBエラー・外部API失敗など）の流れが書かれているか
  • すべての分岐にYes/No両方の行き先があるか（行き止まりになっていないか）
  • 処理の開始と終了が明確に示されているか

擬似コード（ロジックが正しいか）¶

  • 処理の手順に番号を振り、実行順序が明確か
  • 分岐条件は「具体的な値・状態」で書かれているか（「正常な場合」などの曖昧な表現がないか）
  • すべてのエラーにエラーコードとメッセージを定義しているか
  • トランザクションが必要な箇所でトランザクション開始・コミット・ロールバックを明示しているか
  • レスポンスに含めてはいけない情報（パスワードハッシュなど）を明記しているか
  • 複数テーブルを更新する場合、更新順序と失敗時のロールバック対象を定義しているか

バッチ処理（バッチがある場合）¶

  • 実行スケジュールと契機（自動/手動）を定義したか
  • リトライ方針（回数・間隔・上限）を定義したか
  • 多重起動防止（排他制御）の方法を定義したか
  • 大量データ処理時の想定件数・処理時間を見積もったか
  • バッチ失敗時の通知・アラートと対応手順を定義したか

実装可能性（製造担当者が迷わないか）¶

  • 用語・ID・名称が設計書内で統一されているか（表記ゆれがないか）
  • 曖昧な表現（「適切に」「必要に応じて」など）が残っていないか
  • 他の機能・処理と連携している部分に参照先（設計書名・機能ID）を記載したか

成果物例：社員管理システム¶

対象機能：社員情報登録処理（EMP-PROC-001）¶

STEP 1: 処理フロー図¶

flowchart TD
    A([開始]) --> B[リクエストから社員情報を取得]
    B --> C{入力バリデーション}
    C -- エラーあり --> D[エラー返却: 400 Bad Request]
    C -- エラーなし --> E[DBから社員番号の重複チェック]
    E --> F{重複あり？}
    F -- Yes --> G[エラー返却: 409 Conflict「社員番号が重複しています」]
    F -- No --> H[DBからメールアドレスの重複チェック]
    H --> I{重複あり？}
    I -- Yes --> J[エラー返却: 409 Conflict「メールアドレスが重複しています」]
    I -- No --> K[トランザクション開始]
    K --> L[employeesテーブルにINSERT]
    L --> M{INSERT成功？}
    M -- No --> N[ロールバック → エラー返却: 500]
    M -- Yes --> O[employee_departmentsテーブルにINSERT]
    O --> P{INSERT成功？}
    P -- No --> N
    P -- Yes --> Q[コミット]
    Q --> R[登録社員情報をレスポンスとして返却: 201 Created]
    R --> S([終了])
    D --> S
    G --> S
    J --> S
    N --> S

STEP 2: 擬似コード（プログラム設計書）¶

関数: 社員情報登録（リクエストデータ）

1. 入力値バリデーション
   1-1. 社員番号（emp_code）が空の場合 → エラー（400）「社員番号は必須です」を返す
   1-2. 社員番号が「E」＋6桁数字の形式でない場合 → エラー（400）「社員番号の形式が不正です」を返す
   1-3. 姓（last_name）が空または20文字超の場合 → エラー（400）「姓の入力値が不正です」を返す
   1-4. 名（first_name）が空または20文字超の場合 → エラー（400）「名の入力値が不正です」を返す
   1-5. メールアドレス（email）が空またはメール形式でない場合 → エラー（400）「メールアドレスの形式が不正です」を返す
   1-6. 入社日（hire_date）が空または無効な日付の場合 → エラー（400）「入社日の形式が不正です」を返す
   1-7. 入社日が本日より未来の場合 → エラー（400）「入社日は本日以前の日付を指定してください」を返す
   1-8. 部署ID（department_id）が空の場合 → エラー（400）「部署は必須です」を返す

2. DBの employeesテーブルから emp_code が一致するレコードを検索する
   2-1. レコードが存在する場合 → エラー（409）「社員番号がすでに使用されています」を返す。処理終了。

3. DBの employeesテーブルから email が一致するレコードを検索する
   3-1. レコードが存在する場合 → エラー（409）「メールアドレスがすでに登録されています」を返す。処理終了。

4. DBの departmentsテーブルから department_id が一致するレコードを検索する
   4-1. レコードが存在しない場合 → エラー（404）「指定された部署が存在しません」を返す。処理終了。

5. トランザクション開始

6. employeesテーブルに社員情報をINSERTする
   - emp_code：リクエストの社員番号
   - last_name / first_name：リクエストの姓名
   - email：リクエストのメールアドレス
   - hire_date：リクエストの入社日（YYYY-MM-DD形式に変換）
   - employment_type：リクエストの雇用区分
   - status：「在籍中」固定
   - created_at / updated_at：現在日時

7. employee_departmentsテーブルに所属部署をINSERTする
   - employee_id：手順6で発行されたID
   - department_id：リクエストの部署ID
   - start_date：入社日

8. 手順6・7が両方成功した場合 → コミット
   いずれかが失敗した場合 → ロールバックし、エラー（500）「社員情報の登録に失敗しました」を返す

9. 登録した社員情報（employee_idを含む）をレスポンス（201 Created）として返す。処理終了。
