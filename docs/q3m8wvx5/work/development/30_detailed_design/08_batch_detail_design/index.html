バッチ詳細設計¶

このドキュメントの目的¶

バッチ詳細設計は、基本設計で定義したバッチ処理の概要・スケジュール・エラー方針を受けて、製造担当者がコードを書ける粒度まで処理ロジックを具体化する作業です。

基本設計との違い:
基本設計では「何を・いつ・どんな方針で動かすか」を決めます。
詳細設計では「処理の各ステップ・分岐条件・SQL・排他制御・リトライロジック」まで落とし込みます。

作業の流れ¶

1. 処理フロー図の作成
2. 擬似コード（プログラム設計書）の作成
3. SQL設計
4. 排他制御・多重起動防止の設計
5. リトライ・タイムアウト設計
6. ログ出力設計

STEP 1: 処理フロー図の作成¶

バッチ処理全体の流れを図で表現します。特に以下の分岐を明示します。

  • 前提条件チェック（先行バッチの完了確認・入力ファイルの存在確認など）
  • メインループ（対象データを1件ずつ・チャンク単位で処理する流れ）
  • エラー時の分岐（致命的エラーで中断 / 非致命的エラーでスキップ）
  • 終了処理（ログ出力・通知・ロック解放）

フロー図の例：日次売上集計バッチ（BAT-SAL-001）¶

flowchart TD
    A([開始]) --> B[ロックファイルの取得]
    B --> C{ロック取得できたか？}
    C -- No --> D[多重起動エラーをログに記録して終了]
    C -- Yes --> E[batch_executionsに実行開始レコードをINSERT]
    E --> F[前日分の確定注文を全件取得]
    F --> G{対象件数が0件か？}
    G -- Yes --> H[警告ログを出力して正常終了]
    G -- No --> I[対象件数が前日比50%以下か？]
    I -- Yes --> J[警告メールを送信して処理続行]
    I -- No --> K[カテゴリ別・日別に集計クエリを実行]
    J --> K
    K --> L{クエリ成功？}
    L -- No --> M[致命的エラー：中断・batch_executions更新・アラートメール]
    L -- Yes --> N[sales_summaryにUPSERT]
    N --> O{UPSERT成功？}
    O -- No --> M
    O -- Yes --> P[batch_executionsを完了に更新]
    P --> Q[ロックファイルを解放]
    Q --> R[完了ログを出力]
    R --> S([正常終了])
    D --> T([異常終了])
    M --> Q2[ロックファイルを解放]
    Q2 --> T
    H --> Q

STEP 2: 擬似コード（プログラム設計書）の作成¶

処理の手順を、言語に依存しない日本語＋構造化記述で表現します。

擬似コードの例：日次売上集計バッチ（BAT-SAL-001）¶

バッチ: BAT-SAL-001 日次売上集計
引数: target_date（省略時は実行日の前日）

1. ロックファイル（/tmp/bat-sal-001.lock）の取得を試みる
   1-1. 既にロックファイルが存在する場合
        → ERRORログ「多重起動を検知しました」を出力して異常終了（exit 1）

2. batch_executionsテーブルにレコードをINSERTする
   - batch_id: 'BAT-SAL-001'
   - executed_at: 現在日時
   - status: 'running'
   - target_date: target_date

3. target_dateを対象として、ordersテーブルから以下の条件でレコードを取得する
   - status = 'confirmed'
   - DATE(confirmed_at) = target_date

4. 取得件数が0件の場合
   → WARNログ「処理対象データが0件でした」を出力
   → 手順8（正常終了処理）へ

5. 取得件数が前回実行時（batch_executionsテーブルの直前の成功レコード）の件数の50%以下の場合
   → WARNログ「処理件数が前回の50%以下です（前回: {前回件数}件 / 今回: {今回件数}件）」を出力
   → 担当者に警告メールを送信する（処理は続行）

6. ordersテーブルとorder_items・productsテーブルをJOINし、カテゴリ別・日別に金額を集計するクエリを実行する
   （詳細は「STEP 3: SQL設計」を参照）
   6-1. クエリが失敗した場合
        → ERRORログ「集計クエリが失敗しました: {エラー内容}」を出力
        → 手順9（異常終了処理）へ

7. 集計結果をsales_summaryテーブルにUPSERTする（1レコードずつ）
   7-1. UPSERTが失敗した場合
        → ERRORログ「UPSERTに失敗しました: {エラー内容}」を出力
        → 手順9（異常終了処理）へ

8. 正常終了処理
   8-1. batch_executionsのstatusを'success'に更新する（success_count・error_countも更新）
   8-2. INFOログ「BAT-SAL-001 完了（件数: {件数} / 経過時間: {秒数}秒）」を出力
   8-3. ロックファイルを削除する
   8-4. exit 0 で終了

9. 異常終了処理
   9-1. batch_executionsのstatusを'failed'に更新する（messageにエラー詳細を記録）
   9-2. 開発チームにアラートメールを送信する（件名: 「[ALERT] BAT-SAL-001 異常終了」）
   9-3. ロックファイルを削除する
   9-4. exit 1 で終了

STEP 3: SQL設計¶

バッチ処理で使用するSQLを設計します。インデックスの利用状況・実行計画も確認します。

設計するSQLの種類¶

種別 説明
データ取得 処理対象レコードを取得するSELECT文
集計 GROUP BY・SUM等を使った集計クエリ
更新・挿入 UPSERT・INSERT・UPDATE文（冪等性を考慮）
実行履歴管理 batch_executionsテーブルへのINSERT・UPDATE

SQL設計の例¶

処理対象データ取得¶

-- BAT-SAL-001: 前日の確定注文を取得
-- 使用インデックス: idx_orders_status_confirmed_at（status, confirmed_at の複合インデックス）
SELECT
    o.id          AS order_id,
    o.confirmed_at,
    oi.product_id,
    oi.quantity,
    oi.unit_price,
    p.category_id
FROM orders o
    INNER JOIN order_items oi ON o.id = oi.order_id
    INNER JOIN products    p  ON oi.product_id = p.id
WHERE
    o.status       = 'confirmed'
    AND DATE(o.confirmed_at) = :target_date  -- バインドパラメータ（例: '2025-01-31'）
ORDER BY
    p.category_id, o.id;

集計クエリ¶

-- カテゴリ別・日別に売上を集計する
SELECT
    DATE(o.confirmed_at)  AS sale_date,
    p.category_id,
    COUNT(DISTINCT o.id)  AS order_count,
    SUM(oi.quantity * oi.unit_price) AS total_amount
FROM orders o
    INNER JOIN order_items oi ON o.id = oi.order_id
    INNER JOIN products    p  ON oi.product_id = p.id
WHERE
    o.status       = 'confirmed'
    AND DATE(o.confirmed_at) = :target_date
GROUP BY
    DATE(o.confirmed_at), p.category_id;

UPSERTによる冪等な書き込み¶

-- ✅ 同じ日付・カテゴリで再実行しても重複が発生しない
INSERT INTO sales_summary (sale_date, category_id, order_count, total_amount, created_at, updated_at)
VALUES (:sale_date, :category_id, :order_count, :total_amount, NOW(), NOW())
ON DUPLICATE KEY UPDATE
    order_count  = VALUES(order_count),
    total_amount = VALUES(total_amount),
    updated_at   = VALUES(updated_at);

インデックス設計の確認ポイント:
- WHERE句の条件カラムにインデックスが存在するか
- 結合条件（JOIN ON句）のカラムにインデックスが存在するか
- EXPLAIN / EXPLAIN ANALYZEで実行計画を確認したか

STEP 4: 排他制御・多重起動防止の設計¶

バッチの二重起動は、データの重複・不整合を引き起こします。必ず防止策を設計します。

主な排他制御の方法¶

方法 概要 向いているケース
ロックファイル 起動時にファイルを作成し、終了時に削除する 単一サーバーで動作するバッチ
DBテーブルロック batch_executionsのstatusを確認して二重起動を防ぐ 複数サーバーで動作する分散環境
スケジューラーの排他設定 cronやジョブスケジューラー側で多重起動を禁止する スケジューラーに機能がある場合

ロックファイルによる排他制御の設計¶

ロックファイルパス: /var/lock/bat-sal-001.lock

起動時:
  1. ロックファイルが存在するかチェックする
  2. 存在する場合 → PIDが実際に動作中かを確認する
     - 動作中なら二重起動エラーとして異常終了する
     - 動作していない（前回の強制終了等でロックが残存）なら古いロックを削除して続行する
  3. 存在しない場合 → ロックファイルを作成して処理を開始する

終了時（正常・異常いずれも）:
  1. ロックファイルを削除する
  ※ シグナルハンドラ（SIGTERM・SIGINT）でもロックを解放するよう実装する

DBテーブルによる排他制御の設計¶

-- 起動時に実行中ステータスのレコードが存在しないかチェック
SELECT COUNT(*) AS running_count
FROM batch_executions
WHERE batch_id = 'BAT-SAL-001'
  AND status = 'running'
  AND executed_at >= NOW() - INTERVAL 2 HOUR;  -- 2時間以上前の'running'は異常終了とみなす

-- running_count > 0 の場合は二重起動エラーとして終了する

STEP 5: リトライ・タイムアウト設計¶

リトライ設計¶

定義項目 内容
リトライ対象エラー 一時的な障害が原因と考えられるもの（DBの一時的な接続エラー・タイムアウト等）
リトライ対象外エラー データ不正・ロジックエラーなど、リトライしても解決しないもの
最大リトライ回数 3回（最大4回実行）
リトライ間隔 指数バックオフ（1回目: 30秒、2回目: 60秒、3回目: 120秒）
最大リトライ後の動作 異常終了・アラートメール送信

リトライの擬似コード:

関数: DBクエリを実行する（クエリ, 最大試行回数=3）

  試行回数 = 0
  WHILE 試行回数 < 最大試行回数:
    TRY:
      クエリを実行する
      成功したら RETURN 結果
    CATCH 一時的な接続エラー:
      試行回数 += 1
      待機秒数 = 30 × 2^(試行回数 - 1)  -- 指数バックオフ
      WARNログ「DBエラー。{待機秒数}秒後にリトライします（{試行回数}/{最大試行回数}）」
      {待機秒数}秒スリープする
    CATCH その他のエラー:
      リトライしない → そのまま例外を上位に投げる

  最大試行回数を超えた場合 → 例外を上位に投げる

タイムアウト設計¶

タイムアウト種別 設定値 超過時の動作
バッチ全体のタイムアウト 60分 強制終了・異常終了処理を実行・アラートメール
DBクエリのタイムアウト 5分 クエリを中断・リトライ対象エラーとして扱う
外部API呼び出しのタイムアウト 30秒 リトライ対象エラーとして扱う

バッチ全体のタイムアウト実装方法:

# cronでの設定例（timoutコマンドを使用）
0 1 * * * timeout 3600 /opt/batch/bat-sal-001.sh >> /var/log/bat-sal-001.log 2>&1

STEP 6: ログ出力設計¶

ログに必ず含める情報¶

項目 説明 例
タイムスタンプ ミリ秒まで記録する 2025-01-31 01:00:00.123
ログレベル INFO / WARN / ERROR INFO
バッチID 複数バッチが同一ログファイルに出力される場合に必要 BAT-SAL-001
メッセージ 何が起きたかを記録する 日次売上集計バッチ 開始
件数・パラメータ 処理件数・対象日など 対象件数: 1,234件 / target_date: 2025-01-31

ログフォーマット¶

[タイムスタンプ] [レベル] [バッチID] メッセージ

出力例:

[2025-01-31 01:00:00.001] INFO  BAT-SAL-001 バッチ開始 (target_date=2025-01-30)
[2025-01-31 01:00:00.045] INFO  BAT-SAL-001 ロック取得完了
[2025-01-31 01:00:00.102] INFO  BAT-SAL-001 対象件数: 1,234件
[2025-01-31 01:00:00.110] WARN  BAT-SAL-001 処理件数が前回の50%以下です (前回: 2,501件 / 今回: 1,234件)
[2025-01-31 01:00:05.320] INFO  BAT-SAL-001 集計クエリ完了 (カテゴリ数: 12)
[2025-01-31 01:00:08.540] INFO  BAT-SAL-001 UPSERT完了 (12件)
[2025-01-31 01:00:08.600] INFO  BAT-SAL-001 バッチ正常終了 (成功: 1,234件 / エラー: 0件 / 経過時間: 8.5秒)

ログレベルの基準¶

レベル 使用場面
INFO 開始・終了・処理件数など正常な記録
WARN 処理は続行するが注意が必要な状態（件数異常・データ品質の警告など）
ERROR 処理を中断する致命的なエラー

成果物チェックリスト¶

処理フロー図¶

  • 前提条件チェック（先行バッチの完了確認・ファイルの存在確認）が含まれているか
  • メインループ（対象データの取得〜更新）が図示されているか
  • 致命的エラー（中断）と非致命的エラー（スキップ）の分岐が明確か
  • 終了処理（ログ出力・ロック解放）が記載されているか

擬似コード¶

  • 処理ステップに番号が振られており、順番が明確か
  • 各分岐の条件が「具体的な値・状態」で書かれているか
  • 正常終了・異常終了の両パターンが記述されているか
  • 引数（対象日・対象範囲）の受け取り方が定義されているか

SQL設計¶

  • 使用するSQLがすべて記載されているか（SELECT・UPSERT・INSERT等）
  • UPSERTまたはDELETE+INSERTで冪等性が担保されているか
  • SQLにバインドパラメータを使用しているか（SQLインジェクション対策）
  • WHERE句の条件カラムにインデックスが設定されているか確認したか
  • EXPLAIN等で実行計画を確認したか（大量データを扱う場合）

排他制御¶

  • 多重起動防止の方法（ロックファイル / DBテーブル / スケジューラー設定）が定義されているか
  • 強制終了時にロックが残存した場合の対処が定義されているか

リトライ・タイムアウト¶

  • リトライ対象エラーとリトライ対象外エラーが区別されているか
  • リトライ回数・間隔が定義されているか
  • バッチ全体のタイムアウト時間と超過時の動作が定義されているか

ログ¶

  • ログフォーマット（タイムスタンプ・レベル・バッチID・メッセージ）が統一されているか
  • 開始・終了・処理件数・エラー詳細がログに記録されるか
  • ログ出力先（ファイルパス）が定義されているか

完成判定チェックリスト¶

設計書を書き終えた後、以下の観点で見直しを行い、すべてにチェックが入った状態を完成とします。

1. 基本設計との整合性¶

  • 基本設計で定義したバッチID・バッチ名と一致しているか
  • 基本設計で定めたスケジュール（起動時刻・頻度）と矛盾していないか
  • 基本設計で定めたエラー方針（致命的/非致命的の分類）と一致しているか
  • 基本設計で定めた依存バッチ・先行条件が処理フローに反映されているか

2. 処理フローの完全性¶

  • フロー図と擬似コードの分岐・ステップが一致しているか（どちらかに存在してもう一方にない分岐がないか）
  • すべての終端（正常終了・異常終了・早期終了）がフロー図に存在するか
  • エラー発生時に必ずロック解放・履歴更新・ログ出力が行われる経路になっているか
  • 0件データの場合の動作が定義されているか（警告か正常終了か）

3. SQL・データ設計の妥当性¶

  • 処理対象の取得条件が基本設計の要件（集計範囲・対象ステータス等）と一致しているか
  • 再実行（リカバリ）したときに重複・不整合が発生しない構造になっているか（冪等性）
  • SQLのWHERE句・JOIN条件に使用するカラムにインデックスが存在するか確認したか
  • 大量データを扱うSQLについてEXPLAINで実行計画を確認したか

4. 実装可能性の確認¶

  • 擬似コードの各ステップを読んだ担当者が、設計書を見ながらコードを書けるレベルに具体化されているか
  • 「詳細はXXX参照」などの未定義の参照先がすべて解決されているか
  • テーブル名・カラム名・バインドパラメータ名が実際のスキーマと一致しているか

5. 運用・障害対応の考慮¶

  • リトライの最大待機時間（回数×間隔）がバッチ全体のタイムアウトを超えないか計算したか
  • 強制終了・サーバー障害でロックが残った場合の対処手順が定義されているか
  • 再実行（リカバリ）の手順（引数の指定方法・実行前の確認事項）が明確になっているか
  • 監視チームやオンコール担当者がアラートを受け取ったときに何をすべきかが設計書から読み取れるか

6. ログ・追跡可能性¶

  • 異常終了時のログからエラー原因が特定できるだけの情報が記録されるか（エラーメッセージ・対象データ）
  • batch_executionsテーブル等で実行履歴が追跡できるか
  • 処理件数が0件・極端に少ない場合に検知できる仕組みがあるか

よくある失敗¶

SQLが冪等でない¶

INSERT だけで実装すると再実行時に重複レコードが発生する。設計時に UPSERT（ON DUPLICATE KEY UPDATE）または DELETE+INSERT のどちらにするかを明示する。

ロック解放を finally に入れていない¶

例外で終了したときにロックファイルが残ると、次回以降すべて多重起動エラーになる。正常終了・異常終了のどちらでも必ずロックを解放する設計にする。

リトライとタイムアウトが組み合わさると合計待機時間が計算されていない¶

「3回リトライ・各120秒待機」＋「1回あたり最大5分のDBタイムアウト」の場合、最悪 25分以上かかる。バッチ全体のタイムアウトより合計待機時間が長くならないか確認する。

ログに件数が出力されない¶

正常終了しているのに処理件数が0件（データ取得条件のミスなど）でも気づけない。必ず処理件数をINFOログに出力し、0件の場合は警告として出力する設計にする。

引数（対象日）を受け取れない設計になっている¶

固定で「昨日」を計算するだけの実装では、過去日のリカバリ再実行ができない。対象日を引数で指定できる設計にしておく。
