STEP 4　要件定義書テンプレートと記載ガイド¶

このドキュメントの目的¶

STEP 4「要件定義書の作成」を行うためのテンプレートと、各セクションの書き方ガイドを提供します。 要件定義書を初めて書く方でも、このテンプレートに沿って記入することで一定品質のドキュメントを作成できます。

要件定義書とは何か¶

要件定義書は、顧客と開発チームの「合意文書」 です。

「何を作るか」について双方が合意した内容を正式に文書化したものであり、 後工程（設計・開発・テスト）のすべての基盤になります。

要件定義書が果たす役割¶

役割 説明
合意の証拠 「言った・言わない」のトラブルを防ぐ
開発の基準 設計・実装の拠り所になる
テストの根拠 「要件を満たしているか」を確認する基準
スコープの管理 後から追加要求が来たときに判断基準になる

記載ガイド¶

各セクションの書き方のポイントを説明します。

セクション1：プロジェクト概要¶

1.1 背景・目的¶

良い書き方： 「なぜ（Why）」を明確にする

例： 現在、受注管理はExcelで行っているが、社員数が増加した結果、1日あたりの入力件数が200件を超え、Excelの管理限界に達している。転記ミスによる顧客クレームも月2〜3件発生しており、業務効率化と品質向上が急務となっている。本システムは、受注管理を専用Webシステムに移行することで、入力ミスの撲滅と処理時間の50%削減を目的とする。

避けるべき書き方： - 「業務効率化のため」（抽象的すぎる） - 「既存システムの老朽化のため」（何が問題かが不明）

1.4 ゴール・成功基準¶

重要： ゴールは「測定可能な形」で書くことが原則です。

良い書き方 避けるべき書き方
「受注入力を1件5分以内で完了できる」 「受注入力を効率化する」
「月次のミス件数をゼロにする」 「ミスを減らす」
「承認に要する時間を3日から4時間以内にする」 「承認を早くする」

セクション3：機能要件¶

機能詳細の書き方¶

機能詳細は「ユーザーがどう操作するか、システムがどう反応するか」をステップで書きます。

書き方の例（受注登録機能）：

処理内容：
1. ユーザーが「受注登録」画面を開く
2. 顧客名・商品・数量・納期を入力する
3. 「登録」ボタンをクリックする
4. システムが入力内容をバリデーションする
   - 顧客名：必須、100文字以内
   - 商品：選択必須
   - 数量：1以上の整数
   - 納期：今日以降の日付
5. バリデーションエラーがある場合：エラーメッセージを表示し、入力に戻る
6. バリデーションが通った場合：受注データを登録し「登録完了」画面を表示する
7. 承認担当者にメール通知を送信する

バリデーションは必ず記載する

「何を入力できるか」「何は入力できないか」を明確にします。 仕様の曖昧さは開発者が自分で判断して実装してしまい、後から「そうじゃない」となりがちです。

セクション4：非機能要件¶

非機能要件は「具体的な数値」で記載します。

避けるべき書き方 良い書き方
「応答が速いこと」 「通常の検索操作は3秒以内に結果を表示する」
「セキュリティを強化すること」 「パスワードは最低8文字・英数字記号の混在を必須とし、bcryptでハッシュ化して保存する」
「高可用性であること」 「平日9時〜18時の稼働率を99.5%以上とする」

セクション5：画面一覧¶

画面一覧は、システムに存在するすべての画面を一覧化したものです。 設計フェーズのUI設計の出発点になります。

画面一覧の例：

画面ID 画面名 概要 利用者
SC001 ログイン画面 ID/パスワードでシステムにログインする 全ユーザー
SC002 ダッシュボード 受注状況のサマリを表示する 全ユーザー
SC003 受注一覧画面 受注データを一覧・検索する 営業・上長
SC004 受注登録画面 新規受注データを登録する 営業担当
SC005 受注詳細画面 受注データの詳細を確認する 全ユーザー
SC006 承認画面 受注の承認・差し戻しを行う 上長
SC007 請求書一覧画面 請求書の一覧を表示・管理する 経理担当

セクション9：対象外事項¶

「対象外事項」は、要件定義書の中でも特に重要なセクションです。

なぜ重要か： 「書いていないから対応範囲内だと思っていた」というトラブルを防ぐため。

書き方のポイント： - 「将来対応予定」のものも「今回は対象外」として明記する - 顧客が希望したが実現しないものも記載し、理由を添える

よくある品質問題と対策¶

問題 対策
機能の説明が曖昧で開発者が解釈に迷う 処理の流れを番号付きステップで記載する
バリデーション（入力チェック）が未記載 必ず「入力」セクションに必須/任意・形式を書く
非機能要件が「〜すること」のような抽象表現 数値・条件を具体的に記載する
画面が洗い出せていない 業務フロー（To-Be）の各ステップに対応する画面を考える
対象外が書かれていない 「今回はやらないこと」を明示するセクションを作る

成果物チェックリスト¶

STEP 4 完了時に以下が揃っているか確認してください。

  • 要件定義書（全セクション記載済み）
  • 画面一覧（全画面を網羅）
  • ユースケース図 / ユースケース記述（必要な場合）
  • 機能要件：各機能の処理内容・入出力・バリデーション・エラー処理を記載

  • 対象外事項を明記
  • 顧客にレビュー依頼前に、チーム内で内部レビューを実施済み

| 最終更新日 | 2026年6月10日 |
| 作成者 | PM 佐藤 |
| 承認者 | 人事部長 田中様 |
| ステータス | レビュー中 |

## 1. プロジェクト概要

### 1.1 背景・目的

現在、社員情報の管理は人事部・総務・各部門がそれぞれ独自のExcelファイルで行っており、
社員数が150名に増加した結果、以下の問題が顕在化している。

- 複数ファイル間のデータ不整合（月平均5件の修正作業が発生）
- 入社・退職・異動のたびに複数ファイルへの手入力が必要（1件あたり約30分）
- 在宅勤務時はVPN接続が必要でアクセスが困難
- 変更履歴が残らず、誰がいつ変更したか追跡できない

本システムは社員情報をWebシステムで一元管理することで、データ不整合の撲滅と
手続き処理時間の大幅削減を目的とする。

### 1.2 対象ユーザー

| ユーザー種別 | 説明 | 人数（目安） |
|---|---|---|
| 人事担当 | 全社員情報の登録・編集・管理を行う | 3名 |
| 部門マネージャー | 自部門メンバーの情報を閲覧する | 15名 |
| 一般社員 | 自身の情報を閲覧する | 132名 |

### 1.3 システムの全体像

Webブラウザからアクセスできる社員情報管理Webアプリケーション。
ロールに応じたアクセス制御により、人事担当は全社員の情報を管理でき、
部門マネージャーは自部門のみ、一般社員は自分の情報のみ参照できる。
入社・退職・異動の手続きをシステム上で行うと、関係者に自動でメール通知が送信される。

### 1.4 ゴール・成功基準

- 社員情報の入力は1か所のみとし、データ不整合件数をゼロにする
- 入社・退職・異動手続きの処理時間を1件30分から5分以内に削減する
- VPNなしでブラウザからアクセスできる（在宅勤務対応）
- 2026年9月30日までに本番稼働を開始する

---

## 2. 業務フロー

### 2.1 To-Be 業務フロー図

（別添「To-Beフロー図_社員管理_v1.0.pdf」を参照）

### 2.2 業務フローの説明

入社手続きを例にとると、人事担当がシステムに社員情報を1回登録するだけで、
部門マネージャー・IT担当への通知が自動で送信される。
各担当者はシステムにアクセスして情報を確認でき、手動の転記作業は不要となる。

### 3.2 機能詳細

---

#### F001: 社員情報登録

**概要：**
人事担当が新規入社社員の情報をシステムに登録する。

**アクター（誰が使うか）：**
- 人事担当

**前提条件：**
- 人事担当としてログイン済みであること

**処理内容：**
1. ユーザーが「社員登録」画面を開く
2. 以下の情報を入力する（入力欄は「入力」セクション参照）
3. 「登録」ボタンをクリックする
4. システムが入力内容をバリデーションする
5. バリデーションエラーがある場合：エラーメッセージを表示し、入力画面に戻る
6. バリデーションが通った場合：社員情報を登録し「登録完了」画面を表示する
7. 設定された関係者（部門マネージャー・IT担当）に入社通知メールを自動送信する

**入力：**
| 項目名 | 形式 | 必須/任意 | 説明 |
|---|---|---|---|
| 氏名（姓） | テキスト（50文字以内） | 必須 | |
| 氏名（名） | テキスト（50文字以内） | 必須 | |
| 氏名カナ（姓） | テキスト（50文字以内） | 必須 | 全角カタカナのみ |
| 氏名カナ（名） | テキスト（50文字以内） | 必須 | 全角カタカナのみ |
| 生年月日 | 日付 | 必須 | 過去の日付のみ |
| 入社日 | 日付 | 必須 | |
| 所属部門 | 選択（マスタから） | 必須 | |
| 役職 | 選択（マスタから） | 必須 | |
| 雇用形態 | 選択（正社員/契約社員/パート） | 必須 | |
| メールアドレス | メール形式 | 必須 | 重複不可 |
| 内線番号 | 数字（5桁以内） | 任意 | |
| 住所 | テキスト（200文字以内） | 任意 | |
| 緊急連絡先 | 電話番号形式 | 任意 | |

**出力：**
| 項目名 | 形式 | 説明 |
|---|---|---|
| 社員ID | 数値（自動採番） | システムが自動付与する一意のID |
| 登録完了メッセージ | テキスト | 「社員情報を登録しました」 |

**バリデーション（入力チェック）：**
- 氏名（姓・名）・入社日・所属・役職・雇用形態・メールアドレスは必須
- メールアドレスはメール形式であること
- メールアドレスは既存社員と重複しないこと
- 生年月日は今日より過去の日付であること
- 氏名カナは全角カタカナのみ受け付ける

**例外処理・エラー時の動作：**
- 重複メールアドレス：「このメールアドレスはすでに登録されています」と表示
- 必須項目未入力：該当項目の下部に「この項目は必須です」と表示
- システムエラー時：「処理に失敗しました。しばらくしてから再度お試しください」と表示し、ログに記録する

---

#### F008: ロールベースアクセス制御

**概要：**
ユーザーのロール（役割）に応じて、アクセスできる機能・データ範囲を制限する。

**ロール定義：**

| ロール | アクセスできる機能 | 閲覧できる社員の範囲 |
|---|---|---|
| 人事担当 | 全機能（登録・編集・無効化・CSV出力・ログ閲覧） | 全社員 |
| 部門マネージャー | 社員情報閲覧のみ | 自部門メンバーのみ |
| 一般社員 | 自身の情報閲覧のみ | 自分のみ |

## 4. 非機能要件

### 4.1 性能要件

| 項目 | 要件 | 補足 |
|---|---|---|
| 画面応答時間 | 通常操作時：3秒以内 | 検索・一覧表示を含む |
| CSV出力時間 | 150件の全社員データを10秒以内に出力 | |
| 同時接続ユーザー数 | 最大50名 | 人事イベント時（4月等）を想定 |

### 4.2 可用性要件

| 項目 | 要件 | 補足 |
|---|---|---|
| 稼働時間 | 平日7時〜22時（入退社対応のため早朝・夜間も考慮） | |
| 稼働率目標 | 99.0%以上（月間） | |
| 計画停止 | 毎週日曜日 0時〜6時を許容停止時間とする | |
| データバックアップ | 毎日1回（深夜帯） | |
| 障害時のリカバリ目標（RTO） | 4時間以内に復旧 | |

### 4.3 セキュリティ要件

| 項目 | 要件 |
|---|---|
| 認証方式 | ID/パスワード認証（パスワードは8文字以上・英数字混在必須） |
| 認可（アクセス制御） | ロールに応じてアクセス可能な機能・データを制限する（F008参照） |
| データ暗号化 | 通信はHTTPS、パスワードはbcryptでハッシュ化して保存 |
| ログ | 操作ログを1年間保持する |
| 個人情報 | 個人情報保護法に準拠した取り扱いを行う |
| セッション | ログイン後30分操作がない場合は自動ログアウト |

### 4.4 保守性要件

| 項目 | 要件 |
|---|---|
| ログ出力 | エラー発生時に原因特定できる情報（スタックトレース・ユーザーID・操作内容）をログに記録する |
| 障害対応 | 障害発生から4時間以内に原因を特定できること |

### 4.5 移行要件

| 項目 | 内容 |
|---|---|
| 移行対象データ | 現行Excelファイル（社員マスタ.xlsx）の社員情報 150件 |
| 移行方法 | CSVインポート機能を使用してシステムに一括登録する |
| 移行スケジュール | 本番稼働の1週間前（2026年9月22日〜）にデータ移行・確認作業を実施 |
| データクレンジング | 移行前に顧客側でデータの重複・空欄・フォーマット統一を実施する |

## 10. 用語定義

| 用語 | 定義 |
|---|---|
| 社員 | 本システムで管理対象となる在籍中・退職済みを含む全従業員 |
| 無効化 | 退職した社員のアカウントをシステム上で非アクティブ状態にすること。物理削除は行わない |
| ロール | ユーザーに付与される役割（人事担当・マネージャー・一般社員）。ロールによってアクセス権が変わる |
| 変更履歴 | 社員情報が更新された際に記録される、変更前・変更後・変更者・変更日時のログ |
| 論理削除 | データをデータベースから物理的に消さず、「削除済み」フラグを立てることで無効化する手法 |
