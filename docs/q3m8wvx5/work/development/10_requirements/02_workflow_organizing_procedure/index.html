STEP 2　業務フロー整理の手順・表記ルール¶

このドキュメントの目的¶

STEP 2「業務フロー整理」を実施するための具体的な手順と、フロー図の書き方を説明します。 フロー図を書いたことがない方でも、このドキュメントをもとに As-Is / To-Be フロー図を作成できるようになることを目指しています。

業務フロー整理とは何か¶

ヒアリングで収集した情報をもとに、業務の流れを図（フロー図）として可視化する作業です。

「文章で書いても分かりにくい」「人によって理解が違う」という問題を防ぐために、図として表現します。

作成する図は2種類あります。

図の種類 目的
As-Is フロー（現状） 今の業務の流れを正確に把握する
To-Be フロー（将来） システム導入後の理想の業務の流れを設計する

業務フロー整理の全体ステップ¶

1. ヒアリング情報の整理
    ↓
2. As-Is フロー図の作成
    ↓
3. 課題・ムダの洗い出し
    ↓
4. To-Be フロー図の作成
    ↓
5. 顧客との内容確認

ステップ別 詳細手順¶

1. ヒアリング情報の整理¶

フロー図を書く前に、ヒアリングで収集した情報を整理します。

以下の情報を把握してからフロー作成に入ります。

  • 業務に関わる登場人物（役割）は誰か
  • 業務の開始トリガー（何がきっかけで始まるか）
  • 業務の終了条件（何をもって完了とするか）
  • 業務の主な手順（大まかな流れ）
  • 業務中に発生する判断・分岐（条件によって処理が変わる箇所）
  • 業務の中で使われているシステム・ツール・帳票

情報が不足している場合は、この段階で顧客に追加確認します。

2. As-Is フロー図の作成¶

As-Is フローを書く目的¶

「今の業務がどうなっているか」を正確に把握することが目的です。 問題点を見つける前に、まず現実を正確に把握します。

理想や改善案は入れず、今の実態をそのまま図にするのがポイントです。

書き方の手順¶

  1. 登場人物（スイムレーン）を決める 業務に関わる役割（人・部署・システム）を横に並べる

  2. 業務の開始点を書く 何がきっかけで業務が始まるかを「開始」で表す

  3. 処理・作業を順番に書く 誰が何をするかを、該当するスイムレーンに記載する

  4. 判断・分岐を書く 「〜の場合は〇〇、そうでなければ〇〇」という分岐点を菱形で表す

  5. 終了点を書く 業務がどこで完了するかを「終了」で表す

3. 課題・ムダの洗い出し¶

As-Is フローが完成したら、その中から課題・問題・ムダを洗い出します。

以下の視点でフローを見直します。

視点 確認内容 例
無駄な手順 本当に必要な作業か？ 毎回手入力している情報が別システムにある
待ち時間 承認を待っている時間が長い箇所はないか 上長承認に平均3日かかっている
ミスが起きやすい箇所 手作業・手入力が多い場所はないか Excel転記のミスが月3〜4件発生している
重複作業 同じ情報を複数箇所で入力していないか 顧客情報を受注・請求・出荷の3箇所で手入力
担当が不明瞭な箇所 誰がやるか決まっていない作業はないか 「確認する人がいない」ため漏れが発生

課題一覧の書き方¶

No 課題 該当箇所（フローのどこか） 影響 優先度
1 受注データをExcelと基幹システムに二重入力している 受注処理 入力ミス・工数増加 高
2 上長承認に平均3日かかっており業務が止まる 承認フロー リードタイム増加 中

4. To-Be フロー図の作成¶

課題をもとに、システム導入後の理想の業務フローを設計します。

To-Be フローを書く前に確認すること¶

  • どの課題をシステムで解決するか（スコープの確認）
  • システムが自動化・代替する処理はどれか
  • 人間が引き続き行う作業はどれか
  • 新しいフローで新たに発生するリスク・課題はないか

As-Is との変更点を明確にする¶

To-Be フローには、As-Is から変わった点が分かるようにしておくと顧客への説明がしやすくなります。

例：「この処理はシステムが自動で行います」「この承認フローはメール通知に変わります」

5. 顧客との内容確認¶

作成したフロー図は必ず顧客と一緒に確認します。

なぜ確認が必要か： 作成者の理解が間違っていた場合、フローをベースにして作成するすべての成果物（要件定義書・設計書）が間違った方向に進んでしまいます。

確認のポイント¶

  • 「この業務フローは現状の流れを正しく表していますか？」
  • 「この図の中で違和感のある箇所はありますか？」
  • 「To-Be フローを見て、この流れで問題はありますか？」
  • 「システムが自動化する部分の認識は合っていますか？」

確認後に変更があった場合は、フロー図を修正して再度確認を取ります。

フロー図の表記ルール¶

使用する図形（フローチャート記号）¶

以下の記号を使います。ツール（Draw.io、Lucidchart、PowerPoint など）に関わらず共通です。

図形 形状 意味 使用例
開始・終了 丸角四角（角丸長方形） フローの始まりと終わり 「業務開始」「業務完了」
処理 四角形 作業・処理 「受注データを入力する」
判断 菱形（◇） 条件分岐（Yes/No） 「在庫はあるか？」
ドキュメント 波底四角 帳票・書類・ファイル 「受注伝票」「請求書」
データ 平行四辺形 データの入出力 「顧客マスタから取得」
矢印 → 処理の流れ（順序） 処理から次の処理へ

スイムレーン（水泳レーン図）の書き方¶

複数の役割が登場する業務フローにはスイムレーンを使います。 誰が何をするかが一目でわかります。

┌─────────────┬─────────────┬─────────────┐
│  営業担当    │   上長       │  経理担当    │
├─────────────┼─────────────┼─────────────┤
│             │             │             │
│ [受注データ  │             │             │
│  入力]      │             │             │
│      ↓      │             │             │
│ [承認申請]──→│             │             │
│             │ [承認/却下]  │             │
│             │      ↓      │             │
│             │ 承認？       │             │
│             │ Yes↓  No→  │             │
│             │      ├──────→│             │
│ ←──────────│      │      │ [請求処理]  │
│ [修正]      │             │             │
└─────────────┴─────────────┴─────────────┘

表記の統一ルール¶

ルール 理由
処理の名前は「〜する」で統一する 動詞で書くことでアクションが明確になる
判断の条件は「〜か？」で統一する Yes/Noで分岐しやすくなる
矢印にはYes/Noまたは条件を書く 分岐の意味が明確になる
一つの図形に複数の処理を詰め込まない 1図形1アクションを原則とする
略語・社内用語には注釈を付ける 読む人全員が理解できるようにする

推奨ツール¶

ツール 特徴 向いている場面
Draw.io（diagrams.net） 無料・ブラウザで使える 一般的な業務フロー
Miro リアルタイム共同編集 顧客と一緒に作る場合
Lucidchart テンプレートが豊富 整ったドキュメントが必要な場合
PowerPoint / Excel どこでも使える 顧客がITツールを使えない場合

As-Is / To-Be フロー図の記述例¶

As-Is フロー（例：受注処理業務）¶

開始：顧客から注文を受ける

[営業担当]
1. 電話/FAXで注文内容を受け取る
2. 注文内容をExcelに手入力する
3. 在庫を確認する（電話で倉庫に問い合わせ）
   ↓
   在庫あり？
   → Yes: 上長に承認申請（メール）
   → No:  顧客に在庫なしを連絡

[上長]
4. メールを確認する（毎日16時ごろ）
5. 承認 or 差し戻し
   → 承認: 経理にExcelを転送
   → 差し戻し: 営業担当に差し戻し連絡

[経理担当]
6. Excelを受け取り、請求システムに転記する
7. 請求書を発行する

終了：請求書を顧客に郵送

課題洗い出し（例）¶

No 課題 影響
1 注文内容のExcel手入力に時間がかかる（1件15分） 工数・ミスリスク
2 在庫確認を電話で行っており情報に遅延がある リードタイム増加
3 上長の承認確認が1日1回のため待ち時間が長い リードタイム増加
4 ExcelからシステムへのExcel転記で入力ミスが発生 ミスリスク・工数

To-Be フロー（例：受注処理業務）¶

開始：顧客から注文を受ける

[営業担当]
1. Webフォームまたは電話で注文内容を受け取る
2. 受注管理システムに注文を入力する
3. システムが自動で在庫確認を行う
   ↓
   在庫あり？
   → Yes: 承認申請をシステム上で行う（自動で上長に通知）
   → No:  システムが顧客にメールで通知（自動）

[上長]
4. スマートフォン/PCで承認通知を受け取る（リアルタイム）
5. システム上で承認 or 差し戻し
   → 承認: 経理に自動通知・請求データも自動連携

[経理担当]
6. システムから請求データを確認する（手入力不要）
7. 請求書を発行する（システムから自動生成）

終了：請求書をメール/郵送で顧客に送付

成果物チェックリスト¶

STEP 2 完了時に以下が揃っているか確認してください。

  • As-Is 業務フロー図（顧客確認済み）
  • 課題一覧（優先度付き）
  • To-Be 業務フロー図（顧客確認済み）
  • フロー図について顧客から「内容に合意している」という確認を取得済み

成果物例：社員管理システム¶

As-Is フロー（現状：入社手続き業務）¶

開始：新入社員の入社が決定する

[人事担当]
1. 採用決定通知を受け取る（紙またはメール）
2. 「社員マスタ.xlsx」を開き、新入社員情報を手入力する
   （氏名・生年月日・住所・所属・役職・入社日 など）
3. 総務用Excelにも同じ情報を転記する
4. 各部門マネージャーにメールで入社情報を通知する

[部門マネージャー]
5. メールを受け取り、部門独自の「メンバー管理.xlsx」に手入力する
6. 座席・備品・アカウント申請を別途メールで各所に依頼する

[IT担当]
7. メールを受け取りシステムアカウントを手動作成する

終了：全ての入力・通知が完了（完了の確認手段なし）

課題一覧¶

No 課題 該当箇所 影響 優先度
C001 社員情報を人事・総務・部門の3か所に手入力している 入社手続き全般 入力工数増加・データ不整合 高
C002 入力完了の確認手段がなく、漏れが発生している 手続き完了確認 情報漏れ・対応遅延 高
C003 在宅勤務時はVPN接続が必要でExcelにアクセスしにくい 日常の情報参照 業務効率低下 中
C004 変更履歴が残らず、誰がいつ何を変更したか追えない 情報変更時 監査対応困難・ミス発見遅延 中
C005 退職者情報の無効化が手動のため反映漏れが発生する 退職手続き セキュリティリスク 高

To-Be フロー（将来：入社手続き業務）¶

開始：新入社員の入社が決定する

[人事担当]
1. 社員管理システムにログインする
2. 「社員登録」画面で新入社員情報を1回入力する
   （氏名・生年月日・住所・所属・役職・入社日 など）
3. 「登録」ボタンをクリックする
   → システムが自動で関係者（部門マネージャー・IT担当）にメール通知を送信

[部門マネージャー]
4. 通知メールを受け取る
5. システムにログインして社員情報を確認する（手入力不要）
6. 座席・備品の手配を行う

[IT担当]
7. 通知メールを受け取る
8. システムからアカウント作成に必要な情報をCSVで出力する
9. アカウントを作成する

終了：全ての通知・確認がシステム上で完結（完了ステータスをシステムで管理）
