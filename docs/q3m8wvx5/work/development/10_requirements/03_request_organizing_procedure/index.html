STEP 3　要求の整理と優先度付けの方法¶

このドキュメントの目的¶

STEP 3「要求の整理と優先度付け」を実施するための手順を説明します。 ヒアリングや業務フロー整理で出てきた「やりたいこと」を、システムで実現する機能として整理・分類・優先付けする方法を学びます。

要求の整理とは何か¶

ヒアリング・業務フロー整理を経て、顧客からさまざまな「要求」が出てきています。

しかしこれらをそのままシステム要件にすることはできません。 理由は以下の通りです。

  • 同じことを別の言葉で言っている場合がある
  • 要求の中には「必須」のものと「あれば良い」ものが混在している
  • 予算・期間・技術的な制約からすべては実現できない場合がある
  • 「機能（何をするか）」と「品質（どうあるべきか）」が混在している

このステップでは、要求を整理・分類・優先度付けし、今回のシステムで「何を実現するか」を明確にします。

ステップ別 詳細手順¶

1. 要求の一覧化¶

まず、ヒアリングや業務フロー整理で出てきた要求をすべてリストアップします。 この段階では精査・取捨選択は行いません。漏れなくすべて書き出すことが重要です。

要求の収集元¶

  • ヒアリングシート・議事録
  • As-Is フローで見つかった課題一覧
  • 顧客から直接出てきた「欲しい機能」の発言
  • 現行システムの不満点

要求一覧の書き方¶

No 要求元（誰が言ったか） 要求の内容（言葉のまま） 備考
R001 営業担当Aさん 受注データをシステムに入力したい 現状はExcel
R002 上長Bさん 承認作業をスマホでできるようにしたい 出張中でも承認したい
R003 経理Cさん 請求書を自動で生成したい 転記ミスを減らしたい
R004 IT担当Dさん 既存の顧客マスタシステムと連携したい

2. 機能要件と非機能要件の分類¶

集めた要求を「機能要件」と「非機能要件」に分類します。

機能要件とは¶

システムが「何をするか（What）」を定義するもの

ユーザーが操作したときに「できること」「行われる処理」です。

例： - ユーザーがログインできる - 受注データを新規登録・編集・削除できる - 承認者にメールで通知が届く - 請求書PDFを自動生成できる

非機能要件とは¶

システムが「どうあるべきか（How）」を定義するもの

機能そのものではなく、システムの品質・制約・性能に関する要件です。

例： - 画面の応答時間は3秒以内 - 99.9%の稼働率を維持する - データは暗号化して保存する - 100人が同時に操作しても遅延が起きない

非機能要件の分類¶

カテゴリ 内容 例
性能（パフォーマンス） 処理速度・応答時間・スループット 「検索結果を2秒以内に表示する」
可用性 システムの稼働率・停止許容時間 「平日9〜18時は99.9%稼働すること」
セキュリティ 認証・暗号化・アクセス制御 「パスワードはハッシュ化して保存する」
保守性 運用・障害対応のしやすさ 「障害発生時に原因特定できるようにログを出力する」
拡張性 将来的な機能追加のしやすさ 「将来的に海外拠点にも展開できる構造にする」
移行性 既存データの移行・切り替え 「現行Excelのデータを初期データとして取り込む」
法令・規制準拠 法的な制約への対応 「個人情報は個人情報保護法に基づき管理する」

分類のコツ¶

迷ったときは以下で判断します。

  • 「〜できる」「〜する」 → 機能要件
  • 「〜以内に」「〜以上」「〜しないこと」 → 非機能要件

3. 優先度付け（MoSCoW法）¶

要求が出揃ったら、それぞれに優先度を付けます。 よく使われるのが MoSCoW（モスコウ）法 です。

MoSCoW法とは¶

分類 意味 判断基準
Must（必須） これがなければシステムが成立しない 法的要件・業務上絶対に必要な機能
Should（重要） 強く必要だが、なくても一時的には回る 優先度は高いが代替手段がある
Could（あれば良い） 余裕があれば実装したい あると便利だが優先度は低い
Won't（今回は対象外） 今回はやらない（将来対応の可能性あり） スコープ外・予算・期間の制約

MoSCoW法の使い方¶

顧客と一緒に、各要求が Must / Should / Could / Won't のどれに該当するかを議論します。

重要：顧客は最初「全部Must」と言いがちです。 以下の質問をすることで正直な優先度を引き出します。

  • 「もしこれがなかったら、業務は停止しますか？」→ Noなら Must ではない
  • 「予算・期間が半分になったとしたら、どれを先に作りますか？」
  • 「この機能、リリース後1ヶ月は使わなくても大丈夫ですか？」

優先度付き要求一覧の例¶

No 要求 種別 優先度（MoSCoW） 理由
R001 受注データを登録・編集・削除できる 機能 Must 基本業務に必須
R002 承認をスマホから実施できる 機能 Should PCからは可能にする
R003 請求書PDFの自動生成 機能 Must 転記ミス防止が最優先課題
R004 顧客マスタシステムとの連携 機能 Should 初回はCSV取込で代替可能
R005 ダッシュボードで売上集計を見る 機能 Could 手動集計で当面対応可能
R006 応答時間3秒以内 非機能 Must ユーザー満足度・業務効率に直結
R007 操作ログの記録 非機能 Should 将来の監査対応に備え早めに対応したい

4. スコープの確定¶

優先度付けをもとに、今回のシステムで対応する範囲（スコープ） を確定します。

スコープを明確にすることで、後から「この機能も対応するはずだった」というトラブルを防げます。

スコープ定義書の書き方¶

対象範囲（In Scope）：今回のシステムで実現すること

No 機能・対応内容 優先度
1 受注データの登録・編集・削除 Must
2 承認フロー（PC/スマホ対応） Must / Should
3 請求書PDFの自動生成 Must

対象外範囲（Out of Scope）：今回は対応しないこと

No 内容 理由
1 ダッシュボードによる売上集計 予算・期間の制約により次フェーズ対応
2 スマートフォン専用アプリの開発 ブラウザ対応のみで要件を満たせるため

重要： 「対象外」を明記することが非常に重要です。 「書いていないから対応してくれると思っていた」というトラブルを防ぎます。

よくある問題と対処法¶

問題1：要求が多すぎてまとめられない¶

→ まず重複・似たものを統合する → Must の絞り込みを顧客に依頼する（「絶対に必要なもの5つだけ選んでください」）

問題2：顧客が全部「Must」と言う¶

→ 「もしこれがなかったらどうなりますか？」と聞く → 「予算が半分になったら何を先にやりますか？」で優先順位を引き出す

問題3：非機能要件が後回しになる¶

→ ヒアリングの段階で意識的に確認する → 「性能・セキュリティ・稼働時間の要件はありますか？」と明示的に質問する

問題4：スコープが曖昧なまま進んでしまう¶

→ 「対象外事項」を明記することを徹底する → 「〜は対象外ですが、認識合っていますか？」と顧客に確認する

成果物チェックリスト¶

STEP 3 完了時に以下が揃っているか確認してください。

  • 要求一覧（全要求をリストアップ済み）
  • 機能要件・非機能要件に分類済み
  • MoSCoW法で優先度付き
  • スコープ定義書（対象範囲・対象外範囲）
  • 顧客からスコープについて合意を取得済み

スコープ定義書¶

対象範囲（In Scope）

No 機能・対応内容 優先度
1 社員情報のCRUD（論理削除を含む） Must
2 入社・退職・異動の手続き処理および関係者への自動メール通知 Must
3 変更履歴の記録・閲覧 Must
4 ロールベースのアクセス制御（人事・マネージャー・一般社員） Must
5 社員情報のCSVエクスポート Must
6 Webブラウザ経由でのアクセス（VPN不要） Must
7 一般社員による自身の情報閲覧 Should
8 操作ログの記録・確認 Should
9 誕生日・入社記念日アラート Could

対象外範囲（Out of Scope）

No 内容 理由
1 給与システムとのAPI連携 予算・期間の制約。フェーズ2で対応

4 スマートフォン専用ネイティブアプリ ブラウザ対応（レスポンシブ対応）で要件を満たせるため
