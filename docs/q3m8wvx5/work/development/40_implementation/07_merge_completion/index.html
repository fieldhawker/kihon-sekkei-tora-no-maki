STEP 7 & 8: マージ・完了確認手順¶

このSTEPの目的¶

レビューで承認されたコードを本流（mainブランチ）に取り込み、
正しく動作することを確認して作業を完了します。

マージとは¶

マージとは、自分の作業ブランチの変更を main ブランチに取り込む操作のことです。

【マージのイメージ】

main: A - B - C
                \
feature/login:   D - E - F（自分の実装）

↓ マージ後

main: A - B - C - D - E - F

マージが完了すると、自分の変更がチーム全員の共有コードに反映されます。

マージの前提条件¶

マージを行う前に、以下がすべて満たされていることを確認します。

□ レビュアーからの承認（Approve）をもらっている
□ 全ての指摘に対応済みで、レビュアーに確認してもらっている
□ 単体テストが全て合格している
□ CIチェック（自動テスト・Lintなど）がパスしている

承認なしにマージしない。
レビューが終わっていない状態でマージすると、品質未確認のコードが本流に入ります。

マージの手順¶

1. マージ前の最終確認¶

マージ直前に、自分のブランチが main の最新状態と乖離していないか確認します。

# mainの最新を取得
git fetch origin

# mainに対して自分のブランチが最新かどうか確認
git log origin/main..HEAD

main が先に進んでいる場合は、最新の main を取り込んでから（リベースまたはマージして）、再確認します。
操作方法はプロジェクトの方針に従ってください。

2. マージする（GitHubでのマージ）¶

GitHubのプルリクエスト画面から「Merge pull request」ボタンをクリックします。

マージ方法の種類（プロジェクトによって指定される）:

マージ方法 特徴
Merge commit 全コミット履歴 + マージコミットを保持。デフォルト
Squash and merge ブランチの全コミットを1つにまとめてマージ
Rebase and merge コミット履歴を直線状に保ってマージ

どの方法を使うかはプロジェクトのルールに従ってください。
わからない場合はチームリーダーに確認してください。

3. マージ後の動作確認¶

マージが完了したら、開発環境（チーム共有環境）で動作を確認します。

確認する内容: - 自分が実装した機能が正しく動くか - 他の既存機能が壊れていないか（デグレード確認）

ポイント: ローカル環境で動いていても、サーバーに展開すると動かないことがあります。
必ず共有環境での確認まで行ってください。

4. 完了報告する¶

作業完了を担当者・チームに報告します。

報告する内容: - タスク管理ツール（Jira / Backlog / Redmineなど）でタスクのステータスを「完了」に変更する - チームリーダーや関係者に完了を報告する（プロジェクトのルールによる） - マージした内容で問題があればすぐ連絡できるよう、しばらく動作を注視する

5. 作業ブランチを削除する¶

マージ済みのブランチは不要なので削除します。
リポジトリを整理整頓した状態に保つためです。

GitHubでの削除:
マージ後に「Delete branch」ボタンが表示されるのでクリックします。

ローカルでの削除:

# mainブランチに戻る
git checkout main

# mainを最新化する
git pull origin main

# ローカルの作業ブランチを削除
git branch -d feature/login-page

コンフリクト（競合）が発生した場合¶

マージ時に、同じファイルの同じ箇所を自分と他のメンバーが変更していると「コンフリクト（競合）」が発生します。

コンフリクトが起きたファイルの表示例:

<<<<<<< HEAD（mainブランチの内容）
String errorMessage = "ログインに失敗しました";
=======
String errorMessage = "メールアドレスまたはパスワードが正しくありません";
>>>>>>> feature/login-page（自分のブランチの内容）

解消の手順:

  1. コンフリクトが発生しているファイルを開く
  2. <<<<<<< HEAD、=======、>>>>>>> の3つの記号を目印に、
    どちらの内容を採用するか（または両方を組み合わせるか）を判断する
  3. 記号を削除して、正しい内容だけ残す
  4. ファイルを保存してコミットする
# コンフリクトを解消したファイルをステージングに追加
git add src/login/LoginService.java

# コンフリクト解消のコミット
git commit -m "fix: mainブランチとのコンフリクトを解消"

重要: どちらの内容を残すか判断に迷う場合は、1人で決めずに必ず相手（または担当SE）に確認してください。
誤った判断でコンフリクト解消すると、誰かの実装を消してしまう可能性があります。

完了確認チェックリスト¶

□ レビュアーからの承認をもらっている
□ 全指摘の対応が完了している
□ マージした
□ マージ後に開発環境で動作確認した
□ 既存機能が壊れていないことを確認した（デグレード確認）
□ タスク管理ツールでタスクを完了ステータスにした
□ 必要に応じて完了報告した
□ 作業ブランチを削除した（ローカル・リモート両方）

実装全体の成果物まとめ¶

STEP 1〜8を通じて、以下の成果物が揃っていることを確認します。

# 成果物 対応STEP
1 ソースコード（mainにマージ済み） STEP 4
2 単体テスト結果（記録表またはテストコード） STEP 5
3 プルリクエスト（差分・レビューコメント・承認記録） STEP 6〜7
4 マージ済みブランチ（削除済み） STEP 8

よくある失敗¶

承認前にマージしてしまう¶

「レビューの返信を待つのが面倒」で承認前にマージすると、品質未確認のコードが本流に入ります。
承認が来るまで待ちましょう。

マージ後の動作確認を省略する¶

「ローカルで動いていたから大丈夫」という思い込みは危険です。
必ずマージ後の環境で確認します。

ブランチを削除し忘れる¶

不要なブランチが大量に残るとリポジトリが管理しにくくなります。
マージ完了後はすぐに削除するクセをつけましょう。

コンフリクトを独断で解消する¶

相手の実装を理解せずにコンフリクト解消すると、誰かのコードが消える可能性があります。
判断に迷ったら必ず確認してください。
