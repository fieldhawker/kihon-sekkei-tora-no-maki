STEP 4: コーディング手順・規約・セキュリティ¶

このSTEPの目的¶

詳細設計書の内容をコードに変換します。
「動く」だけでなく「読みやすく・安全で・保守しやすい」コードを書くことが目標です。

コーディングの基本的な進め方¶

1. 設計書を読んで実装箇所を特定する¶

設計書を見ながら、どの処理をどの順番で実装するかを把握します。

実装前の確認ポイント: - 処理の流れ（フローチャート・シーケンス図） - バリデーション条件（入力値の制限） - データベース操作（取得・更新・削除するテーブルとSQLの条件） - エラー時の動作（何をログに残して、何をユーザーに返すか）

2. 小さな単位で実装・確認を繰り返す¶

全部書いてから動かすのではなく、小さく作って、動かして、確認するを繰り返します。

【悪い進め方】
全機能を一気に書く → 動かしてみる → 大量のエラー → 原因不明 → 詰む

【良い進め方】
バリデーションだけ書く → 動かして確認 → OK
  ↓
DB検索処理を書く → 動かして確認 → OK
  ↓
結果を返す処理を書く → 動かして確認 → OK

3. こまめにコミットする¶

動作確認が取れたタイミングでコミットします（詳細は 03_branch_management.md を参照）。

コーディング規約¶

プロジェクトには「コーディング規約」という「コードの書き方のルール」があります。
チーム全員が同じルールで書くことで、誰が書いても読めるコードになります。

ルールブックのイメージ: 学校の作文ルール（句読点の使い方、段落の分け方など）と同じです。
自分流で書いてしまうと、チームメンバーが読みにくくなります。

よくある規約の例¶

規約の種類 例
命名規則 変数名はキャメルケース（userName）、定数は大文字スネークケース（MAX_RETRY_COUNT）
インデント スペース2個 または スペース4個（タブは使わない）
1関数の長さ 1つの関数は〇行以内（長すぎる関数は分割する）
コメントの書き方 「何をしているか」ではなく「なぜそうしているか」を書く
ファイル・クラスの命名 機能名+役割（例: LoginService.java, userController.js）

規約の確認場所: - プロジェクトの CONTRIBUTING.md または CODING_STANDARDS.md - チームのWiki - 先輩エンジニアに確認

良いコードの書き方¶

意味のある名前をつける¶

変数名・関数名は「その変数が何を表しているか」がわかる名前にします。

// 悪い例
int x = getUserAge();
String s = "yamada@example.com";
boolean f = true;

// 良い例
int userAge = getUserAge();
String userEmail = "yamada@example.com";
boolean isLoginSuccessful = true;

重複を避ける（同じ処理は1か所にまとめる）¶

同じ処理を複数の場所にコピーして書くと、修正が必要なときに全箇所を直さなければなりません。

// 悪い例（同じバリデーションが複数箇所にある）
// ログイン画面
if (email == null || email.isEmpty() || !email.contains("@")) {
    throw new ValidationException("メールアドレスが不正です");
}

// 会員登録画面
if (email == null || email.isEmpty() || !email.contains("@")) {
    throw new ValidationException("メールアドレスが不正です");
}

// 良い例（共通メソッドにまとめる）
private void validateEmail(String email) {
    if (email == null || email.isEmpty() || !email.contains("@")) {
        throw new ValidationException("メールアドレスが不正です");
    }
}

エラー処理を書く¶

エラーが起きたときに「何もしない」コードは危険です。
エラーをログに残し、適切なメッセージを返します。

// 悪い例（エラーを無視している）
try {
    user = userRepository.findById(userId);
} catch (Exception e) {
    // 何もしない
}

// 良い例
try {
    user = userRepository.findById(userId);
} catch (DatabaseException e) {
    logger.error("ユーザー取得中にエラーが発生しました。userId={}", userId, e);
    throw new SystemException("データの取得に失敗しました");
}

セキュリティの注意点¶

絶対にやってはいけないこと¶

実装ミスが原因でシステムが攻撃されるケースがあります。
以下は必ず守ってください。

SQLインジェクション対策（必須）¶

ユーザーの入力値をそのままSQLに埋め込むことを「SQLインジェクション」の脆弱性といいます。
攻撃者に悪用されると、DBのデータが丸ごと盗まれたり、削除されたりします。

-- 悪い例（入力値を直接SQL文字列に埋め込んでいる）
String sql = "SELECT * FROM users WHERE email = '" + inputEmail + "'";

-- 攻撃例: inputEmail に ' OR '1'='1 を入れられると全件取得されてしまう

// 良い例（プレースホルダー/バインド変数を使う）
String sql = "SELECT * FROM users WHERE email = ?";
PreparedStatement stmt = connection.prepareStatement(sql);
stmt.setString(1, inputEmail);  // 入力値はここでセットする

必ずプレースホルダー（?）またはバインド変数を使ってください。
文字列を + でSQL文に結合するのは絶対に禁止です。

XSS（クロスサイトスクリプティング）対策¶

ユーザーの入力値をHTMLにそのまま出力すると、悪意のあるスクリプトが実行される可能性があります。

<!-- 悪い例 -->
<p>ようこそ、<%= userName %>さん</p>
<!-- userNameに <script>alert('XSS')</script> を入力されると実行される -->

<!-- 良い例（テンプレートエンジンのエスケープ機能を使う） -->
<p>ようこそ、<%= h(userName) %>さん</p>  <!-- Railsの例 -->
<p>ようこそ、{{ userName }}さん</p>      <!-- Vue.js/Angularは自動エスケープ -->

フレームワークのエスケープ機能を必ず使ってください。
HTMLを直接文字列として組み立てるのは避けてください。

パスワードの保存（必須）¶

パスワードはそのままDBに保存せず、必ずハッシュ化します。

// 悪い例（平文で保存）
user.setPassword(inputPassword);

// 良い例（ハッシュ化して保存）
String hashedPassword = BCrypt.hashpw(inputPassword, BCrypt.gensalt());
user.setPassword(hashedPassword);

エラー情報の出力¶

エラーの詳細（スタックトレースなど）をユーザーの画面に表示してはいけません。
攻撃者にシステムの内部情報を教えることになります。

// 悪い例（スタックトレースをそのまま表示）
response.write(e.toString());  // 絶対NG

// 良い例
logger.error("システムエラーが発生しました", e);  // ログには詳細を残す
response.write("エラーが発生しました。しばらく後に再試行してください");  // ユーザーには簡潔なメッセージだけ

コーディング中のセルフチェック¶

実装した機能を自分でコードレビューする習慣をつけましょう。

□ 設計書の仕様通りに実装できているか
□ 変数名・関数名は意味がわかるか
□ 同じ処理のコピーがないか（重複がないか）
□ SQLにバインド変数を使っているか（文字列結合でSQLを作っていないか）
□ HTMLへの出力をエスケープしているか
□ パスワードをハッシュ化しているか
□ エラー処理を書いているか（エラーを握りつぶしていないか）
□ ログを適切に出力しているか
□ コーディング規約に従っているか

よくある失敗¶

設計書と違う仕様で実装してしまう¶

「たぶんこうだろう」と思い込みで実装し、テスト工程でNGになる。
設計書と照らし合わせながら実装し、不明点はすぐに確認する。

コードが動いたら満足してしまう¶

「動く」と「正しく動く」は別物。エラーケースや境界値も確認する。

セキュリティ対策を「後でやる」¶

後でやると忘れる。SQLやHTMLを扱う時点で必ず対策を入れる。

成果物の例（社員管理システム）¶

以下は「社員管理システム」の「社員情報登録機能」を実装した場合のコード例です。
実際の成果物（ソースコード）がどのような形になるかを示しています。

EmployeeRegisterController.java（コントローラー）¶

@Controller
@RequestMapping("/employees")
public class EmployeeRegisterController {

    private final EmployeeRegisterService employeeRegisterService;

    public EmployeeRegisterController(EmployeeRegisterService employeeRegisterService) {
        this.employeeRegisterService = employeeRegisterService;
    }

    /** 社員登録画面を表示する */
    @GetMapping("/new")
    public String showRegisterForm(Model model) {
        model.addAttribute("employeeForm", new EmployeeForm());
        model.addAttribute("departments", employeeRegisterService.getAllDepartments());
        return "employee/register";
    }

    /** 社員登録を実行する */
    @PostMapping("/register")
    public String register(
            @Valid @ModelAttribute EmployeeForm form,
            BindingResult bindingResult,
            RedirectAttributes redirectAttributes,
            Model model) {

        // バリデーションエラーがある場合は登録画面に戻る
        if (bindingResult.hasErrors()) {
            model.addAttribute("departments", employeeRegisterService.getAllDepartments());
            return "employee/register";
        }

        try {
            employeeRegisterService.register(form);
            redirectAttributes.addFlashAttribute("successMessage", "社員情報を登録しました");
            return "redirect:/employees";

        } catch (DuplicateEmployeeNumberException e) {
            bindingResult.rejectValue("employeeNumber", "duplicate",
                    "この社員番号はすでに使用されています");
            model.addAttribute("departments", employeeRegisterService.getAllDepartments());
            return "employee/register";
        }
    }
}

EmployeeForm.java（バリデーション定義）¶

public class EmployeeForm {

    @NotBlank(message = "社員番号は必須です")
    @Pattern(regexp = "\\d{6}", message = "社員番号は半角数字6桁で入力してください")
    private String employeeNumber;

    @NotBlank(message = "氏名は必須です")
    @Size(max = 50, message = "氏名は50文字以内で入力してください")
    private String fullName;

    @NotNull(message = "部署は必須です")
    private Long departmentId;

    @NotNull(message = "入社日は必須です")
    @PastOrPresent(message = "入社日は今日以前の日付を入力してください")
    @DateTimeFormat(pattern = "yyyy-MM-dd")
    private LocalDate hireDate;

    @Email(message = "メールアドレスの形式が正しくありません")
    @Size(max = 255, message = "メールアドレスは255文字以内で入力してください")
    private String email;

    // getter / setter 省略
}

EmployeeRegisterService.java（ビジネスロジック）¶

@Service
public class EmployeeRegisterService {

    private final EmployeeRepository employeeRepository;
    private final DepartmentRepository departmentRepository;

    public EmployeeRegisterService(EmployeeRepository employeeRepository,
                                   DepartmentRepository departmentRepository) {
        this.employeeRepository = employeeRepository;
        this.departmentRepository = departmentRepository;
    }

    public void register(EmployeeForm form) {
        // 社員番号の重複チェック
        if (employeeRepository.existsByEmployeeNumber(form.getEmployeeNumber())) {
            throw new DuplicateEmployeeNumberException(form.getEmployeeNumber());
        }

        Employee employee = new Employee();
        employee.setEmployeeNumber(form.getEmployeeNumber());
        employee.setFullName(form.getFullName());
        employee.setDepartmentId(form.getDepartmentId());
        employee.setHireDate(form.getHireDate());
        employee.setEmail(form.getEmail());

        employeeRepository.save(employee);
    }

    public List<Department> getAllDepartments() {
        return departmentRepository.findAll();
    }
}

EmployeeRepository.java（DBアクセス）¶

@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    /** 社員番号が存在するか確認する（重複チェック用） */
    boolean existsByEmployeeNumber(String employeeNumber);
}

セキュリティポイント:
Spring Data JPAを使用しているため、SQLはフレームワークが生成し、
SQLインジェクションは自動的に防止されています。
入力値のバリデーションは @Valid + BindingResult で統一して処理しています。
