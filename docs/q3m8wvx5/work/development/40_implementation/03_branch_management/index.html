STEP 3: ブランチ作成・バージョン管理¶

このSTEPの目的¶

Gitのブランチを使って、自分の変更を他の作業と分離します。
これにより「誰が何を変更したか」が追えるようになり、問題が起きたときに元に戻せます。

Gitとブランチとは（基本の確認）¶

Git とは、コードの変更履歴を記録・管理するツールです。
チームで同じコードを同時に編集しても衝突しないように管理できます。

ブランチ とは、コードの「作業用コピー」のようなものです。

【ブランチのイメージ】

main（本流）
  │
  ├── feature/login-page（Aさんの作業）
  │     └── ログイン画面を実装中...
  │
  └── feature/user-profile（Bさんの作業）
        └── プロフィール画面を実装中...

→ それぞれ独立して作業できる
→ 完成したらmainに取り込む（マージ）

ポイント: main ブランチは「完成・動作確認済みのコード」を置く場所です。
直接 main に変更を加えるのは禁止。必ず作業ブランチを作ります。

ブランチ作業の手順¶

1. 最新状態に更新する¶

作業を始める前に、リモートの最新コードを取得します。

# リモートの最新コードをローカルに取得
git pull origin main

なぜ重要か: 他のメンバーが昨日マージした変更を取り込まないまま作業すると、
後でコードをマージするときに「コンフリクト（衝突）」が発生しやすくなります。
毎日作業開始前に git pull するのを習慣にしましょう。

コンフリクトが起きたら: 同じ箇所を別々のブランチで変更した場合、マージ時にコンフリクト（衝突）が発生します。コンフリクトが起きても慌てる必要はありません。解決手順は STEP 7: マージ完了 を参照してください。

2. 作業用ブランチを作成する¶

自分の担当機能に対応した名前でブランチを作成します。

# ブランチを作成して、そのブランチに移動する
git checkout -b feature/login-page

または（新しいGitの書き方）:

git switch -c feature/login-page

ブランチ名のルール（プロジェクトによって異なる場合あり）:

種類 ブランチ名の形式 例
新機能の追加 feature/機能名 feature/login-page / feature/user-search
バグ修正 fix/修正内容 fix/login-error / fix/null-pointer
設定・環境変更 chore/変更内容 chore/update-dependencies
タスク番号がある場合 feature/TASK-123-login-page タスク番号を含める

ポイント: ブランチ名は作業内容がひと目でわかる名前にします。
test や work といった曖昧な名前はNGです。

3. 現在のブランチを確認する¶

自分が今どのブランチにいるか確認できます。

git branch

出力例: 

  main
* feature/login-page   ← *がついているのが現在のブランチ

コーディング中のバージョン管理（コミット）¶

コミットとは¶

コミットとは、変更をGitに「記録」する操作です。
「ここまでの変更を保存する」というイメージです。

コミットの手順¶

1. 変更したファイルを確認する

git status

出力例: 

Changes not staged for commit:
  modified:   src/login/LoginForm.java
  modified:   src/login/LoginService.java

Untracked files:
  src/login/LoginValidator.java  ← 新しく作ったファイル

2. コミットするファイルをステージングに追加する

# 特定のファイルを追加（推奨）
git add src/login/LoginForm.java
git add src/login/LoginService.java
git add src/login/LoginValidator.java

# または変更・追加されたファイルをまとめて追加
git add .

注意: git add . はすべての変更ファイルを追加します。
テスト用の一時ファイルや .env などを含めないよう注意してください。

3. コミットメッセージを書いてコミットする

git commit -m "feat: ログイン画面のバリデーション処理を追加"

コミットメッセージの書き方¶

コミットメッセージは「何をしたか」が伝わるように書きます。

形式（Conventional Commits）:

種類: 内容の要約
種類 使う場面 例
feat 新機能の追加 feat: ログイン機能を追加
fix バグ修正 fix: ログインエラー時のメッセージが表示されない問題を修正
refactor 動作を変えないコード改善 refactor: バリデーション処理を共通メソッドに整理
test テストコードの追加・修正 test: ログイン機能の単体テストを追加
chore ビルド設定・依存パッケージなど chore: eslintの設定を追加

良いコミットメッセージ・悪いコミットメッセージ:

悪い例 良い例
修正 fix: パスワード未入力時のエラーメッセージを修正
作業中 feat: ログインフォームのUI実装（バリデーション除く）
変更 refactor: LoginServiceのメソッド名をキャメルケースに統一

こまめにコミットする¶

「全部できてからコミット」はNGです。こまめにコミットすることで: - 問題が起きたときに「どの変更でおかしくなったか」を特定できる - 誤った変更を取り消しやすくなる

コミットの粒度の目安:

良い例（1機能 = 数コミット）:
  commit 1: feat: ログインフォームのHTML/CSSを実装
  commit 2: feat: ログインフォームの入力バリデーションを追加
  commit 3: feat: ログインAPIとの通信処理を実装
  commit 4: fix: バリデーションエラーメッセージの位置がずれる問題を修正

悪い例:
  commit 1: ログイン機能  ← 1コミットに何もかも詰め込んでいる

チェックリスト（ブランチ・コミット）¶

□ 作業前に git pull で最新化した
□ main ブランチ上に直接コードを書いていない
□ 作業内容を表す名前のブランチを作成した
□ こまめにコミットしている
□ コミットメッセージは内容が伝わる形で書いている
□ .env や一時ファイルをコミットに含めていない

よくある失敗¶

main ブランチに直接コミットしてしまった¶

# 対処法: 誤ってmainにコミットした場合はすぐチームリーダーに相談する
# 慌てて操作すると悪化することがあるため、1人で解決しようとしない

ブランチを切り忘れて作業した¶

気づいたタイミングで作業ブランチを作成し、チームメンバーに相談してください。

コミットをまとめすぎてしまった¶

1回のコミットに複数の機能変更が入っていると、後から追うのが困難になります。
「1つの意味のある変更 = 1コミット」を意識しましょう。

成果物の例（社員管理システム）¶

以下は「社員管理システム」の「社員情報登録機能」を実装する場合のブランチ・コミット記録の例です。

ブランチ作成（実行例）¶

# mainを最新化
$ git pull origin main
Already up to date.

# 作業ブランチを作成
$ git checkout -b feature/TASK-201-employee-register
Switched to a new branch 'feature/TASK-201-employee-register'

# ブランチを確認
$ git branch
  main
* feature/TASK-201-employee-register

コミット履歴（記録例）¶

社員情報登録機能を段階的に実装した場合のコミット履歴です。

$ git log --oneline

a1b2c3d feat: 社員登録画面のHTMLテンプレートを追加
b2c3d4e feat: 社員登録フォームの入力バリデーションを実装
c3d4e5f feat: 社員番号の重複チェック処理を追加
d4e5f6g feat: 社員情報をDBに保存するリポジトリ処理を実装
e5f6g7h feat: 登録成功・失敗時のリダイレクト処理を実装
f6g7h8i fix: 入社日の未来日付バリデーションが効いていない問題を修正
g7h8i9j test: 社員登録機能の単体テストを追加

コミットメッセージの具体例¶

# バリデーション追加時
git commit -m "feat: 社員登録フォームの入力バリデーションを実装

- 社員番号: 半角数字6桁の形式チェックを追加
- 氏名: 必須・最大50文字チェックを追加
- 入社日: 必須・日付形式・過去日付チェックを追加
- メールアドレス: RFC準拠の形式チェックを追加（任意項目）"

# バグ修正時
git commit -m "fix: 入社日に未来の日付を入力してもエラーにならない問題を修正

LocalDate.now() との比較条件が逆になっていたため修正。
isAfter → isBefore に変更。"
