STEP 5: 単体テストの進め方¶

このSTEPの目的¶

自分が書いたコードが設計書の通りに正しく動くかを自分で確認します。
「たぶん動いている」ではなく、「確認した」状態でコードを渡すことが責任です。

なぜ単体テストが必要か¶

バグは発見が遅れるほど修正コストが上がります。

【バグ発見のコスト（相対的な比較）】

実装中に発見     → コスト: 1
単体テストで発見  → コスト: 3
結合テストで発見  → コスト: 10
本番リリース後に発見 → コスト: 100以上

自分でテストできる範囲を確認せずに渡すのは、「後の工程で誰かが大変な目に遭う」ことを意味します。
単体テストはチームへの責任でもあります。

単体テストで確認する3つのパターン¶

1. 正常系（ハッピーパス）¶

正しい入力を与えたときに、期待通りの結果になるかを確認します。

テスト対象 入力 期待する結果
ログイン機能 正しいメールアドレスとパスワード ログイン成功 / トップページに遷移
商品検索 キーワード「テレビ」 テレビに関連する商品一覧が表示される
金額計算 商品価格1000円 × 数量3 合計金額3000円が表示される

2. 異常系（エラーケース）¶

不正な入力や異常な状態を与えたときに、適切なエラーになるかを確認します。

テスト対象 入力 期待する結果
ログイン機能 誤ったパスワード エラーメッセージが表示される
入力フォーム 必須項目が空 バリデーションエラーが表示される
検索機能 存在しないキーワード 「検索結果はありません」と表示される
ファイルアップロード 上限サイズを超えたファイル エラーメッセージが表示される

ポイント: 異常系は「正しく失敗するか」を確認します。
エラーが起きたときに何も表示されない・壊れた画面が表示される、は不合格です。

3. 境界値（エッジケース）¶

「ちょうど上限/下限の値」で正しく動くかを確認します。
バグはこの「境界」で起きやすいです。

例: パスワードの文字数制限（8文字以上 / 32文字以下）

テストケース 入力 期待する結果
下限より1つ少ない 7文字のパスワード バリデーションエラー
ちょうど下限 8文字のパスワード OK（エラーなし）
中間 16文字のパスワード OK（エラーなし）
ちょうど上限 32文字のパスワード OK（エラーなし）
上限より1つ多い 33文字のパスワード バリデーションエラー

テストの進め方¶

手順1: テストケースを設計書から洗い出す¶

設計書の「バリデーション条件」「処理フロー」「エラーパターン」を見て、
テストすべきケースをリストアップします。

テストケース一覧の例（ログイン機能）:

No. | 種別    | 入力内容                        | 期待する結果
----|---------|--------------------------------|------------------
1   | 正常系  | 正しいメール・正しいパスワード   | ログイン成功
2   | 異常系  | 正しいメール・誤ったパスワード   | エラーメッセージ表示
3   | 異常系  | 存在しないメールアドレス         | エラーメッセージ表示
4   | 異常系  | メールアドレスが空               | バリデーションエラー
5   | 異常系  | パスワードが空                   | バリデーションエラー
6   | 境界値  | パスワード7文字                  | バリデーションエラー
7   | 境界値  | パスワード8文字                  | ログイン成功
8   | 境界値  | パスワード32文字                 | ログイン成功
9   | 境界値  | パスワード33文字                 | バリデーションエラー

手順2: テストを実施する¶

手動テスト（画面操作）の場合:

  1. アプリケーションを起動する
  2. テストケースを上から順番に実施する
  3. 期待する結果と実際の結果を比較する
  4. 結果を記録する

テスト結果の記録例:

No. | 期待する結果              | 実際の結果               | 合否 | 備考
----|--------------------------|------------------------|------|------
1   | ログイン成功・トップ遷移  | ログイン成功・トップ遷移  | ○    |
2   | エラーメッセージ表示      | エラーメッセージ表示      | ○    |
6   | バリデーションエラー      | バリデーションエラーなし  | ✗    | 7文字でもログインできてしまう → バグ

手順3: バグを修正して再テストする¶

テストでNGになったケースは、コードを修正して再度テストします。
修正前と修正後で、影響範囲が変わっていないか確認します。

バグ発見 → コード修正 → 修正したケースを再テスト → 関連する他のケースも再テスト → 全OK

自動テスト（ユニットテスト）について¶

プロジェクトによっては、テストコードを書いて自動でテストする場合があります。

自動テストのメリット: - 手動テストより素早く実施できる - 同じテストを何度でも同じ条件で実行できる - コード変更後の「デグレード（壊れていないか）確認」が簡単

テストコードの例（Java / JUnit）:

@Test
void ログイン成功_正しいメールとパスワード() {
    // テスト対象のメソッドを呼び出す
    LoginResult result = loginService.login("test@example.com", "password123");

    // 期待する結果と比較する
    assertEquals(LoginStatus.SUCCESS, result.getStatus());
}

@Test
void ログイン失敗_パスワードが空の場合() {
    // パスワードが空の場合はバリデーション例外が投げられるはず
    assertThrows(ValidationException.class, () -> {
        loginService.login("test@example.com", "");
    });
}

プロジェクトで自動テストが求められる場合は、テストコードも成果物のひとつです。
要件はチームリーダーに確認してください。

テスト結果の記録・提出¶

テストが完了したら、結果を記録してコードレビュー依頼時に提示します。

提出物（プロジェクトによって異なる）: - テストケース一覧と実施結果（Excel・Googleスプレッドシートなど） - テストコード（自動テストの場合） - スクリーンショット（必要に応じて）

チェックリスト（単体テスト完了の確認）¶

□ 設計書のバリデーション条件を全てテストした
□ 正常系のテストケースを全て実施した
□ 異常系のテストケースを全て実施した
□ 境界値のテストケースを実施した
□ 全テストが合格（○）になった
□ テスト結果を記録した

よくある失敗¶

正常系だけテストして満足する¶

「メールアドレスとパスワードを入れてログインできた」だけで完了にするのはNG。
異常系・境界値のテストが省略されると、後工程で大量のバグが見つかります。

テスト結果を記録しない¶

「テストしました」と言っても、結果の記録がないと信用してもらえません。
何をテストして何がOKだったかを記録に残しましょう。

バグを見つけたのに放置する¶

「後で直す」は忘れます。バグを発見したらすぐに修正して、再テストしてから進めましょう。

バグ報告（記入例）¶

【バグ報告】

バグNo.  : BUG-011
発見日   : 2025-06-10
発見者   : 山田 太郎
対応テスト: No.11（入社日を未来の日付にして登録する）

【現象】
入社日に明日の日付（2025-06-11）を入力して登録ボタンを押すと、
バリデーションエラーが発生せず、そのまま登録が完了してしまう。

【期待する動作】
「入社日は今日以前の日付を入力してください」というエラーが表示されること。

【原因の調査結果】
EmployeeForm.java の hireDate フィールドに付与したアノテーションが
@PastOrPresent ではなく @Past になっており、「今日」が弾かれていた。
また、@DateTimeFormat の pattern が "yyyy/MM/dd" になっており、
フォームの入力形式 "yyyy-MM-dd" と不一致のため、日付のパースが失敗し
バリデーションがスキップされていた。

【修正内容】
- @Past → @PastOrPresent に変更
- @DateTimeFormat(pattern = "yyyy/MM/dd") → "yyyy-MM-dd" に変更

【修正後の確認】
修正後に再テスト実施 → ✅ 期待通りエラーが表示されることを確認

自動テストコードの例（JUnit / Spring Boot）¶

@SpringBootTest
@AutoConfigureMockMvc
class EmployeeRegisterControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void 正常系_全項目入力して登録成功() throws Exception {
        mockMvc.perform(post("/employees/register")
                .param("employeeNumber", "001234")
                .param("fullName", "山田 太郎")
                .param("departmentId", "1")
                .param("hireDate", "2020-04-01")
                .param("email", "yamada@example.com")
                .with(csrf()))
                .andExpect(status().is3xxRedirection())
                .andExpect(redirectedUrl("/employees"))
                .andExpect(flash().attribute("successMessage", "社員情報を登録しました"));
    }

    @Test
    void 異常系_社員番号が空の場合はバリデーションエラー() throws Exception {
        mockMvc.perform(post("/employees/register")
                .param("employeeNumber", "")  // 空
                .param("fullName", "山田 太郎")
                .param("departmentId", "1")
                .param("hireDate", "2020-04-01")
                .with(csrf()))
                .andExpect(status().isOk())
                .andExpect(view().name("employee/register"))
                .andExpect(model().attributeHasFieldErrors("employeeForm", "employeeNumber"));
    }

    @Test
    void 境界値_氏名が50文字の場合は登録成功() throws Exception {
        String fiftyChars = "あ".repeat(50);
        mockMvc.perform(post("/employees/register")
                .param("employeeNumber", "001235")
                .param("fullName", fiftyChars)
                .param("departmentId", "1")
                .param("hireDate", "2020-04-01")
                .with(csrf()))
                .andExpect(status().is3xxRedirection());
    }
}
