STEP 6: コードレビューの依頼・対応手順¶

このSTEPの目的¶

自分では気づきにくいバグ・設計のズレ・品質の問題を、他の人の目でチェックしてもらいます。
コードレビューはコードを「直してもらう」場ではなく、「品質を上げるための協力関係」です。

コードレビューとは¶

コードレビューとは、自分が書いたコードを他のエンジニアに読んでもらい、
問題がないか・改善できる点はないかを確認してもらう作業です。

なぜ必要か: - 自分ではミスに気づきにくい（書いた本人は「正しい」と思い込んでいる） - 別の視点から見ることで設計の問題・セキュリティの穴が見つかる - チーム全体でコードの品質基準を統一できる

気持ちの準備: 指摘を受けることはよくあります。
「自分を否定された」ではなく「良いコードに育ててもらっている」と捉えてください。
積極的に指摘を求める姿勢が成長を早めます。

コードレビュー依頼の手順¶

1. コードをリモートにプッシュする¶

自分のブランチをGitHubなどのリモートリポジトリにアップロードします。

# 作業ブランチをプッシュ
git push origin feature/login-page

初回プッシュ時は -u オプションを付けることで、以降の git push が簡略化されます。

git push -u origin feature/login-page

2. プルリクエスト（PR）を作成する¶

GitHub / GitLab / Bitbucket などのツールでプルリクエストを作成します。

プルリクエストとは:
「自分のブランチの変更を main ブランチに取り込んでください」という申請のことです。
レビュアーはこのプルリクエスト上でコードを確認し、コメントを残します。

プルリクエストの作成手順（GitHub の例）:

  1. GitHubでリポジトリを開く
  2. 「Pull requests」タブ → 「New pull request」をクリック
  3. base: main、compare: 自分のブランチ を選択
  4. 「Create pull request」をクリック
  5. タイトルと説明を入力して「Create pull request」

3. プルリクエストの内容を記載する¶

プルリクエストの説明欄に以下を書きます。

記載テンプレート（参考）:

## 変更の概要
ログイン機能のバリデーション処理と認証APIとの通信処理を実装しました。

## 実装内容
- メールアドレス・パスワードの入力バリデーション
- ログインAPIへのPOSTリクエスト処理
- ログイン成功時のトップページへのリダイレクト
- ログイン失敗時のエラーメッセージ表示

## 関連するタスク
- TASK-123: ログイン機能の実装

## 確認してほしいポイント
- バリデーションの条件が設計書の仕様と合っているか
- エラーハンドリングの方法でよいか（例外をキャッチする粒度）

## テスト結果
| テストケース | 結果 |
|---|---|
| 正しいメール・パスワードでログイン | ○ |
| 誤ったパスワードでログイン | ○ |
| 空のパスワードでログイン | ○ |
| パスワード7文字 | ○ |
| パスワード8文字 | ○ |

記載のポイント: - 「何を実装したか」を箇条書きで書く - 自信がない箇所・判断が難しかった箇所を正直に書く - テストしたケースと結果を書く

「完璧になってから依頼しよう」と思わなくて大丈夫です。
早めに方向性を確認するための「Draft（ドラフト）PR」としての活用も有効です。

4. レビュアーをアサインする¶

プルリクエストのレビュアー（レビューしてほしい人）を指定します。

  • チームリーダーや先輩エンジニアを指定する
  • 誰を指定すべきか迷ったら確認する

アサインしたら、Slackやチャットツールで「PRを作成しました、確認をお願いします」と一声かけると親切です。

レビュー指摘への対応手順（STEP 7）¶

1. 指摘内容を理解する¶

指摘コメントを読んで、なぜその指摘が来たのかを理解してから修正します。

指摘: 「このSQLはプレースホルダーを使っていません。SQLインジェクションの危険があります」

→ 理解: 入力値をSQL文字列に直接連結しているため、攻撃に使われる可能性がある
→ 修正: バインド変数（?）を使ったPreparedStatementに書き換える

理解できない場合: 「〇〇の指摘について、△△と理解しましたが合っていますか？」と返信して確認してください。
わからないまま修正するのが一番危険です。

2. 修正してコミット・プッシュする¶

指摘内容を修正して、コミット・プッシュします。

git add src/login/LoginService.java
git commit -m "fix: SQLインジェクション対策のためPreparedStatementに変更"
git push origin feature/login-page

プッシュすると、プルリクエストに自動的に変更が反映されます。

3. 修正内容をレビュアーに返信する¶

指摘コメントに対して「修正しました」とコメントを返します。

返信の例:

> このSQLはプレースホルダーを使っていません。

修正しました。PreparedStatementを使ったバインド変数方式に変更しています。
ご指摘ありがとうございます。

返信のポイント: - どのように修正したかを簡潔に説明する - 指摘への感謝を伝える（義務ではないが、チームの雰囲気が良くなる） - 指摘に対して「なぜそうしたか」の説明が必要な場合は補足する

4. 再レビューを依頼する¶

修正が完了したら、レビュアーに「修正しましたので再確認をお願いします」と連絡します。

GitHub の場合: - 「Re-request review」ボタンでレビュー依頼を再送できます

レビュー指摘の種類を理解する¶

指摘には重さがあります。すべての指摘が「必ず直してください」ではない場合もあります。

種類 意味 対応
Must / [必須] 必ず修正が必要 修正してから再依頼
Should / [推奨] できれば修正してほしい 対応するか理由を説明する
Nit / [細かい] 些細なこと（好み・スタイル） 対応しなくても承認される場合がある
Question / [質問] 確認・質問 回答するだけでよい

チェックリスト（コードレビュー依頼）¶

□ 単体テストが全ケース合格している
□ コードを自分でセルフレビューした（明らかなミスがない）
□ ブランチをリモートにプッシュした
□ プルリクエストを作成した
□ プルリクエストの説明欄に変更概要・確認ポイント・テスト結果を記載した
□ レビュアーをアサインした
□ レビュー依頼の連絡をした（チャットツールなど）

よくある失敗¶

説明欄を空にしてPRを出す¶

「コードを見ればわかる」は相手の時間を奪います。
何を実装したか・何を確認してほしいかを書くのはレビュー依頼者の義務です。

指摘に反論する¶

「私はこれが正しいと思います」という姿勢で反論するのはNGです。
まず指摘の意図を理解してから、自分の意図を伝えてください。
話し合いは歓迎ですが、最終的にはチームの方針に従います。

指摘を直してもレビュアーに連絡しない¶

「プッシュしたからいずれ気づくだろう」ではなく、修正が完了したら必ず連絡しましょう。

成果物の例（社員管理システム）¶

以下は「社員管理システム」の「社員情報登録機能」に対するコードレビューの成果物例です。

プルリクエスト説明文（記入例）¶

## 変更の概要

社員情報登録機能（TASK-201）を実装しました。

## 実装内容

- 社員登録画面（`/employees/new`）のHTMLテンプレート追加
- 入力バリデーション（EmployeeForm.java）の実装
  - 社員番号: 必須・半角数字6桁
  - 氏名: 必須・50文字以内
  - 部署: 必須（プルダウン選択）
  - 入社日: 必須・今日以前の日付のみ
  - メールアドレス: 任意・RFC準拠形式・255文字以内
- 社員番号の重複チェック処理（EmployeeRegisterService.java）
- DBへの保存処理（EmployeeRepository.java）
- 登録成功時: 社員一覧にリダイレクト＋フラッシュメッセージ
- 登録失敗時: 同画面に戻り、エラー箇所をハイライト表示

## 関連タスク

- TASK-201: 社員情報登録機能の実装

## 確認してほしいポイント

1. `EmployeeRegisterService.register()` の例外処理の粒度が適切かどうか  
   （DuplicateEmployeeNumberException を独立させるか、DataIntegrityViolationException をキャッチするか迷いました）

2. バリデーションアノテーションの組み合わせ（特に `@PastOrPresent` + `@DateTimeFormat` の併用）で  
   問題がないか確認をお願いします

## テスト結果

| No | テストケース                         | 結果 |
|----|--------------------------------------|------|
| 1  | 全項目入力して登録成功               | ○    |
| 2  | メールアドレス空で登録成功（任意）   | ○    |
| 3  | 社員番号空でバリデーションエラー     | ○    |
| 4  | 社員番号5桁でバリデーションエラー    | ○    |
| 5  | 重複社員番号でエラーメッセージ表示   | ○    |
| 6  | 氏名50文字で登録成功                 | ○    |
| 7  | 氏名51文字でバリデーションエラー     | ○    |
| 8  | 入社日を未来日付でバリデーションエラー | ○    |
| 9  | 不正メールアドレスでバリデーションエラー | ○  |

全14件テスト実施・全件合格を確認しています。

レビューコメントと対応の例¶

レビュアーからの指摘（例）:

[Must] EmployeeRegisterService.java:34

employeeRepository.save(employee); の後にログ出力がありません。
登録完了のログを追加してください。
本番環境でのトラブル調査時に、いつ誰が登録したかを追跡できるようにするためです。

対応コメント（例）:

ご指摘ありがとうございます。

ログ出力を追加しました。
logger.info("社員情報を登録しました。employeeNumber={}, registeredBy={}", ...) の形で
ログレベルINFOで出力しています。
ご確認よろしくお願いします。

レビュアーからの指摘（例2）:

[Nit] EmployeeForm.java:18

変数名 email よりも userEmail の方がクラスの他フィールドと統一感があるかもしれません。
（対応は任意です）

対応コメント（例2）:

ご指摘ありがとうございます。
他のフォームクラスでは email という名前で統一されているため、今回はこのままにさせていただきます。
もしチームで統一方針があれば別途合わせます。
