実装（製造） 作業ガイド¶

実装とは¶

詳細設計で「プログラムをどう書くか」が決まった後、実際にコードを書いてシステムを動かす工程です。
設計書を忠実にコードへ変換しながら、品質・保守性・セキュリティに配慮した実装を行います。

要件定義 → 基本設計 → 詳細設計 → 実装 ← 今ここ → テスト

実装とは何をすることか¶

実装（製造）とは、設計書を見てコードを書くだけではありません。以下の3つを同時に行う作業です。

やること 説明
コーディング 詳細設計書の通りにコードを書く
単体テスト 自分が書いたコードが正しく動くか自分で確認する
品質管理 読みやすく・安全で・後から変更しやすいコードを書く

イメージ: 詳細設計が「施工図」なら、実装は「大工が材料を使って実際に建てる作業」です。設計図通りに、かつ丁寧に仕上げることが求められます。

実装の全体フロー¶

STEP 1: 作業開始前の確認・準備
STEP 2: 開発環境のセットアップ
STEP 3: ブランチ作成（バージョン管理）
STEP 4: コーディング
STEP 5: 単体テスト
STEP 6: コードレビュー依頼
STEP 7: 指摘対応・修正
STEP 8: マージ・完了確認

各STEPの概要¶

STEP 1: 作業開始前の確認・準備¶

目的: 「何を作るか」を正しく把握してから手を動かす。

確認するもの 説明
詳細設計書 自分が担当する機能の設計書を読み込む
タスク定義 担当範囲・期限・完了条件を確認する
不明点リスト 設計書を読んで曖昧な点をリストアップし、担当SEに確認する

ポイント:
コードを書き始める前に必ず設計書を読む。「たぶんこうだろう」で進めると後で大きな手戻りになる。
疑問はコーディング前に解消するのが鉄則。

STEP 2: 開発環境のセットアップ¶

目的: 自分のPC上で開発・確認できる環境を整える。

作業内容:

  1. リポジトリの取得
    プロジェクトのソースコードをローカルに取得（clone / pull）する。

  2. 依存パッケージのインストール
    プロジェクトが使用するライブラリ・ツールをインストールする。

  3. 環境変数・設定ファイルの配置
    DBの接続情報・APIキーなどをローカル用の設定ファイルに記入する。
    ※設定ファイルには機密情報が含まれる場合があるため、Gitにコミットしない。

  4. 動作確認
    既存機能が手元で動くことを確認してから、開発を開始する。

ポイント:
環境構築手順はプロジェクトごとに異なる。README.md や手順書を必ず確認する。
うまく動かない場合はチームメンバーに早めに相談する（1人で長時間悩まない）。

STEP 3: ブランチ作成（バージョン管理）¶

目的: 自分の変更を他の作業と分離し、後から確認・取り消しができる状態を作る。

作業内容:

  1. 最新状態に更新する
    作業前に必ずリモートの最新コードを取得する。
git pull origin main
  1. 作業用ブランチを作成する
    自分の担当機能に対応したブランチ名をつける。
git checkout -b feature/login-page

ブランチ名の例: | 種類 | 例 | |---|---| | 新機能追加 | feature/ログイン機能 / feature/login | | バグ修正 | fix/ログインエラー修正 / fix/login-error | | 設定変更 | chore/DB設定変更 |

ポイント:
main ブランチに直接コードを書かない。必ず作業ブランチを切ってから開発する。

STEP 4: コーディング¶

目的: 詳細設計書に基づいて、動作するコードを書く。

作業内容:

  1. 設計書の内容を順番に実装する
    設計書のロジック・バリデーション・SQL・エラー処理をコードに落とし込む。

  2. コーディング規約に従う
    プロジェクトのコーディング規約（命名ルール・インデントなど）を守る。

  3. こまめにコミットする
    ある程度まとまった単位でコミットし、変更履歴を残す。

git add .
git commit -m "feat: ログイン画面のバリデーション処理を追加"

良いコードのポイント:

観点 悪い例 良い例
読みやすさ int x = getUserAge(); で x を使い回す int userAge = getUserAge(); と意味のある名前をつける
重複を避ける 同じ処理を複数箇所にコピーして書く 関数にまとめて呼び出す
セキュリティ URLパラメータをそのままSQLに埋め込む プレースホルダー（バインド変数）を使う
エラー処理 エラーを握りつぶして何もしない エラーをログに記録し、適切なエラーレスポンスを返す

絶対にやってはいけないこと:

  • SQLに入力値をそのまま埋め込む（SQLインジェクションの原因）
  • パスワードを平文でDBに保存する
  • エラーの詳細（スタックトレースなど）をユーザーの画面に表示する
  • 動作確認もせずにコミット・プッシュする

STEP 5: 単体テスト¶

目的: 自分が書いたコードが、設計書の通りに正しく動くことを自分で確認する。

なぜ必要か:
テストなしでコードを渡すと、後のテスト工程で発覚したバグの修正に何倍もの時間がかかる。
バグは早期発見・早期修正が基本。自分でテストできるものは自分で確認してから渡す。

確認するパターン:

パターン 説明 例
正常系 正しい入力で期待通りの結果になるか 正しいメールアドレス・パスワードでログインできる
異常系 不正な入力でエラーになるか 空のパスワードでエラーメッセージが表示される
境界値 上限・下限の値で正しく動くか パスワード8文字でOK、7文字でエラー

単体テストの種類:

  • 手動確認: 画面を操作して目視で確認する
  • 自動テスト（ユニットテスト）: テストコードを書いて自動で確認する（プロジェクトの方針による）

ポイント:
正常系だけ確認して満足しない。エラーになるべきケースで本当にエラーになるかを必ず確認する。

STEP 6: コードレビュー依頼¶

目的: 自分では気づきにくいバグ・設計のズレ・品質の問題を、他者の目でチェックしてもらう。

作業内容:

  1. プッシュする
    作業ブランチをリモートにプッシュする。
git push origin feature/login-page
  1. プルリクエスト（PR）を作成する
    GitHubなどのツールでプルリクエストを作成する。

プルリクエストに記載する内容: | 項目 | 内容 | |---|---| | タイトル | 変更内容がひと目でわかる名前 | | 概要 | 何を実装したか・なぜその方法を選んだか | | 確認してほしいポイント | 自信がない箇所・判断が難しかった箇所 | | テスト結果 | 確認したテストパターンと結果 |

  1. レビュアーをアサインする
    レビューしてほしい担当者（チームリーダーや先輩エンジニア）を指定する。

ポイント:
「完璧になったらレビューを依頼する」と思わなくて良い。早めに依頼して方向性を確認するのも有効。

STEP 7: 指摘対応・修正¶

目的: レビューで指摘された内容を正しく理解し、適切に修正する。

作業内容:

  1. 指摘内容を理解する
    なぜその指摘が来たのかを理解してから修正する。
    わからない場合は「〇〇だと理解しましたが合っていますか？」と確認する。

  2. 修正してコミット・プッシュする
    修正したコードをコミットし、再度プッシュする。

  3. 修正内容を返信する
    「〇〇の指摘を受けて、△△のように修正しました」とレビュアーに返答する。

  4. 再レビュー依頼する
    修正が完了したら承認をもらう。

ポイント:
指摘はコードへの批判ではなく、品質を上げるための協力。素直に受け入れ、学びに変える姿勢が大切。

STEP 8: マージ・完了確認¶

目的: 自分の変更を本流のブランチに取り込み、完了を確認する。

作業内容:

  1. マージする
    承認されたプルリクエストを main ブランチにマージする（操作方法はプロジェクトによる）。

  2. 動作確認する
    マージ後の環境（開発環境・ステージング環境など）で動作を確認する。

  3. 完了報告する
    担当のタスク・チケットを完了ステータスに変更し、必要に応じてリーダーに報告する。

  4. ブランチを削除する
    マージ済みの作業ブランチは削除する（リポジトリを綺麗に保つため）。

成果物まとめ¶

# 成果物 対応STEP
1 ソースコード（実装済みコード一式） STEP 4
2 単体テスト結果（手動確認表 または テストコード） STEP 5
3 プルリクエスト（差分・説明・レビュー記録） STEP 6〜7
4 マージ済みブランチ STEP 8

よくある失敗と注意点¶

設計書を読まずにコーディングを始める¶

「なんとなく作れそう」と思ってコードを書き始めると、後から大幅な手直しが必要になることが多い。
必ず設計書を読んでから作業を開始する。

単体テストをしないまま渡す¶

「動いてるはず」という思い込みで渡すと、後のテスト工程でバグが大量に発覚する。
自分でテストできる範囲は必ず確認してから渡す。

コミットをまとめすぎる・細かくしすぎる¶

  • まとめすぎ: 1つのコミットに複数の機能の変更が混在 → 後から追えない
  • 細かすぎ: 1行変えるたびにコミット → 履歴がノイズだらけ

目安は「1つの意味のある変更 = 1コミット」。

不明点を1人で抱える¶

詰まったときに誰にも相談せず何時間も悩むのは時間の無駄。
30分〜1時間考えてもわからなければ、チームメンバーや先輩に相談する。

セキュリティの確認を省略する¶

「動けばOK」という意識でセキュリティを後回しにすると、深刻な問題につながる。
特にユーザー入力を扱う処理は、SQLインジェクション・XSSに注意して実装する。

実装中に意識する品質の観点¶

観点 チェック内容
正確性 設計書の仕様通りに動くか
可読性 他の人が読んでも理解できるコードか
セキュリティ 入力値の検証・SQLインジェクション対策はできているか
エラー処理 異常系・例外ケースが適切に処理されているか
パフォーマンス 不要なループ・重い処理がないか
