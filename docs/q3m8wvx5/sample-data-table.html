テーブル・一覧画面サンプル

一覧表示に必要な機能を網羅したデータテーブルの設計パターン

⚠️ 重要： 全件取得して並べるだけは危険。データ量が増えると画面が固まる・DBに負荷がかかる。 ページング・ソート・フィルタは必ずサーバーサイドで実装する。 フロントエンドでやるのはあくまで「見た目の切り替え」だけ、が原則。

📋一覧画面に必要な要素

機能 なぜ必要か 実装時の注意
ページネーション 全件表示するとDBとブラウザ双方に負荷。1000件表示は事故のもと 1ページあたりの件数は変更可能にする。デフォルト20〜50件が目安
ソート ユーザーが「日付順に見たい」「金額の大きい順に見たい」という需要は必ずある デフォルトソート順を仕様書に明記する。UIでは現在のソート方向を▲▼で示す
検索・フィルタ 一覧から目的のデータを探す手段がないとユーザーが困る 検索はサーバーサイドで。フロントのJSだけで絞り込むのは全件取得した場合のみ可
チェックボックス選択 「複数選択して一括削除」「複数件エクスポート」に必要 ページ遷移時に選択状態をどう扱うか仕様化する（リセット or 保持）
空状態（0件）表示 何も表示されないと「バグか？」とユーザーが思う 「条件に一致するデータがありません」など理由を示す。初回アクセス時と検索0件時で文言を変えるとより良い
ローディング状態 データ取得中に何も表示しないと壊れてると思われる スピナーかスケルトンスクリーンを表示。ローディング中はソート・フィルタ操作を無効化する
件数表示 「全○件中 N〜M件を表示」で現在地を把握できる フィルタ適用後の件数も表示する
行クリック / アクション 詳細表示・編集・削除といった行への操作 行全体をクリック可能にする場合はfocus/keyboard対応も必要

💡ページングはなぜサーバーサイドか

フロントエンドページング（クライアントサイドページング）の問題点：
  • 全件データをAPIで取得してから絞り込むため、データ量が多いとAPIレスポンスが遅い・重い
  • ブラウザのメモリに大量データを保持するためメモリ不足・フリーズのリスク
  • 検索エンジンが全データをインデックスできない（SEOが必要なページの場合）

サーバーサイドページングの正しい実装：
  • APIリクエストに ?page=2&per_page=20&sort=amount&order=desc&q=田中 のようなパラメータを渡す
  • APIは { data: [...], total: 1234, page: 2, per_page: 20 } を返す
  • フロントはそのページ分だけ描画する
💻APIパラメータの例
// 一覧取得APIの呼び出し例
const response = await fetch('/api/orders?' + new URLSearchParams({
  page: currentPage,      // 現在のページ番号
  per_page: 20,           // 1ページの件数
  sort: sortColumn,       // ソート列名
  order: sortDirection,   // 'asc' or 'desc'
  q: searchQuery,         // 検索キーワード
  status: statusFilter,   // ステータスフィルタ
}));

// APIレスポンスの構造
{
  "data": [
    { "id": "ORD-0042", "customer": "田中 太郎", ... }
  ],
  "meta": {
    "total": 1234,      // 全件数（ページング表示に使用）
    "page": 2,
    "per_page": 20,
    "last_page": 62
  }
}
