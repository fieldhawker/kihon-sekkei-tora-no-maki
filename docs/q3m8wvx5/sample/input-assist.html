💡 入力補助・オートコンプリート

デバウンス・キーボード操作・アクセシビリティまで含めた入力支援UI設計パターン

⚠️ 入力のたびにAPIを叩いてはいけない
「a」「ap」「app」「appl」「apple」と5文字入力するだけで5回APIリクエストが飛ぶ。 高速タイピングでは100ms以内に複数回入力が起こる。デバウンス（300ms待機）で最後の入力から一定時間後だけAPIを呼ぶ。 また、マウス操作者だけでなくキーボード操作にも必ず対応する（アクセシビリティ要件）。

// デバウンス関数の実装（汎用）
function debounce(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);                 // 前のタイマーをキャンセル
    timer = setTimeout(() => {
      fn.apply(this, args);             // delay ms 後に実行
    }, delay);
  };
}

// 使い方
const handleInput = debounce(async (value) => {
  const results = await searchAPI(value);
  renderDropdown(results);
}, 300); // 300ms 待つ

input.addEventListener('input', (e) => handleInput(e.target.value));

♿5. アクセシビリティ対応

💡 スクリーンリーダーへの配慮が必要な理由
視覚障害者はドロップダウンが開いたことをマウスで確認できない。aria-* 属性を使って「候補リストが表示された」「現在〇番目を選択中」などをスクリーンリーダーが読み上げるよう実装する必要がある。 WCAG 2.1（Web Content Accessibility Guidelines）では Level AA の達成が求められるケースが増えている。
ARIA属性設定場所値の例役割
role="combobox" 入力欄（input） — 「コンボボックス（入力+候補リスト）」であることをSRに伝える
aria-autocomplete 入力欄 "list" 入力に連動して候補リストが表示されることを示す
aria-expanded 入力欄 "true" / "false" ドロップダウンが開いているかどうかをSRに伝える
aria-controls 入力欄 候補リストのid この入力欄がどのリストを制御しているかを示す
aria-activedescendant 入力欄 選択中の候補のid 現在フォーカスされている候補をSRに伝える（フォーカスは入力欄のまま）
role="listbox" 候補リスト（ul/div） — 候補リストであることを示す
role="option" 各候補アイテム — リスト内の選択肢であることを示す
aria-selected 各候補アイテム "true" / "false" 現在選択（ハイライト）されているアイテムを示す
aria-live="polite" 候補件数表示エリア — 候補件数の変化をSRが読み上げる（"3件見つかりました" など）

📢6. キーボード操作の実装まとめ

キー動作実装上の注意
↓ 候補リストの次の項目へ移動 リスト末尾では先頭に戻るか、末尾で止めるかをUXとして決める。preventDefault() でページスクロールを防ぐ。
↑ 候補リストの前の項目へ移動 リスト先頭で↑を押した場合に入力欄に戻る実装が自然。
Enter 選択中の候補を確定 フォームのsubmitが発火しないよう preventDefault() が必要。候補選択後は入力欄にフォーカスを戻す。
Escape ドロップダウンを閉じる 入力値はクリアしない（ユーザーの入力を消さない）。フォーカスは入力欄に留める。
Tab ドロップダウンを閉じて次の要素へ 候補未選択のままTabで離れた場合、入力値を保持するか空にするかを決める。
Backspace タグ入力: 最後のタグを削除 入力欄が空の場合のみ動作させる。入力中はBackspaceの通常動作（文字削除）を優先。
