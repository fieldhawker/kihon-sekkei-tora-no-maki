Sample Screen — Error Handling Patterns

⚠️ エラーハンドリングパターン

HTTPステータスコードとエラーレスポンスの設計パターンを示します。適切なエラーレスポンスはAPIを使うフロントエンド・クライアントの開発効率を大きく左右します。

🚫
やってはいけない 3大NG
(1) 全エラーに200を返す — クライアントがエラー判定できず、ログ監視も機能しない
(2) スタックトレースをそのまま返す — 内部構造・フレームワーク情報が攻撃者に渡る
(3) エラーメッセージが「システムエラー」だけ — 利用者もサポートも原因が全く分からない

1エラーレスポンスの標準構造

プロジェクト全体で統一したエラー構造を定めることで、クライアントの実装が単純になります。

{
  "error": {
    "code": "INSUFFICIENT_STOCK",        // 機械可読なエラーコード。クライアントがswitch文で処理
    "message": "商品の在庫が不足しています",  // ユーザー向けメッセージ（日本語OK）
    "details": [                         // フィールドエラーなど複数の詳細（省略可）
      {
        "field": "items[0].quantity",
        "message": "在庫数(2)を超えた数量(5)が指定されました",
        "code": "EXCEEDS_STOCK"
      }
    ],
    "errorId": "err_20260520_abc123"     // サポート問い合わせ用の一意ID。ログと紐付け可能
  }
}
✅
codeフィールドを設ける理由
HTTPステータスコードだけでは粒度が粗い。INSUFFICIENT_STOCK vs ITEM_DISCONTINUED のように業務ロジックのエラーを区別し、クライアントがエラー種別に応じた処理（リトライ・再入力誘導など）を実装できる。
✅
errorIdを発行する理由
ユーザーが「エラーID: err_20260520_abc123 が出ました」とサポートに連絡すると、サーバーログから即座に詳細（スタックトレース・リクエスト内容）を特定できる。スタックトレースを返さずに問題を追跡できる。

2HTTPステータスコード別パターン

各ステータスコードの使いどころとレスポンス例を確認できます。

📌
400 Bad Request — リクエスト自体が不正
クライアントが送ったデータが不正な場合。フィールドごとのエラー詳細を返すことで、フロントエンドが各入力欄にエラーを表示できる。
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力内容に誤りがあります",
    "details": [
      { "field": "email",    "message": "メールアドレスの形式が正しくありません", "code": "INVALID_FORMAT" },
      { "field": "password", "message": "パスワードは8文字以上で入力してください",  "code": "TOO_SHORT", "min": 8 },
      { "field": "age",      "message": "年齢は0〜150の整数で入力してください",    "code": "OUT_OF_RANGE" }
    ],
    "errorId": "err_20260520_val001"
  }
}
📌
401 Unauthorized — 認証が必要 / トークンが無効
未ログインとトークン期限切れは code で区別する。クライアントは TOKEN_EXPIRED の場合はリフレッシュトークンで再取得を試みられる。
// パターンA: トークン期限切れ（リフレッシュで対応可能）
{
  "error": {
    "code": "TOKEN_EXPIRED",
    "message": "セッションの有効期限が切れました。再ログインしてください",
    "expiredAt": "2026-05-20T13:00:00Z"
  }
}

// パターンB: 未認証（ログインが必要）
{
  "error": {
    "code": "NOT_AUTHENTICATED",
    "message": "このAPIはログインが必要です"
  }
}
📌
403 Forbidden — 認証済みだが権限不足
401と混同しやすい。401は「誰ですか？」、403は「あなたには権限がありません」。必要なロールを明示することでクライアントが適切なメッセージを出せる。
{
  "error": {
    "code": "PERMISSION_DENIED",
    "message": "この操作を行う権限がありません",
    "details": {
      "requiredRole": "ADMIN",
      "currentRole":  "VIEWER",
      "resource":     "/api/admin/users",
      "action":       "DELETE"
    }
  }
}
📌
404 Not Found — リソースが存在しない
「何が」見つからないかを具体的に伝える。IDを含めることでクライアントがデバッグしやすくなる。
{
  "error": {
    "code": "ORDER_NOT_FOUND",
    "message": "注文が見つかりません",
    "details": {
      "resourceType": "Order",
      "resourceId":   "ord_20260520_999",
      "hint":         "削除済みまたはIDが誤っている可能性があります"
    }
  }
}
📌
409 Conflict — 楽観的ロック競合 / 重複登録
「誰かが先に更新した」ことを知らせる。version情報を返すことでクライアントが最新データを取得して再試行できる。
// パターンA: 楽観的ロック競合
{
  "error": {
    "code": "OPTIMISTIC_LOCK_CONFLICT",
    "message": "他のユーザーが先にデータを更新しました。最新のデータを確認してください",
    "details": {
      "yourVersion":   3,
      "currentVersion": 4,
      "lastUpdatedBy": "user_0421",
      "lastUpdatedAt": "2026-05-20T14:31:55Z"
    }
  }
}

// パターンB: ユニーク制約違反（重複登録）
{
  "error": {
    "code": "DUPLICATE_EMAIL",
    "message": "このメールアドレスはすでに登録されています"
  }
}
📌
422 Unprocessable Entity — 業務ルール違反
400との違い: 400はデータ形式の問題、422は業務ロジックに違反している場合。「形式は合っているが業務上受け付けられない」状況に使う。
// 在庫不足
{
  "error": {
    "code": "INSUFFICIENT_STOCK",
    "message": "在庫が不足しているため注文できません",
    "details": {
      "itemId":        "item_456",
      "itemName":      "ワイヤレスイヤホン Pro",
      "requestedQty":  5,
      "availableQty":  2
    }
  }
}

// クーポン期限切れ
{
  "error": {
    "code": "COUPON_EXPIRED",
    "message": "クーポンの有効期限が切れています",
    "details": {
      "couponCode": "SPRING2026",
      "expiredAt":  "2026-03-31T23:59:59Z"
    }
  }
}
📌
429 Too Many Requests — レート制限超過
Retry-After ヘッダーと残カウントをレスポンスに含めることで、クライアントが適切に待機して再試行できる。
// HTTP Headers
X-RateLimit-Limit:     100
X-RateLimit-Remaining: 0
X-RateLimit-Reset:     1716210600
Retry-After:           60

// Response Body
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "リクエスト制限を超えました。しばらく待ってから再試行してください",
    "details": {
      "limit":          100,
      "windowSeconds":  3600,
      "retryAfterSeconds": 60,
      "resetAt":        "2026-05-20T15:30:00Z"
    }
  }
}
📌
500 Internal Server Error — サーバー内部エラー
スタックトレースは絶対に返さない。errorIdのみ返してサーバーログと紐付ける。ユーザー向けメッセージは「時間をおいて再試行」など一般的なものに留める。
// ユーザーに返すレスポンス（スタックトレースは含めない）
{
  "error": {
    "code": "INTERNAL_SERVER_ERROR",
    "message": "サーバー内部でエラーが発生しました。しばらくしてから再度お試しください。",
    "errorId": "err_20260520_srv_abc123"
  }
}

// サーバーログには詳細を記録（外部には出さない）
{
  "level": "ERROR",
  "errorId": "err_20260520_srv_abc123",
  "message": "NullPointerException at OrderService.java:142",
  "stackTrace": "java.lang.NullPointerException\n  at ...",
  "requestBody": { ... }
}
📌
503 Service Unavailable — メンテナンス中 / 一時的に利用不可
Retry-Afterヘッダーとメンテ終了予定時刻を必ず返す。ロードバランサーレベルで返すことが多い。
// HTTP Headers
Retry-After: 3600

// Response Body
{
  "error": {
    "code": "SERVICE_UNAVAILABLE",
    "message": "システムメンテナンス中です",
    "details": {
      "maintenanceStart":    "2026-05-20T02:00:00Z",
      "scheduledEndTime":    "2026-05-20T05:00:00Z",
      "maintenancePageUrl":  "https://status.example.com",
      "contactEmail":        "support@example.com"
    }
  }
}

3ユーザー向けエラー画面の例

APIエラーをフロントエンドで受けた際に表示するエラー画面の例です。

⚠️ 入力エラー
400
入力内容を確認してください
• メールアドレスの形式が正しくありません
• パスワードは8文字以上で入力してください
🔐 ログインが必要です
401
セッションの有効期限が切れました
セキュリティのため、30分操作がない場合は自動的にログアウトされます。再度ログインしてください。
🚫 アクセス権限なし
403
このページへのアクセス権限がありません
管理者ページにアクセスするには管理者権限が必要です。権限が必要な場合はシステム管理者にお問い合わせください。
🔍 見つかりません
404
ページが見つかりません
URLが間違っているか、削除された可能性があります。URLをご確認いただくか、トップページからお探しください。
💥 サーバーエラー
500
システムエラーが発生しました
申し訳ありません。処理中にエラーが発生しました。しばらくしてから再度お試しください。

解決しない場合はエラーID err_20260520_abc をサポートにお伝えください。
🔧 メンテナンス中
503
ただいまメンテナンス中です
システムの改善作業を実施しています。

終了予定: 2026-05-20 05:00
状況はこちらで確認できます

4よくある実装ミス

NGパターン なぜ問題か 正しい対応
全エラーに200を返す ログ監視ツールがエラーを検知できない。HTTPクライアントのエラーハンドリングが機能しない 適切なHTTPステータスコードを返す。4xx/5xxを正しく使い分ける
スタックトレースをレスポンスに含める フレームワーク名・バージョン・内部パスが露出し、攻撃の手がかりになる errorIdのみ返し、詳細はサーバーログで管理する
エラーメッセージが英語（Exception名そのまま） 一般ユーザーが理解できない。サポートコストが増加する ユーザー向けは日本語で具体的に。codeで機械可読なキーを提供する
404と403を混同する リソースが存在するが権限がない場合に404を返すのはセキュリティ上の判断だが、一般業務システムでは混乱を招く セキュリティ要件に応じて使い分けを設計段階で決定し、一貫性を保つ
バリデーションエラーを1件ずつしか返さない ユーザーが修正 → 再送信 → 次のエラー、を繰り返す羽目になる 400/422のdetailsに全フィールドのエラーをまとめて返す
エラーレスポンスの構造がAPIによってバラバラ フロントエンドが各APIのエラー構造を個別に対応しなければならない プロジェクト全体でエラーレスポンスの構造を標準化し、共通ミドルウェアで整形する
