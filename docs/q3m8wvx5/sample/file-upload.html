📁 ファイルアップロード

ファイルアップロードUIの設計パターンとサーバーサイド連携を示すサンプルです。

⚠️ セキュリティの大原則： フロントエンドのファイル検証はあくまでUX向上のため。ユーザーはブラウザの検証を簡単に迂回できます。 サーバーサイドでも必ず再検証してください（Content-Type偽装・拡張子偽装対策）。

📋アップロードの基本設計

ファイルアップロード機能は、フロントとサーバーの両方で検証することが必須です。各検証がどちらで行われるべきかを整理します。

検証項目 フロント サーバー 理由
ファイルサイズ 実施 実施 フロントはUX向け（早期エラー表示）、サーバーは攻撃対策（大容量DoS防止）
拡張子チェック 実施 実施 拡張子のみでは不十分。MIMEタイプと合わせてサーバーで検証する
MIMEタイプ（Content-Type） 参考程度 必須 フロントのMIMEはブラウザ推定値。サーバーでマジックバイトを読んで確認する
ファイル名サニタイズ 不要 必須 パストラバーサル（../../etc/passwd）対策。サーバーでUUIDなどに置換する
ウイルス・マルウェアスキャン 不可 推奨 ClamAVなどのスキャナを組み込む。特にユーザー投稿系では必須
画像の解析・リサイズ 任意 実施 ピクセルフラッド攻撃（極端に大きな解像度）を防ぐためサーバー側で制限する

📊アップロード進捗デモ

「アップロード開始」ボタンを押すと各ファイルの模擬アップロードが開始されます。進捗バーでリアルタイムに状態を確認できます。完了・失敗はランダムに発生します（デモ用）。失敗したファイルは個別にリトライできます。

🔒セキュリティ上の注意点

⚠️ 実装時に必ず対処すべき5つのリスク
  1. パストラバーサル攻撃：ファイル名に ../../etc/passwd のようなパスが含まれると、サーバーの任意パスに書き込まれる危険があります。ファイル名はUUID等に強制置換してください。
  2. ファイル名サニタイズ：日本語・スペース・特殊文字を含むファイル名はOSやシェルで問題を起こすことがあります。サーバー側で安全な文字列のみに変換してください。
  3. 実行可能ファイルの禁止：.exe / .sh / .php / .js 等の実行可能ファイルを受け付けてはいけません。許可リスト方式（白リスト）で拡張子を制限してください。
  4. ストレージパスの非公開：保存先のファイルパスをURLに直接使用してはいけません。DBにファイルIDを保存し、IDでファイルを返すAPIを経由してください（直リンク禁止）。
  5. ストレージ容量の制限：ユーザー単位・システム全体のストレージ上限を設け、DoS攻撃（無限アップロード）を防いでください。
💡 大容量ファイルの設計パターン： 10MB超のファイルを扱う場合、ブラウザで直接POSTするのではなく、署名付きURL（Presigned URL）をサーバーから発行し、クライアントがS3等に直接アップロードする方式が一般的です。これによりサーバー負荷を大幅に削減できます。
