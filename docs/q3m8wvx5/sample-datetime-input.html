📅 日付・時刻入力

日付・時刻入力UIの設計パターンとよくある落とし穴を解説するサンプルです。

⚠️ タイムゾーンを軽視すると本番で重大なバグになります： 日本にいるユーザーが入力した「2026-05-20 00:00」をそのまま保存すると、タイムゾーンが異なるサーバーでは前日の15:00として扱われることがあります。 保存はUTC、表示はローカルタイム（+9:00）への変換が基本です。

📋日付の扱い方の基本

場面推奨形式NGパターン理由
DB保存（日時） TIMESTAMP WITH TIME ZONE / UTC VARCHARで文字列保存 文字列では大小比較・計算ができない。DBの型を使うことで自動的に比較・演算が可能
API送受信 ISO 8601: 2026-05-20T12:00:00Z 2026/5/20 12:00 や Unixタイムスタンプ ISO 8601はRFC準拠・世界標準。末尾のZでUTCを明示する
フロントエンド表示 Intl.DateTimeFormat でロケール変換 文字列をそのまま表示 サーバーからはUTC文字列を受け取り、ブラウザのタイムゾーンで表示変換する
日付のみ（誕生日等） DATE型: 2026-05-20 DATETIMEで00:00:00を付加 時刻なしの日付にDATETIMEを使うとタイムゾーン変換で日付がずれる危険がある
期間計算（締め切り等） 終了日時をUTCで保存し、差分をjs/DBで計算 ユーザー入力をそのまま減算 DST（サマータイム）切り替え時に1時間ずれが発生することがある

⚠️よくある実装ミス

❌ 具体的な失敗パターン集
  • 文字列比較でバグ： "2026/1/5" > "2026/1/10" → true（5 > 1 の文字列比較になる）。 必ず new Date() でオブジェクトに変換してから比較する。
  • 月が0始まり： new Date(2026, 5, 20) は5月ではなく6月。JSの月は0〜11。 new Date(2026, 4, 20) が5月20日。
  • タイムゾーンなし文字列のパース： new Date("2026-05-20") はUTCとして解釈されるため、日本時間では2026-05-20 09:00になる。 日付のみの入力をJST深夜0時として扱いたい場合は new Date("2026-05-20T00:00:00+09:00") と明示する。
  • うるう年・月末の計算ミス： 「1ヶ月後」を date.setMonth(date.getMonth() + 1) で計算すると、 1月31日の1ヶ月後が3月3日（うるう年以外）になることがある。ライブラリで処理する。
  • DB側のTIMESTAMP WITHOUT TIME ZONEの罠： PostgreSQLの TIMESTAMP（タイムゾーンなし）はタイムゾーン情報を一切持たないため、 サーバーのロケール設定が変わると解釈が変わる。必ず TIMESTAMPTZ を使う。

🗄️DBとAPIでの日付の扱い方

場面推奨型サンプル値
レコードの作成日時（PostgreSQL） TIMESTAMPTZ DEFAULT NOW() 2026-05-20 03:00:00+00（UTC）
