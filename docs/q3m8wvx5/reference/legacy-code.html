Reference — レガシーコード対策

🏚️ レガシーコードからの脱却
Working Effectively with Legacy Code — 実務適用ガイド

Michael C. Feathers 著「レガシーコード改善ガイド」で提言されている、 テストのないコードを安全に変更するための考え方・技法・手順を実務の観点でまとめた資料です。 既存システムの保守・機能追加を担当するSEが、リスクを最小化しながら変更を進める際の指針として活用してください。

⚠️
レガシーコードの定義（Feathers の定義）
一般的に「古いコード」と思われがちだが、Feathers の定義は明快：
「テストのないコードはレガシーコードである」
— 記述された日付や使用言語は関係ない。自動テストによる安全網がなければ、変更のたびにリグレッションリスクを負い続ける。

🔄 1. コード変更の4タイプ

Feathers はコードへの変更を4種類に分類する。どのタイプかを意識するだけで、リスクの見積もりとアプローチが変わる。

変更タイプ 目的 外部動作の変化 リスク
機能追加
Feature Addition		 新しい振る舞いを追加する。既存動作は壊さない 新機能の振る舞いのみ追加 中
バグ修正
Bug Fix		 誤った振る舞いを修正する 一部の動作が変わる（意図的） 高
リファクタリング
Refactoring		 外部動作を変えずに内部構造を改善する 変えない（変えてはいけない） 中
パフォーマンス改善
Performance Optimization		 外部動作を変えずに速度・リソースを改善する 変えない（変えてはいけない） 中
💡 実務ポイント： バグ修正は「既存の動作を変える」ため、影響範囲の特定とテストが特に重要。 「直したつもりが別のバグを生む」スパイラルを防ぐには、修正前に現状動作を記録する特徴付けテストが有効。

🔁 2. フィードバックループの確立

なぜフィードバックループが必要か 安全網

変更が正しいかどうかをすぐに確認できる環境がなければ、変更のたびに手動確認が必要になる。これが積み重なると変更コストが指数的に増加する。

❌ フィードバックループなし
  • 変更 → 手動テスト → デプロイ → 動確
  • 確認に30分〜数時間かかる
  • 副作用の検出が遅れる
  • リファクタリングが怖くてできない
  • 「とりあえず動けばいい」思考が広がる
✅ フィードバックループあり
  • 変更 → テスト実行（数秒〜数分）
  • リグレッションを即座に検出
  • 小さく安全な変更を繰り返せる
  • リファクタリングが安心してできる
  • コードの品質改善サイクルが回る

テストハーネス構築の優先順位 テスト戦略

一度に全テストを書こうとしない。変更を加える箇所から始め、徐々に広げていく。

優先度対象アプローチ
1 これから変更する箇所 変更前に特徴付けテストを書く。変更後は新たな振る舞いのテストも追加する
2 変更箇所が呼び出す依存クラス 依存を断ち切るリファクタリングを行い、モック/スタブで分離する
3 バグが頻発するモジュール 再発防止のためにテストを追加し、修正を安全に進める
4 その他のモジュール 機能追加・バグ修正のタイミングに合わせてテストを徐々に充実させる

✂️ 3. 継ぎ目（Seam）の概念

📖
継ぎ目（Seam）とは
「コードを書き換えることなく、プログラムの動作を変えられる箇所」のこと。 テスト可能にするためには、本番コードと依存先の間に継ぎ目を見つけ（または作り）、 テスト時だけ動作を差し替えられるようにする。

継ぎ目の種類と実装例 Seam

継ぎ目の種類説明Java/TS の典型例
オブジェクト継ぎ目 依存オブジェクトをコンストラクタ・セッター・引数で差し替えられるようにする（DI） インタフェースを導入しモックを注入する。SpringのDIコンテナが典型
リンク継ぎ目 コンパイル・リンク時に別の実装を差し込む Javaのクラスパス差し替え、TypeScript の module mock（Jest）
前処理継ぎ目 マクロや条件コンパイルで動作を切り替える Cのプリプロセッサ。Java/TSでは通常不要
✗ 継ぎ目なし — テスト不可能
class OrderService {
  processOrder(order: Order) {
    // 直接 new しているため差し替えられない
    const mailer = new SmtpMailer();
    const repo = new OrderRepository();
    repo.save(order);
    mailer.send(order.userEmail, "注文確定");
  }
}
✓ 継ぎ目あり — DI でモック注入可能
class OrderService {
  constructor(
    private readonly repo: IOrderRepository,
    private readonly mailer: IMailer
  ) {}

  processOrder(order: Order) {
    this.repo.save(order);
    this.mailer.send(order.userEmail, "注文確定");
  }
}
// テストではモックを渡すだけ
const svc = new OrderService(mockRepo, mockMailer);

🔬 4. 特徴付けテスト（Characterization Tests）

📖
特徴付けテストとは
「コードが現在どう動いているかを記録するテスト」。 正しい動作を先に検証するのではなく、現状の動作を観察し固定する。 変更後に同じテストを実行することで、意図しない動作変化（リグレッション）を即座に検出できる。

特徴付けテストの作成手順 手順

1変更対象のメソッド・クラスを特定し、呼び出しに必要な最低限の依存を洗い出す
2テストを書く（期待値はまだ正しいかわからないのでダミーで書く）
3テストを実行し、実際の出力値・例外・副作用を観察する
4観察結果をそのまま期待値として記録する（これが「現状の振る舞い」の固定）
5テストが安定して通ることを確認したら、変更作業を開始する
6変更後にテストを実行し、意図しない差異がないか確認する
特徴付けテストなしで変更
// 変更前のコードを理解しないまま
// 「たぶんこうだろう」で修正する
// → どこかで壊れても気づかない
class TaxCalculator {
  calculate(price: number): number {
    // 複雑な条件分岐...
    // 変更してみたが本当に大丈夫?
  }
}
特徴付けテストで現状を固定してから変更
// まず現状の動作を記録する
describe("TaxCalculator (characterization)", () => {
  it("1000円の場合 110 を返す", () => {
    expect(calc.calculate(1000)).toBe(1100);
  });
  it("軽減税率品 1000円の場合 1080 を返す", () => {
    expect(calc.calculate(1000, true)).toBe(1080);
  });
});
// テストが通ったら安心して変更できる

🔗 5. 依存関係を断ち切る技法

テストが書けない最大の原因は「依存関係が絡み合っていること」。 以下の技法は、コードを大きく変えずに依存を切り離し、テスト可能にするためのリファクタリングパターン。

① メソッドの抽出（Extract Method）
Extract Method
長大なメソッドの中から、テストしたいロジックの塊を別メソッドとして切り出す。 切り出したメソッドは単独でテストできるようになる。 最も基本的かつ安全なリファクタリング。
👉 使いどき：長大メソッドの一部をテストしたいとき
変更前 — テストできない巨大メソッド
processOrder(order) {
  // 100行の複合処理
  // 税計算
  const tax = order.price * 0.1;
  const total = order.price + tax;
  // DBへの保存
  this.db.save({...order, total});
  // メール送信
  this.mailer.send(order.email);
}
変更後 — 税計算だけテスト可能に
processOrder(order) {
  const total = this.calculateTotal(order);
  this.db.save({...order, total});
  this.mailer.send(order.email);
}

// 単独でテスト可能
calculateTotal(order: Order): number {
  const tax = order.price * 0.1;
  return order.price + tax;
}
② クラスの抽出（Extract Class）
Extract Class
肥大化したクラスから、単一の責務を持つクラスを切り出す。 切り出したクラスはインタフェースを通じてモックに差し替えられるようになる。
👉 使いどき：クラスが複数の責務を持っていてテストが難しいとき
③ 新芽メソッド（Sprout Method）
Sprout Method
既存の複雑なメソッドに機能追加する際、既存コードには手を触れず、 新しいメソッドとして追加し、既存メソッドから呼び出す。 新芽メソッドはテストが書きやすい状態で作れる。
👉 使いどき：既存の危険なメソッドを変更せずに機能追加したいとき
既存の複雑なメソッドを直接変更
// 既存の複雑なメソッドに条件追加
// → テスト困難な部分が増える
updateInventory(items) {
  // 複雑な既存処理...（触りたくない）

  // ここに新処理を追加してしまう
  if (items.some(i => i.qty < 0)) {
    throw new Error("在庫不足");
  }
}
新芽メソッドで安全に追加
updateInventory(items) {
  // 既存の複雑な処理（触らない）
  this.validateStockLevels(items); // 追加呼び出しのみ
}

// 新しいメソッドは単独でテスト可能
validateStockLevels(items: Item[]): void {
  if (items.some(i => i.qty < 0)) {
    throw new InsufficientStockError();
  }
}
④ 新芽クラス（Sprout Class）
Sprout Class
新芽メソッドより大きな追加が必要なとき、独立した新クラスとして実装し、既存クラスから利用する。 既存クラスへの変更を最小限にしつつ、新しい機能を安全にテスト可能な形で導入できる。
👉 使いどき：複数のメソッドを追加するような大きな機能拡張のとき
⑤ ラップメソッド（Wrap Method）
Wrap Method
既存メソッドを別名にリネームし、元の名前のメソッドで既存処理と新処理を両方呼び出す。 既存の呼び出し側コードを変更せずに振る舞いを追加できる。
👉 使いどき：既存メソッドの前後に処理を追加したいが、呼び出し側を変えたくないとき
⑥ コンストラクタ注入 / セッター注入（Dependency Injection）
Constructor / Setter Injection
new でオブジェクトを生成している箇所をコンストラクタ引数またはセッターで受け取るよう変更する。 テスト時はモックを注入でき、本番ではDIコンテナが依存を解決する。 最も基本的で効果が高いテスト容易性改善。
👉 使いどき：DB・メール・外部APIなど I/O 依存をモックに差し替えたいとき

🛡️ 6. 安全な変更の進め方

レガシーコードを変更する際の基本フロー フロー

変更の種類（機能追加 / バグ修正）によってアプローチが異なる。いずれもテストを先行させることが核心。

機能追加の場合
1追加箇所の周辺コードを読み、現状の動作を把握する
2継ぎ目を探す。なければ依存を断ち切るリファクタリングを行う
3新芽メソッド/クラスとして新機能を実装し、テストを書く
4既存コードから新芽を呼び出すよう結合する
5全テストを実行し、リグレッションがないことを確認する
バグ修正の場合
1バグを再現するテストを書く（まず失敗することを確認）
2変更対象の周辺に特徴付けテストを追加し、現状動作を固定する
3バグを修正する（最小限の変更で）
4バグ再現テストが通ることを確認する
5特徴付けテストが壊れていないことを確認する
⚠️
「動いているコードを壊したくない」というプレッシャーへの対処
変更への恐れは、テストがないから生まれる。 「テストを書く時間がない」という状況こそ技術的負債が蓄積した証拠。 小さな変更でも変更した部分にテストを1本追加するだけで、次の変更が格段に楽になる。 1回の変更に1本のテスト追加を習慣にするだけで、半年後のコードベースは別物になる。

👃 7. テスト困難にするコードの臭い

以下のパターンがあると、テストを書くことが困難になる。設計レビュー・コードレビューの際に意識する。

臭いの名前症状対処法
神クラス（God Class） 1つのクラスが数百〜数千行、あらゆる処理を持つ 責務ごとにクラスを抽出（Extract Class）
長大メソッド 1メソッドが数十〜数百行、複数の処理が混在 ロジックを抽出（Extract Method）
隠れた依存 メソッド内で new して依存を生成。外から差し替え不可 コンストラクタ注入 / DI パターンを導入
グローバル状態 static フィールド・シングルトンを多用し、テスト間で状態が汚染される 状態をインスタンス変数に移し、テスト毎に初期化する
I/O と計算の混在 ファイル読み書き・DB操作とビジネスロジックが同一メソッドに混在 純粋関数（計算）と副作用（I/O）を分離する
コンストラクタの副作用 コンストラクタがDB接続・ファイル読み込みなどを行う コンストラクタは初期化のみに留め、副作用は初期化メソッドに移す
継承の濫用 深い継承ツリーで基底クラスの変更が多くのクラスに波及する 継承よりコンポジション。共通処理はインタフェース＋委譲に変更

✅ 8. 実務チェックリスト

変更前チェック
確認項目
☐変更の種類（機能追加 / バグ修正 / リファクタリング / パフォーマンス改善）を明確にしてある
☐変更対象のコードに対して特徴付けテストを追加し、現状動作を固定してある
☐変更による影響範囲（呼び出し側・呼び出される側）を洗い出してある
☐テストが書けない場合、継ぎ目（Seam）の導入計画がある
変更中チェック
確認項目
☐1回のコミットは「テストを追加する」または「コードを変更する」のいずれか1つに絞っている
☐リファクタリング中に外部動作を変えていない（テストが全て通り続けている）
☐新芽メソッド/クラスを使い、既存の危険な部分に無用な変更を加えていない
☐モック/スタブを過度に使わず、ビジネスロジックの本質はモックなしでテストしている
変更後チェック
確認項目
☐変更前に追加した特徴付けテストが全て通っている
☐バグ修正の場合、バグを再現するテストが通っている
☐機能追加の場合、新機能の正常系・異常系テストが通っている
☐テストカバレッジが変更前より低くなっていない
☐コードの臭い（神クラス・長大メソッド等）を新たに生み出していない
