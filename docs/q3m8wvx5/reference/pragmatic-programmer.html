🎯 1. 達人の姿勢 — 責任・誠実さ・積極性

"Take Responsibility"
自分のキャリア、学習、プロジェクトの成否に責任を持つ
達人は言い訳をしない。問題の原因がどこにあるかを問わず、解決策を提示できるよう行動する。 「○○のせいでできません」ではなく「こうすれば前進できます」という姿勢が達人を定義する。

猫がソースコードを食べた？— 言い訳の代わりに選択肢を提示する 言い訳禁止

問題が起きたとき、「できない理由」を並べるのではなく「どうすれば前進できるか」を提示するのが達人の姿勢。

✗ 言い訳型 — 問題をそのまま投げる
// 開発中の報告例
「テストが間に合いません。
 仕様変更が多すぎたので。」

「本番でバグが出ました。
 レビューで指摘がなかったので。」

「ドキュメントが古くて
 理解に時間がかかりました。」
✓ 選択肢型 — 解決案を添えて相談する
// 達人的な報告例
「テスト完了が金曜になります。
 優先度を下げる機能はありますか？」

「本番バグの原因と対処案を
 3パターン用意しました。」

「ドキュメントを最新化しました。
 今後は変更時に合わせて更新します。」

十分によいソフトウェア — 完璧主義とのバランス 現実的完璧主義

「完璧」を求めるあまり何もリリースしないのは悪。ユーザーが今必要な品質で届け、フィードバックを得て改善し続ける。

💡 実務ポイント： 品質のトレードオフを意識的に決定する。「このリリースでは○○の品質レベルを保証するが、△△は次のスプリントで対応する」と ステークホルダーと合意することが達人的アプローチ。品質の妥協を「見えない形で」行うのは悪、明示的に議論するのは良。

🔁 2. DRY原則 — Don't Repeat Yourself

"Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."
システム内のあらゆる知識は、単一の・明確な・信頼できる表現を持たなければならない
DRY はコードの重複を避けることだけではない。知識の重複を避けることが本質。 同じ情報が複数の場所に存在すると、一方を変更したときに他方が古くなり、矛盾が生まれる。

DRY違反の4パターン 知識の重複

パターン説明対処法
コードの重複 同じロジックが複数のクラス・メソッドに散在する 関数・クラスに抽出して一元化する
コメントとコードの重複 コードに書いてある内容をコメントで繰り返す コメントは「なぜ」だけ書く。「何」はコードを読めばわかる
データ構造の重複 DBスキーマ・APIスキーマ・コードの構造が別々に管理される スキーマを単一のソースから自動生成する（OpenAPI → DTO、Prisma → 型定義など）
ドキュメントとコードの重複 設計書に書いた仕様がコードに二重に存在する コードから設計書を生成する。または設計書をコードに近い場所に置く（ADR等）
✗ DRY違反 — 税率が3箇所に散在
// OrderService.java
double tax = price * 0.10;

// InvoiceService.java
double taxAmount = subtotal * 0.10;

// ReportService.java
BigDecimal tax = amount.multiply(
  new BigDecimal("0.10")
);
// → 税率変更時に3箇所を漏れなく直す必要がある
✓ DRY準拠 — 税率の定義は1箇所
// TaxConstants.java
public final class TaxConstants {
  public static final BigDecimal STANDARD_RATE
      = new BigDecimal("0.10");
}

// 各サービスは定数を参照するだけ
BigDecimal tax = price.multiply(
  TaxConstants.STANDARD_RATE
);
// → 税率変更は TaxConstants の1箇所だけ

📐 3. 直交性 — Orthogonality

"Eliminate Effects Between Unrelated Things."
関係のないものどうしが影響し合わないようにする
直交性とは「ある機能を変更しても他の機能に影響しない」という性質。 コンポーネントが直交していると、変更の影響範囲が局所化され、テストが容易になり、再利用性が高まる。 逆に「どこかを直したら別のところが壊れた」は直交性の欠如が原因。

直交性を確認するテスト質問 直交テスト

設計レビュー・コードレビュー時に以下を自問する。

Q1. 変更の局所性
「この機能を変えるとき、変更が必要なファイルはいくつか？」 → 1〜3ファイルで収まるなら直交性が高い
Q2. 独立性
「このモジュールを単独でテストできるか？」 → モックなしでテストできるなら直交性が高い
Q3. 再利用性
「このコンポーネントを別のシステムで使えるか？」 → 使えるなら直交性が高い
✗ 直交性なし — UIとビジネスロジックが密結合
// UI コンポーネントの中に
// ビジネスロジックが直接書かれている
function OrderButton({ orderId }) {
  const handleClick = async () => {
    // 税計算がUIに直接ある
    const tax = price * 0.1;
    const total = price + tax;
    // DB操作もUIに直接ある
    await db.orders.update(orderId, { total });
    alert("注文確定: " + total + "円");
  };
}
✓ 直交性あり — UI・ロジック・データが分離
// UI はイベントを発火するだけ
function OrderButton({ orderId }) {
  const { confirmOrder } = useOrderService();
  const handleClick = () => confirmOrder(orderId);
  return <button onClick={handleClick}>確定</button>;
}

// ビジネスロジックは独立したサービスに
class OrderService {
  async confirmOrder(orderId: string) {
    const order = await this.repo.findById(orderId);
    const total = this.calc.calculateTotal(order);
    await this.repo.confirm(orderId, total);
  }
}

🪟 4. 壊れ窓理論 — No Broken Windows

"Don't leave broken windows (bad designs, wrong decisions, or poor code) unrepaired."
壊れた窓（悪い設計・誤った判断・粗悪なコード）を放置するな
犯罪学の「壊れ窓理論」と同じ原理。ビルの窓が1枚割れたまま放置されると、 やがて全ての窓が割られる。コードベースでも「少し汚いコード」「TODO コメントの放置」「テストのスキップ」が 蓄積すると、チーム全体の規律が崩れていく。

開発現場における「壊れ窓」の例 品質維持

壊れ窓の例放置した場合の悪化直し方
// TODO: あとで直す の放置 誰も直さないまま増殖し、コードの意図が読めなくなる 期限を設定してチケット化するか、即座に修正する
テストを @Ignore / skip したまま放置 「テストをスキップしてもいい」という文化が定着する スキップの理由をコメントに書き、期限付きで復活させる
警告（Warning）を無視し続ける 警告が増えて本当に重要な警告が埋もれる CI でゼロ警告を強制する。or 警告を抑制した理由をコメントする
設計書と実装の乖離を放置 設計書は参照されなくなり、口伝えの知識だけが残る 変更時に必ず設計書も更新するルールをチームで合意する
命名規則の例外を「一時的に」許可 例外が標準になり、規則が形骸化する 例外を認めるときは理由をコメントに残し、技術的負債として記録する
💡 実務ポイント： 壊れ窓を「今すぐ直せない」場合は、意図的に汚くしたことをコメントで明示する。 // FIXME: 暫定対応（CR-025 で正式対応予定 2026-06-30） のように、 「誰かがちゃんと認識している問題」にするだけで、品質劣化のスピードが大きく変わる。

🎯 5. トレーサー弾 — Tracer Bullets

"Use Tracer Bullets to Find the Target."
トレーサー弾を使って目標を見つけよ
機能を1つ1つ完璧に作りこんでから結合するのではなく、 エンドツーエンドで薄く動くものを最初に作る。 UI → API → DB まで貫通する「骨格」を先に完成させ、 そこにフィードバックを得ながら肉付けしていく手法。

プロトタイプとの違い 開発戦略

🎯 トレーサー弾
  • 捨てない。本番コードとして育てる
  • エンドツーエンドで動く最小限の実装
  • 本番品質のコード（テスト・エラー処理含む）
  • フィードバックを得て機能を追加していく
  • 例：注文機能の画面・API・DB保存が最低限動く状態
🔬 プロトタイプ
  • 捨てる。学習のための使い捨て
  • 特定のリスク・不確実性を検証するため
  • 品質は問わない（エラー処理・テスト不要）
  • 検証が終わったら破棄し、本番コードを書き直す
  • 例：新しいライブラリのAPIを試すためのコード
💡 実務ポイント： スプリントの最初に「薄く全層を貫通するストーリー」を選ぶ。 「商品一覧を表示する」だけでもUI→API→DBが全部通ると、 環境構築・デプロイパイプライン・チーム間の役割分担の問題が早期に露見する。 後半の「いざ結合したら動かない」を防ぐ最も有効な手段。

📜 6. 契約による設計 — Design by Contract

"Design with Contracts."
契約によって設計せよ
すべての関数・メソッドには「事前条件（呼び出し側の責任）」と「事後条件（実装側の保証）」と 「不変条件（常に成立すべき状態）」がある。 これを明示的に設計・実装することで、バグの原因特定が格段に速くなる。

事前条件・事後条件・不変条件の実装 DbC

✗ 契約なし — どこでエラーが起きたか不明
// 何でも受け入れ、どこかで失敗する
function calculateDiscount(
  price: number,
  discountRate: number
): number {
  return price - (price * discountRate);
}
// price が負でも、rate が 2.0 でも動いてしまう
// 結果がおかしくてもどこが問題か不明
✓ 契約あり — 違反箇所が即座に明確になる
function calculateDiscount(
  price: number,       // 事前条件: price > 0
  discountRate: number // 事前条件: 0 <= rate <= 1
): number {
  // 事前条件チェック（呼び出し側の責任を強制）
  if (price <= 0) throw new Error("price must be > 0");
  if (discountRate < 0 || discountRate > 1)
    throw new Error("rate must be 0..1");

  const result = price - (price * discountRate);

  // 事後条件チェック（実装の保証を検証）
  if (result < 0 || result > price)
    throw new Error("postcondition violated");

  return result; // 不変条件: 0 <= result <= price
}
契約の種類意味誰の責任か実装例
事前条件（Precondition） 関数を呼び出す前に満たされていなければならない条件 呼び出し側（クライアント） 引数バリデーション・Guard Clause
事後条件（Postcondition） 関数が完了した後に保証される状態 実装側（サプライヤー） 戻り値のアサーション
不変条件（Invariant） オブジェクトの生存期間中に常に成立すべき状態 クラス設計者 コンストラクタ・各メソッドの終了時に検証

⏱️ 7. 見積もりの技法

"Estimate to Avoid Surprises."
驚きを避けるために見積もれ
見積もりは「予言」ではなく「精度付きの予測」。重要なのは精度の幅を明示すること。 「2週間でできます」より「うまくいけば1週間、通常は2週間、問題が出れば1ヶ月」の方が 正直かつ有用な情報。

見積もり単位と精度の関係 精度

答える単位によって精度の期待値を調整できる。「130日」と言うより「約4ヶ月」と言う方が適切な精度を伝えられる。

答える単位暗示する精度使いどき
1〜15 分±5分程度「ちょっと見てくる」レベルのタスク
1〜数 時間±30%程度1日以内で終わるタスク
1〜3 日±1日程度機能単位のタスク
1〜数 週±数日程度小機能・バグ修正スプリント
1〜数 ヶ月±1〜2週間機能リリース・フェーズ
数 ヶ月〜半年±1ヶ月以上プロジェクト全体の概算

見積もりを改善するPRACTICE 見積技法

① 何を聞かれているか理解する
「実装」だけか、「テスト・レビュー・デプロイ」も含むか確認してから答える
② 類似タスクの実績を参照する
過去の類似案件の実績を残しておき、根拠として使う（ベロシティの活用）
③ 不確実な要素を明示する
「○○の仕様が確定していないため±1週間のブレがある」と添える
④ 見積もりを分解する
「画面実装3日＋API実装2日＋テスト2日」のように積み上げで見積もる
⑤ 実績を記録して精度を上げる
見積もりと実績の差を記録し、次の見積もりに活かす継続的改善
⑥ 「わからない」と言う勇気
不確実なまま答えるより「○○を調査してから答えます」が誠実で正確

🤖 8. 自動化の哲学 — Shell Games & Power Editing

"Don't Use Manual Procedures."
手作業の手順を使うな
人間は繰り返し作業でミスをする。手順書を読みながら手動でやる作業は、必ずいつかミスをする。 繰り返し行う作業はスクリプト化・自動化し、人間の判断が必要なことだけに集中する。
自動化すべき作業自動化の手段期待される効果
ビルド・テスト実行 CI/CD パイプライン（GitHub Actions, GitLab CI 等） 毎回確実に実行され、人的ミスがなくなる
コードフォーマット Prettier, google-java-format, spotless（コミット時フック） フォーマットの議論がなくなり、レビューに集中できる
静的解析・Lint ESLint, Checkstyle, SpotBugs（PR時に自動実行） コードレビューでの指摘を機械的なものに費やさなくなる
DBマイグレーション Flyway, Liquibase（デプロイパイプラインに組込み） 手順漏れによるスキーマ不整合を防ぐ
リリース手順 デプロイスクリプト、Infrastructure as Code（Terraform等） 環境差分・手順抜けによる障害を防ぐ
ドキュメント生成 OpenAPI → API仕様書、Javadoc → リファレンス コードと仕様書の乖離（DRY違反）を防ぐ
💡 実務ポイント： 「自動化する時間がない」は逆。自動化しない方が長期的なコストは高い。 1回10分の手作業が週3回あれば月1時間。それを5人のチームで1年続けると60時間の損失。 自動化への投資は最も ROI の高い投資のひとつ。

🧪 9. テストへの姿勢

"Test Early, Test Often, Test Automatically."
早くテストし、頻繁にテストし、自動でテストせよ
テストはコードを書いた後にするものではない。設計・実装と並行して、可能なら先に書く。 バグは早期発見するほど修正コストが低い。本番で見つかるバグは、開発中に見つかるバグの100倍のコストがかかるとも言われる。

達人のテスト哲学 テスト文化

原則内容
テストは仕様書である テストコードは「コードがどう動くべきか」を記述した生きたドキュメント。テストを読めば仕様がわかるように書く
テストしやすいコードを書く テストが書きにくいコードは設計が悪いサイン。テスト容易性は良い設計の指標
自分のコードをテストするのは自分 「テストチームがやる」ではなく、実装者自身が単体テストを書く責任を持つ
バグが出たらテストを追加する バグを修正するだけでなく、同じバグが再発しないようにテストを追加する。バグはテストの不足を示している
100% カバレッジは目的ではない 数値を追うのではなく、ビジネスロジックの重要なパスと境界値を必ずテストする

📈 10. 知識のポートフォリオ

"Invest Regularly in Your Knowledge Portfolio."
知識のポートフォリオに定期的に投資せよ
エンジニアの知識は金融資産と同じ。定期的に投資し、リスク分散し、リバランスする。 特定の技術だけに依存すると、技術の陳腐化とともにキャリアが危うくなる。 「今使っていない技術」を学ぶことが、将来の選択肢を広げる。

ポートフォリオ構築の指針 継続学習

📅 定期的に投資する
毎週/毎月「新しいことを学ぶ時間」を意図的に確保する。モチベーションに頼らず習慣にする
📊 多様化する
メイン言語の深掘りだけでなく、異なるパラダイム（関数型・宣言型）や隣接分野（インフラ・DB）も学ぶ
⚖️ リスク管理する
枯れた技術（低リスク・低リターン）と新興技術（高リスク・高リターン）をバランスよく学ぶ
🔄 定期的に見直す
半年ごとに学習内容を振り返り、時代遅れになったスキルと伸ばすべきスキルを棚卸しする
📚 1年に1冊は新しい言語を学ぶ
新しい言語を学ぶことで、使い慣れた言語の「当たり前」を疑う思考力が鍛えられる
💬 コミュニティに参加する
読書・個人学習だけでなく、勉強会・OSS 貢献・技術ブログで「教える」ことで理解が深まる

批判的思考 — 情報を鵜呑みにしない 批判的思考

「○○が最高のフレームワーク」「△△は使うべきでない」という主張を、文脈なしに信じない。

情報に対して問うべきこと
☐ 「誰が言っているのか？」— その人の経験・立場・利害関係は？
☐ 「なぜそう言っているのか？」— ベンダー推薦か、実体験か、理論か？
☐ 「いつの話か？」— 3年前に最適だった技術が今も最適とは限らない
☐ 「自分のコンテキストに適用できるか？」— GAFA の事例がスタートアップに当てはまるとは限らない
☐ 「実際に試したか？」— 読んだ知識と動かした知識は全く異なる
