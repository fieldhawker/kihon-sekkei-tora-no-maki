Reference — 命名規則

📖 命名規則ガイド
リーダブルコードの原則を実務に活かす

「The Art of Readable Code（リーダブルコード）」で提言されている命名の原則を、 変数・関数・クラス・DB・API・設計書ファイル名まで、実務の具体例と対比形式でまとめた資料です。 「なんとなく」の命名から脱出し、チームメンバーが読んで迷わないコード・設計書を作る習慣を身につけましょう。

⚡
リーダブルコード 命名の7原則（早見）
  1. 具体的な名前をつける — 汎用語（data, manager, info）を避け、何を表すか一目で分かる名前にする
  2. 誤解を招かない名前にする — 読み手が「こういう意味かな」と別の意味に解釈できる名前は使わない
  3. 長さより情報量 — 短くても意味が失われるくらいなら、少し長くても情報が多い名前を選ぶ
  4. 省略は慎重に — チーム内で自明な略語のみ使用し、それ以外は書き下す
  5. スコープに合わせた長さ — ループ変数など狭いスコープは短くてよい。広いスコープの変数は詳細な名前にする
  6. コメントは「なぜ」を書く — 名前で表現できない理由・背景・制約だけをコメントにする（「何」はコードを読めばわかる）
  7. 否定形・フラグ名は慎重に — disabled/notFound などは二重否定になりやすい。肯定形で設計する

📦 1. 変数名

原則①：汎用語を避け、具体的な名前を使う 具体性

data・value・info・result は「何でも入る箱」に見える。何のデータかを名前で示す。

✗ NG — 読んでも何かわからない
// 何の data? value?
const data = await fetchApi();
const value = data.result;
let info = {};
let temp = [];
✓ OK — 内容が名前から伝わる
// 何を取得したか一目瞭然
const orderList = await fetchApi();
const totalPrice = orderList.totalAmount;
let userProfile = {};
let pendingOrders = [];

原則②：スコープに合わせた長さにする スコープ

ループの添字 i は慣習として許容される。しかし複数のループが入れ子になるときは意味のある名前を使う。

✗ NG — i/j が何を指すか混乱する
for (let i = 0; i < orders.length; i++) {
  for (let j = 0; j < orders[i].items.length; j++) {
    process(orders[i].items[j]);
  }
}
✓ OK — 変数名がループの意図を示す
for (const order of orders) {
  for (const item of order.items) {
    processOrderItem(item);
  }
}

原則③：単位・型ヒントを名前に含める 単位明示

時間・金額・バイト数など単位を持つ値は、名前に単位を入れるとバグを防げる。

✗ NG — 単位が不明でバグの温床に
const timeout = 30;       // 秒? ミリ秒?
const fileSize = 1024;    // バイト? KB?
const price = 5000;       // 円? ドル?
const start = Date.now(); // 変数名だけでは用途不明
✓ OK — 単位込みで誤用できない
const timeoutMs = 30_000;        // ミリ秒
const fileSizeBytes = 1024;      // バイト
const priceYen = 5000;           // 円
const sessionStartedAt = Date.now(); // 開始時刻
💡 実務ポイント： Spring Boot の application.yml では timeout: 30 が秒かミリ秒か毎回迷う。 プロパティキーを timeout-ms: 30000 にするだけでレビュー時の質問が消える。

⚙️ 2. 関数・メソッド名

原則①：動詞＋目的語で「何をする関数か」を示す 動詞先頭

get/fetch/calculate/validate/send などの動詞で始め、目的語で対象を示す。

✗ NG — 動詞なし・曖昧
function user() { ... }          // 取得? 生成?
function order(id) { ... }       // 検索? 削除?
function data(input) { ... }     // 何の処理?
function doProcess(x) { ... }    // do は何も言ってない
✓ OK — 動詞＋目的語で明確
function fetchCurrentUser() { ... }
function cancelOrder(orderId) { ... }
function validateEmailFormat(email) { ... }
function calculateTotalPrice(items) { ... }

原則②：get vs fetch vs find の使い分け 一貫性

チーム内で動詞の意味を統一する。混在するとルールが読み取れない。

動詞推奨用途例
get同期・即時・ローカルから取得（キャッシュ・プロパティ等）getOrderStatus()
fetch非同期・外部API・DBへのI/OfetchUserProfile(userId)
findコレクションを検索して1件返す（見つからない場合は null）findOrderById(orderId)
listコレクションを検索して複数件返すlistActiveOrders()
create新規リソースの生成（DBへの INSERT を含む）createUserAccount(dto)
update既存リソースの更新（DB UPDATE）updateShippingAddress(orderId, address)
delete / remove削除。delete は物理削除、archive は論理削除で区別すると明確deleteExpiredSessions()
validateバリデーションのみ（副作用なし）。失敗時は例外 or falsevalidatePasswordStrength(raw)
calculate計算のみ（副作用なし）calculateDiscountAmount(price, rate)
send外部へ送信（メール・通知・Webhook等）sendPasswordResetEmail(user)

原則③：誤解を招く名前を避ける 誤解防止

filter は「除外」か「絞り込み」か読み手によって解釈が割れる。

✗ NG — 解釈が割れる
// filterUsers — 有効ユーザーだけ残す?
//              無効ユーザーを除去する?
const result = filterUsers(users);

// clipText(100) — 100文字で切る?
//                 100文字の先だけ残す?
const s = clipText(body, 100);
✓ OK — 動作が一意に読める
// 有効なユーザーのみ抽出（残すほう）
const activeUsers = selectActiveUsers(users);

// 先頭100文字を返す（切り取るほう）
const preview = truncateToLength(body, 100);

🏗️ 3. クラス・型・インタフェース名

原則①：名詞で「何であるか」を示す。役割サフィックスを統一する 名詞・役割

パターン用途例
XxxServiceビジネスロジック層OrderService, UserAuthService
XxxRepositoryDB アクセス層（DAO相当）OrderRepository, ProductRepository
XxxControllerHTTPリクエスト受付（MVC）OrderController, AuthController
XxxDtoAPI入出力専用データ構造CreateOrderRequestDto, UserSummaryDto
XxxEntityDBテーブルに対応するドメインオブジェクトOrderEntity, UserEntity
XxxMapperEntity ↔ Dto 変換OrderMapper
XxxException独自例外クラスOrderNotFoundException, PaymentFailedException
XxxConfig設定クラスSecurityConfig, DatabaseConfig
IXxx / XxxInterfaceインタフェース（言語慣習に合わせる）Java: UserRepository（Iなし）, TypeScript: IUserRepository
✗ NG — 役割がわからない・混在している
class UserManager { ... }      // Service? Repository?
class OrderHelper { ... }      // 何を助けるの?
class DataProcessor { ... }    // 何の data?
class AuthUtil { ... }         // 関数の寄せ集め?
✓ OK — 役割がサフィックスで一目瞭然
class UserAuthService { ... }  // 認証ビジネスロジック
class OrderRepository { ... }  // 注文DB操作
class InvoiceGeneratorService { ... }
class JwtTokenConfig { ... }

🔀 4. 真偽値・フラグ名

原則①：is / has / can / should で始める is/has/can

真偽値は質問文の形にすると if (isLoggedIn) のように自然な英文になる。

✗ NG — true/false が読み取れない
let login = true;          // ログイン中? ログイン処理?
let adminFlag = false;     // Flag は何を意味する?
let permission = true;     // 許可? 権限オブジェクト?
let deleted = true;        // 論理削除? 物理削除済み?
✓ OK — if 文で自然に読める
let isLoggedIn = true;
let isAdmin = false;
let canEditOrder = true;
let isDeleted = true;      // 論理削除フラグ

原則②：二重否定を避ける 否定回避

!isNotFound や !isDisabled のような二重否定はバグの原因になる。肯定形で設計する。

✗ NG — 二重否定で意味が逆になりやすい
if (!isNotFound) { ... }
// !isNotFound = 見つかった、ということ?

let isDisabled = false;
if (!isDisabled) { ... }
// 有効なとき、という意味だが頭を使う
✓ OK — 肯定形で直感的に読める
if (isFound) { ... }
// 見つかったとき、と一目でわかる

let isEnabled = true;
if (isEnabled) { ... }
// 有効なとき、とそのまま読める

🔒 5. 定数名

原則：UPPER_SNAKE_CASE で「変えてはいけない値」であることを明示する 大文字

定数は変数と視覚的に区別できると、誤って代入してしまうリスクが減る。

✗ NG — 変数と見分けがつかない・意味が不明
const maxRetry = 3;
const key = "X-Api-Key";
const timeout = 5000;
const v1 = "/api/v1";
✓ OK — 定数とわかり・意味も明確
const MAX_RETRY_COUNT = 3;
const API_KEY_HEADER_NAME = "X-Api-Key";
const REQUEST_TIMEOUT_MS = 5_000;
const API_V1_BASE_PATH = "/api/v1";
よく使う定数の命名パターン（Java / TypeScript 共通）
用途NG例OK例
HTTPヘッダ名authHeaderAUTHORIZATION_HEADER_NAME
DB接続タイムアウトdbTimeoutDB_CONNECTION_TIMEOUT_MS
最大アップロードサイズmaxSizeMAX_UPLOAD_SIZE_BYTES
パスワード最小長minLenPASSWORD_MIN_LENGTH
ロール名adminROLE_ADMIN / ROLE_VIEWER
APIバージョンv1API_VERSION_V1

🗄️ 6. DB テーブル・カラム名

原則①：テーブル名は複数形 snake_case・カラムは snake_case で統一 snake_case

DBはJavaのcamelCaseルールが使えない。チーム内で表記を統一することが重要。

対象ルールNG例OK例
テーブル名複数形・snake_case・英単語userInfo, Orderusers, orders
中間テーブル親_子 の形（アルファベット順）role_useruser_roles
主キーid 固定（BIGINT）user_id, seqid（テーブル内では不要）
外部キー参照テーブル名_iduserId, uiduser_id
フラグ系カラムis_xxx / has_xxx（BOOLEAN）deleted, adminFlgis_deleted, is_admin
日時カラムxxx_at（timestamp）/ xxx_date（date型）create_time, updatedcreated_at, updated_at
ステータスカラムxxx_status（ENUM or VARCHAR）status（単独は曖昧）order_status, payment_status
✗ NG — 表記ゆれ・用途不明
CREATE TABLE UserInfo (
  seq        BIGINT PRIMARY KEY,
  userName   VARCHAR(100),
  adminFlg   TINYINT,
  createTime DATETIME,
  updateDate DATETIME
);
✓ OK — 統一・意図が明確
CREATE TABLE users (
  id         BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_name  VARCHAR(100) NOT NULL,
  is_admin   BOOLEAN NOT NULL DEFAULT FALSE,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL
);

原則②：意味が重複するカラム名を避ける 重複排除

テーブル名が orders のとき、カラムに order_id（自テーブルの主キー）は冗長。外部キーのみ xxx_id とする。

✗ NG — テーブル名と重複
-- orders テーブル
order_id     BIGINT PK  -- orders.order_id は冗長
order_status VARCHAR    -- order_ も冗長
order_total  DECIMAL
✓ OK — テーブル名との重複なし
-- orders テーブル
id           BIGINT PK  -- テーブル内では id で十分
status       VARCHAR    -- テーブルが order のコンテキスト
total_amount DECIMAL

🔌 7. API エンドポイント名

原則：リソース名（名詞・複数形）で表現し、動詞はHTTPメソッドに任せる RESTful

URLに動詞を入れると /getUser//fetchUser//retrieveUser のように揺れが生まれる。名詞＋HTTPメソッドで統一する。

HTTPNG例OK例意味
GET/getUsers/usersユーザー一覧取得
GET/getUserById?id=1/users/{id}ユーザー1件取得
POST/createUser/usersユーザー作成
PUT/updateUser/1/users/{id}ユーザー全体更新
PATCH/changePassword/users/{id}/passwordパスワード変更（部分更新）
DELETE/deleteUser/1/users/{id}ユーザー削除
POST/doLogin/auth/sessionsログイン（セッション生成）
DELETE/logout/auth/sessions/currentログアウト（セッション削除）

原則②：パスパラメータとクエリパラメータの使い分け パラメータ

種別用途NGOK
パスパラメータ {id} 特定1件を一意に識別するID /orders?orderId=123 /orders/123
クエリパラメータ ?key=val 絞り込み・ソート・ページング /orders/active/page/2 /orders?status=active&page=2

📄 8. 設計書・ファイル名の命名

原則①：プレフィックス＋連番で「種別と順序」を示す プレフィックス

ファイル名だけで「何フェーズの・何の文書か」がわかると、Slack で共有したときも迷わない。

プレフィックスフェーズ命名パターン例
B- 基本設計 B-{連番}_{概要}_{yyyymmdd}.xlsx B-01_画面一覧_20260401.xlsx
D- 詳細設計 D-{連番}_{概要}_{yyyymmdd}.xlsx D-03_API仕様_注文系_20260415.yaml
C- 共通ライブラリ C-{連番}_{ライブラリ名}.md C-01_認証ライブラリ設計.md
ADR- 意思決定記録 ADR-{連番}_{決定事項}.md ADR-003_認証方式の選定.md
WBS- スケジュール WBS_{プロジェクト名}_{yyyymm}.xlsx WBS_受注管理システム_202604.xlsx
CR- 変更要求 CR-{連番}_{変更概要}_{yyyymmdd}.xlsx CR-005_消費税計算仕様変更_20260520.xlsx

原則②：バージョン管理は日付 or v番号を末尾につける バージョン

「最終版」「最終版(2)」「本当に最終版」は絶対禁止。ルールを決めてブレをなくす。

✗ NG — バージョン管理が破綻する
基本設計書.xlsx
基本設計書_最終.xlsx
基本設計書_最終版(2).xlsx
基本設計書_本当に最終.xlsx
基本設計書_レビュー後修正版.xlsx
✓ OK — 日付 or v番号で一意
B-01_画面一覧_20260401.xlsx    ← 初版
B-01_画面一覧_20260408.xlsx    ← レビュー後更新
B-01_画面一覧_20260415.xlsx    ← 最終承認版

-- または semver で管理する場合 --
B-01_画面一覧_v1.0.0.xlsx
B-01_画面一覧_v1.1.0.xlsx

原則③：ソースコードのファイル名・クラス名は「役割」を含める 役割名

種別NG例（曖昧）OK例（役割が明確）
Javaパッケージcom.example.utilcom.example.order.service
com.example.user.repository
JavaクラスファイルOrderManager.javaOrderRegistrationService.java
TypeScript ファイルuser.tsuser-profile.service.ts
user-auth.repository.ts
FlywayマイグレーションV1__init.sqlV1__create_users_table.sql

テストクラスOrderTest.javaOrderRegistrationServiceTest.java

💬 9. コメントの書き方（命名を補う手段として）

⚠️
コメントで補う前に、まず命名を改善する
コードにコメントが必要になったとき、まず「変数名・関数名を改善すればコメント不要にできるか？」を考える。 名前で表現できる情報はコメントにしない。コメントが必要なのは「名前では伝えられない理由・背景・制約」だけ。

原則：コメントは「なぜ（Why）」を書く。「何（What）」はコードを読めばわかる Why

✗ NG — コードを日本語に訳しているだけ
// ユーザーを取得する
const user = await userRepository.findById(id);

// ループで注文を処理する
for (const order of orders) {
  process(order); // 処理する
}

// nullチェック
if (user == null) {
  throw new Error("not found");
}
✓ OK — 読んでもわからない「理由」だけ書く
// キャッシュヒット率向上のため findById より
// findByIdWithCache を使う（平均 40ms → 5ms）
const user = await userRepository.findByIdWithCache(id);

// 解約済み注文も再送対象に含める仕様（CR-012参照）
for (const order of orders) {
  processForResend(order);
}

// findById は存在しない場合 null を返す仕様
// Optional を返すよう変更予定（ADR-009参照）
if (user == null) {
  throw new UserNotFoundException(id);
}

コメントが「必要な」4つのケース

① 非自明な制約・仕様
// 消費税は受注時点の税率を固定する
// 変更しても過去の注文に影響しない
② 意図的に非直感的な実装
// パフォーマンスのため N+1 を許容
// バッチサイズ 100件以下のため問題なし
③ 外部ドキュメントへの参照
// 仕様: ADR-003 参照
// 変更: CR-012 承認済み
④ TODO / FIXME
// TODO: v2.1 で Redis キャッシュに変更
// FIXME: タイムゾーン変換が未実装
⑤ ライブラリのバグ回避
// jackson v2.14 のバグ回避
// https://github.com/.../issues/123
⑥ 難解なアルゴリズムの概要
// Luhn アルゴリズムでカード番号を検証
// 詳細は https://... 参照

✅ 命名レビュー チェックリスト

コードレビュー・設計書レビュー時に以下を確認する。全項目 OK であれば命名起因のバグ・誤解は大幅に減る。

確認項目NG の兆候
☐変数名・引数名に data/value/info/result/temp を使っていない何でも入る「なんとなく名前」が多い
☐関数名が「動詞＋目的語」の形になっているuser()/order()/doProcess()
☐チーム内で get/fetch/find の使い分けが統一されている同じ意味の動詞が混在している
☐真偽値が is/has/can/should で始まっているloginFlag/adminFlg/permission
☐二重否定の変数・条件式がない!isNotFound/!isDisabled
☐時間・金額・サイズなど単位が名前に含まれているtimeout/size/price（単位不明）
☐DBカラム名が snake_case で統一されているuserName/createTime/adminFlg
☐APIエンドポイントに動詞が入っていない/getUser//createOrder//doLogin
☐設計書ファイル名が「プレフィックス＋連番＋概要＋日付」になっている設計書_最終版(2).xlsx
☐コメントが「何（What）」ではなく「なぜ（Why）」を説明しているコードをそのまま日本語に訳したコメント
