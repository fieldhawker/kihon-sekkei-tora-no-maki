🗄️ SQLアンチパターン

Bill Karwin著。データベース設計・SQLクエリで「やってしまいがちな罠（アンチパターン）」を24個挙げ、なぜ問題になるのか・どう解決するかをわかりやすく解説した実践書。「動いているからOK」ではなく、長期的にメンテナンスしやすいDB設計を身につけるための必読書です。

🧭
このページの使い方（初心者向け）
アンチパターンとは「よかれと思ってやってしまう、実は問題のある設計・実装」のことです。まず「なぜ問題なのか」を理解し、次に「どう直すか」を覚えるのが近道です。全部一気に覚えなくてOK。設計・実装で迷ったときに辞書として引いてください。

📚 アンチパターンの全体像

🗂️

24のアンチパターンは4カテゴリに分類される

本書のアンチパターンは以下の4つに分類されています。このページでは特に初心者がハマりやすいものを厳選して解説します。

カテゴリ内容代表的なアンチパターン
① 論理設計テーブル・カラムの設計上の問題ジェイウォーク、EAV、IDリクワイアドなど
② 物理設計インデックスや型の設定上の問題インデックスショットガン、丸め誤差など
③ クエリSQLの書き方の問題スパゲッティクエリ、暗黙の列など
④ アプリケーション開発アプリとDBの連携上の問題パスワード平文保存、SQLインジェクションなど

🏗️ 論理設計のアンチパターン

🚶

① ジェイウォーク（Jaywalking）— カンマ区切りリストを1カラムに詰め込む

「複数の値を1つのカラムに入れれば楽」と思ってカンマ区切りなどで詰め込んでしまう設計です。横断歩道を無視して道を渡る（Jaywalk）ことになぞらえています。

😰 アンチパターンの例

-- タグをカンマ区切りで1カラムに保存
CREATE TABLE articles (
  id    INT PRIMARY KEY,
  title VARCHAR(255),
  tags  VARCHAR(255)  -- "tech,db,sql"
);
  • 特定タグで検索するのが困難・遅い
  • タグの追加・削除でアプリ側の文字列操作が必要
  • タグ数の制限やバリデーションができない

😊 正しい設計（中間テーブル）

-- 多対多の関係は中間テーブルで表現
CREATE TABLE tags (
  id   INT PRIMARY KEY,
  name VARCHAR(50)
);
CREATE TABLE article_tags (
  article_id INT REFERENCES articles(id),
  tag_id     INT REFERENCES tags(id),
  PRIMARY KEY (article_id, tag_id)
);
  • インデックスが効いて高速検索できる
  • JOIN で柔軟に組み合わせられる
💡
見分け方：カラムに「,（カンマ）」や「/（スラッシュ）」「;（セミコロン）」で区切った値を入れようとしたら要注意。「もしかして多対多の関係では？」と自問しましょう。
🌀

② EAV（Entity-Attribute-Value）— 汎用テーブルで属性を縦持ちにする

「将来どんな属性が増えても対応できるように」とすべての属性を「名前・値」の形で縦に並べる設計です。柔軟に見えて、実際には非常に扱いにくくなります。

😰 アンチパターンの例

-- すべての属性をkey-valueで管理
CREATE TABLE user_attributes (
  user_id        INT,
  attribute_name VARCHAR(50),
  value          VARCHAR(255)
);
-- ("1", "name",  "田中太郎")
-- ("1", "email", "tanaka@example.com")
-- ("1", "age",   "30")

😊 正しい設計

-- 属性はカラムとして定義する
CREATE TABLE users (
  id    INT PRIMARY KEY,
  name  VARCHAR(100) NOT NULL,
  email VARCHAR(255) NOT NULL,
  age   INT
);

属性が頻繁に増える場合はJSON型カラムや、サブクラス設計（テーブル継承）も検討する。

⚠️
EAVの問題点：型のバリデーション不可（数値も文字列もVARCHAR）、NOT NULL制約が使えない、JOIN が複雑で読みにくい、パフォーマンスが悪い。「柔軟性」と引き換えに失うものが多すぎます。
🆔

③ IDリクワイアド（ID Required）— とにかく id カラムを追加する

「テーブルには必ず id という名前の連番主キーを付ける」というルールを盲目的に適用してしまうアンチパターンです。

😰 問題になるケース

-- 中間テーブルに不要な id を付ける
CREATE TABLE article_tags (
  id         INT PRIMARY KEY AUTO_INCREMENT, -- 不要
  article_id INT,
  tag_id     INT
);
-- → article_id + tag_id の重複を防げない！

😊 自然キーや複合主キーを使う

-- 複合主キーで重複を防ぐ
CREATE TABLE article_tags (
  article_id INT REFERENCES articles(id),
  tag_id     INT REFERENCES tags(id),
  PRIMARY KEY (article_id, tag_id)
  -- 同じ組み合わせの重複登録を DB が防ぐ
);
💡
主キーの選び方：「一意性を保証する自然なキーがあるか」を先に考える。なければサロゲートキー（連番 id）を使う。とにかく id を付けるのが目的ではなく、データの一意性を保証することが目的です。
🔑

④ キーレスエントリ（Keyless Entry）— 外部キー制約を使わない

「後で入れればいい」「パフォーマンスが下がる気がする」などの理由で外部キー制約（FOREIGN KEY）を設定しないアンチパターンです。

⚠️
外部キーなしの問題：
  • 存在しない親レコードを参照するデータが入り込む（孤児データ）
  • 親を削除しても子が残り続ける（ゴミデータ）
  • アプリ側でのチェックは漏れやすく、バグの温床になる

正しい対応

外部キー制約を設定し、ON DELETE CASCADE（親削除時に子も削除）や ON DELETE RESTRICT（参照されている親は削除不可）を意識して選択する。データの整合性はアプリではなくDBに守らせるのが基本原則。

⚙️ 物理設計のアンチパターン

🔫

⑤ インデックスショットガン — インデックスを闇雲に付ける（または全く付けない）

インデックスの理解が浅いまま「なんとなく全部に付ける」または「パフォーマンスを意識しないで全く付けない」というアンチパターンです。

😰 よくある失敗

  • 全カラムにインデックスを付ける
    → INSERT/UPDATE が遅くなる
  • WHERE句で使うカラムにインデックスなし
    → 全件スキャンで遅い
  • カーディナリティが低い列（フラグ等）にインデックス
    → ほぼ効果がない

😊 インデックスの考え方

  • 検索条件（WHERE）・JOIN・ORDER BY に使うカラムに付ける
  • 値の種類が多い（カーディナリティが高い）カラムに付ける
  • EXPLAIN でクエリの実行計画を確認する
  • 付け過ぎは更新コストが上がるので注意
💸

⑥ 丸め誤差（Rounding Errors）— 金額や精密な数値に FLOAT を使う

浮動小数点型（FLOAT / DOUBLE）は2進数で数値を表現するため、10進数の値を正確に表現できません。金額計算など精度が求められる場面で使うと誤差が発生します。

😰 アンチパターン

CREATE TABLE orders (
  id     INT PRIMARY KEY,
  amount FLOAT  -- NG: 誤差が出る
);

例：0.1 + 0.2 が 0.30000000000000004 になる現象

😊 正しい型を使う

CREATE TABLE orders (
  id     INT PRIMARY KEY,
  -- NUMERIC(精度, 小数桁) で正確に管理
  amount NUMERIC(10, 2)  -- OK
);

DECIMAL（= NUMERIC）は10進数を正確に保持できる。

🔍 クエリのアンチパターン

👻

⑦ 暗黙の列（Implicit Columns）— SELECT * を使う

SELECT * はすべてのカラムを取得するため「楽」に見えますが、長期的には多くの問題を引き起こします。

😰 SELECT * の問題

  • 不要なカラムまで取得してパフォーマンスが悪化
  • カラムが追加されたとき、アプリ側で予期しない動作が起きる
  • どのカラムを使っているかコードから読み取れない
  • JOINしたとき同名カラムが衝突する

😊 必要なカラムを明示する

-- NG
SELECT * FROM users WHERE id = 1;

-- OK: 必要なカラムだけ指定
SELECT id, name, email
FROM users
WHERE id = 1;
🌫️

⑧ 曖昧なグループ（Ambiguous Groups）— GROUP BY の誤用

GROUP BY の結果に、グループ化していないカラムをそのまま SELECT しようとするアンチパターンです。MySQLなど一部のDBは警告なく動いてしまうため、特に注意が必要です。

😰 アンチパターン

-- department_id でグループ化しているのに
-- name（非集計カラム）を SELECT している
-- → どの name が返るか不定！
SELECT department_id, name, MAX(salary)
FROM employees
GROUP BY department_id;

😊 正しい書き方

-- GROUP BY に含めるか、集計関数を使う
SELECT department_id, MAX(salary)
FROM employees
GROUP BY department_id;

-- 名前も欲しい場合はサブクエリや JOIN を使う
🍝

⑨ スパゲッティクエリ — 1つのクエリで全部やろうとする

複数の集計・結合を1つの巨大なクエリにまとめようとするアンチパターンです。「SQLは1文で書けるだけ書いたほうが賢い」と誤解されがちですが、複雑すぎると正しい結果が得られないことがあります。

⚠️
デカルト積（Cartesian Product）の罠： 関係のない2テーブルを JOIN 条件なしで結合すると、行数が掛け算で爆発します（10行 × 10行 = 100行）。複雑なクエリほどこのミスが起きやすく、気づきにくいです。

対策：分割して考える

  • まず小さなクエリで意図した結果が出るか確認する
  • 複雑な処理は WITH句（CTE: Common Table Expression）で段階的に書く
  • 読みやすさ・メンテナンス性を「賢さ」より優先する
❓

⑩ 未知への恐怖（Fear of the Unknown）— NULL の誤解

NULL は「値がない・不明」を表す特別な値です。ところが NULL = NULL は TRUE ではなく NULL（不明）になるなど、直感に反する動作があります。

😰 よくあるミス

-- NULL と = で比較しても常に偽になる
WHERE deleted_at = NULL   -- NG（常に偽）
WHERE deleted_at != NULL  -- NG（常に偽）

-- NOT IN に NULL が含まれると全件除外
WHERE id NOT IN (1, 2, NULL)
-- → 結果が0件になる！

😊 正しい書き方

-- IS NULL / IS NOT NULL を使う
WHERE deleted_at IS NULL      -- OK
WHERE deleted_at IS NOT NULL  -- OK

-- NOT IN の代わりに NOT EXISTS か
-- LEFT JOIN ... WHERE x IS NULL を使う

🖥️ アプリケーション開発のアンチパターン

💉

⑪ SQLインジェクション — ユーザー入力を直接クエリに埋め込む

ユーザーが入力した値を検証せずにSQLに埋め込むと、悪意のある入力によってデータが破壊・漏洩する最も危険なアンチパターンです。

😰 危険なコード例（PHP）

// ユーザー入力をそのまま埋め込む
$sql = "SELECT * FROM users
        WHERE name = '$name'";
// name に「' OR '1'='1」と入力されると
// 全ユーザーが取得できてしまう

😊 プリペアドステートメントを使う

// プレースホルダーを使う
$stmt = $pdo->prepare(
  "SELECT * FROM users WHERE name = ?"
);
$stmt->execute([$name]);
// 入力値は値として扱われ、SQL構文にならない
🚨
絶対に守ること：ユーザー入力・外部からのデータは必ずプリペアドステートメント（バインド変数）を使う。文字列をエスケープするだけでは不十分な場合があります。
🔓

⑫ 読めるパスワード（Readable Passwords）— パスワードを平文で保存する

パスワードをそのままDBに保存するアンチパターンです。DBが漏洩した瞬間に全ユーザーのパスワードが流出します。

😰 やってはいけない

  • パスワードをそのまま保存（例："password123"）
  • MD5やSHA-1などの旧来ハッシュを使う（解析が容易）
  • ハッシュだけでソルトなし（レインボーテーブル攻撃に脆弱）

😊 正しい方法

  • bcrypt / Argon2 / scrypt などのパスワード専用ハッシュ関数を使う
  • ソルト（ランダム値）を付与してハッシュ化する（多くのライブラリが自動処理）
  • フレームワークの認証ライブラリに任せるのが最も安全
👻

⑬ ファントムファイル（Phantom Files）— ファイルパスをDBに保存してファイル本体はDBの外で管理

画像などのファイルをDBの外（ファイルシステムやS3等）に置き、DBにはそのパスだけ保存する設計は広く使われています。ただし、ファイルとDBの整合性管理を怠るとゴーストデータが溜まります。

⚠️
よくある問題：
  • DBのレコードを削除したがファイルが残り続ける（ストレージの無駄）
  • ファイルが削除されたのにDBにパスが残り、アクセス時に404エラー
  • バックアップでDBとファイルの整合性が取れない

対策

  • レコード削除時のファイル削除をアプリ層で確実に実装する（または非同期ジョブで処理）
  • 定期的にDBとファイルシステムの整合性チェックバッチを走らせる
  • 重要データはDBのBLOB型に保存する選択肢も検討する

🌱
アンチパターンを学ぶ意味：これらは「やってしまいがちな落とし穴」であり、知っているだけで多くの問題を未然に防げます。最初から完璧な設計は難しいですが、「これはアンチパターンかも？」と立ち止まる習慣が、長期的に保守しやすいシステムを作ります。
