🐧 UNIXという考え方

Mike Gancarz著「The UNIX Philosophy」。UNIXが50年以上にわたって生き続けている理由は、コードの書き方ではなく「ものの考え方」にあります。小さく・シンプルに・組み合わせる——この哲学は、現代のソフトウェア開発にもそのまま通用します。

🧭
このページの使い方（初心者向け）
UNIXの哲学は「コマンドラインの話」ではありません。「どう設計するか・どう分割するか・どう組み合わせるか」というソフトウェア設計の普遍的な原則です。9つのルールをひとつずつ、自分のコードや設計に当てはめて考えてみましょう。

🌍 なぜ「UNIX的考え方」が今も通用するのか

⏳

50年以上生き続けた設計思想

UNIXは1969年にAT&Tベル研究所で誕生しました。その後、Linux・macOS・AndroidなどあらゆるOSの土台となり、現代のクラウドサーバーの大半で動いています。これほど長く生き残った理由のひとつが、シンプルさを徹底した設計哲学にあります。

❌ 複雑さを積み重ねるソフトウェア

  • 機能が増えるほど重くなる
  • 変更するたびに壊れるリスクが上がる
  • 誰も全体を把握できなくなる
  • やがて誰も触れなくなる

✅ UNIX的なソフトウェア

  • 小さく・ひとつのことに集中する
  • シンプルなインターフェースで繋がる
  • 組み合わせることで大きな力を発揮する
  • 何十年も使い続けられる

1️⃣ 小さいものは美しい

🔬

Small is Beautiful

ひとつのプログラム・関数・クラスはできるだけ小さく保つべきです。小さいコードは読みやすく、テストしやすく、再利用しやすい。大きなコードはその反対です。

😰 やりがちな失敗

// 1つの関数が何でもやる
function processOrder(order) {
  // バリデーション(20行)
  // 在庫確認(30行)
  // 金額計算(25行)
  // メール送信(40行)
  // DB保存(35行)
  // ログ出力(15行)
} // 合計165行！

😊 UNIX的なアプローチ

// 小さな関数を組み合わせる
function processOrder(order) {
  validate(order);
  checkStock(order);
  const total = calcTotal(order);
  saveOrder(order, total);
  sendConfirmEmail(order);
  logOrder(order);
} // 各関数は10〜20行
💡
目安：関数が画面に収まらなくなったら、分割を検討するサインです。「この関数は何をするか？」が一文で言えない場合も同様です。

2️⃣ ひとつのことをうまくやれ

🎯

Do One Thing and Do It Well

これはUNIX哲学の中で最も有名なルールです。プログラムも関数も、ひとつの仕事に集中するべきです。複数のことをやろうとすると、どれも中途半端になります。

UNIXコマンドの例

ls ファイル一覧を表示する。それだけ。
grep テキストを検索する。それだけ。
sort 並び替える。それだけ。
wc 文字数・行数を数える。それだけ。

どれも単純ですが、組み合わせることで強力な処理ができます。

⚠️
「そして（and）」が出たら分割のサイン：関数の説明に「〜して、かつ〜する」が入る場合、それは複数の責任を持っている証拠です。
例：「ユーザーを保存して、メールも送る」→ saveUser() と sendWelcomeEmail() に分ける

3️⃣ できる限り早くプロトタイプを作れ

🚀

Build a Prototype as Soon as Possible

完璧な設計を紙の上で考え続けるより、動くものを早く作ってフィードバックを得る方が最終的に良いものができます。プロトタイプは「使い捨て」でよく、試行錯誤のための道具です。

プロトタイプの目的

  • 技術的に実現可能かを素早く確かめる
  • ユーザーに見せて「これでいいですか？」と確認する
  • 設計の誤りを早期に発見する
  • チームの認識を揃える
💡
「完璧」は「完成」の敵：プロトタイプは汚くていい。後で作り直すことを前提に、まず動かすことを優先しましょう。早く試すほど、早く学べます。

4️⃣ 移植性を大切にせよ

🌐

Portability over Efficiency

特定の環境・OS・ハードウェアに依存したコードを書くと、環境が変わったときに動かなくなります。特定の環境に縛られないコードは、長く使い続けられます。

移植性が低い例

  • Windowsのパス区切り文字 \ をハードコード
  • 特定のDBのSQLの方言に依存
  • 特定のブラウザのAPIを直接使う

移植性が高い例

  • パス操作ライブラリを使う（path.join() など）
  • 標準SQLを使い、方言はORMに任せる
  • 抽象化レイヤーを挟んで実装を隠す

5️⃣ データはテキストファイルに格納せよ

📄

Store Data in Flat Text Files

バイナリ形式やプロプライエタリな形式より、人間が読めるテキスト形式を優先しましょう。テキストデータはどんな環境でも開け、バージョン管理でき、デバッグもしやすい。

形式メリットデメリット
テキスト（JSON, YAML, CSV） 人間が読める・gitで差分が見える・ツールを選ばない バイナリより少し大きい
バイナリ・独自形式 処理が速い・サイズが小さいことがある 専用ツールがないと読めない・デバッグが難しい
💡
現代での応用：設定ファイルはYAMLやJSON・TOMLに。ログはテキスト形式で出力。インフラ定義はコードとして管理（Infrastructure as Code）。これらはすべて「テキストを大切にする」UNIX哲学の現代的な表れです。

6️⃣ ソフトウェアのレバレッジを使え

⚙️

Use Software Leverage to Your Advantage

「ゼロから作る」より既存の道具・ライブラリ・プログラムを最大限に活用するべきです。車輪の再発明をしている時間は、本当に価値を生む部分に使いましょう。

レバレッジを使う具体例

  • 認証機能 → OAuth・JWTライブラリを使う（自前で暗号化を実装しない）
  • 日付計算 → dayjs・date-fns などのライブラリを使う
  • データ変換 → jq・pandas など専門ツールを使う
  • テスト → JUnit・pytest などのフレームワークを使う
⚠️
注意：ライブラリ選定でも「小さくシンプルなもの」を選ぶのがUNIX的。何でもできる巨大フレームワークより、ひとつのことをうまくやる小さなライブラリの方が長く使えることが多いです。

7️⃣ シェルスクリプトを使って力を高めよ

💻

Use Shell Scripts to Increase Leverage and Portability

繰り返し行う作業は自動化するのがUNIX的です。手でやっていることはミスが起きます。スクリプトにすれば毎回同じ結果が得られ、他の人も使えます。

自動化すべきよくある作業

  • ビルド・デプロイ手順
  • テストの実行
  • ログファイルの整理
  • データのバックアップ
  • 環境のセットアップ
  • 定型レポートの生成
💡
初心者へ：「シェルスクリプト」が難しければ、まずnpm scripts・Makefile・CI/CDのワークフロー定義から始めましょう。「手順をコードで記録する」という発想自体がUNIX哲学です。

8️⃣ 対話的インターフェースを避けよ

🔗

Avoid Captive User Interfaces

ユーザーに次のステップを強制する「対話的なインターフェース」は、自動化の妨げになります。入力と出力を標準化し、他のプログラムと繋がりやすくする設計を目指しましょう。

自動化しにくい設計

  • 毎回「次へ」ボタンを押さないと進まないウィザード
  • パスワードを対話的に入力させるCLI
  • 出力が固定幅テキストで解析しにくい

自動化しやすい設計

  • 引数・環境変数で全パラメータを受け取れる
  • 出力はJSON・CSV など解析しやすい形式
  • 終了コードで成功・失敗を伝える
💡
APIに置き換えて考えると：良いAPIは「どんな引数を渡せばどんな結果が返るか」が明確で、他のシステムから呼び出しやすい。これがUNIX哲学の「繋がりやすさ」の現代版です。

9️⃣ すべてのプログラムはフィルタとして設計せよ

🔄

Make Every Program a Filter

UNIXの強力さは「パイプ（|）」——プログラムの出力を別のプログラムの入力に繋ぐ仕組みから来ています。データを受け取り、変換して、渡す「フィルタ」として設計すると、組み合わせの自由度が飛躍的に上がります。

パイプで繋ぐとこんなことができる

# エラーログを集計して多い順に上位10件を表示する
cat app.log | grep "ERROR" | awk '{print $5}' | sort | uniq -c | sort -rn | head -10

cat（読む）→ grep（絞る）→ awk（加工）→ sort（並べる）→ uniq -c（数える）→ sort -rn（再並べ）→ head（先頭だけ）という小さなフィルタの連鎖

💡
関数にも同じ発想を：副作用が少なく、入力を受け取って出力を返す関数は「フィルタ」です。こういった関数は組み合わせやすく、テストもしやすい。関数型プログラミングの考え方とも通じています。

📋 9つのルール まとめ

📌

UNIX哲学 チートシート

#ルール一言で言うと
1小さいものは美しいコードは小さく保つ
2ひとつのことをうまくやれ関数の責任はひとつだけ
3できる限り早くプロトタイプを作れ動くものを早く試す
4移植性を大切にせよ環境依存を減らす
5データはテキストファイルに格納せよ人が読める形式を優先する
6ソフトウェアのレバレッジを使え車輪の再発明をしない
7シェルスクリプトを使って力を高めよ繰り返し作業を自動化する
8対話的インターフェースを避けよ自動化しやすいインターフェースを作る
9すべてのプログラムはフィルタとして設計せよ組み合わせられる設計にする

🌱 UNIX哲学を現代開発に活かす

🏗️

マイクロサービス・クリーンアーキテクチャとのつながり

「小さく・ひとつのことに集中し・組み合わせる」というUNIX哲学は、現代のソフトウェア設計にも形を変えて生き続けています。

マイクロサービス

大きなアプリを「ひとつのことをうまくやる小さなサービス」に分割する考え方。UNIXの「小さいプログラムを繋げる」発想そのものです。

単一責任の原則（SRP）

「クラスは変更の理由をひとつだけ持つべき」——「ひとつのことをうまくやれ」のオブジェクト指向版です。

CI/CD パイプライン

「コードを取得 → ビルド → テスト → デプロイ」と処理を繋いでいく仕組み。まさにUNIXのパイプ思想の現代的な表れです。

Infrastructure as Code

インフラ設定をテキストファイル（YAMLなど）で管理する手法。「データはテキストで格納せよ」の直接的な応用です。

🎓 明日からできること

🚀

UNIX哲学をコードに活かすファーストステップ

#今日からできること対応するルール
1「長すぎる」と感じた関数を分割してみる小さいものは美しい
2「〜して、かつ〜する」関数を探して責任を分けるひとつのことをうまくやれ
3手作業で繰り返している操作をスクリプト化する自動化・レバレッジ
4設定値をコードにハードコードせずファイルや環境変数に移す移植性・テキスト格納
5既存ライブラリで代替できる自作コードを探すレバレッジを使え
🌱
UNIX哲学の本質：これらのルールは「こうしなければならない」という縛りではなく、50年間のソフトウェア開発者たちが学んだ知恵の結晶です。「なぜこのルールがあるのか」を理解すれば、状況に応じて柔軟に応用できます。シンプルさへの追求は、今日の開発にも変わらず有効です。
