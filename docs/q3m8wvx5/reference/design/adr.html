Architecture Decision Records

🏛️ ADR サンプル集

アーキテクチャ決定記録（ADR）は、「なぜその技術・構成を選んだか」を将来の自分やチームに伝えるための文書です。7つのサンプルを収録しています。

🧭
このページの使い方（初心者向け）
1) 「迷った/揉めた/後で変えにくい」判断が出たらADRを書く → 2) 結論だけでなく理由（制約・代替案・トレードオフ）を書く → 3) 破壊的変更が起きたら Superseded で後続ADRへつなぐ。
ADRがあると「なぜそうしたか」が共有でき、設計レビューや引き継ぎが楽になります。
💡
ADR（Architecture Decision Record）とは？
アーキテクチャに関する重要な意思決定を記録するドキュメントです。「何を決めたか」だけでなく「なぜそう決めたか」「他に何を検討したか」「どんなトレードオフがあるか」を記録することで、後から決定を変更する際の判断材料になります。

📁 ADRの管理方法

📝

推奨フォーマット（Markdownファイル）

# ADR-NNN: [タイトル]

**ステータス**: Accepted / Proposed / Superseded by ADR-NNN

## コンテキスト
なぜこの決定が必要だったか。背景・課題・制約を記述する。

## 決定事項
何を決めたか。

## 根拠
なぜこの選択をしたか。

## 検討した代替案
| 選択肢 | メリット | デメリット | 採用/不採用 |
|------|--------|---------|----------|

## 影響・トレードオフ
この決定による影響（良いこと・悪いこと）を記述する。

## 関連ADR
ADR-NNN: [タイトル]

ADRはGitリポジトリの docs/adr/ フォルダに配置し、コードと一緒にバージョン管理することを推奨します。

📋 ADRサンプル一覧（ADR-001〜007）

ADR-001 バックエンドフレームワーク選定 Accepted
コンテキスト

Webシステムのバックエンド開発において、適切なフレームワークを選定する必要がある。チームは Java に習熟しており、大規模なトランザクション処理と外部API連携が要件に含まれる。

決定事項

Spring Boot (Java) を採用する

検討した代替案
選択肢メリットデメリット判定
Spring Bootチーム習熟度高、エコシステム充実、エンタープライズ実績起動が重い、学習コストがある✅ 採用
FastAPI (Python)高速、シンプル、AI/MLとの親和性チームのPython経験不足❌ 不採用
Express (Node.js)軽量、フロントと同言語大規模での型安全性が課題❌ 不採用
影響・トレードオフ
  • チームのスキルセットと合致しており、即戦力で開発開始できる
  • JVM起動時間が長いため、Lambda等のサーバーレス利用時は要検討
  • Spring Securityにより認証・認可の実装が標準化できる
ADR-002 データベース選定 Accepted
コンテキスト

業務データの永続化に使用するデータベースを選定する。トランザクション整合性が必要な業務データ（申請・承認フロー）が中心であり、複雑なJOINが発生する。

決定事項

PostgreSQL を採用する

検討した代替案
選択肢メリットデメリット判定
PostgreSQLACID準拠、JSON対応、豊富な型、OSS—✅ 採用
MySQL枯れた技術、Web実績多数JSONサポートがPostgresより弱い❌ 不採用
DynamoDBスケーラビリティ高、フルマネージド複雑クエリが苦手、JOINなし❌ 不採用
ADR-003 認証方式選定（セッション / JWT / OAuth2） Accepted
コンテキスト

Webシステムの認証方式を決定する。社内向けシステムで、外部サービス連携なし。スマートフォン対応は不要。

決定事項

セッションベース認証（サーバーサイドセッション + Cookie）を採用する

検討した代替案
選択肢向いているケース今回の判断
セッション認証社内Webアプリ、ステートフルでOK✅ 採用：シンプル・無効化が即時
JWTスケールアウト、マイクロサービス❌ 不採用：トークン即時無効化が困難
OAuth2 + OIDC外部サービス連携、SSO❌ 不採用：外部連携が不要のため過剰
ADR-004 フロントエンドフレームワーク選定（React / Vue.js / Next.js） Accepted
決定事項

React + TypeScript を採用する

検討した代替案
選択肢メリットデメリット判定
React + TypeScriptコミュニティ最大、型安全、豊富ライブラリ設定が多い、学習コスト✅ 採用
Vue.js 3学習コスト低、日本語資料豊富大規模プロジェクト実績が少ない❌ 不採用
Next.jsSSR・SSG対応、SEO有利SEO不要の社内ツールに過剰❌ 不採用
ADR-005 アーキテクチャパターン選定（モノリス vs マイクロサービス） Accepted
決定事項

モジュラモノリス（モノリシックアーキテクチャ、ただしモジュール分割を明確化）を採用する

根拠
  • チーム規模が小さく（3名）、マイクロサービスの運用コストが見合わない
  • 初期リリースはシンプルな構成で素早く価値提供し、スケールが必要になった時点で分割を検討する
  • モジュール境界を明確に設計しておくことで、将来のマイクロサービス化を容易にする
ADR-006 キャッシュ戦略（インプロセス / Redis / CDN の使い分け） Accepted
決定事項

セッションキャッシュにRedis、静的コンテンツにCDN（CloudFront）、頻繁に読む参照データにインプロセスキャッシュ（Caffeine）を組み合わせる

使い分け方針
キャッシュ対象採用技術理由
セッションRedis複数サーバー間でセッション共有が必要
コードマスタ等の参照データCaffeine（インプロセス）更新頻度低い、サーバー内完結
静的ファイル（JS/CSS/画像）CloudFront CDNエッジ配信でグローバル低レイテンシ
ADR-007 API設計スタイル選定（REST / GraphQL / gRPC） Accepted
決定事項

RESTful API（OpenAPI 3.0仕様書付き）を採用する

検討した代替案
選択肢向いているケース今回の判断
REST汎用的なWeb API、HTTP標準に準拠✅ 採用：最も広く理解されており、ツール資産が豊富
GraphQLクライアントが必要なデータを選択、多様なクライアント❌ 不採用：Webのみでオーバーフェッチが問題にならない
gRPCマイクロサービス間通信、高スループット❌ 不採用：ブラウザクライアント対応が複雑
