# API作成チェックリスト

## 設計・インターフェース

- [ ] エンドポイントのURLが REST の命名規則に沿っている（名詞・複数形）
- [ ] HTTPメソッドが操作の意味と一致している（GET/POST/PUT/PATCH/DELETE）
- [ ] リクエスト・レスポンスのスキーマ（型・必須項目）が定義されている
- [ ] HTTPステータスコードが適切に使い分けられている（200/201/400/401/403/404/409/500）
- [ ] エラーレスポンスのフォーマットが統一されている
- [ ] APIバージョニングの方針が定義されている（URLパス・ヘッダーなど）
- [ ] OpenAPI（Swagger）などのAPI仕様書が用意されている

## バリデーション

- [ ] リクエストパラメータの必須チェックが実装されている
- [ ] 各パラメータの型・形式・文字数の検証が実装されている
- [ ] 相関チェック（複数項目をまたぐ条件）が実装されている
- [ ] 不正な入力値に対して400エラーと具体的なエラーメッセージが返る
- [ ] SQLインジェクション・コマンドインジェクション対策が実施されている

## 認証・認可・セキュリティ

- [ ] 認証方式が定義されている（JWT・APIキー・セッションなど）
- [ ] 認証トークンの有効期限・更新方法が定義されている
- [ ] 操作ごとに必要な権限（ロール）が定義されている
- [ ] 認可チェックがサーバーサイドで実装されている
- [ ] HTTPS通信が強制されている
- [ ] センシティブな情報（パスワード・トークン）がレスポンスに含まれない
- [ ] CORS設定が適切に行われている

## パフォーマンス・信頼性

- [ ] レスポンスタイムの目標値が定義されている
- [ ] N+1問題が考慮されている
- [ ] 必要に応じてキャッシュ戦略が設計されている（キャッシュキー・TTL）
- [ ] ページネーション（limit/offset or カーソル）が実装されている（一覧系）
- [ ] レート制限（Rate Limiting）の要否が検討されている
- [ ] タイムアウト設定が行われている

## データ整合性・トランザクション

- [ ] 更新系操作のトランザクション範囲が適切に設計されている
- [ ] 楽観的ロック・悲観的ロックの要否が検討されている
- [ ] 冪等性が考慮されている（POST系でも再送で重複しないか）
- [ ] 非同期処理が必要な操作は非同期ジョブとして切り出されている

## ログ・モニタリング

- [ ] リクエスト・レスポンスのログが出力される（リクエストID付き）
- [ ] エラー時のスタックトレース・詳細がサーバーログに記録される
- [ ] 処理時間の計測・ログ出力が行われている
- [ ] ヘルスチェックエンドポイントが用意されている

## テスト

- [ ] 正常系のリクエスト・レスポンスが動作確認されている
- [ ] 各バリデーションエラーが適切なステータスコード・メッセージで返ることが確認されている
- [ ] 認証なし・権限不足のアクセスが拒否されることが確認されている
- [ ] 境界値・大量データでの動作が確認されている
- [ ] 同時リクエスト・並行実行時の整合性が確認されている
- [ ] 単体テスト・結合テストのカバレッジが確保されている
