# バッチ作成チェックリスト

## 設計・要件定義

- [ ] 実行トリガーが明確になっている（定時・手動・イベント駆動）
- [ ] 実行スケジュール（頻度・時刻・タイムゾーン）が定義されている
- [ ] 処理対象のデータ範囲・条件が明確になっている
- [ ] 処理件数の上限・想定件数が見積もられている
- [ ] 冪等性が考慮されている（同一バッチを複数回実行しても結果が変わらない）
- [ ] 再実行（リラン）の仕様が定義されている
- [ ] 処理の前提条件・依存関係（先行バッチなど）が整理されている

## エラーハンドリング

- [ ] エラー発生時の動作（中断 or スキップ）が定義されている
- [ ] エラーログの出力仕様が定義されている（ログレベル・出力先・フォーマット）
- [ ] アラート通知の仕様が定義されている（通知先・条件・内容）
- [ ] 部分失敗時のロールバック・補償処理が考慮されている
- [ ] タイムアウト時間が設定されている
- [ ] デッドロック・ロック競合の考慮がされている

## パフォーマンス・スケーラビリティ

- [ ] 大量データのチャンク分割処理が考慮されている
- [ ] DBへの負荷（大量SELECT/UPDATE）が考慮されている
- [ ] インデックスの利用が最適化されている
- [ ] トランザクションの粒度が適切に設計されている
- [ ] 並列処理の可否と実装方針が定義されている
- [ ] メモリ使用量の上限が考慮されている

## 実行管理・運用

- [ ] 実行ログ（開始・終了・処理件数）が出力される
- [ ] 実行状態（成功・失敗・実行中）を管理する仕組みがある
- [ ] 二重起動防止の仕組みがある
- [ ] 処理進捗を確認できる手段がある
- [ ] バッチ実行の履歴が残る仕組みがある
- [ ] 手動実行・強制停止の手順が定義されている

## セキュリティ

- [ ] 実行権限・実行ユーザーが適切に設定されている
- [ ] 処理する個人情報・機密データのマスキング・暗号化が考慮されている
- [ ] 外部ファイル入出力がある場合、ファイルの入力値検証が行われる
- [ ] 接続情報（DB・API）が安全に管理されている（ハードコードしない）

## テスト

- [ ] 正常系テストケースが用意されている（全件処理、0件処理）
- [ ] 異常系テストケースが用意されている（データ不備、DB障害）
- [ ] 境界値テスト（最大件数、チャンク境界）が実施されている
- [ ] 再実行テストが実施されている
- [ ] 本番相当のデータ量での性能テストが実施されている
