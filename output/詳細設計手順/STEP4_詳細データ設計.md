### 3-4. 詳細データ設計

**このステップの目的**: 基本設計の論理データ設計を、実際にDBに作れる物理設計まで落とし込みます。ここでの設計がシステムの性能に直結します。

- [ ] ER図（物理レベル）を作成した
  > 物理レベル＝実際のDB上のテーブル名・カラム名・データ型をそのまま表現したER図
- [ ] テーブル定義書を物理設計レベルで完成させた（全カラムの型・桁数・制約・コメント）
- [ ] 主要なクエリ（SELECT文）を設計し、実行計画を検討した
  > 実行計画＝DBがクエリをどう処理するかの計画。性能問題の早期発見に重要
- [ ] インデックスの設計（どのカラムにインデックスを張るか）を確定した
  > インデックス＝検索を高速化する仕組み。張りすぎると更新が遅くなるためバランスが重要
- [ ] テーブルのパーティション設計が必要か検討した
  > パーティション＝大量データのテーブルをデータ範囲で分割して性能を保つ設計
- [ ] 文字コード・照合順序（コレーション）を決定した
  > 照合順序＝文字の比較・ソートのルール。日本語では `utf8mb4_unicode_ci` などを使う
- [ ] シーケンス（自動採番）の設計方針を決定した
  > 例：IDはDB自動採番 / UUIDを使う / アプリケーション側で採番する
- [ ] N+1問題が起きやすい箇所を特定し、対策（JOINまたは一括取得）を設計した
  > N+1問題＝1件の親データに対してN件の子データを個別に取得する非効率なクエリパターン
- [ ] データ量見積もりを基に、テーブルごとの初期サイズ・1年後の想定サイズを確認した
- [ ] マイグレーションスクリプト（CREATE TABLE / ALTER TABLE）を最新状態に更新した
