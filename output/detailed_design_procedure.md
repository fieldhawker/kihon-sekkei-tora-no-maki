# 基本設計→詳細設計 移行手順書

> **対象読者**: システム開発の詳細設計を担当するSE。詳細設計の経験が浅い方でも使えるよう、用語解説・理由・注意点を随所に記載しています。
> **対象システム**: 汎用的なWebシステム開発

---

## はじめに ― この手順書の使い方

この手順書は、**基本設計が完了してから詳細設計を終えるまで**の作業を、抜け漏れなく進めるための道具です。

| こんなときに使う | 参照箇所 |
|----------------|---------|
| 全体の流れを把握したい | [1章 作業全体像](#1-作業全体像) |
| 何から手をつけるか迷っている | [2章 作業フロー](#2-作業フロー各ステップの順序と依存関係) |
| 各作業で何を作ればよいか確認したい | [3章 チェックリスト](#3-各作業のチェックリスト) |
| 基本設計書に何が揃っているか確認したい | [4章 引き継ぎ確認](#4-基本設計からの引き継ぎ確認チェックリスト) |
| 作るべき成果物の一覧を見たい | [5章 成果物一覧](#5-成果物一覧) |
| 実装フェーズに進んでよいか判断したい | [付録A 移行判定](#付録a-移行判定チェックリスト実装フェーズ移行前の最終確認) |
| ありがちな失敗を事前に知りたい | [付録B よくある抜け漏れ](#付録b-よくある抜け漏れパターン) |
| 用語の意味を確認したい | [付録C 用語集](#付録c-用語集) |

> **チェックリストの使い方**: 各項目を完了したら `[x]` に変更してください。プロジェクトによっては不要な項目もあります。その場合は `[n/a]` として理由をコメントに残しましょう。

> **AIへの相談テンプレートについて**: 各作業セクションの末尾に「💬 AIに相談するときのコンテキスト」を折りたたみ表示で掲載しています。クリックして展開し、`【】`の中を自分のプロジェクトに合わせて書き換えてから、そのままAI（Claude・ChatGPTなど）に貼り付けてください。

## 1. 作業全体像

```
基本設計フェーズ（インプット）
        │
        ▼
┌───────────────────────────────────────────┐
│  移行準備                                  │
│  ・基本設計成果物のレビュー・承認確認       │
│  ・体制・開発環境・スケジュールの確定       │
└───────────────────┬───────────────────────┘
                    │
                    ▼
┌───────────────────────────────────────────┐
│  詳細設計フェーズ                          │
│                                           │
│  [1] 詳細機能設計（処理ロジック）          │
│       ↓  ←── [2] 詳細画面設計 は並行可    │
│  [3] 詳細API設計                          │
│       ↓  ←── [4] 詳細データ設計 は並行可  │
│  [5] 詳細外部インタフェース設計            │
│       ↓                                   │
│  [6] 詳細セキュリティ設計                  │
│       ↓                                   │
│  [7] テスト設計（単体・結合）              │
│       ↓                                   │
│  [8] 開発環境・コーディング規約            │
└───────────────────┬───────────────────────┘
                    │
                    ▼
        詳細設計レビュー・承認
                    │
                    ▼
        実装（コーディング）フェーズへ移行
```

**詳細設計とは何か？**
> 基本設計で「どのように作るか」の骨格が決まったあと、「具体的にどう実装するか」をコーダーが迷わず書けるレベルまで細かく設計するフェーズです。詳細設計書を見ればプログラマーがコードを書けるくらいの粒度が目標です。基本設計との違いは「粒度（細かさ）」にあります。

## 2. 作業フロー（各ステップの順序と依存関係）

| ステップ | 作業名 | 前提（依存） | 主な担当 | 難易度目安 |
|---------|--------|------------|--------|----------|
| 0 | 基本設計成果物の受領・確認 | 基本設計の承認完了 | PL / SE | ★☆☆ |
| 1 | 詳細機能設計（処理ロジック・擬似コード） | ステップ0 | SE | ★★★ |
| 2 | 詳細画面設計（全項目・バリデーション・エラー） | ステップ0 | SE / UIデザイナー | ★★☆ |
| 3 | 詳細API設計（OpenAPI完全版） | ステップ1、ステップ2 | SE | ★★★ |

| 5 | 詳細外部インタフェース設計 | ステップ3 | SE | ★★☆ |
| 6 | 詳細セキュリティ設計 | ステップ1〜5 | SE / セキュリティ担当 | ★★★ |
| 7 | テスト設計（単体テスト・結合テスト方針） | ステップ1〜6 | SE / テスト担当 | ★★☆ |
| 8 | 開発環境構築・コーディング規約 | ステップ0 | SE / インフラ担当 | ★★☆ |
| 9 | 詳細設計書の取りまとめ・レビュー | ステップ1〜8 | PL / SE | ★★☆ |

> **並行作業のポイント**
> - ステップ1（機能）とステップ2（画面）は画面に紐づく処理ロジックが明確になっている部分から並行して進められます
> - ステップ3（API）は機能設計と画面設計が揃ってから着手すること。APIのインプット・アウトプットは機能と画面の両方に依存します
> - ステップ8（開発環境）は早期に着手してもよく、実装開始前に必ず完了させること

## 3. 各作業のチェックリスト

### 3-0. 基本設計成果物の受領・確認

**このステップの目的**: 詳細設計を始める前に、インプット（基本設計書）が詳細設計を進めるのに十分な状態かを確認します。基本設計の曖昧さは詳細設計で必ず問題になります。

- [ ] 基本設計書に正式な承認（サイン・捺印・電子承認）があることを確認した
- [ ] システム構成図（論理・物理）の最終版を入手した
- [ ] 機能一覧（設計版）の最終版を入手し、全機能に機能IDが付与されていることを確認した
- [ ] 画面一覧・画面遷移図の最終版を入手した
- [ ] API一覧（概要版）の最終版を入手した
- [ ] ER図（論理レベル）・テーブル定義書の最終版を入手した
- [ ] 非機能要件設計書を入手し、性能目標の数値が明記されていることを確認した
- [ ] セキュリティ設計書・権限マトリクスの最終版を入手した
- [ ] 外部インタフェース設計書（連携先との仕様合意済み）を入手した
- [ ] 運用設計書・移行設計書の最終版を入手した
- [ ] 未決事項（TBD）一覧を確認し、詳細設計に影響する未決事項がないことを確認した
  > 未決事項が残っている場合は、詳細設計着手前に解消してください。設計途中での仕様変更は手戻りの最大要因です
- [ ] 詳細設計フェーズの体制（担当者アサイン）が確定していることを確認した
- [ ] 詳細設計フェーズのスケジュール（マイルストーン・レビュー日程）が確定していることを確認した

### 3-1. 詳細機能設計（処理ロジック・擬似コード）

**このステップの目的**: 基本設計で定義した「機能の概要」を、プログラマーがコードを書けるレベルの「処理手順」まで細かく落とし込みます。ここが詳細設計の核心です。

- [ ] 機能一覧の全機能について、処理フロー（フローチャートまたは処理手順書）を作成した
  > フローチャート＝処理の流れを図で表したもの。「もし〇〇なら△△する」という分岐も含めて表現します
- [ ] 各機能の正常系・異常系・例外系の処理をすべて定義した
  > 正常系＝想定どおりの操作。異常系＝入力エラー等ユーザー操作による問題。例外系＝システムエラー・タイムアウト等
- [ ] ビジネスルール（業務上の計算・判定ロジック）を具体的な条件式で定義した
  > 例：「申請金額が1万円以上の場合は部長承認が必要」→ `amount >= 10000 AND role == "部長"` のように明示する
- [ ] バッチ処理について、処理対象の抽出条件・処理順序・エラー発生時の継続/中断方針を定義した
- [ ] 帳票・レポートについて、集計ロジック・並び順・出力条件を定義した
- [ ] 各機能の前提条件（事前に満たすべき状態）と事後条件（処理後に保証される状態）を定義した
  > 例：「申請作成機能」の前提条件＝「ログイン済みであること」、事後条件＝「申請レコードがdraft状態で保存されていること」
- [ ] トランザクション境界（どこからどこまでを1つのDB操作のまとまりにするか）を定義した
  > トランザクション＝「全部成功するか全部失敗するか」を保証するDB操作のまとまり。途中で失敗したときにデータが中途半端な状態にならないようにする仕組み
- [ ] 非同期処理・バックグラウンド処理がある場合、処理方式と状態管理方法を定義した
- [ ] 機能間の呼び出し関係（どの機能がどの機能を使うか）を整理した

### 3-2. 詳細画面設計

**このステップの目的**: 基本設計のワイヤーフレームをベースに、画面の全項目・動作・エラー表示をコーダーが実装できるレベルまで細かく定義します。

- [ ] 各画面の全入力項目について、以下を定義した
  - 項目名（論理名・物理名）
  - 入力形式（テキスト・数値・日付・選択肢・ファイル等）
  - 文字種制限（全角・半角・英数字等）
  - 文字数制限（最小・最大）
  - 必須/任意
  - 初期値・デフォルト値
  - 数字型の場合、単位を確定する
- [ ] 全バリデーションルール（入力チェック）を項目ごとに定義した
  > バリデーション＝入力値の正しさを確認するチェック。例：「金額は半角数字のみ、1〜9桁」
- [ ] バリデーションエラー時のメッセージ文言を全パターン定義した
  > 例：「金額を入力してください」「金額は数字で入力してください」「金額は9桁以内で入力してください」
- [ ] 各ボタンの活性/非活性の切り替え条件を定義した
  > 例：「送信ボタンは必須項目がすべて入力されるまで非活性にする」
- [ ] 画面ロード時の初期表示ロジック（データ取得・表示条件）を定義した
- [ ] ページネーション・ソート・検索機能がある場合、その動作仕様を定義した
  > ページネーション＝一覧を複数ページに分けて表示する機能
- [ ] モーダル・ダイアログがある場合、表示条件・操作結果を定義した
- [ ] レスポンシブ対応が必要な場合、ブレークポイントごとのレイアウト変化を定義した
  > ブレークポイント＝画面サイズに応じてレイアウトが切り替わる境界値（例：768px以下はスマホ表示）
- [ ] 操作の確認ダイアログ（「削除してよいですか？」等）が必要な箇所を定義した
- [ ] タイムアウト・セッション切れ時の画面動作を定義した
- [ ] アクセス権限がないユーザーが画面にアクセスした場合の動作を定義した

### 3-3. 詳細API設計

**このステップの目的**: 基本設計のAPI概要版を、実装できるレベルの完全な仕様に仕上げます。フロントエンドとバックエンドの約束事をここで確定させます。

- [ ] 全APIについて、リクエストの詳細（ヘッダー・パスパラメータ・クエリパラメータ・リクエストボディの全項目・型・必須/任意）を定義した
- [ ] 全APIについて、レスポンスの詳細（成功時・エラー時の全項目・型・説明）を定義した
- [ ] 全APIのHTTPステータスコードの使い分けを定義した
  > 例：200 OK（取得成功）、201 Created（作成成功）、400 Bad Request（入力エラー）、401 Unauthorized（未認証）、403 Forbidden（権限なし）、404 Not Found（存在しない）、500 Internal Server Error（サーバーエラー）
- [ ] エラーレスポンスのフォーマット（エラーコード・メッセージの構造）を統一・定義した
- [ ] 認証が必要なAPIと不要なAPIを明示した（認証ヘッダーの方式も定義）
- [ ] ページネーションが必要なAPIについて、パラメータ形式と上限件数を定義した
- [ ] ファイルアップロード・ダウンロードAPIがある場合、ファイル形式・サイズ上限・Content-Typeを定義した
- [ ] APIのリクエスト・レスポンス例（サンプルJSON）を全APIに用意した
- [ ] OpenAPI（YAML）仕様書を完全版に更新した
  > 基本設計フェーズの概要版から、全項目・全型・全サンプルを含む完全版に肉付けする
- [ ] APIのべき等性（同じリクエストを複数回送っても同じ結果になるか）を設計した
  > べき等性＝「ボタンを2回押してしまっても二重登録にならない」ための設計。POSTは原則べき等でないため、二重送信対策が必要
- [ ] レート制限（APIの呼び出し回数制限）が必要な場合、制限値を定義した

### 3-6. 詳細セキュリティ設計

**このステップの目的**: 基本設計で方針を決めたセキュリティ設計を、実装レベルの具体的な仕様に落とし込みます。後から追加するのが最も困難な領域のため、ここで確実に設計します。

- [ ] CSRF対策（CSRFトークンまたはSameSite Cookie）を設計した

- [ ] ファイルアップロードの検証方法（MIME型・拡張子・ウイルスチェック）を定義した
- [ ] ログに出力してはいけない情報（パスワード・クレジットカード番号等）を明示した

## 付録B. よくある抜け漏れパターン

| パターン | 影響 | 対策 |
|---------|------|------|
| 処理フローの異常系・例外系を定義していない | 実装者が独自判断でエラー処理を書く → 動作がバラバラになる | 正常系と同時に異常系・例外系も必ず設計する |
| 画面項目のバリデーションが「要確認」のまま | 実装者が独自ルールで実装 → テストで大量指摘 | バリデーションルールは詳細設計で全項目確定させる |
| APIの二重送信対策（べき等性）を後回し | 本番でデータ二重登録が発生 | APIごとに二重送信対策の要否を設計時に判断する |
| インデックス設計を実装後に検討する | 性能問題が発覚してから対処 → テーブル変更が困難 | 詳細設計フェーズでクエリとセットでインデックスを設計する |
| テスト設計を実装後に行う | テスト観点が抜ける・工数が膨らむ | 詳細設計フェーズでテスト観点を洗い出してから実装に入る |
| 開発環境の構築手順を口頭のみで共有する | メンバー追加時に再現できない | セットアップ手順書を必ずドキュメント化してGitに含める |
| セキュリティ設計を実装後にチェックする | 設計レベルの問題は修正コストが大きい | 各機能設計でセキュリティ観点を都度確認する |
| 外部連携テスト環境の準備を後回しにする | テスト工程直前に調整が必要と判明 | 詳細設計フェーズ中にモック/スタブ/サンドボックスを確保する |
| コーディング規約を後から決める | コードレビューで大量の書き直し指摘 | 実装開始前に規約をチームで合意・Linterで自動化する |
| N+1問題を実装後に発見する | 性能改善のためのリファクタリングが必要 | 詳細設計でクエリを設計するタイミングで一緒に検討する |
| 共通ライブラリ（エラー処理・ログ・認証）を各機能担当が個別に実装する | 実装が微妙に異なるコードが乱立し、保守性が著しく低下する | 機能実装の前に共通ライブラリ（resource⑤参照）を先行して設計・実装する |
| CDパイプラインを設定せず手動デプロイで運用する | リリース手順の属人化・ミス・工数増大 | 詳細設計フェーズでCDパイプラインの設定方針を決め、ステージング環境へのデプロイを自動化する |

## 付録C. 用語集

| 用語 | 説明 |
|-----|------|
| N+1問題 | 1件の親データに対してN件の子データを個別クエリで取得する非効率なパターン。ループ内でSELECTが走る状態 |
| ORM | Object-Relational Mapper。JavaのJPAやPythonのSQLAlchemyなど、オブジェクトとDBを自動マッピングするライブラリ |
| REST API | HTTPを使ってリソース（データ）を操作するAPI設計スタイル。GET/POST/PUT/DELETEの使い分けが特徴 |
| XSS | クロスサイトスクリプティング。悪意のあるスクリプトをページに埋め込まれる脆弱性 |
| CSRF | クロスサイトリクエストフォージェリ。ユーザーを騙して意図しない操作をさせる攻撃 |
| SQLインジェクション | 悪意のある入力値でSQLを書き換え、DBを不正操作される脆弱性 |
| CI | Continuous Integration（継続的インテグレーション）。コードをプッシュするたびに自動テスト・ビルドを実行する仕組み |
| CD | Continuous Delivery/Deployment（継続的デリバリー/デプロイ）。テスト通過後にコードを自動でステージング・本番環境に配置する仕組み |
| DDL | Data Definition Language。テーブルを作成・変更・削除するSQL文（CREATE TABLE / ALTER TABLE等）の総称 |
| DML | Data Manipulation Language。データを操作するSQL文（SELECT / INSERT / UPDATE / DELETE）の総称 |
| JWT | JSON Web Token。ログイン情報をJSON形式で暗号化したトークン。APIの認証に広く使われる |
| Linter | コードの書き方の問題（バグになりそうな箇所・規約違反）を自動検出するツール |
| OpenAPI | REST APIの仕様を記述するための標準フォーマット。YAML形式が主流。Swagger UIで表示できる |
| インデックス | データベースの検索を高速化する仕組み。張りすぎると更新が遅くなる |
| カバレッジ | テストがコードのどれくらいの範囲を実行しているかの割合 |
| シーケンス図 | システム間・コンポーネント間のメッセージのやり取りを時系列で表した図 |
| スタブ | テスト時に本物の外部システムの代わりに用意する「偽物のシステム」 |
| トランザクション | 「全部成功するか全部失敗するか」を保証するDB操作のまとまり |
| ハッシュ化 | 元に戻せない形式にデータを変換すること。パスワードの安全な保存に使う |
| パーティション | 大量データのテーブルを範囲で分割してI/O性能を向上させるDB機能 |
| フォーマッター | コードの字下げ・改行・スペースを自動で整形するツール |
| べき等性 | 同じ操作を何度繰り返しても結果が変わらない性質。二重送信対策の設計原則 |
| マイグレーション | DBのスキーマ（構造）変更をバージョン管理しながら適用する仕組み・スクリプト |
| モック | テスト時に本物の依存コンポーネントの代わりを演じるオブジェクト（ソフトウェア内部） |
| 結合テスト | 複数の機能・コンポーネントを組み合わせた状態での動作確認テスト |
| 境界値 | バリデーションの最小・最大・ちょうど境目の値。テストで特に重要なデータ |
| 照合順序 | DBにおける文字の比較・ソートのルール（コレーションとも呼ぶ） |
| 単体テスト | 機能の最小単位（関数・メソッド）を個別に動作確認するテスト |
| 物理設計 | 実際のDBに作るテーブル名・カラム名・型・インデックスまで含めた設計レベル |
| 擬似コード | 特定のプログラミング言語に依存しない、処理の流れを日本語混じりで書いた記述 |

## 付録D. 経験の浅いSEからの改善提案（読みやすさ向上のために）

> この付録は、詳細設計の経験が浅いSEとして本手順書を実際に使ってみた際に「ここをもっとこうしてほしかった」と感じた点をまとめたものです。手順書の改訂や教育資料の整備に活用してください。

### D-1. 全体構成について

**① 「詳細設計書」の完成イメージ（サンプル）へのリンクが欲しい**

本手順書はチェックリスト形式で「何を決めるか」は丁寧に書かれていますが、「完成した詳細設計書がどのような見た目か」のサンプルがないと、初心者はゴールのイメージが持ちにくいです。`output/` フォルダ内の他ファイルや外部サンプルへのリンクを冒頭に案内してもらえると助かります。

**② 「基本設計との違い」を表で1ページ目に示してほしい**

本手順書の説明文には「詳細設計とは何か？」のセクションがありますが、基本設計とのすみ分けが文章だけではイメージしにくいです。以下のような対比表を1章の冒頭に置くと全体感を掴みやすくなります：

| 設計項目 | 基本設計での記載レベル | 詳細設計での記載レベル |
|---------|------------------|------------------|
| API設計 | エンドポイント・目的の概要 | 全パラメータ・型・エラーコード・OpenAPI完全版 |
| 画面設計 | ワイヤーフレーム・遷移図 | 全項目のバリデーション・エラーメッセージ・ボタン制御 |
| データ設計 | ER図（論理）・テーブル概要 | 物理ER図・全カラムの型/桁数・インデックス・SQL |

---

**③ ステップ間の「成果物の受け渡し」が見えにくい**

各ステップが何のアウトプットを次のステップに渡すのかが明示されていません。例えば「ステップ1（詳細機能設計）の成果物がステップ3（詳細API設計）のインプットになる」という関係を、作業フロー表に「主な成果物（アウトプット）」列として追加すると流れが追いやすくなります。

### D-2. 各ステップの説明について

**④ 3-1（詳細機能設計）の「擬似コード」の記載例を本文に1つ載せてほしい**

擬似コードの説明は用語集にありますが、実際にどのように書くのかのサンプルが本文にありません。以下のような短い例を1つでも示してもらえると、初心者がどの粒度で書けばよいかが掴みやすくなります：

```
【擬似コード例：申請提出機能】

6. 画面に「申請しました」を表示する
```

**⑤ 3-3（詳細API設計）の「HTTPステータスコードの使い分け」が多く覚えにくい**

200/201/400/401/403/404/500の説明が一行ずつ並んでいますが、初心者が混乱しやすい「400と422の違い」「401と403の違い」について補足があると助かります。特に「401 = 未ログイン」「403 = ログイン済みだが権限なし」の違いは実装時に迷いやすいポイントです。

---

**⑥ 3-4（詳細データ設計）の「N+1問題」の説明をもう少し具体的にしてほしい**

用語集に定義はありますが、初心者には「どんなコードを書くとN+1問題になるのか」がイメージできません。以下のような1〜2行の具体例を添えるだけで理解が深まります：

> 例：申請一覧を表示する際に、各申請に対して個別に「申請者のユーザー名を取得するSELECT文」を発行してしまうと、申請が100件あれば101回のSQL（1 + 100）が実行される。これがN+1問題。対策は申請取得時にJOINして一度に取得すること。

---

**⑦ 3-6（詳細セキュリティ設計）の「ストレッチング」の意味が分かりにくい**

「ハッシュ処理を繰り返して総当たり攻撃を遅くする技法」とありますが、初心者には「なぜ繰り返すと遅くなるのか」が分かりません。「1回のハッシュ計算は一瞬だが、10万回繰り返すことで攻撃者が総当たり試行するのに時間がかかるようにする」という仕組みを一文で添えると理解しやすくなります。

---

### D-3. 成果物・チェックリストについて

**⑧ 5-2（詳細設計の成果物一覧）に「作成工数目安」の列が欲しい**

初めて詳細設計を担当する場合、「D-01 詳細機能設計書」が1機能あたり何時間かかるのか、全くイメージが湧きません。プロジェクト規模によって変わることは承知のうえで、「画面1枚あたり○〜○時間」のような粗い目安が一覧にあると、スケジュール立案の参考になります。

---

**⑨ 付録A（移行判定チェックリスト）の各項目に「未達の場合の影響」を添えてほしい**

現在はYes/Noのチェックリストのみですが、「なぜこれが必要か」が書かれていないため、初心者は未達のまま進んでしまうリスクがあります。付録Bの「よくある抜け漏れパターン」と組み合わせて、各チェック項目に対応するリスクパターンへの参照番号を添えると、重要性が伝わりやすくなります。

---

### D-4. その他

**⑩ 「詳細設計レビューで先輩SEがよく指摘するポイント」を別表で欲しい**

付録Bの「よくある抜け漏れパターン」は設計者視点ですが、レビュアー（先輩SE・PL）が指摘しがちな観点を別途まとめた表があると、自己レビューの精度が上がります。例：「画面項目の型・桁数が未定義」「エラーメッセージが全画面で統一されていない」「トランザクション境界の記載がない」など。
