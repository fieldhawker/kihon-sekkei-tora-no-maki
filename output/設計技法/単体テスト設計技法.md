# 単体テスト設計技法まとめ

単体テスト（Unit Test）は、プログラムを構成する最小単位（関数やメソッドなど）が、設計通りに正しく動作するかを検証するテストです。
テストケースを場当たり的に作成すると、考慮漏れによるバグの見逃しや、無駄なテストケースの増大を招きます。

以下に、単体テストを効率的かつ網羅的に行うための代表的なテスト設計技法（ブラックボックス・ホワイトボックス）と、それぞれの具体的な使用事例をまとめます。

---

## 1. 入力データに着目した技法（ブラックボックステスト）
プログラムの内部構造を意識せず、仕様書（インプットとアウトプット）に基づいてテストケースを設計します。

### 1.1 同値分割法 (Equivalence Partitioning)
入力値を「同じように処理されるグループ（同値クラス）」に分け、各グループから代表値を1つ選んでテストする手法です。

*   **使用事例**: 「パスワードの文字数チェック（8文字以上、16文字以下）」のテスト
    *   **無効同値（短すぎる）**: 7文字以下のグループから、代表値として `5文字` をテスト。
    *   **有効同値（正常）**: 8〜16文字のグループから、代表値として `10文字` をテスト。
    *   **無効同値（長すぎる）**: 17文字以上のグループから、代表値として `20文字` をテスト。
    *   **効果**: 無数にある入力パターンのうち、重複するテストを省き、最小限のケースで効率よく全体をカバーできます。

### 1.2 境界値分析 (Boundary Value Analysis)
バグが発生しやすい「同値クラスの境界（しきい値とその前後の値）」を狙ってテストする手法です。同値分割法とセットで使われます。

*   **使用事例**: 「年齢による入場制限（18歳以上のみ入場可）」のテスト
    *   **境界となる値**: 18
    *   **テストケース**:
        *   `17歳` (境界のすぐ下：入場不可となるか)
        *   `18歳` (境界値：入場可となるか)
        *   `19歳` (境界のすぐ上：入場可となるか)
    *   **効果**: プログラムの `>=` と `>` の書き間違い（オフバイワンエラー）などの、ありがちな境界のバグを効果的に検出できます。

## 2. 条件と結果の組み合わせに着目した技法

### 2.1 デシジョンテーブルテスト (Decision Table Testing)
複数の入力条件の組み合わせによって出力（結果）が変化するロジックに対して、考えられる全ての組み合わせを表にしてテストする手法です。

*   **使用事例**: 「ECサイトの割引判定」のテスト
    *   **条件**: 「プレミアム会員か」「購入額5000円以上か」
    *   **テストケース（組み合わせ表）**:
        *   会員=Yes, 5000円=Yes ➔ 結果: 割引適用
        *   会員=Yes, 5000円=No  ➔ 結果: 割引適用外
        *   会員=No,  5000円=Yes ➔ 結果: 割引適用外
        *   会員=No,  5000円=No  ➔ 結果: 割引適用外
    *   **効果**: 複雑な `if-else` 条件の組み合わせにおける、仕様の矛盾やテストの抜け漏れを確実に防ぐことができます。

### 2.2 ペアワイズテスト / 直交表 (Pairwise Testing)
多数のパラメータがある場合、全ての組み合わせをテストすると膨大な数になるため、「任意の2つのパラメータの組み合わせが必ず1回以上登場する」ようにケースを絞り込む手法です。

*   **使用事例**: 「OS、ブラウザ、言語の設定による画面表示」のテスト
    *   パラメータ: OS(Win/Mac)、ブラウザ(Chrome/Edge/Safari)、言語(日/英)
    *   全組み合わせは 2 × 3 × 2 = 12通りですが、ペアワイズ用のツールを利用することで、より少ない回数（例：6回程度）で「OS×ブラウザ」「ブラウザ×言語」等の全ての「2ペアの組み合わせ」を網羅できます。
    *   **効果**: テストの実行回数を大幅に削減しつつ、ソフトウェアのバグの大半は「1〜2つのパラメータの組み合わせ」で発生するという経験則に基づき、高い欠陥検出率を維持できます。

## 3. プログラムの内部構造に着目した技法（ホワイトボックステスト）
プログラムのソースコード（内部の処理ルート）を読み解き、すべてのルートが正しく実行されるかを確認します。

### 3.1 制御フローテスト / パステスト (Control Flow Testing)
コード内の命令や分岐をどれだけ通過したか（カバレッジ：網羅率）を基準にテストを設計します。

*   **代表的な基準**:
    *   **C0 (命令網羅 / ステートメントカバレッジ)**: すべての「行（命令）」を少なくとも1回は実行する。
    *   **C1 (分岐網羅 / ブランチカバレッジ)**: すべての `if`文などの「分岐（True/False）」を両方とも1回は実行する。（単体テストで一般的に目標とされる水準）
    *   **C2 (条件網羅 / コンディションカバレッジ)**: 分岐の中の「複数条件（A && B）」のそれぞれのTrue/Falseの組み合わせを網羅する。
*   **使用事例**: `if (A > 0 || B > 0)` というコードのC1（分岐網羅）テスト
    *   Trueルートを通すテスト: `A=1, B=0`
    *   Falseルートを通すテスト: `A=0, B=0`
    *   **効果**: コードの中に「絶対に実行されない無駄なコード（デッドコード）」がないか、想定したルートを正しく通るかを保証できます。

## 4. 状態の変化や経験則に着目した技法

### 4.1 状態遷移テスト (State Transition Testing)
システムが特定の「状態（ステータス）」を持っており、イベントによって状態が変わる場合に、その遷移が正しく行われるかをテストします。

*   **使用事例**: 「動画プレイヤー（停止・再生・一時停止）」のテスト
    *   **テストケース**:
        *   「停止中」に「再生ボタン」を押す ➔ 「再生中」になるか
        *   「一時停止中」に「再生ボタン」を押す ➔ 「再生中」になるか
        *   「停止中」に「一時停止ボタン」を押す ➔ 無視されるか（またはエラー処理されるか）
    *   **効果**: 不正なタイミングでの操作や、想定外の状態のまま処理が進んでしまうバグを検出できます。

### 4.2 エラー推測 (Error Guessing)
テストエンジニアの経験や直感に基づいて、「開発者がミスしやすそうな箇所」や「過去にバグが出たパターン」を狙い撃ちにする手法です。

*   **使用事例**: 入力フォームのテスト
    *   **狙い撃ちのテストデータ**:
        *   空文字列（空のまま送信）
        *   半角スペースだけの入力
        *   全角・半角の混在
        *   非常に長い文字列（数万文字）
        *   特殊文字（SQLインジェクションを狙った `'` や `;` など）
    *   **効果**: 定型的なテスト技法（同値分割や境界値分析）では見逃されがちな、システムの「弱点」を突くことができます。
