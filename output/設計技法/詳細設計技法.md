# 詳細設計技法まとめ

詳細設計は、基本設計で定義した「システムが何をするか（What）」を元に、「それをプログラムでどう実現するか（How）」を具体化し、プログラマーが迷いなくバグの少ないコードを実装できる状態にするためのフェーズです。

以下に、詳細設計で特に有用な設計技法と、それぞれの具体的な使用事例をまとめます。

---

## 1. ロジックと処理の流れを整理する技法
複雑な計算やデータの加工処理を正確に伝えるための手法です。

### 1.1 IPO記述（Input - Process - Output）
モジュールや関数に対して、「何を入力し」「どのような処理を行い」「何を出力するか」を明確に定義する手法です。

*   **使用事例**: 「割引料金計算関数」の設計
    *   **Input**: ユーザーの年齢（数値）、購入金額（数値）、クーポンコード（文字列）
    *   **Process**:
        1. クーポンコードが有効かDBで検証する。
        2. 年齢が65歳以上ならシニア割引（10%オフ）を適用する。
        3. 購入金額が5000円以上なら送料無料フラグを立てる。
    *   **Output**: 最終請求金額（数値）、送料無料フラグ（真偽値）

### 1.2 アクティビティ図（UML）
処理の分岐やループに加え、並行処理（マルチスレッド）などの複雑な処理の流れを視覚的に表現するフローチャートの発展形です。

*   **使用事例**: 「ECサイトの注文確定バッチ処理」の設計
    *   **流れ**:
        1. 注文データを受信（開始）
        2. 【並行処理開始】
            *   在庫管理APIを叩いて在庫を引き当てる
            *   決済代行APIを叩いてクレジットカードの与信枠を確保する
        3. 【並行処理待ち合わせ】両方の処理が成功したか判定する
        4. 成功なら注文確定ステータスへ更新し、完了メールを送信（終了）

## 2. データと状態を整理する技法
データがどう保存され、どう変化していくかを定義します。

### 2.1 物理ER図 / テーブル定義書
システムで扱うデータを、実際のデータベースのテーブル構造、カラムのデータ型、制約（NOT NULL、外部キーなど）まで具体化した設計です。

*   **使用事例**: 「ユーザー（users）テーブル」と「注文（orders）テーブル」の設計
    *   **設計内容**: `orders`テーブルの`user_id`カラムは、`users`テーブルの`id`カラムに対する外部キー（Foreign Key）制約を設定する。
    *   **効果**: これにより「存在しないユーザーIDで注文が作られる」といったデータの不整合をデータベースレベルで強固に防ぐことができます。

### 2.2 状態遷移表（モジュールレベル）
特定のオブジェクトやデータが持つ状態（ステータス）が、どのようなイベント（操作）によってどう変化するかを網羅的に定義します。

*   **使用事例**: 「釣行記録アプリの投稿ステータス」の設計
    *   **状態**: `[下書き]`, `[公開中]`, `[削除済]`
    *   **遷移ルールの例**:
        *   `[下書き]` の時に「公開する」イベント発生 ➔ `[公開中]` へ遷移
        *   `[削除済]` の時に「公開する」イベント発生 ➔ **エラー（無効な遷移）**
    *   **効果**: プログラム内で「既に削除された投稿を無理やり公開しようとする」などの不正操作に対する例外処理を漏れなく実装できます。

## 3. プログラムの構造と連携を整理する技法
どのコードが、どういう役割を持って、誰と通信するのかを定義します。

### 3.1 シーケンス図（UML）
オブジェクトやコンポーネント間で「いつ」「誰が」「どのAPIやメソッドを呼び出し」「どう返ってくるか」を時系列で図解する手法です。

*   **使用事例**: 「スマホアプリからのログイン処理」の設計
    *   **時系列の動き**:
        1. アプリ(UI)が、バックエンド(API)の `/login` へPOSTリクエストを投げる。
        2. APIサーバーが、DBへパスワードハッシュの照会（SELECT）を行う。
        3. DBが、APIサーバーに照会結果を返す。
        4. APIサーバーが、セッショントークンを発行し、アプリへ `200 OK`（トークン付き）を返す。
        5. アプリがトークンを保存し、ホーム画面へ遷移する。
    *   **効果**: 非同期処理やAPI連携が連続する処理において、タイミングによるバグ（処理完了前に次へ進んでしまう等）を防ぐための必須ツールです。

### 3.2 クラス図（UML）
システムを構成するクラス（データと振る舞いのまとまり）の構造と、クラス間の関係性（継承、依存など）を定義します。

*   **使用事例**: 「釣り具の管理機能」の設計
    *   **構造**:
        *   親クラス: `Tackle`（釣り具：共通プロパティとして「名前」「メーカー」を持つ）
        *   子クラス1: `Rod`（ロッド：Tackleを継承し、専用プロパティ「長さ」を持つ）
        *   子クラス2: `Reel`（リール：Tackleを継承し、専用プロパティ「ギア比」を持つ）
    *   **効果**: 共通部分を親クラスにまとめることでコードの重複を防ぎ、将来「すべての釣り具に『重量』プロパティを追加したい」といった仕様変更が起きた際も1箇所の修正で済むようになります。

## 4. 外部とのインターフェースを取り決める技法
フロントエンドとバックエンドの連携をスムーズにするための手法です。

### 4.1 API仕様書（OpenAPI / Swagger 等）
システム間で通信するためのルール（エンドポイントURL、リクエストパラメータ、レスポンスのデータ構造）を厳密に定義します。

*   **使用事例**: 「釣り場のお天気情報取得API」の設計
    *   **定義内容**:
        *   エンドポイント: `GET /api/v1/weather/{location_id}`
        *   リクエスト: パスパラメータ `location_id` (整数) を必須とする。
        *   レスポンス: 成功時は `{ "temp": 25.5, "weather": "sunny" }` のJSONを返す。
    *   **効果**: OpenAPI等で記述することで、仕様書から「テスト用のモックサーバー」を自動生成できます。これにより、バックエンドの実装完了を待たずにフロントエンド開発を並行して進めることが可能になります。

---

## 5. 複雑な条件分岐の整理（テスト兼用）

### 5.1 デシジョンテーブル（決定表）
複数の条件が複雑に絡み合うビジネスロジックにおいて、条件の全組み合わせに対する動作をマトリクス状に整理します。

*   **使用事例**: 「送料無料の判定ロジック」の設計
    *   **条件**:
        1. プレミアム会員か？ (Y/N)
        2. 購入金額が5000円以上か？ (Y/N)
        3. キャンペーン期間中か？ (Y/N)
    *   **結果**: 送料無料にするか？ (実行/非実行)
    *   **効果**: 全ての組み合わせを可視化することで、「一般会員で5000円未満だがキャンペーン中の場合はどうするか？」といった仕様の抜け漏れや矛盾をコードを書く前に発見できます。作成した表はそのままテストケースとして網羅的なテストに活用できます。
