# STEP 2: ブランチ・PR・コードレビュールール

**このステップの目的**: チーム全員が同じルールで開発できるよう、コードの管理方法を統一します。実装開始前に全員で合意してください。

---

## ブランチ命名規則

### 推奨パターン（GitHub Flow ベース）

```
main                        ← 常にデプロイ可能な状態を維持
  └── feature/F-001-user-register   ← 機能実装ブランチ
  └── feature/F-002-login
  └── fix/login-validation-error    ← バグ修正ブランチ
  └── chore/update-dependencies     ← 依存関係更新・設定変更
```

### 命名ルール

| プレフィックス | 使用場面 | 例 |
|------------|--------|---|
| `feature/` | 新機能の実装 | `feature/F-001-user-register` |
| `fix/` | バグ修正 | `fix/login-null-pointer` |
| `chore/` | ライブラリ更新・CI設定等 | `chore/update-spring-boot-3.2` |
| `docs/` | ドキュメントのみの変更 | `docs/update-api-spec` |

**ルール**:
- 機能Issueのブランチ名には機能ID（F-001等）を含める（設計書との対応が一目でわかる）
- 英数字・ハイフンのみ。スペース・スラッシュの多重使用は禁止
- ブランチ名は短く（40文字以内）、何をするブランチか分かるように

## コミットメッセージ規則

### 推奨フォーマット（Conventional Commits）

```
<type>(<scope>): <概要（英語または日本語）>

# 例
feat(user): ユーザー登録APIエンドポイントを追加
fix(auth): JWTトークンの有効期限検証エラーを修正
test(user): ユーザー登録APIの単体テストを追加
docs(api): OpenAPIスペックにユーザー登録を追記
chore: Spring Bootを3.2.0にアップグレード
```

| type | 使いどころ |
|------|---------|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `test` | テスト追加・修正 |
| `docs` | ドキュメントのみ |
| `refactor` | 機能変更なしのリファクタリング |
| `chore` | ビルド設定・依存関係 |

## PRの作り方とテンプレート

### PRを出す前のセルフチェック

- [ ] ブランチ名が命名規則に従っている
- [ ] CIが通っている（ビルド・テストが全てグリーン）
- [ ] Linterエラーがゼロ
- [ ] 自分でコードを読み返した（typo・コメントアウトの残り等）
- [ ] 本番環境に不要なデバッグコード（`console.log`・`System.out.println` 等）が残っていない
- [ ] テストを追加した（既存テストだけでなく、今回の変更に対応したテストがある）

### PRの説明文テンプレート

リポジトリの `.github/PULL_REQUEST_TEMPLATE.md` に以下を配置しておきます。

```markdown
## 変更内容
<!-- 何をどう変えたか、1〜3行で -->

## 対応Issue
Closes #（Issue番号）

## テスト方法
<!-- レビュアーが動作確認する手順。「○○の画面で△△を入力して□□を確認」まで書く -->
1. 
2. 

## スクリーンショット（画面変更がある場合）
<!-- before/after の画像を貼る -->

## レビュアーへの補足
<!-- 設計判断の理由・迷った点・特に見てほしい箇所 -->
```

## コードレビュー観点チェックリスト

レビュアーはこの観点でレビューしてください。「動いていればOK」ではありません。

### 設計との整合性

- [ ] 設計書（D-01〜D-18）の仕様どおりに実装されているか
- [ ] APIのリクエスト/レスポンス構造が OpenAPI 仕様（D-03）と一致しているか
- [ ] エラーレスポンスの形式が共通ライブラリ（C-02）の規約に従っているか
- [ ] バリデーションが設計書（D-02の項目定義表）の内容をカバーしているか

### コードの品質

- [ ] コーディング規約書（D-15）に違反していないか
- [ ] 変数名・関数名が意図を表しているか（`tmp`・`data`・`flag` は避ける）
- [ ] 1つの関数/メソッドが1つの責務に収まっているか（長すぎる関数は分割する）
- [ ] 重複したコードが共通化できていない状態で残っていないか
- [ ] ハードコードされた値（マジックナンバー・URLなど）が定数や設定値になっているか

### セキュリティ

- [ ] SQL は ORM / プリペアドステートメントを使っており、文字列連結による SQL 組み立てがないか
- [ ] ユーザー入力値を出力する際に HTML エスケープ処理があるか（XSS対策）
- [ ] 認証が必要なエンドポイントに認証チェックが適用されているか
- [ ] ログに個人情報・パスワード・トークンを出力していないか
- [ ] エラーメッセージにスタックトレース等の内部情報をクライアントに返していないか

### テスト

- [ ] 単体テストが追加されているか
- [ ] 正常系・異常系・境界値がテストされているか
- [ ] テスト名が何をテストしているか分かるか

### その他

- [ ] TODO コメントが残っていないか（残す場合は Issue 番号を付ける）
- [ ] ログ出力が適切なレベル（DEBUG / INFO / WARN / ERROR）で追加されているか

## レビュー指摘の対応方法

| 指摘の種別 | 対応方法 |
|---------|--------|
| **Must**（必ず修正） | コードを修正してコミットし、レビュアーに再確認を依頼する |
| **Should**（できれば修正） | 修正するか、しない理由をコメントに記載してレビュアーに確認 |
| **Nit**（細かい指摘・任意） | 対応するかどうかは作成者の判断でよい |

> **指摘への返信**: すべての指摘に対して「修正しました」または「〇〇のため修正しません」と返信してから「Resolve」にする。無言のResolveは禁止。
