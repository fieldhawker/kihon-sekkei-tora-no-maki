# 付録: よくある実装トラブルと対処

---

## よくある実装トラブルパターン

| トラブル | 原因 | 対処 |
|--------|-----|-----|
| 「設計書に書いていないケース」が実装中に頻出する | 設計の詳細度が不足していた | SEに即座に確認し、設計書を更新してから実装する |
| ローカルでは動くが他の端末で動かない | 環境依存のコード（絶対パス等）や設定漏れ | Docker Composeで環境を統一する。設定値はすべて環境変数経由にする |
| マージしたらCIが落ちた | ブランチ間の競合を解決せずにマージした | PRを出す前に `main` の最新をローカルにpullしてマージしてから確認する |
| レビューで大量指摘が来て手戻りが多い | PRが大きすぎる（1PRに複数Issueの変更が混在） | 1PR = 1Issueを原則にする。変更行数が500行を超えるPRは分割を検討する |
| バグ修正で別のバグを出した | テストがない状態で修正した | 修正前に再現テストを書いてから修正する |
| 本番とステージングで挙動が違う | 環境変数や設定が異なる | 環境間の設定差分を `.env.example` に全て明示する |
| マイグレーション実行が失敗した | スクリプトに構文エラーがある、または実行順序の問題 | Flywayのバージョン番号を確認し、ローカルでロールバック→再適用して動作確認する |
| APIのレスポンス形式がフロントエンドの期待と違う | 基本設計時のAPI設計書とOpenAPI（D-03）に齟齬がある | OpenAPI仕様を正とし、バックエンド・フロントエンド双方が仕様書を確認する |

## コードレビューでよく指摘される問題

### セキュリティ系

```java
// NG: SQL文字列を直接連結している（SQLインジェクション脆弱性）
String query = "SELECT * FROM users WHERE id = " + userId;

// OK: プリペアドステートメントを使う
PreparedStatement stmt = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
stmt.setInt(1, userId);
```

```javascript
// NG: 入力値をそのままHTMLに埋め込んでいる（XSS脆弱性）
element.innerHTML = userInput;

// OK: テキストとして扱う
element.textContent = userInput;
```

### 設計品質系

```java
// NG: 一つのメソッドが複数の責務を持つ
public void registerUser(UserDto dto) {
    // バリデーション
    if (dto.getName() == null) throw new Exception("...");
    // DB保存
    userRepository.save(dto);
    // メール送信
    mailService.sendWelcomeMail(dto.getEmail());
    // ログ出力
    logger.info("User registered: " + dto.getEmail());
}

// OK: 責務を分離する
public void registerUser(UserDto dto) {
    validateUser(dto);
    User user = userRepository.save(dto.toEntity());
    mailService.sendWelcomeMail(user.getEmail());
    logger.info("User registered: {}", user.getId());
}
```

## 用語集

| 用語 | 説明 |
|-----|------|
| Issue | タスクや課題を管理する単位。GitHub Issues・Jira・Backlogで管理 |
| PR（Pull Request） | ブランチの変更を main に取り込む前にレビューを依頼する仕組み |
| CI（継続的インテグレーション） | コードをマージするたびに自動でビルド・テストが走る仕組み |
| Linter | コードの書き方が規約に沿っているかを自動チェックするツール |
| マージコンフリクト | 同じファイルの同じ箇所を別ブランチで変更したときに生じる競合 |
| バックログ | 着手待ちのIssue一覧 |
| スプリント | 1〜2週間の開発サイクル。スプリント内でどのIssueを消化するかを計画する |
| テクニカルデット | 「後で直す」として先送りした技術的な問題の累積 |
| ペアプロ | 2人1組でコードを書く手法。片方が書き、もう片方がリアルタイムでレビューする |
| シードデータ | テスト・開発用に初期投入するサンプルデータ |
