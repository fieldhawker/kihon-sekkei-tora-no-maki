# STEP 3: 実装進捗管理

**このステップの目的**: 実装フェーズを通じて、遅延・品質問題・設計変更を早期に発見して対処します。

---

## 進捗管理の基本サイクル

実装フェーズは以下のサイクルで管理します。

```
毎日: 進捗確認（デイリースタンドアップ）
毎週: スプリントレビュー・計画（スプリントが1週間の場合）
随時: 設計変更の記録・リスクのエスカレーション
```

---

## デイリースタンドアップ（日次確認）

毎朝10〜15分で以下を確認します。チームが小規模（2〜3名）なら口頭で、リモートならSlackの投稿でも可。

### 各メンバーが報告する3つのこと

1. **昨日やったこと**: 完了したIssue番号・作業内容
2. **今日やること**: 着手するIssue番号・目標
3. **困っていること（ブロッカー）**: 進められない原因があれば共有

### ブロッカーへの対応

| ブロッカーの種類 | 対応方針 |
|-------------|---------|
| 技術的に分からない | その場でペアプロ / 30分調べて解決しなければ上位SEに相談 |
| 設計書の内容が不明確 | 設計者（SE）に即日確認。「たぶんこういう意味だろう」で実装しない |
| 他のタスクの完了待ち | 依存しない別Issueに先行着手 |
| 外部要因（ライブラリのバグ等） | PLに報告・対応策（バージョン変更等）を相談 |

## 進捗状況の可視化

### Issueボードの状態を常に最新に保つ

| ステータス | 定義 |
|---------|-----|
| Backlog | まだ着手していない |
| In Progress | 実装中（担当者が着手したらすぐ移動） |
| In Review | PRを出してレビュー待ち |
| Done | レビュー承認・マージ済み |

**ルール**: 実際の作業状態とボードの状態を一致させる。「実は昨日からやっていた」は禁止。

### 進捗率の計算

```
実装完了率 = Done のIssue数 ÷ 全Issue数 × 100
```

予定スケジュールと実際の完了率を毎週比較します。10%以上の乖離が2週間続く場合はPLにエスカレーションしてください。

## 遅延を検知したときの対処

### 遅延の早期シグナル

以下のサインが出たら、PLに報告して対策を協議してください。

- 同じIssueが3日以上 "In Progress" のまま動いていない
- バグ修正Issueが新規発生より消化が遅い状態が続いている
- 「あとで直す」「後でテスト書く」が積み上がっている
- 設計変更が頻繁に発生していて手戻りが多い

### 遅延時の選択肢

| 選択肢 | 内容 | 注意点 |
|------|------|------|
| スコープ削減 | 「任意」優先度のIssueをリリースから外す | スコープ変更は必ず顧客・PLの承認を得る |
| 増員 | 実装担当者を追加する | 新規参加者のキャッチアップコストで逆に遅れることがある |
| 残業 | 短期的に作業時間を増やす | 品質低下・バーンアウトのリスクがあり、原則は最終手段 |
| リリース延期 | スケジュールを伸ばす | 顧客・PL承認必須。早めに判断するほど影響が小さい |

> **最悪の対応**: 遅延を隠す・「なんとかなる」と楽観視する。問題が小さいうちに報告するのがSEとしての重要な責務です。

## 設計変更の管理

実装中に設計書と異なる実装が必要になることがあります。必ず以下のプロセスを踏んでください。

### 設計変更が発生したときの手順

1. **変更の必要性を確認**: 「設計書が間違っている」のか「自分の理解が間違っている」のかを先に確認する
2. **影響範囲を把握**: 変更によって他のAPI・画面・テストに影響が出ないかを確認する
3. **SE・PLに報告**: 自分の判断で設計を変えない。必ず報告して承認を得る
4. **設計書を更新**: 承認を得たら、設計書の該当箇所を即日更新する
5. **関係者に通知**: 変更箇所に依存している実装者に変更内容を共有する

### 変更管理台帳への記録

設計変更が発生するたびに以下を記録します。

| 変更日 | 変更箇所（設計書名・セクション） | 変更内容 | 理由 | 承認者 | 影響するIssue |
|------|--------------------------|--------|-----|------|------------|
| 2024-XX-XX | D-03 POST /users | レスポンスに`userId`を追加 | フロントで必要と判明 | SE○○ | #45, #67 |

## 実装完了の判定基準

実装フェーズの完了は以下の条件が**すべて**満たされた状態です。

- [ ] すべてのIssueが「Done」になっている
- [ ] 未対応バグがゼロ（または、リリース後対応として記録・承認済み）
- [ ] CI（自動ビルド・テスト）が mainブランチで通っている
- [ ] コードカバレッジが合意した基準（例: 70%以上）を満たしている
- [ ] 設計変更管理台帳が最新の状態になっている
- [ ] 本番デプロイ手順書（D-16に含む）を実際に試してデプロイできることを確認した

> **完了したら**: [テスト実施手順/](../テスト実施手順/) に進んでください。
