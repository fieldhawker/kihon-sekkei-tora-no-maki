# STEP 0: 移行確認（詳細設計 → 実装）

**このステップの目的**: 詳細設計書が「実装を開始できる状態」かを確認します。ここで問題が見つかれば、実装着手前に解消してください。

---

## 移行判定チェックリスト

### 設計書の状態確認

- [ ] 詳細設計書の全セクションが完成し、PLおよびレビュアーの承認を得ている
- [ ] 未決事項（TBD）がゼロ、またはリスク受容として記録・承認されている
- [ ] 設計変更があった場合、最新版の設計書が関係者に共有されている
- [ ] トレーサビリティマトリクス（D-18）が最新状態に更新されている

### 開発環境の状態確認

- [ ] チーム全員の開発端末で、開発環境構築手順書（D-16）の手順が完了している
- [ ] ローカル環境でアプリケーションが起動し、サンプルAPIが動作することを確認した
- [ ] DBのマイグレーションスクリプト（Flyway等）を実行し、ローカルDBが最新スキーマになっている
- [ ] `.env.example` または環境変数設定手順が整備されており、新規参加者がすぐセットアップできる状態
- [ ] テスト用のサンプルデータ（シードデータ）が用意されている

### チーム・ツールの準備

- [ ] GitHub / GitLab などのリポジトリが作成済みで、チーム全員がアクセスできる
- [ ] Issue管理ツール（GitHub Issues / Jira / Backlog等）が決まっており、プロジェクト・ボードが設定済み
- [ ] 開発チャンネル（Slack / Teams 等）が設定されており、全員が参加している
- [ ] コーディング規約書（D-15）とLinter設定がリポジトリに含まれている
- [ ] CI（自動ビルド・テスト）の設定ファイルがリポジトリに含まれており、パイプラインが動作している
- [ ] ブランチ戦略（詳細設計STEP8で決定済み）をチーム全員が理解している

### 共通ライブラリの状態確認

- [ ] C-01〜C-05（認証・エラーハンドリング・バリデーション・ログ・レスポンス形式）の実装が完了している
- [ ] 共通ライブラリのユニットテストが存在し、CIで通っている
- [ ] 共通ライブラリの使い方ドキュメント（または README）が整備されている

---

## 移行前に解消できない問題がある場合

| 問題の種類 | 対応方針 |
|---------|---------|
| 設計書に未完成セクションがある | 実装着手前に設計を完了させる。「実装しながら考える」は手戻りの元 |
| 開発環境が一部の端末でセットアップできない | 実装着手前に解消する。環境問題を抱えたまま実装を始めると全員の作業が止まる |
| 共通ライブラリが未完成 | C-01〜C-05 だけでも先に完成させてから機能実装に入る |
| TBDが残っている | 期限を決めて顧客・PL経由で解消する。TBDを抱えたまま実装した箇所は後で全部やり直しになる |
