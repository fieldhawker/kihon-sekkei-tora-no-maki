# STEP 1: 単体テスト実施

**このステップの目的**: 個々の関数・クラスが詳細設計書（D-01）の仕様どおりに動作するかを確認します。

---

## 単体テストの基本方針

### 単体テストで確認すること

| テスト対象 | 確認内容 | 例 |
|---------|--------|---|
| 正常系 | 期待どおりの入力で期待どおりの結果が返るか | ユーザーID=1 で登録ユーザーのデータが返る |
| 異常系（バリデーション） | 不正入力でエラーが返るか | 空のメールアドレスで登録しようとすると400エラーが返る |
| 異常系（境界値） | 上限・下限・空・null で正しく動くか | 文字数上限ちょうど・上限+1 でそれぞれ動作が正しいか |
| 例外処理 | エラー発生時に正しく例外を投げるか | DB接続エラー時に503ではなく500を返すか |

### 単体テストで確認しないこと（結合テストの範囲）

- 実際のDBとの連携（DBはモック化する）
- 外部APIとの連携（外部APIはモック化する）
- 画面とAPIの連携

---

## テストケースの実行手順

### 手順1: テストコードが実装されていることを確認する

単体テスト設計書（D-13）に記載されたテストケースに対応するテストコードが実装されていることを確認します。

### 手順2: ローカルでテストを実行する

```bash
# Java/Maven の場合
mvn test

# Java/Gradle の場合
./gradlew test

# Python/pytest の場合
pytest

# Node.js/Jest の場合
npm test
```

### 手順3: テスト結果を確認する

- **全テストがグリーン（Pass）になること**が最低条件
- 失敗（Red）があれば、失敗したテスト名・エラーメッセージを確認してバグを修正する
- 修正後は再実行して全テストがグリーンになることを確認する

### 手順4: カバレッジを確認する

```bash
# Java/JaCoCo の場合（target/site/jacoco/index.html で確認）
mvn test jacoco:report

# Python/coverage の場合
pytest --cov=src --cov-report=html

# Node.js/Jest の場合
npm test -- --coverage
```

---

## カバレッジの考え方

| カバレッジ率 | 評価の目安 |
|-----------|---------|
| 80%以上 | リリース品質として合格ライン（詳細設計のSTEP8で合意した基準に従う） |
| 60〜80% | 不足。重要なビジネスロジックにテストが書かれているか確認する |
| 60%未満 | 要注意。バグが出やすい状態。追加テストを優先する |

> **カバレッジ100%が目標ではない**: getter/setterやフレームワーク生成コード等は除外することが多い。重要なのは「ビジネスロジック・バリデーション・エラーハンドリングにテストがあるか」です。

## テスト消化の管理

| 日付 | 実行テスト数 | Pass | Fail | Skip | カバレッジ | 所見 |
|-----|----------|------|------|------|---------|-----|
| XX/XX | 150 | 148 | 2 | 0 | 78% | バリデーション2件失敗→修正中 |
| XX/XX | 152 | 152 | 0 | 0 | 82% | 全グリーン。結合テストへ移行 |

## バグを発見したときの手順

1. **再現手順を確認する**: どの操作・入力で発生するかを確認する
2. **バグ票を起票する**: 付録のテンプレートを使って記録する
3. **担当者にアサインする**: 修正担当者を明確にする
4. **修正後に再テストする**: 修正されたコードで同じテストが通ることを確認する
5. **デグレーション確認**: 修正によって他のテストが壊れていないことを確認する

> **「直ったかもしれない」で終わらせない**: 修正担当者の「直しました」報告後、テスト担当者が必ず再実行して確認してください。
