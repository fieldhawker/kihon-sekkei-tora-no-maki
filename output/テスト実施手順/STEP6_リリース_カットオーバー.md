# STEP 6: リリース・カットオーバー

**このステップの目的**: UAT完了・リリース判定OKを受けて、システムを本番環境へ安全に切り替えます。カットオーバーは「作業が多く・やり直しが困難で・プレッシャーが高い」作業です。事前に手順書を整備し、リハーサルを実施してから本番に臨みます。

---

## カットオーバーの種類

プロジェクトの状況に応じて切替方式を選んでください。

| 方式 | 説明 | メリット | デメリット | 向いているケース |
|-----|-----|---------|---------|-------------|
| **一括切替（ビッグバン）** | 旧システムを停止し、新システムに一斉切替 | シンプル・コストが低い | 切替失敗時のリスクが大きい | 旧システムがない新規開発・規模が小さい |
| **並行稼働** | 一定期間、旧・新を同時運用して徐々に移行 | リスクが低い・比較確認できる | コスト高・二重運用の負担 | 基幹系の置き換え・データ移行が複雑 |
| **段階的切替（フェーズ移行）** | 機能や部署ごとに段階的に切り替える | リスク分散・問題の局所化 | 期間が長い・整合管理が複雑 | ユーザー数が多い・機能が独立している |

> **どの方式を選ぶかは基本設計（STEP 7 運用・移行設計）で決定します。** このSTEPでは選定済みの方式に従って実施します。

## 事前準備チェックリスト

### カットオーバー手順書の整備（X-14日前まで）

- [ ] カットオーバー当日の作業を「分単位」のタイムライン形式でまとめた手順書を作成した
- [ ] 各作業に担当者・所要時間・完了確認方法を明記した
- [ ] バックアウト（切戻し）判定基準と手順を手順書に含めた
- [ ] 手順書をチーム全員・顧客担当者がレビューし合意した

### 本番環境の準備（X-7日前まで）

- [ ] 本番サーバー・クラウドリソースのプロビジョニングが完了している
- [ ] 本番環境の設定ファイル（環境変数・接続先等）が本番用に設定されている
- [ ] SSL/TLS証明書が本番ドメインに設定されており、有効期限を確認した
- [ ] 監視ツール（CloudWatch / Datadog 等）のアラート設定が完了している
- [ ] バックアップの自動取得設定が有効になっている
- [ ] ログ出力先が本番用に設定されており、ログが記録されることを確認した

### リハーサルの実施（X-3日前まで）

- [ ] ステージング環境でカットオーバー手順を通しで実施した（リハーサル）
- [ ] リハーサルで手順の誤り・所要時間のズレ・抜け漏れを洗い出し、手順書を修正した
- [ ] データ移行スクリプトをステージング環境で実行し、移行データの整合性を確認した
- [ ] バックアウト手順もリハーサルで実施し、旧システムへの切戻しが機能することを確認した

### 関係者への周知（X-7日前まで）

- [ ] システム停止時間帯・切替完了予定時刻をユーザーに告知した
- [ ] カットオーバー当日の緊急連絡先・エスカレーションフローを関係者に共有した
- [ ] ヘルプデスク・サポート窓口に、カットオーバー後の問い合わせ増加を事前連絡した

## スモークテスト項目（例）

> **スモークテスト**: カットオーバー直後に「最低限の主要機能が動いているか」を短時間で確認するテスト。全機能を網羅する必要はなく、「これが動かなければ業務が止まる」機能に絞ります。

| ID | 確認内容 | 合否 |
|----|---------|-----|
| SM-001 | ログインできること | □Pass / □Fail |
| SM-002 | トップ画面が表示されること | □Pass / □Fail |
| SM-003 | メインとなる業務機能（一覧・登録・更新）が動作すること | □Pass / □Fail |
| SM-004 | API が正常レスポンスを返すこと（ヘルスチェックエンドポイント） | □Pass / □Fail |
| SM-005 | DBへの書き込み・読み取りが正常に行えること | □Pass / □Fail |
| SM-006 | メール送信機能が動作すること（テストメールを送信） | □Pass / □Fail |
| SM-007 | 管理者機能にアクセスできること | □Pass / □Fail |

## バックアウト（切戻し）判定

カットオーバー中に重大な問題が発生した場合、旧システムへの切戻しを判断します。

### バックアウト判定基準

以下のいずれかが発生した場合、PLが即座にバックアウトを判断します。

- [ ] スモークテストでCritical機能のFailが発生し、XX分以内に解決できない
- [ ] データ移行件数に期待値から XX% 以上の乖離がある
- [ ] 本番DBへのデータ整合性エラーが検出された
- [ ] セキュリティに関わる問題（認証バイパス・データ漏洩リスク）が発覚した

> **バックアウトの判断は「タイムリミット」を事前に決めておくこと。** 問題が発生したとき、焦りと希望的観測から「もう少し待てば解決するかも」と切戻しを先延ばしにしがちです。「XX:XX までに解決しなければ切戻す」を事前に合意しておくことで、冷静な判断ができます。

### バックアウト手順（チェックリスト形式）

- [ ] PL がバックアウト判定を宣言し、全参加者に周知した
- [ ] 新システムへのDNSを旧システムのIPに戻した
- [ ] 新システム上で作成・更新されたデータを旧システムのDBに反映した（移行後に書き込みがあった場合）
- [ ] 旧システムのメンテナンスモードを解除した
- [ ] 旧システムが正常に動作することをスモークテストで確認した
- [ ] バックアウト完了を関係者に通知した
- [ ] バックアウトの原因・対処内容を記録した

## 切替後モニタリング（カットオーバー後 1〜2週間）

カットオーバー直後は問題が発生しやすい時期です。以下を継続的に監視してください。

| 監視項目 | 確認頻度 | 異常の目安 | 対応 |
|--------|--------|---------|-----|
| エラーログ件数 | 1日2回（朝・夕） | 前日比 XX% 増 | 原因調査・緊急対応 |
| APIレスポンス時間 | 1日1回 | 非機能要件の閾値を超えた場合 | チューニング検討 |
| サーバーリソース（CPU / メモリ / ディスク） | 1日1回 | 閾値（例: 80%）を超えた場合 | スケールアップ・アラート確認 |
| ユーザーからの問い合わせ件数 | リアルタイム | 急増した場合 | 内容の分類・FAQ作成・緊急修正 |
| DB接続数・スロークエリ | 1日1回 | スロークエリが増加した場合 | インデックス追加・クエリ最適化 |

## よくある問題と対処

| 問題 | 主な原因 | 対処 |
|-----|---------|-----|
| データ移行後に件数が合わない | 移行スクリプトのバグ・文字コード不整合 | 差分を特定して手動補正。次回から移行リハーサルで事前検証を徹底する |
| SSL証明書エラーがユーザーに表示される | DNSのTTL残留・証明書設定ミス | TTLの短縮設定を事前に行う。証明書を再確認 |
| カットオーバー後にログインできないユーザーがいる | セッションCookieのドメイン設定ミス・キャッシュ残留 | ブラウザキャッシュ削除を案内。設定ミスは即時修正 |
| 本番環境でのみパフォーマンスが遅い | 本番データ量の差・インデックス未設定 | スロークエリを特定してインデックスを追加。DBのEXPLAINで確認 |
| 切戻し後にデータが失われた | バックアウト時のデータ同期漏れ | バックアウト前のDBダンプを元に手動復旧。切戻し手順にデータ同期ステップを追加 |
