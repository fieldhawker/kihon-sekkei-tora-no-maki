# STEP 5: 受入テスト（UAT）

**このステップの目的**: 顧客・エンドユーザーがシステムを実際に操作し、要件定義で合意した要件を満たしているかを確認します。UATは「開発側が承認する」ではなく「顧客が承認する」テストです。

---

## UATとは

> **UAT（User Acceptance Testing）**: エンドユーザーまたは顧客が実施するテスト。システムテストが「仕様どおりか」を確認するのに対し、UATは「業務で使えるか・要件の意図どおりか」を確認します。UATで承認を得ることが、本番リリースの前提条件です。

| 比較軸 | システムテスト | UAT |
|------|------------|-----|
| **実施者** | 開発側SE | 顧客・エンドユーザー |
| **参照基準** | 基本設計書・詳細設計書 | 要件定義書・業務フロー |
| **確認観点** | 仕様どおりに動くか | 業務で使えるか・要件の意図どおりか |
| **合否判定** | SE・PL | 顧客・プロダクトオーナー |
| **完了の証拠** | テスト結果報告書 | UAT完了確認書（顧客署名） |

## 事前準備チェックリスト

### UATシナリオの準備

- [ ] 要件定義書の全機能要件を網羅したUATシナリオ一覧を作成した
- [ ] 各シナリオが「業務の流れ」として自然につながっており、顧客が自分で操作できる粒度になっている
- [ ] 異常系シナリオ（入力ミス・権限なし操作等）も含めている
- [ ] 顧客がシナリオの内容をレビューし、合意している

### UAT環境・データの準備

- [ ] UATはステージング環境（本番相当）で実施することを顧客と合意している
- [ ] 本番に近いテストデータ（マスタデータ・初期データ）をUAT環境に投入した
- [ ] 個人情報を含む本番データを使用する場合、マスキングまたは匿名化の対応をした
- [ ] UATで使用するアカウント（ロール別）を作成し、顧客に払い出した

### 顧客への引き渡し準備

- [ ] ユーザーマニュアルまたは操作説明書を顧客に事前共有した
- [ ] UAT開始前に操作説明会（デモ）を実施した
- [ ] 問題発生時の報告方法（バグ票の書き方・連絡先）を顧客に案内した
- [ ] UATの実施期間・完了判定基準を顧客と合意した

## UAT実施中の管理

### 指摘事項の分類

UATで顧客から受けた指摘はすべて記録し、以下に分類します。

| 分類 | 定義 | 対応方針 |
|-----|-----|---------|
| **バグ（仕様不一致）** | 設計どおりに動いていない | 修正してUATに差し戻し |
| **仕様変更要望** | 要件定義時にない機能の追加・変更要求 | 変更管理プロセスで対応（スコープ外として工数・スケジュールを交渉） |
| **操作方法の質問** | 機能は正しいが顧客が操作方法を理解していない | マニュアルの補足・操作説明で解決。必要に応じてUI改善を検討 |
| **環境・データの問題** | UATデータの誤りや環境設定の問題 | データ修正・環境修正で対応。バグとは区別する |

> **「バグ」と「仕様変更」の区別が重要**: 要件定義書・設計書に記載がある動作と異なる場合はバグです。記載がない追加機能の要望は仕様変更として扱い、スコープ・工数・スケジュールへの影響を顧客と協議します。

### UAT進捗の追跡

- シナリオ消化率（実施件数 ÷ 総件数）を毎日記録する
- 未解決の指摘件数を重大度別に集計し、顧客・PLに定期報告する

---

## UAT完了判定チェックリスト

- [ ] 全UATシナリオを実施し、消化率100%（未実施のシナリオがある場合は顧客の承認があること）
- [ ] Critical・High の指摘がゼロ（または顧客の書面合意でリリース後対応を確認済み）
- [ ] 仕様変更として分類した指摘事項の対応方針が顧客と合意済み
- [ ] 顧客がUAT完了確認書（または相当する書面）に署名している
- [ ] UATの指摘対応で発生した設計変更が設計書に反映されている

## よくある問題と対処

| 問題 | 原因 | 対処 |
|-----|-----|-----|
| UATが終わらない | シナリオが多すぎる・指摘が次々発生する | 優先度の高い業務シナリオに絞り込む。指摘の分類を徹底し「仕様変更はスコープ外」を顧客に説明する |
| 顧客が操作方法を理解できない | マニュアル不足・UI設計の問題 | 操作説明の追加実施、マニュアルの改善。根本的なUIの問題はバグとして修正を検討 |
| システムテストで通ったのにUATで落ちた | テストデータが本番と異なる・業務フローの理解不足 | UATデータを本番に近い状態で整備する。次回から業務フローを設計段階でより詳細に確認する |
| 顧客が「仕様変更要望」を「バグ」と主張する | 要件定義時の合意内容が曖昧 | 要件定義書・議事録を根拠に丁寧に説明する。合意内容の記録の重要性を次プロジェクトに活かす |
