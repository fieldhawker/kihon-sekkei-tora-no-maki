## バグ票テンプレート

バグを発見したら、以下のフォーマットで起票してください。GitHub Issues・Jira・Excel等に記録します。

```markdown
## バグ票 BUG-XXXX

### 基本情報
- **発見日**: YYYY-MM-DD
- **発見者**: 氏名
- **対象環境**: ローカル / ステージング / 本番
- **重大度**: Critical / High / Medium / Low
- **ステータス**: 未対応 / 対応中 / 修正済み / 確認済み / 保留

### 再現手順
1. （操作ステップ1）
2. （操作ステップ2）
3. （操作ステップ3）

### 期待する結果
（正しい動作の説明）

### 実際の結果
（バグの症状の説明）

### スクリーンショット / ログ
（証拠となる画像・エラーログを添付）

### 修正担当者
（担当するエンジニアの名前）

### 修正内容（修正後に記入）
（何をどう直したかの説明）

### 修正確認者・確認日
（テスト担当者の名前・確認日）
```

## テスト消化管理台帳テンプレート（Excel / スプレッドシート）

以下の列構成で管理します。

| 列名 | 内容 |
|-----|-----|
| テストケースID | ST-001 等の採番（設計書のケースIDと一致させる） |
| テストレベル | 単体 / 結合 / システム |
| 機能カテゴリ | 認証・ユーザー管理・購入等 |
| テスト内容 | 何を確認するか（1行で） |
| 実施予定日 | YYYY-MM-DD |
| 実施日 | YYYY-MM-DD（実施後に記入） |
| 実施者 | 担当者名 |
| 結果 | ✅Pass / ❌Fail / ⏭Skip / — 未実施 |
| バグ票No. | Failの場合に BUG-XXX を記入 |
| 備考 | スキップ理由・特記事項 |

### 進捗集計用の数式例（Googleスプレッドシート）

```
総件数    =COUNTA(F2:F1000)-1
Pass件数  =COUNTIF(H2:H1000,"✅Pass")
Fail件数  =COUNTIF(H2:H1000,"❌Fail")
未実施件数=COUNTIF(H2:H1000,"—")
消化率    =（Pass件数+Fail件数）÷ 総件数 × 100
```

## よくあるバグパターン（経験から学ぶ）

| パターン | 原因 | 再発防止策 |
|--------|-----|---------|
| null参照エラー（NullPointerException） | 入力がない場合・DBから取得できない場合のnullチェックが漏れ | バリデーションで必須チェックを追加・Optional/null安全な書き方を使う |
| 文字コード化け | DBの文字コードがUTF-8になっていない・接続文字列にcharsetが未指定 | 環境構築手順書にDB文字コード設定を明記する |
| タイムゾーン問題 | サーバー・DB・アプリでタイムゾーンが統一されていない | アーキテクチャ設計時にタイムゾーンをUTCに統一する |
| ページネーションのオフバイワン | ページ番号の0始まり/1始まりが混在している | API設計書で0始まり/1始まりを明記し、実装と合わせる |
| セッション切れのエラーハンドリング漏れ | JWTトークン期限切れ時のフロントエンドの処理がない | 401レスポンス時の共通ハンドラーをフロントエンドに実装する |
| 並行更新の競合（競合状態） | 複数ユーザーが同じデータを同時に更新する設計が考慮されていない | 楽観的ロック（バージョン番号）または排他ロックを設計に組み込む |
| テスト環境で再現しない本番バグ | データ量・環境設定が本番と異なる | ステージング環境を本番相当に保つ。本番データ量でのテストを計画する |
