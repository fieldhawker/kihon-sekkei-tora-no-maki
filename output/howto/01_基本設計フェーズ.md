## PHASE 1: 基本設計

### フェーズの目標

「どのように作るか」の骨格を決めます。詳細な実装方法はまだ決めません。このフェーズが終わると、「何をどう作るかの方針」が関係者全員で合意された状態になります。

### 主な参照資料

- **資料①** [`基本設計手順/`](../基本設計手順/) ― 各STEPのチェックリストと進め方（STEP0〜STEP8 + 付録）
- **資料②** [`基本設計_成果物ガイド/`](../基本設計_成果物ガイド/) ― 各成果物で使うツールの選び方（B-01〜B-19）

---

### STEP 1-0: インプット確認

**やること**: 要件定義成果物が基本設計を始めるのに十分な状態かを確認する

**参照資料**: 資料① → `3-0. 要件定義成果物の受領・確認`

**手順**
1. 資料①の `3-0` のチェックリストを開く
2. 項目を1つずつ確認する
3. 未入手・未確定の項目があれば、**設計着手前に解消する**

> **ここで時間をかけることが後工程の手戻りを防ぐ最大の投資です。**「なんとなく揃っている」では進めないようにしましょう。

---

### STEP 1-1: システムアーキテクチャ設計

**やること**: システムの「骨格」を決める。後続のすべての設計の前提になる

**参照資料**
- 資料① → `3-1. システムアーキテクチャ設計`（チェックリスト）
- 資料② → `B-01 システム構成図`、`B-02 技術スタック選定書`（ツール選定）
- 資料⑥ → `adr_sample/`（技術選定の意思決定をADRとして記録する）

**作成する成果物**

| 成果物 | ツール | 参照 |
|------|------|-----|
| システム構成図（論理・物理） | Draw.io | 資料② B-01 |
| 技術スタック選定書（ADR形式） | Markdown + Git | 資料② B-02 |

**AIへの相談**
> 資料①の `3-1` 末尾の「💬 AIに相談するときのコンテキスト」を開いてテンプレートを使ってください。

**重要な判断ポイント**

このステップで以下を確定させます。迷ったらAIや先輩SEに相談してください。

- クラウド構成（AWS / Azure / GCP / オンプレ）
- フロントエンド技術（React / Vue.js / 従来型 など）
- バックエンド技術（Java/Spring Boot / Python/FastAPI / Node.js など）
- データベース（MySQL / PostgreSQL / DynamoDB など）
- 認証方式（JWT / セッション / SSO など）

### STEP 1-5: 非機能要件設計

**やること**: 性能・可用性・スケーラビリティ等の品質基準を具体的な数値で設計に落とし込む

**参照資料**
- 資料① → `3-5. 非機能要件設計`
- 資料① → `4-2. 非機能要件`（要件定義書に数値があるか確認するチェックリスト）
- 資料② → `B-13 非機能要件設計書`

**必ず数値で定義する項目**

| 項目 | 悪い例（NG） | よい例（OK） |
|-----|-----------|-----------|
| レスポンスタイム | 「なるべく速く」 | 「95パーセンタイルで3秒以内」 |
| 同時接続数 | 「多くても大丈夫なように」 | 「同時接続50名を想定」 |
| 稼働率 | 「できるだけ止めない」 | 「年間稼働率99.9%以上」 |
| RTO | 「早めに復旧する」 | 「障害から4時間以内に復旧」 |

### STEP 1-7: 運用・移行設計

**やること**: リリース後に「誰が・どのように・何を使って」運用するかを決める

**参照資料**
- 資料① → `3-7. 運用・移行設計`
- 資料② → `B-16 運用設計書`、`B-17 移行設計書`

> **よくある失敗**: 運用設計を「開発が終わってから考えよう」と後回しにすると、本番切替直前に大混乱になります。基本設計フェーズ内で完了させてください（資料① 付録B参照）。

---

### STEP 1-7b: テスト戦略（基本方針）の策定

**やること**: リリース品質を担保するための「何をどこまでテストするか」の基本方針を決める

**参照資料**
- 資料① → `3-7b. テスト戦略（基本方針）の策定`

**決めること（最低限）**

| 項目 | 内容 |
|-----|------|
| テストレベルの割り当て | 単体テスト・結合テスト・システムテストの担当者と目的 |
| 性能テストの要否 | ユーザー数・処理量から必要かを判断する |
| テスト環境の方針 | 本番とどの程度近い構成を用意するか |
| テスト完了基準 | リリースGOを出すための判定条件 |

> **なぜ基本設計で決めるか**: テスト工数はプロジェクト全体の20〜30%を占めることもあります。スケジュールにテスト工数を確保するためにも、基本設計の段階でテスト戦略を確定させる必要があります。
