## PHASE 2: 詳細設計

### フェーズの目標

「具体的にどう実装するか」をプログラマーがコードを書けるレベルまで細かく定義します。このフェーズが終わると、「この設計書を見れば誰でもコードが書ける」状態になります。

### 主な参照資料

- **資料③** [`詳細設計手順/`](../詳細設計手順/) ― 各STEPのチェックリストと進め方（STEP0〜STEP9 + 付録）
- **資料④** [`詳細設計_成果物ガイド/`](../詳細設計_成果物ガイド/) ― 各成果物で使うツールの選び方（D-01〜D-18）
- **資料⑤** [`共通ライブラリ/`](../共通ライブラリ/) ― 先行して作る共通ライブラリの設計書（C-01〜C-17）

---

### STEP 2-0: インプット確認

**やること**: 基本設計書が詳細設計を始めるのに十分な状態かを確認する

**参照資料**: 資料③ → `3-0. 基本設計成果物の受領・確認`

**特に確認すること**

- 非機能要件の数値が明記されているか（「なるべく速く」等の曖昧表現が残っていないか）
- 未決事項（TBD）がゼロになっているか
- 外部連携先の仕様書を入手済みか

### STEP 2-先: 共通ライブラリの先行設計・実装

**やること**: 全機能から使われる共通処理を、機能実装の前に設計・実装しておく

**参照資料**: 資料⑤ [`共通ライブラリ/`](../共通ライブラリ/) 全体

**なぜ先行するのか**
> 各機能担当が個別にログ・エラーハンドリング・バリデーションを実装すると、「同じ処理が微妙に異なる実装で乱立」します。先に共通化しておくことで品質が均一になり、機能実装の速度も上がります。

**最優先で実装する共通ライブラリ（機能実装前に必須）**

| ライブラリ | 資料⑤の参照先 | 理由 |
|---------|-----------|-----|
| C-01 認証・認可 | `C-01. 認証・認可ライブラリ` | 認証なしにAPIが実装できない |
| C-02 エラーハンドリング | `C-02. エラーハンドリングライブラリ` | エラーレスポンスの形式を先に統一する |
| C-03 バリデーション | `C-03. バリデーションライブラリ` | 共通バリデーターを先に作ると機能実装が速くなる |
| C-04 ログ出力 | `C-04. ログ出力ライブラリ` | ログフォーマットを最初から統一する |
| C-05 レスポンス形式 | `C-05. レスポンス統一フォーマット` | APIのレスポンス構造を最初に決める |

**2〜3機能目の前までに追加する共通ライブラリ**

| ライブラリ | 資料⑤の参照先 |
|---------|-----------|
| C-06 ページネーション | `C-06. ページネーションライブラリ` |
| C-07 ファイル操作 | `C-07. ファイル操作ライブラリ` |
| C-08 メール送信 | `C-08. メール送信ライブラリ` |
| C-12 DB共通処理 | `C-12. DB共通処理ライブラリ` |
| C-13 コードマスタ管理 | `C-13. コードマスタ管理ライブラリ` |

## フェーズをまたぐ継続的な作業

以下の作業はフェーズに関わらず、設計の進行と並行して継続的に行ってください。

### アーキテクチャ上の意思決定をADRで記録する

技術的な決断を下したとき（「なぜこの技術を選んだか」「代替案として何を検討したか」）は、そのつどADRファイルを作成してGitに保存してください。

**参照資料**: 資料② → `ADRとは何か`・`ベストなADR管理方法`

**ADRインデックス（README.md）の自動生成**
> 資料② → `README.md の自動生成方法` を参照してください。`generate-adr-index.bat` をダブルクリックするだけで一覧が更新されます。

### トレーサビリティの維持

要件IDと設計書の対応表（トレーサビリティマトリクス）を更新し続けてください。設計が抜け漏れていないかを確認するための重要な資料です。

**参照資料**: 資料② → `B-19 トレーサビリティマトリクス`、資料④ → `D-18 更新版トレーサビリティマトリクス`
