## コンテキスト

Webアプリケーションのフロントエンドフレームワークを決定する必要がある。

- チームのフロントエンド経験: JavaScript は全員経験あり。TypeScript経験者は1名
- 画面数: 約20画面
- 複雑なUIインタラクションは少ない（CRUD操作が中心）
- SSR（サーバーサイドレンダリング）の要件はなし（SEO対策不要の社内システム）

---

## 検討した選択肢

### 選択肢1: React + Vite

**メリット**: 最も広く使われており、ライブラリエコシステムが豊富。採用情報・学習資料が多い。TypeScriptとの相性が良い。
**デメリット**: Vue.jsに比べてテンプレートの書き方が独特（JSX）。State管理の方法が複数あり迷いやすい。

### 選択肢2: Vue.js 3 + Vite

**メリット**: テンプレート構文が HTMLに近く学習コストが低い。Options APIとComposition APIの両方が使える。
**デメリット**: Reactより求人・参考資料がやや少ない。大規模アプリでの実績がReactより少ない。

### 選択肢3: Next.js（React ベース）

**メリット**: SSR・SSGが使いやすい。ファイルベースルーティング。フルスタックでも使える。
**デメリット**: 社内システムのためSSRは不要。Reactより学習コストが高い。

---

## 決定

**選択肢2: Vue.js 3 + Vite + TypeScript** を採用する。

### 理由

- チームメンバーのHTMLテンプレート経験を活かせる学習コストの低さを優先した
- CRUD操作中心のシステムでVue.jsの表現力で十分
- Composition APIにより、コンポーネントロジックの再利用性も確保できる

### 採用するライブラリ構成

| 役割 | ライブラリ | バージョン |
|-----|---------|---------|
| フレームワーク | Vue.js | 3.x |
| ビルドツール | Vite | 5.x |
| 型システム | TypeScript | 5.x |
| ルーティング | Vue Router | 4.x |
| 状態管理 | Pinia | 2.x |
| UIコンポーネント | Vuetify または Element Plus | 最新安定版 |
| HTTP通信 | Axios | 1.x |
| フォーム | VeeValidate | 4.x |

---

## 影響範囲

- 開発環境構築手順書（D-16）にNode.js・npm設定を記載する
- コーディング規約（D-15）にVue.js固有のルールを追記する
- CI設定に `npm run build` と `npm run test` を追加する
