## コンテキスト

バックエンドAPIの設計スタイルを決定する必要がある。

- フロントエンド: SPA（Vue.js 3）
- バックエンド: Java / Spring Boot
- 外部システム連携: あり（REST APIが求められている）
- チームの経験: REST API の経験は全員あり。GraphQL・gRPC の経験者はなし

---

## 検討した選択肢

### 選択肢1: REST API

**概要**: HTTP動詞（GET/POST/PUT/DELETE）とURIリソース設計に基づく標準的なAPI設計スタイル。

**メリット**:
- 最も広く使われており、チームの学習コストが最も低い
- OpenAPI（Swagger）でドキュメント化しやすい
- 外部システムとの連携でも使いやすい
- キャッシュ（HTTPキャッシュ・CDN）との相性が良い

**デメリット**:
- 複雑な関連データを取得するために複数リクエストが必要になることがある（オーバーフェッチ・アンダーフェッチ）

---

### 選択肢2: GraphQL

**概要**: クライアントが必要なデータを宣言的に指定してAPIから取得できるクエリ言語。

**メリット**:
- クライアントが必要なフィールドだけ取得できる（オーバーフェッチの解消）
- 1リクエストで関連データを一括取得できる
- フロントエンド開発が柔軟になる

**デメリット**:
- チーム全員に学習コストが発生する
- キャッシュの設計が複雑
- エラーハンドリングがRESTより複雑
- 外部システム連携では使いにくい

---

### 選択肢3: gRPC

**概要**: Protocol Buffersを使ったバイナリ通信プロトコル。主にサービス間通信に使われる。

**メリット**:
- バイナリ通信のため高速・軽量
- 厳密なスキーマ定義で型安全

**デメリット**:
- ブラウザからの直接通信には向かない（gRPC-Webが必要）
- 社内システム・外部連携での採用実績が少なく、チームの学習コストが高い

---

## 決定

**選択肢1: REST API（RESTful設計）** を採用する。

### 採用するAPI設計規約

| 項目 | 規約 |
|-----|-----|
| ベースURL | `/api/v1/` でバージョン管理 |
| リソース命名 | 複数形の名詞（`/users`・`/products`）|
| HTTP動詞 | GET=取得 / POST=作成 / PUT=全体更新 / PATCH=部分更新 / DELETE=削除 |
| フィルタ・検索 | クエリパラメータで表現（`GET /products?category=food&sort=price`）|
| ページネーション | `page` と `per_page` パラメータを使用 |
| エラーレスポンス | RFC 7807（Problem Details）形式に準拠 |

### ドキュメント化

- OpenAPI 3.0 形式（YAML）で API 仕様書を作成する
- Stoplight Studio を使って YAML を編集・プレビューする
- CI で `redoc-cli` を実行して HTML ドキュメントを自動生成する

---

## 影響範囲

- 詳細API仕様書（D-03）はOpenAPI 3.0形式で作成する
- 全APIに共通レスポンス形式（共通ライブラリ C-05）を適用する
- `GET /products` 等の一覧系APIにはページネーション（共通ライブラリ C-06）を適用する
