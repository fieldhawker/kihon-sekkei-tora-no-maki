# 報告管理

> **このファイルの目的**: SESエンジニアが「自社」と「客先」の両方に対して、何を・いつ・どのように報告すべきかを整理します。報告の質は、契約継続・単価交渉・緊急時対応に直結します。

---

## 報告先と報告の目的

SESエンジニアには2種類の報告相手がいます。それぞれの目的が異なります。

| 報告先 | 主な目的 | 報告しないと起きること |
|------|--------|-----------------|
| **自社担当（営業・PM）** | 契約管理・トラブル対応・キャリア支援 | 問題が起きたときに自社が動けない。評価・単価交渉の材料がなくなる |
| **客先（PL・リーダー）** | 業務の進捗管理・品質確保 | 現場での信頼を失う。問題の発見が遅れる |

---

## 自社担当への報告

### 報告頻度の目安

| 状況 | 頻度 |
|-----|-----|
| 通常時 | 週1回（週次報告） |
| 現場参画直後（最初の1ヶ月） | 週2回程度（現場の状況を細かく共有） |
| 問題・トラブル発生時 | 即時報告 |
| 契約更新の時期（1〜2ヶ月前） | より詳細な状況共有 |

### 週次報告の内容テンプレート

```markdown
## 週次報告 YYYY-MM-DD（第XX週）

### 今週やったこと
- 基本設計書（機能設計パート）の作成：進捗80%
- 設計レビュー（レビュー会議に参加・指摘対応）
- DBスキーマの見直し対応

### 来週の予定
- 基本設計書の完成・提出
- データ設計のレビュー依頼
- 外部IF設計着手

### 稼働時間（今週）
- 合計: XX時間（契約精算幅: 140〜180時間/月 → 今月累計: XXX時間）

### 現場の状況
- プロジェクト全体: 設計フェーズ順調。来週からデータ設計着手予定
- チームの雰囲気: 良好。メンターの鈴木さんに設計方針で相談できている
- 気になること: ○○機能の仕様が未決定のため、来週以降に確認が必要

### スコープ・契約に関する懸念（あれば）
- 特になし / ○○の追加依頼があった（詳細別途相談）

### 困っていること・相談事項
- 特になし / ○○について判断に迷っているので相談したい
```

### 即時報告すべき事項

以下は発生したらすぐに自社担当に連絡してください。

| 事象 | 理由 |
|-----|-----|
| ハラスメント・不当な扱いを受けた | 自社が客先に申し入れる必要があるため |
| 契約外の作業を強く求められた | スコープ・契約交渉が必要なため |
| 客先から直接「契約終了」「更新しない」と言われた | 自社が正式な手続きを行う必要があるため |
| 情報漏洩・セキュリティインシデントが発生した | 迅速な対応が必要なため |
| 大きなバグ・障害が発生した | 自社として把握が必要なため |
| 体調不良・長期休暇が必要になった | 稼働時間・契約への影響があるため |
| 現場でのトラブル（人間関係含む）が深刻化した | 早期に対応するほど解決しやすい |

## 客先（現場の上司・リーダー）への報告

### 報告タイミング

| タイミング | 内容 |
|---------|-----|
| 朝（作業開始時） | 今日のタスク・懸念事項の共有（朝会がある場合） |
| 作業中（必要に応じて） | ブロッカーの発生・仕様の確認・想定外の事態 |
| 終業時 | 今日の進捗・明日の予定・残課題 |
| タスク完了時 | 成果物の提出・レビュー依頼 |
| 問題発生時 | 即時報告（遅延の見込み・バグ発見等） |

### 進捗報告の伝え方

「何%終わった」ではなく「あと何がどのくらいかかるか」を伝える方が実務的に役立ちます。

```
NG: 「基本設計書は今60%くらい完成です」

推奨: 「基本設計書のうち、機能設計パートは完了しました。
     残りはデータ設計との整合確認と非機能要件パートで、
     明日中に完成できる見込みです。
     1点確認させていただきたいのですが、○○の仕様が...」
```

### 問題・遅延を報告するとき

遅延や問題が発生したときの報告は、「謝罪」だけでなく**現状・影響・対処方針**をセットで伝えると信頼を保ちやすいです。

```
「○○機能の設計で予想より複雑な仕様が見つかり、
当初の見積もりより1日遅れる見込みです。

影響: 設計書の提出が木曜 → 金曜になります。
対処: 今日の夕方までに判断が必要な点を整理してご確認いただければ、
     遅れを最小化できます。確認をお願いできますか？」
```

## 稼働時間の管理と報告

### 自己管理の重要性

SESでは稼働時間が直接的に単価・報酬に影響します。正確な記録が重要です。

- 開始・終了時刻を毎日記録する
- 現場の打刻ツールだけでなく、**自分でもメモを残す**
- 月末に精算幅との乖離がないか確認する

### 精算幅に関わる報告

| 状況 | 対応 |
|-----|-----|
| 精算幅の上限（例: 180h）を超えそう | 月の前半の時点で自社担当に報告。超過分の精算条件を確認する |
| 精算幅の下限（例: 140h）を下回りそう | 自社担当に報告。タスク状況・理由を説明する |
| 稼働時間の記録に誤りがあった | 速やかに自社担当・客先に申告する。後で発覚すると信頼を失う |

---

## 報告の記録を残す習慣

口頭での報告だけでなく、**記録として残すこと**が重要です。

- メールやチャットで報告する（口頭のみにしない）
- 重要な合意・決定はその場でメモを取り「○○と認識していますがよろしいでしょうか」と確認する
- 「言った・言わない」のトラブルを防ぐために、重要な指示は文書化してもらう

### 自分のための作業ログ

日々の記録は「こんなに仕事した」という証拠になり、契約更新や単価交渉の材料にもなります。

```
## 作業ログ 2026-04-26（日）

- 基本設計書：機能設計セクション作成 (3h)
  - 機能一覧T-01作成。要件定義書の要件F-001〜F-015を網羅
  - F-008の仕様が不明確。鈴木さんに確認 → 明日回答もらえる

- 設計レビュー参加 (1.5h)
  - アーキテクチャ設計のレビュー。指摘事項3件
  - 対応: DB接続プールの設定を見直す（明日着手）

- 稼働: 9:00〜18:00（休憩1h） → 実稼働 8h
```
