# 日常の立ち回り

> **このファイルの目的**: 現場に入った後、日々の業務をスムーズに進め、自社と客先の双方から信頼を得るための立ち回りを整理します。

---

## SESの「二重の関係」を理解する

SESエンジニアは常に2つの関係の中で動いています。

```
自社（SES会社）
    │ 雇用関係・給与・キャリア管理
    │ 契約・法的な報告義務
    ▼
自 分
    │ 業務の実行
    │ 日常のコミュニケーション
    ▼
客先（常駐先）
    │ 業務上の指示・現場のルール
    │ 評価（契約継続・単価交渉に影響）
```

この二重関係において、**客先に溶け込みながら自社への義務も果たす**バランスが求められます。

| 客先優先にしすぎると | 自社優先にしすぎると |
|-----------------|-----------------|
| 自社の報告が疎かになる | 現場で浮いて信頼を得にくい |
| 契約外作業を断れなくなる | 客先との関係が悪化する |
| 自社のルールを破ってしまう | 契約更新されにくくなる |

## 日次・週次ルーティン

### 毎日やること

- [ ] 作業開始・終了時に打刻（稼働時間の正確な記録）
- [ ] 当日のタスクをチームに共有する（朝会・スタンドアップがある場合は参加）
- [ ] 作業ログを手元メモまたはツールに残す（週次報告の素材になる）
- [ ] 終業前に進捗・ブロッカーを上司・メンターに報告する

### 毎週やること

- [ ] 週次報告を自社担当に送る（[06_報告管理.md](06_報告管理.md) 参照）
- [ ] 来週のタスク・目標を確認する
- [ ] 技術的な疑問・設計上の迷い点を整理し、相談機会を作る

## コミュニケーションの原則

### 報告・連絡・相談（報連相）の徹底

SESエンジニアが「信頼できる」と評価される最大の要因の一つが**報連相の質**です。

| 種類 | タイミング | 内容 |
|-----|---------|-----|
| **報告** | タスク完了時・終業時 | やったこと・結果・次に進むこと |
| **連絡** | 変化が生じたとき | 遅延の見込み・仕様への疑問・不在予定 |
| **相談** | 判断が必要なとき | 方針の選択・困っていること・リスク |

> **「完了してから報告」ではなく「途中でも状況を共有」** する方がトラブルを早期に防げます。特に遅延の見込みが出たときは、早めに伝えることが重要です。

### 確認を取ることの重要性

設計・実装に入る前に、**認識のすり合わせ**を行う習慣をつけてください。

```
NG: 仕様を自分で解釈して実装する → 後で「そういう意図じゃなかった」

推奨: 「この仕様はこういう意味で理解しましたが合ってますか？」
    → 合意を取ってから着手 → 手戻りが最小化される
```

## 二重の指示系統への対処

### 客先と自社の指示が矛盾したとき

| 矛盾のパターン | 対処法 |
|-----------|------|
| 客先が「この設計方針でやって」と言うが、自社のガイドラインと合わない | まず自社担当に相談。現場に合わせる場合は自社の承認を取る |
| 客先が「毎日22時まで残業してほしい」と言う | 稼働時間管理は自社の管轄。自社担当に相談して調整してもらう |
| 客先が「他の案件の手伝いもしてほしい」と言う | 契約外の可能性が高い。自社担当に確認を依頼する |
| 自社担当から「この現場のやり方に合わせてほしい」と指示が来る | 自社担当の指示に従う |

> **原則: 迷ったら自社担当に相談する。** 客先からの要求を勝手に受け入れることで後からトラブルになるより、一度自社を通す方が安全です。

## 現場での「立場」の意識

### SESエンジニアとして注意すること

| 場面 | 推奨しない行動 | 推奨する行動 |
|-----|-----------|----------|
| 設計方針の議論 | 客先エンジニアを否定するような強い意見 | 「こういう観点もあると思いますが、いかがでしょうか」という提案スタイル |
| 他のSES・ベンダーの話題 | 批判・比較 | コメントしない。必要な場合は中立的な立場を保つ |
| 現場の内部情報（組織の課題・人間関係）| 自社担当以外に話す・SNSに書く | 守秘義務の範囲内として扱う |
| 客先からの「愚痴」「内部事情」 | 積極的に聞いて引き込まれる | 必要最低限の受け応えにとどめる |

### 技術的な発言の仕方

SESとして現場に入ると、「外部の人間が言うと重く受け取られる」場面と「外部だから言いやすい」場面の両方があります。

- **プロとしての意見を持つ**: 「言われたことだけやる」姿勢は評価されにくい。設計レビューでは積極的に意見を述べる
- **ただし現場を尊重する**: 現場の判断・決定には「外部の自分が覆すもの」という姿勢は持たない
- **根拠を持って話す**: 「こうした方がいい」ではなく「○○の理由でこちらが適切と考えます」と伝える

## トラブル対応

### 業務上のトラブル（技術的な問題・遅延など）

1. 自分で原因を特定できる範囲を確認する
2. 解決できない場合は**早めに**上司・メンターに状況を報告する
3. 自社担当にも共有する（客先との調整が必要になる可能性があるため）

### 人間関係のトラブル

| トラブルの種類 | 対処の優先順位 |
|-----------|------------|
| 現場の担当者から過剰なプレッシャーをかけられる | ①自社担当に相談 → ②自社担当から客先に対応依頼 |
| 現場でハラスメント（パワハラ・セクハラ等）を受ける | ①自社担当に即報告 → ②自社のハラスメント相談窓口に連絡 |
| チームメンバーと技術的な意見が対立する | 業務上の議論として冷静に対応。感情的にならない |
| 客先から理不尽な要求を繰り返し受ける | 記録を残した上で自社担当に相談 |

> **重要**: ハラスメント・パワハラは「証拠を残す」ことが重要です。発言の日時・内容・その場にいた人をメモしておきましょう。

### 技術的な判断の誤り（バグ・設計ミス）

SESエンジニアが設計や実装でミスをしたとき、「自社に責任がある（準委任だから成果物の完成責任はない）」「客先に責任がある」という議論になることがあります。

- まず誠実に謝罪し、修正対応を速やかに行う
- 損害賠償の話になった場合は**必ず自社担当を交えて対応する**（個人で対応しない）
- 再発防止策を提案することで信頼を回復する

## 体調管理・有給休暇の取り方

### 有給休暇

- 有給は自社の制度に従って取得する
- 現場にも事前に連絡する（何日前までに連絡するかは現場のルールに従う）
- 現場の「空気」で有給が取りにくい場合は自社担当に相談する

### 体調不良・当日欠勤

1. 自社担当に連絡する
2. 現場の直接上司にも連絡する（どちらが先でも可だが、両方に連絡する）
3. 担当タスクの状況（ブロッカーになるものがないか）を簡単に伝える

> **SES特有の注意点**: 欠勤が多いと稼働時間が精算幅を下回り、単価減額になる可能性があります。体調管理も業務の一部として意識してください。
