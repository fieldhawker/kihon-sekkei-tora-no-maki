# キャリア管理

> **このファイルの目的**: SES環境でのキャリア形成・スキルアップ・市場価値の高め方を整理します。「色々な現場を経験できる」SESの利点を最大化するための考え方をまとめます。

---

## SES環境のキャリア上のメリットとリスク

### メリット

| メリット | 内容 |
|--------|-----|
| 多様な現場経験 | 異なる業界・技術スタック・チーム文化を経験できる |
| 技術の幅が広がる | 1社にいると触れない技術・設計手法に触れられる |
| 短期間で環境を変えられる | 合わない現場は次の契約更新で変えられる |
| 市場価値の客観的な確認 | 「どの現場でも通用するか」が試される |

### リスク

| リスク | 対処法 |
|------|------|
| スキルの「広く浅く」になりがち | 意図的に「深める領域」を決める（下記参照） |
| 正社員と比べてキャリアが可視化しにくい | 自分でポートフォリオ・スキル棚卸しをする |
| 「指示待ち」になりやすい | 主体的に設計提案・改善提案をする習慣をつける |
| 現場が変わるたびに積み上げがリセットされる感覚 | 技術スキルと業務スキルは現場をまたいで蓄積されることを意識する |

---

## スキルの棚卸し方法

現場が変わるたびに「自分が何を得たか」を整理する習慣をつけてください。

### スキルマップテンプレート

```markdown
## スキル棚卸し（最終更新: YYYY-MM）

### 技術スキル

| 技術・ツール | レベル | 経験年数 | 経験した現場・業務 |
|-----------|------|---------|--------------|
| Java（Spring Boot） | 実務で設計・実装できる | 3年 | ○○社 ECシステム・△△社 社内システム |
| React | 基本的な実装ができる | 1年 | △△社 管理画面 |
| MySQL | テーブル設計・インデックス設計ができる | 3年 | 複数現場 |
| AWS（ECS/RDS/S3） | インフラ構成設計・構築ができる | 1.5年 | ○○社 |
| 基本設計（要件定義〜設計書作成） | 単独で担当できる | 2年 | △△社・□□社 |

### ビジネス・業務スキル

| スキル | 経験内容 |
|------|--------|
| 顧客折衝 | 仕様確認・要件ヒアリングを担当。非技術者への説明も対応 |
| 設計書レビュー | チーム内レビューのファシリテーション経験あり |
| タスク管理 | GitHub Issues・Jiraでのチケット管理・スプリント運営経験 |

### 業界・ドメイン知識

| 業界 | 業務知識の概要 |
|-----|-----------|
| 小売・EC | 在庫管理・受発注フロー・帳票設計の知識 |
| 金融 | 口座管理の基礎・セキュリティ要件の理解 |
```

## 現場をまたいで「深める」テーマを決める

SESで多くの現場を経験する場合、意識的に「深める軸」を持つことが重要です。

### よくある「深める軸」の選び方

| 軸の種類 | 例 |
|--------|---|
| 技術特化 | バックエンドAPI設計・DBチューニング・クラウドインフラに特化する |
| フェーズ特化 | 基本設計・要件定義フェーズを得意とするSEとして確立する |
| 業界特化 | 金融・医療・流通など特定業界の業務知識を深める |
| 役割特化 | テックリード・設計レビューの専門家として立ち位置を確立する |

> **深めるテーマがあると**: 自己紹介がしやすくなる・単価交渉の根拠になる・「この人に頼みたい」という指名依頼が来やすくなる

## 自分の評価を可視化する

### 現場での評価を把握する

自社の評価者（上司）は現場にいないため、**客先からの評価**が単価・契約継続に直結します。

定期的に以下を確認してください。

- [ ] 現場のリーダー・PLとの1on1の場で「業務の評価・改善点」を直接確認する
- [ ] 契約更新のタイミング（2〜3ヶ月前）に自社担当から客先の評価を聞く
- [ ] 「何が評価されているか」「何を改善すべきか」を自社担当からフィードバックしてもらう

### 成果を記録する

次の現場への参画・単価交渉・転職活動のために、**具体的な成果を記録**しておきます。

```markdown
## 現場成果記録（○○社 ECシステム開発 2025.04〜2026.03）

### 担当した業務
- 基本設計フェーズ: 商品管理・注文管理機能の設計
- 詳細設計フェーズ: 注文APIのOpenAPI仕様書作成
- テストフェーズ: 結合テストのシナリオ設計・実施

### 成果・実績
- 基本設計書（B-03〜B-11 相当）を単独でドラフト作成し、レビュー2回で承認
- 注文APIの設計でレスポンス時間を平均200ms → 80msに改善（インデックス最適化）
- テストシナリオ100件作成、バグ検出率30%（前フェーズ比 +10%）

### 習得したスキル・知識
- ECシステムの在庫管理ロジック（引当・入荷・ロット管理）
- AWSのECSを使ったブルーグリーンデプロイの実施経験
- 大規模テーブル（1億件超）のDBチューニング手法

### 難しかった点・学び
- 外部決済サービスとの連携設計で、エラーハンドリングの網羅が予想以上に複雑だった
  → シーケンス図で全パターンを洗い出してから設計する手法を習得
```

## 単価交渉の考え方

SESの単価は「市場の相場」と「自分の市場価値」のバランスで決まります。

### 単価が上がるタイミング

- 契約更新時（3〜6ヶ月ごと）
- 担当範囲が広がったとき（設計 → 設計＋レビューリード等）
- 新しいスキルを習得し、稼働した証拠があるとき
- 他社からのオファーがあったとき（競合状況として活用できる）

### 単価交渉の準備

1. **自分の市場相場を調べる**: 求人サイト・エージェントに確認する（ITエンジニアの経験年数・スキル別相場）
2. **成果を数値で言えるようにする**: 「設計書を作った」ではなく「画面XX枚・API XX本の基本設計を担当した」
3. **自社担当に交渉を依頼する**: 自分から客先に直接言うのではなく、自社担当を通じて行う

## 次の現場を選ぶときの観点

現場が変わるタイミングは「自分のキャリアに合った現場を選ぶ」チャンスです。

| 確認項目 | 確認方法 |
|--------|--------|
| 担当フェーズ・業務内容 | 自社担当に詳細を確認する |
| 技術スタック | 自分の目指す方向と合っているか |
| チームの規模・文化 | 自社担当経由で現場の雰囲気を聞く |
| 稼働条件（常駐/リモート・稼働時間） | 契約書で確認する |
| 単価 | 現状・目標と比較する |
| 契約期間・更新サイクル | 短期（3ヶ月）か長期（1年〜）か |
| 成長できるか | 「得意なことを活かせる現場」と「新しいスキルが身につく現場」のバランスを考える |

## 技術学習の進め方（SES向け）

現場の業務だけでスキルアップを期待するのは限界があります。業務外でも意識的に学習する習慣が重要です。

### 現場で得た経験を「汎用化」する

特定の現場でしか通じない知識ではなく、他の現場でも使える形に昇華させます。

```
現場での経験 → 汎用化された知識
「○○社のECシステムのDB設計」
    ↓ 抽象化
「高トラフィックなECシステムにおけるRDB設計のベストプラクティス」
    ↓ さらに応用
「大規模テーブルのパフォーマンスチューニング」として次の現場でも活用
```

### 学習リソースの活用

| 種類 | 具体例 |
|-----|------|
| 書籍 | 現場で使った技術の体系的な理解（SQLアンチパターン・Clean Architecture等） |
| 公式ドキュメント | 使ったツール・FWの公式Doc。ブログより信頼性が高い |
| 個人開発 | 現場で学んだ技術をプライベートで試す（GitHubに残す） |
| 資格 | AWS認定・情報処理技術者試験。客観的な指標として単価交渉に使える |
| 勉強会・コミュニティ | connpass等のイベント。現場外の人脈形成にもなる |
