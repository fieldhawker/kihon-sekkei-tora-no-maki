# スコープ管理（作業範囲の管理）

> **このファイルの目的**: SESエンジニアが「どこまでが自分の仕事か」を正しく判断し、契約外作業に巻き込まれないようにするための考え方と対処法をまとめます。

---

## なぜスコープ管理が重要か

SES現場では、下記のような理由で作業範囲が曖昧になりやすいです。

- 「せっかくここにいるから、これもお願い」という口頭依頼
- 現場が忙しくなると「手が空いてるなら手伝って」という圧力
- 契約書の「その他付随する業務」という曖昧な文言
- 現場の担当者が契約内容を知らないケース

スコープ外の作業を断りにくい雰囲気の中で引き受け続けると、次の問題が生じます。

| 問題 | 内容 |
|-----|-----|
| 稼働時間の超過 | 精算幅を超えても追加費用が払われない可能性がある |
| 本来の業務の遅延 | 契約内の業務が疎かになり評価が下がる |
| 責任の拡大 | 「やったこと」に対して責任を問われるリスク |
| 契約の形骸化 | 実態と契約内容がかけ離れ、後でトラブルになる |

---

## 契約範囲の確認方法

まず「自分の契約にどこまでの業務が含まれているか」を正確に把握してください。

### 確認すべき情報源

1. **自社との契約書・作業指示書**: 業務内容の定義
2. **客先との参画条件書・業務内容確認書**: 担当範囲の具体的な記述
3. **自社担当（営業・PM）への直接確認**: 書面に書かれていない場合

### 業務範囲の典型例（基本設計担当の場合）

| 通常スコープ内 | グレーゾーン（要確認） | スコープ外の可能性が高い |
|-----------|----------------|------------------|
| 基本設計書の作成 | 設計書のフォーマット整備 | 他機能の設計書の代理作成 |
| 設計レビューへの参加 | レビュー資料の準備代行 | 後輩エンジニアの教育・メンタリング |
| 顧客との仕様確認 | 他チームとの調整・折衝 | インフラ構築・サーバー設定 |
| 既存設計書の読み込み | 設計書管理ツールの整備 | 他ベンダーとの契約交渉 |

## 追加依頼が来たときの判断フロー

```
客先から「○○もお願いできますか？」という依頼が来た
              │
              ▼
    契約書・参画条件書を確認
              │
    ┌─────────┴──────────┐
    │                    │
  契約内の業務と         判断がつかない
  判断できる             または契約外の可能性がある
    │                    │
    ▼                    ▼
  受ける                 自社担当に相談・確認を依頼する
  （通常どおり進める）        │
                         ┌──┴──┐
                     契約内と    契約外と
                     確認できた  確認できた
                         │          │
                         ▼          ▼
                       受ける      丁寧に断る
                                  （下記の断り方を参照）
```

---

## 断り方のテンプレート

直接「それは契約外なので」と言うと関係が悪化することがあります。以下のように、**自社を挟んで対応する**形が最も角が立ちません。

### パターン1: 確認が必要な場合（一番使いやすい）

```
「ありがとうございます。その作業について、一度自社の担当者に
確認を取らせていただいてもよいでしょうか。
明日には回答できるようにします。」
```

### パターン2: 契約外と確認が取れた後

```
「確認したところ、今回の契約範囲は〇〇の業務と伺っています。
○○の作業については、別途ご相談いただくか、自社の担当者を
交えてご確認いただけますでしょうか。」
```

### パターン3: 今のタスクが優先の場合（完全な断りではないケース）

```
「現在担当している〇〇の作業が△△のスケジュールで動いており、
今週中は対応が難しい状況です。
来週以降であれば対応可能ですが、いかがでしょうか。」
```

> **重要**: 口頭での断りだけでなく、**自社担当に経緯を必ず報告・記録**してください。「断った」「どのように断ったか」を自社が把握していることが、後のトラブル防止になります。

---

## 追加作業を受ける場合の対応

スコープ外であっても、自社と客先の合意があれば受けることができます。その場合は以下を確認してください。

- [ ] 自社担当が客先と合意し、作業内容・工数・単価が調整された
- [ ] 追加作業の内容・期間・完了条件が明確になった
- [ ] 追加作業が既存タスクのスケジュールに影響しないか確認した
- [ ] 追加作業の成果物・責任範囲が明確になった

## よくあるスコープ問題のパターン

### パターン1: 「ちょっとだけ」が積み重なる

```
・「このバグだけ直してほしい」（担当外の機能）
・「この画面のデザインも少し直してほしい」
・「テストも一緒にやってほしい」
↓
気づいたら本来の担当外の業務を大量に抱えている
```

**対処**: 「小さなお願い」でも記録に残す。月単位で自分の作業内訳を振り返り、スコープ外が増えていないか確認する。

### パターン2: 「あなたが詳しいから」という巻き込まれ

```
・「○○のことは田中さん（あなた）が一番詳しいよね」
・「他のメンバーには任せられないから」
↓
「詳しい人」「頼れる人」として追加依頼が集まってくる
```

**対処**: 知識提供（質問への回答・レビューへの参加）と実作業（設計・実装を自分でやる）は区別する。実作業としての追加依頼は自社担当を通じて契約確認する。

### パターン3: 前任者の業務を引き継ぐ形での拡大

```
・「前任の△△さんはこれもやってくれていたんだけど」
↓
前任者が非公式にやっていた業務をデフォルトで引き継ぐ形になる
```

**対処**: 参画時に前任者の実際の作業内容と自分の契約範囲を照合する。差異がある場合は参画初期に自社担当に確認・調整を依頼する。

## スコープ管理の記録テンプレート

自分の作業記録をつけておくと、スコープの証明に役立ちます。

```
## 週次作業記録 YYYY-WW（W: 週番号）

### 契約内業務
- [作業内容] [所要時間]
- 基本設計書（機能設計パート）の作成: 12h
- 設計レビュー参加・修正対応: 4h

### 追加依頼・対応記録
- [日付] [依頼者] [依頼内容] [対応方針]
- 04-22 鈴木さん ○○機能のバグ調査依頼
  → 担当外のため自社担当に確認中

### 今週気になったこと（自社担当共有用）
- ○○の追加依頼が3件来ており、スコープ確認が必要
```
