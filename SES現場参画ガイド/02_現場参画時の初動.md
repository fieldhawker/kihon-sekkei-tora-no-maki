# 現場参画時の初動

> **このファイルの目的**: 客先に初めて入るときの「最初の2週間」でやるべきことを整理します。最初の印象と情報収集が、その後の働きやすさを大きく左右します。

---

## 初動の心構え

> **「わからないことをわかったふりしない」「小さく貢献を積み上げる」**
>
> 新しい現場に入ったばかりのエンジニアに即戦力を期待している現場は少ないです。最初の数週間は「環境理解」と「信頼の種まき」に集中することが、長期的に見て最も効率的です。

---

## Day 1（参画初日）のチェックリスト

### 必ず確認すること

- [ ] 入館・セキュリティの手続き（入館証・ICカード・PC貸与の有無）
- [ ] 座席・作業場所の確認
- [ ] 緊急時・災害時の避難経路・連絡先の確認
- [ ] 直接の上司・メンターにあたる人の確認（現場で誰に質問・報告するか）
- [ ] 稼働時間・休憩時間・打刻ルールの確認
- [ ] 連絡ツール（Slack / Teams / メール等）のアカウント設定

### 環境整備

- [ ] 開発端末（PC）の初期セットアップ（セキュリティポリシーを確認してから行う）
- [ ] VPN・社内ネットワークへの接続確認
- [ ] リポジトリ（GitHub / GitLab等）へのアクセス権の確認
- [ ] 必要なツール・ソフトウェアのインストール（現場の許可ポリシーを確認する）
- [ ] 開発環境のローカル構築手順書の入手

### 挨拶・自己紹介

- [ ] チームメンバー全員への挨拶（名前・所属・担当業務を端的に伝える）
- [ ] 隣接するチーム・連携先への挨拶
- [ ] 自社担当（営業・PMなど）への参画完了報告

## 最初の2週間でやること

### 現場のルール・文化を把握する

現場には「文書化されていない暗黙のルール」が必ずあります。最初の2週間は積極的に観察・質問して把握します。

| 確認項目 | 確認方法 |
|--------|--------|
| コミュニケーションの主なツールと使い分け | メンターに聞く |
| レビューの文化（厳しいか・丁寧か）| コードレビューを観察する |
| 会議の進め方・議事録の有無 | 会議に参加して観察する |
| 設計書・ドキュメントの保存場所とルール | メンターに聞く。Confluence / Notionなど |
| 業務時間外の連絡への対応方針 | 明示的に確認する |
| 有給・体調不良時の連絡手順 | 現場と自社の両方を確認する |

### プロジェクトの現状を把握する

- [ ] プロジェクトの目的・背景・現在のフェーズを把握した
- [ ] 自分が担当する業務範囲を書面または口頭で確認した（曖昧な場合は文書化を依頼する）
- [ ] 現在の課題・問題点・急ぎの対応事項を把握した
- [ ] 直近のマイルストーン・リリース予定日を把握した
- [ ] チームの技術スタック（言語・FW・インフラ）を把握した

### コードベース・設計書を読む

- [ ] リポジトリのREADMEを読んだ
- [ ] 既存の設計書（基本設計書・詳細設計書）の場所を確認し、主要な部分を読んだ
- [ ] アーキテクチャの概要（フォルダ構成・主要クラス・データの流れ）を理解した
- [ ] 自分が担当する機能のコードを読んで動作を理解した
- [ ] ローカル環境でアプリケーションを起動できることを確認した

### 小さなタスクで貢献する

最初の2週間で「0→1の大きな成果」を焦る必要はありません。むしろ小さなタスクを確実にこなすことが信頼につながります。

- バグ修正・軽微な改善タスクを1〜2件完了する
- 手順書の誤字・古い情報を見つけて修正するPRを出す
- 開発環境構築手順でつまずいた箇所を更新する

## 情報収集のコツ

### 効果的な質問の仕方

SES（外部エンジニア）として現場に入るときは、質問の仕方一つで印象が変わります。

| やりがちなNG | 推奨 |
|-----------|-----|
| 「〇〇って何ですか？」と文脈なく聞く | 「〇〇の仕様を確認したくて、Aのファイルを見たのですが、Bの部分が理解できませんでした。教えていただけますか？」と自分で調べたことを示してから聞く |
| 分からないことをそのまま放置する | 「この認識で合っていますか？」と確認を取る |
| 何度も同じことを聞く | メモを取り、聞いたことを記録する |
| 大きな作業をいきなり一人で始める | 着手前に「このように進めようと思いますが問題ありませんか？」と確認する |

### 把握しておくべき関係者マップ

以下を把握しておくと、誰に何を聞けばよいかが明確になります。

| 役割 | 現場での肩書き例 | 自分との関係 |
|-----|-------------|-----------|
| 現場の直接上司 | PL / リーダー / PM | 業務上の指示・相談窓口 |
| 技術メンター | シニアSE / アーキテクト | 技術的な質問窓口 |
| 隣接チームの担当者 | 他機能のSE / インフラ担当 | 連携時の窓口 |
| 自社担当（営業・PM） | 自社の営業 / PM | 契約・トラブル時の窓口 |
| 顧客（発注者） | 顧客PM / 業務担当者 | 要件確認の相手（直接接触するか確認が必要） |

## よくある初動の失敗

| 失敗パターン | 結果 | 回避策 |
|-----------|-----|------|
| 現場のルールを確認せずに自己流で進める | チームのルールと衝突してやり直し | 着手前に「チームの標準的な進め方」を確認する |
| 分からないのに分かったと言ってしまう | 後で大きな手戻りが発生する | 理解があいまいな部分は必ず確認する |
| 最初から大きな設計変更・改善提案をする | 「空気を読まない人」と思われる | まず現場のやり方を尊重して動き、信頼を積んでから提案する |
| 自社担当への報告を怠る | 問題が発生したときに自社が動けない | 週次で状況報告する習慣をつける |
| 客先で政治的な話（他の会社・エンジニアの批判）をする | 信頼を失う | 愚痴・批判は現場でしない。自社担当にのみ伝える |
