# サンプル画面一覧

Webシステム設計・開発の理解を深めるためのサンプル画面候補。
各画面は「実際にどう見えるか・どう動くか」を具体的に示すことを目的とする。
**初心者向け注記**: ★マークの画面は特に基礎的な概念を扱うため、最初に参照することを推奨。

---

## ★ 1. ページレイアウト・ナビゲーション構造 `layout-navigation.html`

**目的**: Webシステムの画面骨格となるレイアウトパターンと、ナビゲーションの組み方を示す

初心者がつまずきやすい点:
- ヘッダー・サイドバー・メインコンテンツ・フッターの役割の違いがわからない
- どこに何を置くべきかの判断基準がない
- ページ遷移とコンポーネント再利用の概念がイメージできない

含めるべきレイアウトパターン:
- ヘッダー固定 + スクロールコンテンツ（最もよく使われる基本形）
- 左サイドバー + メインエリア（管理画面の定番）
- パンくずリスト（現在位置の表示・どの深さまで表示するか）
- タブによるコンテンツ切り替え（URL変化あり vs なし の違い）
- レスポンシブ時のサイドバー折りたたみ（ハンバーガーメニュー）
- フッターの役割（著作権表示・利用規約リンク・バージョン情報）

ナビゲーションの設計ポイント:
- アクティブ状態（現在開いているメニューの強調）
- 権限によるメニュー表示/非表示の制御
- 階層メニュー（2階層まで）の開閉

## ★ 2. UIコンポーネント状態一覧 `ui-components.html`

**目的**: ボタン・入力欄・リンクなどの基本部品が持つ「状態」の種類と視覚的な表現を示す

初心者がつまずきやすい点:
- ボタンに「通常」以外の状態があることを知らない
- 無効（disabled）とローディング中の使い分けが不明
- フォーカス状態のアクセシビリティ上の重要性を知らない

各コンポーネントの状態:
- ボタン: 通常 / ホバー / フォーカス / 押下中 / ローディング中 / 無効（disabled）
- テキスト入力: 通常 / フォーカス / 入力済み / エラー / 無効 / 読み取り専用
- チェックボックス / ラジオボタン: 未選択 / 選択済み / 部分選択（チェックのみ）/ 無効
- セレクトボックス: 通常 / フォーカス / 選択済み / 無効
- リンク: 通常 / ホバー / 訪問済み / フォーカス

ボタン種別の使い分け（初心者向け解説）:
- プライマリ（主アクション・1画面に1つが原則）
- セカンダリ（補助アクション）
- デンジャー（削除・取り消しなど破壊的操作）
- ゴースト（目立たせたくない操作）
- アイコンのみボタン（ツールチップ必須の理由）

## ★ 3. レスポンシブデザイン・ブレークポイント `responsive-design.html`

**目的**: 画面幅によってレイアウトが変化する仕組みと、設計時の考え方を示す

初心者がつまずきやすい点:
- PCで作ったデザインがスマホで崩れる理由がわからない
- ブレークポイントをいつ・どこで設定すべきかの基準がない
- テーブルやフォームのスマホ対応が特に難しい

含めるべきパターン:
- 同一コンテンツのPC / タブレット / スマホ表示比較
- グリッドレイアウトの列数変化（4列→2列→1列）
- テーブルのスマホ対応（横スクロール vs カード形式への変換）
- ナビゲーションのスマホ対応（ハンバーガーメニュー）
- フォームのスマホ対応（ラベルの位置変化）
- 画像のレスポンシブ（アスペクト比の維持）

ブレークポイントの一般的な基準（参考値）:
- スマホ: ~768px
- タブレット: 768px~1024px
- PC: 1024px~

## ★ 4. ログ出力画面 `log-viewer.html`

**目的**: ログの種別・構造・表現パターンを示す

初心者がつまずきやすい点:
- 「エラーが発生しました」のような情報量ゼロのログを書いてしまう
- DEBUGとINFOとWARNの使い分け基準がわからない
- ログを見て何が起きたか追えるようになるための情報が何か知らない

各ログエントリに含めるべき情報:
- タイムスタンプ（ミリ秒まで・タイムゾーン明示）
- ログレベル（DEBUG / INFO / WARN / ERROR / FATAL）
- トレースID・スパンID（同一リクエストのログをまとめて追うための識別子）
- サービス名・ホスト名（マイクロサービス・複数台構成での特定に必要）
- ユーザーID・セッションID（誰の操作か特定するために必要）
- 操作内容・エンドポイント（何をしていたか）
- 処理時間（ms）（パフォーマンス問題の調査に必要）
- 結果コード・エラーコード（プログラムで判別できる分類）
- スタックトレース（エラー時のみ）

ログレベルの使い分け基準:
- DEBUG: 開発中の詳細情報（本番では出力しない）
- INFO: 正常な業務イベント（ログイン・注文・バッチ開始/終了など）
- WARN: 問題ではないが注意が必要な状態（リトライ発生・閾値接近など）
- ERROR: 処理が失敗したが継続可能（1件の処理失敗など）
- FATAL: システム継続不可能な致命的障害

具体的なログパターン例（悪い例→良い例の対比付き）:
- 正常系: ユーザーログイン成功（誰が・どこから・何ms）
  - 悪い例: `ログイン成功`
  - 良い例: `INFO [2024-01-15T10:23:45.123+09:00] user.login.success userId=U1234 ip=192.168.1.1 duration=234ms sessionId=abc123`
- 警告系: DBコネクションプール残数低下（現在値・閾値）
  - 悪い例: `DBの接続数が少なくなっています`
  - 良い例: `WARN pool.connection.low current=3 threshold=5 max=20 host=db-primary.internal`
- エラー系: 外部API呼び出しタイムアウト（URL・タイムアウト値・リトライ回数）
  - 悪い例: `外部APIエラー`
  - 良い例: `ERROR external.api.timeout url=https://api.example.com/v1/items timeout=5000ms retryCount=3 traceId=xyz789`
- 致命的: DBへの接続失敗（接続先・エラー詳細・影響範囲）
- 業務エラー: 在庫不足で注文失敗（商品ID・要求数・在庫数）
- セキュリティ: 連続ログイン失敗によるアカウントロック（IP・試行回数・対象ユーザー）

## 5. エラーハンドリング画面 `error-patterns.html`

**目的**: HTTPステータスコードとエラーレスポンスの設計パターンを示す

初心者がつまずきやすい点:
- すべてのエラーに200や500を返してしまう
- エラーメッセージをユーザーにそのまま見せてしまう（スタックトレースの露出）
- クライアント向けとサーバーログ向けのメッセージを分けていない

エラーレスポンスの基本構造（初心者向け）:
```json
{
  "error": {
    "code": "ITEM_NOT_FOUND",       // プログラムで判別できる機械可読なコード
    "message": "指定された商品が見つかりません", // ユーザーに見せるメッセージ
    "details": "itemId: 'ABC123' は存在しません", // 追加情報（省略可）
    "errorId": "ERR-20240115-001"   // ログと照合するための識別子
  }
}
```

HTTPステータスコードと使い方:
- 400 Bad Request: バリデーションエラー（どのフィールドが・なぜNG・正しい形式）
- 401 Unauthorized: 未認証（トークン期限切れ vs 未ログイン の区別）
- 403 Forbidden: 権限不足（何の権限が必要か明示）
- 404 Not Found: リソース不存在（IDを含めた具体的なメッセージ）
- 409 Conflict: 楽観的ロック競合（誰が・いつ更新したか）
- 422 Unprocessable Entity: 業務ルール違反
- 429 Too Many Requests: レート制限（残回数・リセット時刻）
- 500 Internal Server Error: サーバーエラー（errorId付き・ユーザー向けvs開発者向けの分離）
- 503 Service Unavailable: メンテナンス・過負荷（再試行推奨時刻）

やってはいけないパターン（初心者向け）:
- 全エラーに200を返してbodyにエラー内容を入れる
- スタックトレースをレスポンスに含める
- 「システムエラー」だけで何も手掛かりのないメッセージ
- DBのエラーメッセージをそのまま返す（テーブル名・カラム名の漏洩）

## ★ 6. バリデーション表示画面 `form-validation.html`

**目的**: フォームバリデーションのUXパターンと実装方針を示す

初心者がつまずきやすい点:
- クライアントサイドだけで完結させようとする（サーバーサイドの必要性を知らない）
- エラーメッセージが「入力してください」だけで何を入れればいいか不明
- 入力途中でエラーを出して使いにくいUIを作ってしまう

バリデーションのタイミング（使い分け基準）:
- フォーカスを外れたとき（blur）: 入力完了後に即座にチェック（推奨）
- 入力中（input）: 文字数カウントやリアルタイム強度表示など限定的な用途のみ
- 送信ボタン押下時: 最終チェック（必ず実施）
- サーバーレスポンス受信後: サーバーでしか確認できない内容

エラーメッセージの良い例・悪い例:
- 悪い例: `入力エラー`
- 悪い例: `メールアドレスが正しくありません`
- 良い例: `メールアドレスは「user@example.com」の形式で入力してください`
- 悪い例: `パスワードエラー`
- 良い例: `パスワードは8文字以上で、英字・数字・記号をそれぞれ1文字以上含めてください`

バリデーションパターン:
- フィールド単位のエラー表示（エラーがあるフィールドの近くに表示）
- フォーム全体のサマリーエラー（ページ上部にまとめて表示・アクセシビリティ対応）
- サーバーサイドエラーの表示（クライアントをすり抜けたケース）
- 複合バリデーション（開始日 < 終了日 / パスワード確認一致 など）
- 非同期バリデーション（ユーザー名の重複チェック・チェック中の表示）
- 必須項目の表示方法（「必須」ラベル vs 「任意」ラベルどちらを付けるか）

## ★ 7. マルチステップフォーム（ウィザード） `wizard-form.html`

**目的**: 複数ステップにわたる入力フォームの設計パターンを示す

初心者がつまずきやすい点:
- 1ページに項目が多くなりすぎてUXが悪くなる
- 「戻る」を押したら入力内容が消える
- どのステップにいるか・何ステップあるかをユーザーが把握できない

含めるべきパターン:
- ステップインジケーター（現在位置・完了済み・未完了の視覚的区別）
- 「次へ」押下時のそのステップのバリデーション
- 「戻る」時の入力内容の保持
- 最終確認ステップ（入力内容のサマリー表示）
- 途中離脱の警告（「保存せずに離れますか？」）
- 完了画面（次のアクションの提示）

ステップ設計の考え方:
- 1ステップあたりの項目数は5〜7個が目安
- 関連する項目をグループ化してステップを決める
- 後のステップが前のステップの入力内容に依存する場合の処理

## 8. テーブル・一覧画面 `data-table.html`

**目的**: 一覧表示に必要な機能の網羅と実装パターンを示す

初心者がつまずきやすい点:
- ページネーションを実装せずに全件取得してしまう
- ソートをフロントエンドのみで実装してしまう（大量データで破綻）
- 0件・エラー・ローディングの状態を考慮していない

含めるべき要素:
- ページネーション（件数・現在ページ・総件数・1ページあたりの件数変更）
- ソート（昇順/降順の切り替え・現在のソート状態の表示）
- フィルタリング（テキスト・日付範囲・ステータス・複数条件の組み合わせ）
- 列の表示/非表示（カスタマイズ可能なテーブル）
- 行選択（単一・複数・全選択・ページをまたいだ選択の扱い）
- 一括操作（削除・エクスポート等）
- 空状態（0件時の表示・フィルター結果0件 vs データそのものが0件 の違い）
- ローディング状態（スケルトン表示）
- エラー状態（取得失敗時のリトライ）
- 行の展開（詳細の表示・折りたたみ）
- インライン編集（テーブル上での直接編集）

## 9. 通知・フィードバックパターン画面 `notifications.html`

**目的**: ユーザーへのフィードバック手段の種類と使い分けを示す

初心者がつまずきやすい点:
- 全部モーダルで確認させてしまい操作が重くなる
- 処理成功を何も通知しない（操作が効いたかわからない）
- 通知の重要度に関わらず同じ見た目にしてしまう

種類と使い分け（初心者向け判断基準付き）:
- トースト通知: 操作成功など短命な情報（自動消去・3〜5秒）→ ユーザーの作業を止めないとき
- インラインメッセージ: フォーム周辺の文脈情報 → その場所に関連する情報のとき
- モーダルダイアログ: 確認が必要な操作（削除確認等）→ 取り消せない操作の前に必ず使う
- バナー: サービス全体に関わる重要情報（メンテナンス予告等）→ 全ユーザーに伝えるとき
- バッジ: 未読件数などの数値表現
- プログレス表示: 長時間処理の進捗（パーセント・推定残時間・キャンセル可否）
- 確認ダイアログの文言設計（「はい/いいえ」は使わない → 操作を明示した動詞で）
  - 悪い例: 「削除しますか？」→「はい」「いいえ」
  - 良い例: 「注文ID: #1234を削除します。この操作は取り消せません。」→「削除する」「キャンセル」

## 10. 認証・認可フロー画面 `auth-flow.html`

**目的**: 認証状態の遷移と各状態での画面表現を示す

初心者がつまずきやすい点:
- 認証（Authentication）と認可（Authorization）の違いがわからない
- セッション切れをエラーと同じように扱ってしまう
- 「ログインしていない」ユーザーへの適切な誘導ができない

認証と認可の違い（初心者向け）:
- 認証: 「あなたは誰ですか？」→ 本人確認（ログイン）
- 認可: 「あなたはこれをしていいですか？」→ 権限チェック

想定するパターン:
- ログイン画面（通常・MFA・SSO）
- セッション期限切れ時の再認証（現在の画面を保持したまま・作業内容を失わない）
- 権限不足時のフォールバック表示（何の権限が必要か・誰に申請すればよいか）
- パスワードリセットフロー（メール送信→リンク期限→再設定→完了）
- 「ログインしたまま」（Remember me）の扱い
- ログアウト確認とセッション破棄後の遷移

## 11. 非同期処理・ローディング状態画面 `async-states.html`

**目的**: 非同期処理中のUIパターンと状態管理を示す

初心者がつまずきやすい点:
- APIを呼んでいる間、ボタンを連打できてしまう
- ローディング中の表示が全くない（何かが起きているのかわからない）
- エラー時にただ画面が壊れる（リカバリー手段を提示しない）

状態のパターン（初心者向け: 必ず4状態を意識する）:
- idle（初期状態）
- loading（処理中）
- success（成功）
- error（失敗）

具体的な表現パターン:
- 初期ロード中（スケルトンスクリーン vs スピナー の使い分け）
- 部分更新中（ボタン非活性化・連打防止・インジケーター表示）
- 長時間処理（プログレスバー・推定残時間・バックグラウンド処理への切り替え）
- 処理成功後の画面更新（差分更新 vs フルリロード）
- 処理失敗時のリカバリー（リトライボタン・前状態への復元・エラーメッセージ）
- 楽観的UI更新（先に反映して後から同期・失敗時のロールバック）

## 12. ファイルアップロード `file-upload.html`

**目的**: ファイルアップロードUIの設計パターンとサーバーサイドの連携を示す

初心者がつまずきやすい点:
- ファイルサイズ・拡張子チェックをフロントだけでやればいいと思っている
- アップロード中の進捗を表示しない
- 複数ファイルのアップロード状態の管理が難しい

含めるべきパターン:
- ドラッグ&ドロップ + ファイル選択ボタン（両対応が標準）
- アップロード前バリデーション（ファイル形式・サイズ上限・個数上限）
- アップロード進捗（ファイルごとのプログレスバー）
- アップロード成功/失敗の状態表示
- 失敗したファイルの個別リトライ
- アップロード済みファイルの一覧・削除
- プレビュー表示（画像の場合）

サーバーサイドでも必ずやること（初心者向け）:
- ファイル形式の検証（Content-Typeの偽装に注意）
- ファイルサイズの制限
- ウイルススキャン（要件に応じて）
- 保存先パスのサニタイズ（パストラバーサル攻撃の防止）

## 13. 日付・時刻入力 `datetime-input.html`

**目的**: 日付・時刻に関わる入力UIの設計パターンとよくある落とし穴を示す

初心者がつまずきやすい点:
- テキスト入力だけで日付を受け付けてしまいフォーマットが揃わない
- タイムゾーンを考慮していない
- 日付範囲の入力で「終了日 < 開始日」を弾けない

含めるべきパターン:
- 日付ピッカー（カレンダーUI）
- 時刻ピッカー
- 日付範囲選択（開始日〜終了日の一体型UI）
- 相対日付の入力（「今日」「今月」「先月」ボタン）
- タイムゾーンの表示と入力
- 入力値の正規化（2024/1/5 → 2024-01-05）

DB・APIでの日付の扱い方（初心者向け）:
- 保存はUTC、表示はローカルタイム
- フォーマットはISO 8601（2024-01-15T10:30:00+09:00）を使う
- 「日付のみ」と「日時」は型を分けて扱う

## 14. 検索UI `search-ui.html`

**目的**: 検索機能のUIパターンと検索体験の設計を示す

初心者がつまずきやすい点:
- 検索するたびにAPIを叩いてしまう（デバウンスを知らない）
- 検索結果が0件のときに何も表示しない
- 検索条件をURLに含めないのでリロードで消える

含めるべきパターン:
- シンプルな検索ボックス（デバウンス処理・入力クリアボタン）
- 検索結果のハイライト（検索キーワードの強調表示）
- 詳細検索（複数条件の組み合わせ・絞り込み）
- 検索条件のURL反映（ブックマーク・シェア可能にする）
- 検索中・結果あり・結果なし・エラーの4状態
- 検索履歴・よく使う検索条件の保存
- 件数と検索時間の表示（「〇件が見つかりました（0.03秒）」）

## 15. APIレスポンス設計サンプル画面 `api-response.html`

**目的**: REST APIのレスポンス構造のパターンと設計方針を示す

初心者がつまずきやすい点:
- 成功でも失敗でも `{ "result": "ok" }` のような独自形式を作ってしまう
- 日付や金額の型をAPIごとにバラバラにしてしまう
- ページング情報をどこに入れるかチームで統一できていない

含めるべきパターン:
- 単一リソース取得
- コレクション取得（ページング情報付き）
- 作成・更新のレスポンス（作成後のリソースをそのまま返す）
- 一括操作のレスポンス（部分成功の扱い・何件成功・何件失敗か）
- エラーレスポンスの標準構造（全エンドポイントで統一）
- 日時の形式（ISO 8601統一）
- 金額の形式（数値型 vs 文字列型 の議論）
- enumの表現（文字列 vs 数値）
- nullと省略（undefined）の使い分け

## 16. 権限・ロール別表示パターン画面 `rbac-display.html`

**目的**: ロールベースアクセス制御のUI表現パターンを示す

初心者がつまずきやすい点:
- フロントで非表示にするだけでAPIに直接アクセスすれば実行できてしまう
- 「表示しない」と「グレーアウト」の使い分けがわからない

重要な原則（初心者向け）:
- **フロントの非表示はUXのためであり、セキュリティではない**
- 必ずサーバーサイドでも権限チェックを行う

想定するパターン:
- ボタン・メニューの表示/非表示（存在を知られたくない機能）
- ボタンのグレーアウト（機能の存在は示しつつ使えない状態）
- 読み取り専用フィールドの表示（表示するが編集不可）
- 機能へのアクセス拒否メッセージ（権限申請先の案内付き）
- 同一画面でのロール別レイアウト差分（管理者 vs 一般ユーザー）
- 権限のないデータのマスキング表示（個人情報の部分マスク等）

## 17. 監査ログ・操作履歴画面 `audit-log.html`

**目的**: 誰が・いつ・何を・どう変えたかを記録・表示するパターンを示す

初心者がつまずきやすい点:
- 何を記録すべきかがわからない（全部記録 vs 必要なものだけ）
- 変更前後の差分をどう保存・表示するか
- 監査ログ自体が改ざんされるリスクを考慮していない

ログに含めるべき情報:
- 操作者（ユーザーID・名前・IPアドレス・ユーザーエージェント）
- 操作日時（タイムゾーン明示・サーバー時刻）
- 操作種別（CREATE / UPDATE / DELETE / VIEW / EXPORT）
- 対象リソース（種別・ID・名称）
- 変更前後の値（差分表示・何が変わったかをひと目で把握）
- 操作理由・承認者（業務要件がある場合）

記録すべき操作の考え方:
- 取り消せない操作（削除・確定・送信）は必ず記録
- 個人情報・金融情報の閲覧も記録対象（参照ログ）
- バッチ処理・自動処理も操作者を「SYSTEM」として記録

## 18. ダッシュボード・KPI表示画面 `dashboard.html`

**目的**: 集計データの可視化パターンと更新頻度の設計を示す

初心者がつまずきやすい点:
- データが取得できないとき画面が壊れる
- リアルタイム更新とバッチ更新の違いを意識していない
- 「最終更新日時」を表示しないのでデータが古いかどうかわからない

含めるべき要素:
- リアルタイム更新 vs 定期バッチ更新の違いと明示方法
- 数値の前回比・目標比（増減の方向とアイコン）
- データが取得できない場合の表示（エラー状態・部分的な失敗）
- 期間切り替え（日/週/月/カスタム）
- データの最終更新日時の表示
- グラフの種類の使い分け（折れ線・棒・円・散布図 それぞれ向いている用途）
